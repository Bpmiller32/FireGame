The FireGame Engine

A hand-built 2D Donkey Kong-style platformer engine — a singleton core wiring Three.js rendering, a Rapier2D physics world, and a Vue presentation shell, all glued together by an event bus and a contact-table collision system. This document walks every file, every function, and every hard-won line so you can open any source file and know exactly what it does and how to change it.

TypeScript · strict Three.js r162 · WebGL Rapier2D · @dimforge compat Vue 3 · UI shell mitt · event bus dat.gui + stats.js · debug Vite · bundler

The one idea: a clean engine / game seam

Everything in this codebase serves a single architectural conviction: the engine layer (src/engine/) is a reusable platformer engine that never names a single Donkey-Kong concept, and the game layer (src/game/) supplies all the rules, entities, feel and content, wiring itself into the engine from the outside. The whole guide is organised around that seam.

🧩Singleton core

Experience owns the canvas, scene, camera, renderer, physics, input, resources and debug — created once via GetInstance()/Configure().

📡Event bus

A typed mitt emitter decouples subsystems. The engine emits tick/resize; the game speaks gameStart, gameOver, switchLevel, loading events and more.

💥Contact table

Physics contacts are routed through a generic (typeA,typeB) → handler registry. The game registers concrete rules; the engine never hard-codes a reaction.

🔁Finite-state machines

A tiny generic StateMachine drives the player (idle/run/jump/fall/climb) and the barrels (roll/descend/bounce) through pure handler functions.

🏗️GLB-driven levels

Levels are modelled in Shapr3D/Blender, exported to GLB; a parser turns folder names + glTF extras into typed LevelData the factories spawn.

🎚️Config-driven game

Entity factories, contact rules, the level registry, input bindings and "feel" profiles live in src/game/config & attributes — the control panel.

The decision-log codes. Throughout the source you'll see comments like (C5), (D3), (D28). These reference the author's private decision log — each marks a deliberate, hard-won choice (e.g. C5 = the camera consumes a plain snapshot, never the game Player; D3 = the engine never names a game type, the game injects its palette). This guide surfaces and decodes every one — see the decision log.
