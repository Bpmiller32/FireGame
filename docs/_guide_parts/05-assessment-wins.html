Hard-won victory. A from-scratch 2D physics platformer in under 800 KB gzipped, where roughly a megabyte of that (raw) is a serious deterministic physics solver, is genuinely lean. There is no app to install, no store, no native runtime: the URL is the game. It runs on any device with a modern browser and WebGL2 — desktop, laptop, phone, tablet, a kiosk, an embedded panel — same binary, no porting.

Why "runs anywhere" is true and not marketing: the entire dependency set is pure JavaScript + WebAssembly with zero native or platform-specific packages.

Four dependencies. Rapier's -compat build loads its WASM with an async await RAPIER.init() (physics.ts:31), so it boots in any browser without a build-time WASM step. That is the whole portability story, and it holds up.

The engine/game separation is real, not aspirational

Lots of codebases claim a clean architecture in comments. This one's central invariant — the dependency arrow only ever points game → engine — actually holds when you grep for violations. That is the hardest thing to keep true over a refactor, and it's true here.

Hard-won victory — the grep passes. grep -r "game/" src/engine/ finds nothing. No file under src/engine/ imports anything from src/game/. The engine could be lifted into a different 2D platformer without dragging a single Donkey-Kong concept along. The discipline is even self-documented at resourceLoader.ts:427: "the engine deliberately cannot import the game's EntityType enum (engine names zero game concepts…)".

Three places show this isn't luck — it's a repeated, deliberate pattern of inverting the dependency so the engine receives game knowledge as plain data instead of importing it:

The core abstractions are small, generic, and genuinely reusable

The two load-bearing engine primitives — the FSM and the contact table — are tiny, fully generic, and import nothing game-specific. They're the kind of code you'd copy into the next engine unchanged.

class StateMachine<H extends { State: S }, S extends string>

The whole finite-state-machine engine is one generic class whose Update() is a single line — dispatch to the handler for the host's current state. No engine imports, no singleton, trivially testable. It powers both the player (idle/run/jump/fall/climb) and the barrels (roll/descend/bounce) with zero duplication.

// stateMachine.ts — the entire dispatch
Update() { this.handlers[this.host.State]?.(this.host); }

ContactRegistry.Dispatch(phase, x, y)

A declarative (A, B) → run table that fires symmetric rules exactly once with a stable argument order, allocating nothing on the hot path. This is what replaced the old hand-written collision if/else ladder — and it's the fix that finally killed the barrel/enemy collision bug.

// contactRegistry.ts — two-sided match, fired once, no temporaries
if (rule.a(x) && rule.b(y))      { rule.run(x, y); }
else if (rule.a(y) && rule.b(x)) { rule.run(y, x); }

Two more abstractions punch above their weight:

  • Data-driven level spawning. gameDirector.ts:249 just iterates parsed rows and dispatches ENTITY_FACTORIES[data.type](...) — adding a placeable thing is (mostly) a new factory entry, not director surgery. The engine GLB parser stays game-blind by carrying an uninterpreted meta?: Record<string,string> bag (levelData.ts) that each factory reads only its own keys from.
  • Input as data. The device fills a plain InputState snapshot via CaptureInto(state) (input.ts:273), and the player reads that buffer — not the keyboard directly. The seam is explicitly designed for "keyboard today; network/replay/AI later" (inputState.ts:1). That one decision is what makes AI barrels and future netcode a fill-the-struct-differently problem instead of a rewrite.

Choosing Rapier was the right call

A platformer lives or dies on its collision and movement. Reaching for Rapier2D — a modern, Rust-built, WASM-compiled, determinism-capable solver — instead of hand-rolling AABB sweeps or pulling in a heavier non-deterministic engine is a mature, forward-looking choice.

What it buys you, concretely:

  • A real solver, for free. Continuous collision, sensors/intersections, kinematic character control, collision-group bitmasks — all production-grade, none of it your maintenance burden.
  • A foundation that's now deterministic. Rapier is built for cross-platform determinism, and the engine now exploits it — a fixed-timestep accumulator drives the sim and a seeded PRNG drives gameplay randomness (see Recently Shipped). So replay, deterministic tests, and lockstep netcode are all on the table; a non-deterministic engine would have closed those doors permanently.
  • WASM portability. Same solver, same numerics, every platform — no native build matrix.
Hard-won victory. The timestep hazard was clearly understood, not ignored — and is now resolved with a fixed-step accumulator. The supporting care remains: the camera uses frame-rate-correct exponential smoothing (camera.ts, 1 - Math.exp(-rate * Δt)) and the clock clamps delta and never feeds zero (time.ts). The author knew where the dragons were.

The craft details that show real care

Beyond the big architecture, a dozen small decisions reveal a builder who sweats the right things.

🎚️ The "Feel Lab" — live tuning with paste-back

Easily the best DX touch in the codebase. playerDebug.ts builds writable dat.GUI sliders that handlers read fresh every frame, so a slider mutates the movement feel instantly — and dumpFeelValues() prints a ready-to-paste attributes block, closing the loop from "tweak by hand" to "commit the numbers." That is how good platformer feel actually gets made.

📦 Zero-allocation hot paths

The per-frame code that matters is allocation-aware: the SpriteAnimator early-returns on unchanged state and only touches the texture inside its timing branch; the camera mutates one cached Vector3 rather than newing one each frame; collision resolution is an O(1) collider-handle → GameObject Map lookup, never a scan; and ContactRegistry.Dispatch allocates nothing. (There are still hot spots to fix — see performance — but the instincts are right.)

🧭 Self-documenting decision discipline

The (C5)/(D3)/(D28) decision-log tags aren't decoration — they're a culture of writing down why a shape is deliberate so future-you doesn't "simplify" a hard-won decoupling back into a bug. Combined with comments that name the exact bug a line prevents, this codebase teaches its own reasoning. Most don't.

And the quiet ones: levels auto-register via an eager import.meta.glob (drop a GLB in assets/levels/, it appears), the resource loader is fail-soft so a single 404 advances the bar and still boots the game rather than hanging, and every engine subsystem stores its event-handler refs so it can precisely off() them in Destroy() — the correct pattern for a long-lived global bus.

The bottom line: the architecture is sound, the seams are real, the footprint is tiny, and the foundation can carry far more game than currently sits on it. The next section is about how much farther — read it as a roadmap from someone who likes this engine, not a teardown.
