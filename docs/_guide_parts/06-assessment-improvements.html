What Could Be Better — an Honest Roadmap

Conceptual critique from someone who has read every file and likes where this is going. Two cohorts of fixes have already shipped (see below); what remains is debatable on purpose — agree, disagree, or defer. Each item is grounded in real code, ranked by payoff, and paired with a concrete direction. Severity is about impact on the engine's stated direction (reusable, portable, expandable), not "bug count."

1 · True multi-instance / dependency injection

Bus ownership, teardown wiring, the Configure guard, and a second update phase are all ✅ shipped. What's left is the deep one: the engine is still a global singleton, just a resettable one.

High One Experience + one module-global Emitter per page — no DI

eventBus.ts — const Emitter = mitt(...) exported global | experience.ts — private static instance (now nullable/resettable, but still a singleton)

Problem. Destroy() now resets the instance so a fresh boot works (HMR/restart), but you still can't run two worlds at once or inject a world per test: a headless netcode sim, a server-authoritative tick, an editor preview pane, and split-screen all need >1 Experience and a non-global bus. Every test would also share the one global emitter.

Direction. Make the bus an instance owned by Experience (this.Events = mitt()) and thread the Experience/world handle into subsystems and GameObject instead of GetInstance(). This is the prerequisite for headless/multi-world/testable use — and the largest single change, so do it when one of those futures is actually on the table.

Also here (Low): the two update phases (OnGameUpdate sim + OnRenderUpdate render) cover today's needs; if a third consumer appears (editor overlay, debug HUD, netcode pre/post-physics), promote them to a small ordered, multi-subscriber hook registry with named phases.

2 · The last reusability seam

Assets, level types, and the input source are all injected/abstracted now (✅ shipped). One seam remains.

Medium The engine event-bus module still owns the game's event vocabulary

eventBus.ts lives in engine/ yet enumerates gameStart/gameOver/gameWin/switchLevel and imports GameObject for the gameObjectRemoved payload

Problem. A second game must edit (or fork) an engine file to change its event vocabulary, and the engine bundle imports GameObject for one payload type. The discipline is documented but not enforced by the type system.

Direction. Make the bus generic over a game-supplied event map — engine exports only EngineEventMap (tick/resize/resources*), game/ defines GameEventMap and composes the typed emitter. Mirrors the asset-manifest / level-type / InputSource injections you already did.

3 · Expandability ceilings

Where the current design resists the next features. (Teardown duplication is reduced — levelEntityArrays() shipped — but the rest of these are untouched and not wrong for today's DK clone.)

High A single Player is hardcoded — "multiple players" is a rewrite

gameDirector.ts — public Player!: Player; one camera target, one ladder-state push, one feel apply; contactRules "Enemy→Player = gameOver" is one-shot

Problem. Every player-facing system assumes one instance. Co-op, split-screen, or a ghost replay would touch all of them.

Direction. Promote to Players: Player[] iterated in Update/RenderUpdate; make the camera target a function of the array (or a designated follow target); move lose/win/feel into per-player state.

High GameObject is a god base-class; inheritance forces all of it onto every entity

gameObject.ts — render + physics + transform + interpolation + collision-group bit-math + dispose, all in one base every entity extends

Problem. A pure sensor that needs no mesh, or a future logic-only/pooled entity, still inherits the entire render+physics+dispose surface, and the shape/material unions are closed (a new collider shape edits the base).

Direction. Move toward composition — split Transform, a Renderable (mesh/material/interpolate/dispose), and a PhysicsBody (collider/groups) into components a GameObject has, not is. Opens an ECS path if you ever want one.

Also here (Medium): the typed-array entity store got a light unification (teardown via levelEntityArrays()); a fully generic Map<type, GameObject[]> would make adding an entity purely additive, at the cost of casts (deferred — your explicit typed arrays read better). Runtime-spawned barrels still live outside the factory registry (new Enemy(...) hardcoded) — a Spawner + runtime factory would unify them; enemy "flavor" is a constructor boolean discriminated by FSM state — define enemy archetypes as data (feel + start state + handler map + resource), mirroring FEEL_PROFILES. And there's still no serialization / save-load / scene-snapshot — extract a Scene/World with toSnapshot()/fromSnapshot() to unlock checkpoints, save/load, and (with the fixed step + interpolation now in place) rollback netcode.

4 · Performance & memory

The hot paths are now allocation-free (the player/enemy per-frame churn and the shared-geometry dispose are ✅ shipped). What's left is lower-stakes.

Medium Barrels are compacted but not pooled — spawn/destroy still churns bodies & meshes

SpawnEnemy/SpawnCrazyEnemy build a fresh rigid body + collider + cloned mesh each spawn; Destroy tears them all down (every 2–3s)

Problem. In-place compaction removed the per-frame array churn and the 5s linger, but each barrel's birth and death still allocates/frees a Rapier body, a collider, and a cloned mesh group. At DK barrel rates it's fine; it's the next lever if you push entity counts.

Direction. A small barrel pool: reuse a fixed set of rigid bodies/meshes, re-seeding position + FSM state on spawn instead of constructing/destroying. (More complex than today's compaction — defer until you actually need it.)

Also here (Low): cameraTargetFromPlayer still news a fresh CameraTarget object each frame (reuse one struct on the director); PhysicsDebug rebuilds line geometry + typed arrays and walks the whole scene graph every frame (pre-size + throttle — debug-only); and there's no InstancedMesh/batching for identical barrels (caps swarm-scale).

5 · Correctness

The KCC enter/exit edges, the failed-load rollback, and the type escape hatches are ✅ shipped. One implicit invariant remains.

Medium Contact handlers destroy entities synchronously mid-rule-iteration

contactRules.ts — barrel→TrashCan emits gameObjectRemoved (→ enemy.Destroy()) from inside ContactRegistry.Dispatch, while the physics event drain / KCC diff is still iterating

Problem. Dispatch re-evaluates each rule's predicates fresh, so a later rule acting on a pair after a prior rule destroyed one side is currently safe only because matchesType is null-safe and the current rule set has disjoint pairs. It's an invariant held by luck, not structure.

Direction. Queue gameObjectRemoved for end-of-frame destruction (a deferred-destroy list drained once per tick) instead of destroying synchronously inside the contact dispatch. Cheap, and it makes the invariant explicit.

6 · Developer experience, testing & tooling

The biggest remaining gap, and the one you chose to defer. The fixed timestep + seeded PRNG now make deterministic tests possible for the first time, and the extracted levelNodeTypes map makes a level validator easy.

High Zero tests, zero CI, no typecheck/test/lint scripts

package.json scripts = dev / build / preview only

Problem. Every refactor (the log shows many) is validated only by manually playing the game — no regression net, no gate. (You've been validating these batches by typecheck+build+review by hand; a harness makes that repeatable.)

Direction. Add vitest + a typecheck script (vue-tsc --noEmit) + minimal CI. Seed it on the clean seams: StateMachine routing, ContactRegistry two-sided dispatch, mathHelpers (the seeded PRNG is a one-line golden-sequence test), and the GLB parser.

High State handlers can't be unit-tested without booting the whole app

a handler can only run by booting Experience or faking a Player with `as unknown as Player`; jump math is entangled with rendering + the singleton

Problem. The most valuable thing to test (movement/jump math) is the hardest to reach.

Direction. Define a narrow PlayerMovementContext interface (only what handlers read/write: IsTouching, NextTranslation, Time, Input, the feel scalars, a playAnim(state) callback) and type handlers against that, not the concrete Player. With the fixed timestep, the asserted arc is now stable.

Also here (Medium): binary GLB levels have no in-repo validator — add a tiny Node CLI that imports ParseLevel headless (feed it the extracted levelNodeTypes map) and flags unknown types / dupes / zero-size colliders / unrecognized meta. No eslint/prettier gate. And level-load failure is still surfaced only as a logged string — emit a structured levelLoadFailed event (mirroring resourceLoadFailed) so an embedding app or a test can observe it.

Closing thought. Two batches in, the pattern holds: the decoupling, the contact table, the input-as-data seam, and the FSM are exactly why a fixed timestep, render interpolation, an InputSource interface, and idempotent teardown were small, surgical changes rather than rewrites. Everything left on this list rides the same seams — a test harness, a generic event bus, multiple players, and serialization are each one clean abstraction away.
