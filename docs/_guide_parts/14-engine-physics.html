Physics — the Rapier2D world

Physics wraps a single Rapier2D rigid-body world: it boots the WASM solver, steps it on a fixed timestep, and — the heart of the engine — drains Rapier's per-step collision event queue, resolves each colliding collider back to the owning GameObject, and fans the event out two ways: to the object's own override callbacks AND into the declarative ContactRegistry contact table. It is the engine half of the contact-table system; the game half lives in Config: contactRules. ContactPoints is the tiny per-surface boolean vocabulary the gameplay layer reads off all of this.

This file is the engine's physics subsystem: one RAPIER.World, one RAPIER.EventQueue, a registry mapping collider handles back to game objects, and the dispatch logic that turns raw Rapier events into engine-level callbacks and contact-rule firings. It is owned by the singleton Core: Experience, which does this.Physics = new Physics() and then await this.Physics.Configure() inside its own async Configure() (see Core: Experience), and which drives Physics.Update() first on every tick. Its imports tell the whole story of its job:

  • RAPIER from "@dimforge/rapier2d-compat" — the WASM-backed physics engine. The compat build is chosen deliberately: it bundles the WASM and exposes RAPIER.init() so the world can be created after an await instead of needing a separate fetch/instantiate step.
  • Experience — the singleton back-reference, used to reach Scene (where debug meshes live), Time.Delta (the timestep), and Debug.
  • GameObject — the type the registry resolves collider handles to; the dispatcher reads its OnCollisionEnter/Exit, OnSensorEnter/Exit hooks and IsBeingDestroyed flag (see Entities: GameObject).
  • ContactRegistry — the generic, game-blind contact-rule table this class instantiates and dispatches into (see Entities: ContactRegistry).
  • Debug — optional; only wired when the debug panel is active.

Lifecycle is the engine's standard four-phase shape, but with the constructor replaced by an async Configure() because WASM init cannot be awaited in a real constructor: new Physics() does nothing, Configure() builds the world, Update() steps + drains every frame, Destroy() frees the WASM world.

Line-level notes:

  • World and EventQueue are public because game-side entities reach through Physics.World to create bodies/colliders (e.g. Entities: GameObject's createObjectPhysics calls this.Physics.World.createRigidBody(...)), and Debug: PhysicsDebug walks World.forEachCollider to draw wireframes.
  • Contacts is public so the game can populate it once at startup — Config: contactRules receives experience.Physics.Contacts and calls .Add(...) on it.
  • gameObjectRegistry is the crux of event resolution: Rapier hands you only opaque numeric collider handles in events; this map turns a handle back into the rich GameObject. It is keyed by collider handle, not body handle, because one rigid body may carry several colliders.

public async Configure(): Promise<void>

The real, async constructor. No parameters; returns a promise the caller (Core: Experience) awaits so nothing touches World before the WASM is live. Body, in order:

this.experience = Experience.GetInstance();
this.Scene = this.experience.Scene;

await RAPIER.init();
this.World = new RAPIER.World({ x: 0.0, y: -9.81 });
this.EventQueue = new RAPIER.EventQueue(true);
this.IsPaused = false;

// Initialize the declarative contact-rule table (game fills it at startup)
this.Contacts = new ContactRegistry();

// Initialize GameObject registry for collision event mapping
this.gameObjectRegistry = new Map<number, GameObject>();

// Debug
if (this.experience.Debug.IsActive) {
  this.debug = this.experience.Debug;
  this.debug.InitPhysicsDebug(this);
}
  • await RAPIER.init() — loads/instantiates the Rapier WASM module. Everything below depends on it; this single line is why the whole method (and Experience.Configure) must be async.
  • new RAPIER.World({ x: 0.0, y: -9.81 }) — gravity vector. Negative-Y is "down" in this engine's world space (a standard 2D screen-up convention), at roughly Earth gravity. This is the only place gravity is set; tweak the constant to make the whole game feel heavier/floatier (though most "feel" lives in Attributes: baseFeel via the character controller, not raw gravity).
  • new RAPIER.EventQueue(true) — the boolean is Rapier's autoDrain flag: the queue auto-clears between steps so stale events from a prior frame never leak into the next. We rely on this and never manually clear it.
  • this.Contacts = new ContactRegistry() — created empty here; the engine never adds rules. The game injects them post-configure.
  • The debug block is the seam to Debug: PhysicsDebug: only when Debug.IsActive do we keep a debug reference and call InitPhysicsDebug(this), which hands this (the live Physics) to PhysicsDebug so it can read Scene and World.
Hard-won victory — async over constructor. Rapier's WASM cannot be initialized synchronously. By making the entry point async Configure() instead of a constructor, and having Experience await this.Physics.Configure() before wiring the tick handler, we guarantee no frame ever runs World.step() against an uninitialized solver. This is why new Physics() is a no-op.
Gotcha — ordering. this.experience = Experience.GetInstance() must run first, and this.Scene is read from it immediately, so Experience must have set its Scene before calling Physics.Configure(). It does: this.Scene = new THREE.Scene() is assigned a few lines before this.Physics.Configure() in Experience.Configure.

public RegisterGameObject(gameObject: GameObject): void

Records every collider of a game object's rigid body into gameObjectRegistry so future events can be resolved. Called by Entities: GameObject at the end of createObjectPhysics (after the body + collider exist).

if (!gameObject.PhysicsBody) {
  console.warn("Cannot register GameObject without a physics body");
  return;
}

// Register all colliders for this rigid body
const numColliders = gameObject.PhysicsBody.numColliders();
for (let i = 0; i < numColliders; i++) {
  const collider = gameObject.PhysicsBody.collider(i);
  this.gameObjectRegistry.set(collider.handle, gameObject);
}

Loops over all colliders, not just collider 0, so a multi-collider body (e.g. a sensor collider plus a solid collider) maps every handle to the same owner. The guard returns early (with a warning) if there is no body — registration is meaningless without one.

public UnregisterGameObject(gameObject: GameObject): void

The mirror of registration: deletes every collider handle of the body from the map. Called by Entities: GameObject inside Destroy(), before World.removeRigidBody() — the colliders must still exist to enumerate their handles.

if (!gameObject.PhysicsBody) {
  return;
}

const numColliders = gameObject.PhysicsBody.numColliders();
for (let i = 0; i < numColliders; i++) {
  const collider = gameObject.PhysicsBody.collider(i);
  this.gameObjectRegistry.delete(collider.handle);
}
Gotcha — unregister before removeRigidBody. World.removeRigidBody() destroys the colliders, after which numColliders()/collider(i) can no longer give you the handles to delete. GameObject.Destroy calls UnregisterGameObject(this) first, then removes the body — reverse that order and you leak stale handles in the map.

private getGameObject(colliderHandle: number): GameObject | undefined

The internal one-line lookup used by the event dispatcher: return this.gameObjectRegistry.get(colliderHandle);. Returns undefined for any handle that was never registered (e.g. a body created directly against World without going through RegisterGameObject) — the dispatcher treats that as "skip."

public GetGameObjectFromCollider(collider: RAPIER.Collider): GameObject | undefined

The public sibling of getGameObject — same map lookup, but takes a RAPIER.Collider object and reads its .handle. It exists for one specific, subtle case the comment spells out:

// Resolve a collider to its owning GameObject (or undefined if unregistered).
// Lets a kinematic character controller feed its OWN contacts into the contact
// registry: a kinematic body keeps a skin gap from solids it moves into, so
// Rapier emits no collision event for it as the mover — but the controller's
// computed collisions still report what it touched. The game can resolve those
// to GameObjects and Dispatch them, so the same contact rules fire either way.
public GetGameObjectFromCollider(
  collider: RAPIER.Collider
): GameObject | undefined {
  return this.gameObjectRegistry.get(collider.handle);
}
Hard-won victory — kinematic mover gets no event. The player uses a Rapier kinematic character controller, which moves with a skin gap and resolves penetration itself. Because it never actually overlaps the solids it slides against, Rapier's EventQueue emits no collision event with the player as the mover. The controller's computedCollision(i) list still reports what it touched, though — so the player can call GetGameObjectFromCollider(hit.collider) and Contacts.Dispatch(...) manually, and a rule like "enemy touches player → game over" fires whether the enemy rolled into the player (a real event) or the player walked into the enemy (a computed collision). Same table, both directions. See Architecture: Collisions.

private handleCollisionEvents(): void

The engine's contact-dispatch core, called once per step from Update(). It drains the EventQueue and, for each event, resolves both colliders to game objects, decides sensor-vs-solid, fires the two objects' override hooks, and dispatches the matching declarative contact rules. There is no hard-coded "if player and enemy…" anywhere — that knowledge lives entirely in the game's ContactRegistry rules.

The drain callback signature is Rapier's: (handle1, handle2, started) — two collider handles and a boolean that is true for "contact began" and false for "contact ended."

this.EventQueue.drainCollisionEvents((handle1, handle2, started) => {
  const collider1 = this.World.getCollider(handle1);
  const collider2 = this.World.getCollider(handle2);

  // Skip if colliders don't exist
  if (!collider1 || !collider2) {
    return;
  }

  const gameObject1 = this.getGameObject(handle1);
  const gameObject2 = this.getGameObject(handle2);

  // Skip if either GameObject is not registered or is being destroyed
  if (
    !gameObject1 ||
    !gameObject2 ||
    gameObject1.IsBeingDestroyed ||
    gameObject2.IsBeingDestroyed
  ) {
    return;
  }

Two guard layers before any dispatch:

  • Colliders still alive — World.getCollider(handle) returns falsy if that collider was removed earlier in the same drain (e.g. a contact rule destroyed one body, and a later event still references it). Skipping avoids touching a freed handle.
  • Both registered and neither dying — an unregistered handle resolves to undefined; a GameObject.IsBeingDestroyed === true means it is mid-teardown.
Hard-won victory — the IsBeingDestroyed guard. GameObject.Destroy sets this.IsBeingDestroyed = true as its very first line, before removing the body. Combined with the check here, this prevents the classic bug where a contact handler runs against a half-torn-down object (mesh disposed, body removed) and dereferences something undefined. It is the exact mechanism behind commit b0436b4's "finally fix barrel/enemy bug." It also means dispatch is naturally re-entrancy-safe: a rule that destroys a body during the drain can't trigger a follow-up event against the corpse.

Next, route sensor vs. solid:

// Check if either collider is a sensor to route to appropriate callbacks
const isSensor1 = collider1.isSensor();
const isSensor2 = collider2.isSensor();
const isSensorEvent = isSensor1 || isSensor2;

if (isSensorEvent) {
  // This is a sensor intersection event
  if (started) {
    if (isSensor1 && gameObject1.OnSensorEnter) {
      gameObject1.OnSensorEnter(gameObject2);
    }
    if (isSensor2 && gameObject2.OnSensorEnter) {
      gameObject2.OnSensorEnter(gameObject1);
    }
    // Fire any matching declarative contact rules (two-sided internally)
    this.Contacts.Dispatch("sensorEnter", gameObject1, gameObject2);

    if (this.debug && this.debug.LogSensors) {
      this.debug.LogSensorEvent(gameObject1, gameObject2, "enter");
    }
  } else {
    // ... mirror: OnSensorExit + Dispatch("sensorExit", ...) + LogSensorEvent(..., "exit")
  }
} else {
  // This is a solid collision event
  if (started) {
    if (gameObject1.OnCollisionEnter) gameObject1.OnCollisionEnter(gameObject2);
    if (gameObject2.OnCollisionEnter) gameObject2.OnCollisionEnter(gameObject1);
    this.Contacts.Dispatch("enter", gameObject1, gameObject2);
    // ... LogCollisionEvent(..., "enter")
  } else {
    // ... OnCollisionExit + Dispatch("exit", ...) + LogCollisionEvent(..., "exit")
  }
}

Line-level reasoning for the routing:

  • Sensor detection is a property of the collider, not a separate event stream. In Rapier a sensor collider produces collision events (when armed with ActiveEvents.COLLISION_EVENTS), not a distinct intersection channel — so this one drain loop handles both, and collider.isSensor() is the discriminator. If either collider is a sensor, the whole event is treated as a sensor/intersection event.
  • The override hooks are one-sided, gated on the sensor flag. For sensor events, only the side that is the sensor gets its OnSensorEnter/Exit called (isSensor1 && gameObject1.OnSensorEnter) — the object passing through a trigger should not receive a "you are a sensor" callback. For solid events both sides are notified symmetrically. Every hook is optional (the ? on the method) so the if (…OnCollisionEnter) presence-check is mandatory.
  • The four phase strings — "enter", "exit", "sensorEnter", "sensorExit" — are the bridge to the contact table. They match exactly the ContactPhase union in Entities: ContactRegistry. Contacts.Dispatch(phase, gameObject1, gameObject2) is two-sided internally: the registry tests each rule as both (a,b) and (b,a), so dispatch passes the raw pair in handle order and lets the table sort out direction. That's why a rule fires no matter which body Rapier listed first.
  • Logging is opt-in and game-owned. this.debug.LogSensors / LogCollisions are getters that delegate to a game-registered CollisionLogSink (see Debug: Debug), so the engine forwards log events without knowing the concrete logger. Off by default; toggled in the debug GUI.
Note — enter/exit only, no "stay." Rapier's collision events fire on transition, not per-frame. There is no "still touching" event. The gameplay layer that needs continuous "am I on the ground right now" state does not get it from here — it recomputes per frame from the kinematic controller and writes ContactPoints booleans (the second file in this section). GameObject.enablePhysicsEvents even documents this: "Physics only dispatches enter/exit (no per-frame 'stay')."
Gotcha — events only fire if the collider is armed. A collider emits nothing unless setActiveEvents(RAPIER.ActiveEvents.COLLISION_EVENTS) was called on it. GameObject arms this automatically for any entity that defines an OnCollision*/OnSensor* callback (enablePhysicsEvents), and exposes enableContactEvents() for entities that only want the declarative table. If a contact rule "isn't firing," check the collider is armed before suspecting this dispatcher.

Sets this.IsPaused = paused. That's all it does — and that minimalism is the point.

Note — decoupling seam. The engine deliberately does not subscribe to "gameOver"/"gameWin" on the event bus. The game's GameDirector decides when to pause and calls SetPaused(true); the engine stays ignorant of game concepts. IsPaused is public for reads (e.g. UI), SetPaused is the intent-revealing setter.

public Update(): void

The per-step advance. Experience's onTick now runs a fixed-step accumulator: it calls Physics.Update() 0..N times per frame — once per accumulated 1/60s step, each time before that step's OnGameUpdate — and runs Renderer.Update once at the end of the frame. So game logic and rendering always see a freshly-stepped world.

if (this.IsPaused) {
  return;
}

// Time.Delta is the FIXED simulation timestep (set by the frame loop), so each
// physics step is constant and reproducible. No clamp needed — the frame loop
// bounds how often we step (the old Math.min(..., 0.1) was dead code anyway,
// since Time.Delta was already capped well below 0.1).
this.World.timestep = this.experience.Time.Delta;
this.World.step(this.EventQueue);

// Process collision and sensor events
this.handleCollisionEvents();

// Run debug physics logic if needed
if (this.debug) {
  this.debug.UpdatePhysicsDebug(this);
}
  • Pause short-circuit — when paused, the world is not stepped and no events are drained, freezing the simulation while the render loop still runs.
  • Fixed timestep this.World.timestep = this.experience.Time.Delta — Time.Delta is no longer the raw frame delta; it is the fixed simulation step (1/60s) that Core: Experience's accumulator assigns immediately before each Update(). Every step is the same constant size, so the simulation is frame-rate-independent and reproducible. The old Math.min(Time.Delta, 0.1) cap was deleted as dead code: the accumulator can only ever feed this a constant 1/60, and the wall-clock catch-up is bounded upstream — Core: Time clamps FrameDelta (the value that drives the accumulator, not the integration step) to 1/30. A single giant step can no longer reach Rapier, so nothing tunnels through walls on a frame hitch.
  • World.step(this.EventQueue) — advances the solver and fills the (auto-drained) queue with this step's collision/sensor transitions. Passing the queue is what makes events available to the very next line.
  • handleCollisionEvents() runs immediately after the step, in the same frame, so callbacks and contact rules act on the just-resolved positions.
  • Debug last — UpdatePhysicsDebug(this) rebuilds the collider wireframe and refreshes object counts (see Debug: PhysicsDebug), only if a debug ref was captured in Configure.
Gotcha — never set Time.Delta to 0. A zero timestep stalls the solver, so Core: Time flags the field "Never 0." In practice the frame loop always assigns Delta = Time.FixedTimestep (1/60) before each step, and the constructor seeds it to the same value, so this line never receives 0 — but preserve that invariant if you ever touch the loop.
Hard-won victory — a true fixed timestep. This used to be a variable (clamped real-delta) step. It is now a genuine accumulator-based fixed timestep: Core: Experience's onTick accumulates FrameDelta and runs Physics.Update() (with the matching OnGameUpdate) in 0..N constant 1/60s steps per frame, rendering once, while Rapier always receives exactly that fixed step. The payoff is a simulation that is frame-rate-independent and reproducible — identical inputs play out identically on a 60Hz and a 144Hz display.

public Destroy(): void

Tears down the WASM world and the debug wireframe. Called by Experience.Destroy in its "camera then physics then renderer then resources" order.

// Dispose of the Rapier world (guarded so a Destroy after a half-failed
// Configure — e.g. RAPIER.init() threw before World was built — doesn't cascade).
this.World?.free();

// Tear down the debug wireframe (PhysicsDebug owns its own mesh)
if (this.debug) {
  this.debug.DestroyPhysicsDebug();
}
  • World?.free() releases the WASM memory backing every body, collider, and joint in one call — there is no per-body cleanup loop here because individual entities free their own bodies during their Destroy(); this is the final sweep of whatever remains. The ?. is deliberate: if Configure() aborted before World was built (e.g. RAPIER.init() threw), World is still undefined, and the optional chain makes Destroy() a safe no-op instead of throwing a second error on top of the first.
  • DestroyPhysicsDebug() removes and disposes the wireframe LineSegments mesh, which PhysicsDebug owns (Physics itself holds no debug mesh).
Note — teardown survives a half-failed boot. Guarding with this.World?.free() is part of the engine's broader teardown hygiene: Core: Experience now resets its singleton and uses optional chaining when tearing down subsystems, so a Configure() that died partway through can still be cleanly Destroy()ed and retried. Physics holds up its end of that contract — never assume World exists just because Destroy() was called.
Warning — contact-force events are not used here. Rapier's EventQueue can also surface contact-force events via drainContactForceEvents (impulse magnitudes between solids). This engine never drains them — only drainCollisionEvents is called — because the gameplay only needs enter/exit transitions and trigger overlaps, not force magnitudes. If you ever want "hard landing" damage scaled by impact force, that is where you'd add a second drain pass; today it does not exist, so don't expect a force-events code path.

Modifying this: To add a new kind of contact behavior, you almost never touch this file — add a rule to Config: contactRules (game data) or an OnCollision*/OnSensor* override on the entity. Touch handleCollisionEvents only to change the dispatch mechanism itself (e.g. add a drainContactForceEvents pass, or a new phase string — which must also be added to ContactPhase in Entities: ContactRegistry). To change global feel, edit the gravity vector in Configure or the fixed step (Time.FixedTimestep) that Core: Time defines. Keep this class game-blind: if you find yourself importing an EntityType or checking a specific name here, the logic belongs in the game layer instead.

src/engine/types/contactPoints.ts 34 LOC

A pure TypeScript interface — no runtime code — describing the set of surfaces a character can be touching right now, as eight booleans. It is the engine's contribution to the per-frame "where am I?" vocabulary; the engine defines the shape, the game owns an instance and decides what each flag means for gameplay. Concretely, Player (game layer) declares public IsTouching!: ContactPoints and rebuilds it every frame: the player's kinematic-controller collision pass writes ground/ceiling/leftSide/rightSide/edgePlatform, and the ladder sensor writes ladderCore/ladderTop/ladderBottom. Player state handlers (Player states: Idle/Running/Jumping/Falling/Climbing) read these to decide transitions, and PlayerDebug binds them into the dat.GUI as live checkboxes.

Note — this is the "stay" state the event queue won't give you. Because Rapier only emits enter/exit transitions (see handleCollisionEvents above), continuous "still on the ground" information has to be recomputed each frame. ContactPoints is that recomputed snapshot — the bridge between raw physics events and the gameplay state machine. It is reset to all-false and refilled every tick, so reading a stale value across frames is never a concern.

export default interface ContactPoints {
  // ===== BASIC COLLISION DIRECTIONS =====
  ground: boolean;
  ceiling: boolean;
  leftSide: boolean;
  rightSide: boolean;

  // ===== SPECIAL PLATFORM TYPES =====
  edgePlatform: boolean;

  // ===== LADDER COLLISION ZONES =====
  ladderCore: boolean;
  ladderTop: boolean;
  ladderBottom: boolean;
}

Why these eight, and why split this way:

  • Four cardinal directions are the minimum a 2D platformer needs: ground (jump gating + coyote time), ceiling (head-bonk velocity cancel), and the two walls (movement blocking / wall slide). They come from the player's own per-frame collision query, not from the contact table — this is "self" state, not "who hit me."
  • edgePlatform is separated out because an edge tile is still "ground," but unlocks edge-specific behavior; the player sets it by resolving the ground collider to its Platform and reading platform.IsEdge.
  • The three ladder zones exist because a ladder is not one trigger but three overlapping sensors — a tall core for climbing plus thin top/bottom bands for the hand-off to standing. The state machine combines them, e.g. "in core, and not (at top while pressing up) and not (at bottom while pressing down)" to decide whether to enter the climbing state (see Player states: Running/Idle/Climbing). These three are written by the LadderSensor's sensor callbacks — the engine-side sensor events from this section's handleCollisionEvents flowing through to gameplay booleans.
Tip — naming convention. The fields stay camelCase deliberately (unlike the PascalCase public API elsewhere) because they mirror "real types" the way Rapier's own x/y/z vectors do; PlayerDebug notes this exact rationale where it binds them into the GUI.

Modifying this: Add a boolean here whenever the player needs a new persistent "am I touching X?" surface (e.g. onSlope, inWater). Then do two things or the flag is dead: (1) reset it to false in the player's per-frame reset block, and (2) set it true from whatever detects it — a collision pass or a sensor callback. Because this is just a type, adding a field is free at runtime; the cost is the gameplay code that must populate and read it. Keep it boolean and surface-oriented — numeric per-instance data does not belong here (per Types: userData, that lives as typed fields on entity classes).
