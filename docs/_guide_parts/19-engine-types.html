Engine Types — the shared vocabulary

Nine tiny files, zero runtime behaviour, enormous decoupling power. These are the plain data shapes and string/number enums that let the engine and the game speak to each other without either holding a reference to the other's classes. Every one is either a const-object enum (the same-named const + type idiom), a data-shape interface, a one-method seam interface (InputSource), or both — the enums chosen so the same identifier works as a value (PlayerStates.IDLE) and as a type annotation (: PlayerState). Read this section as the dictionary the rest of the guide is written in.

Note — two flavours of "engine type". Some of these are genuinely engine-pure: CameraTarget was deliberately invented so the Engine Camera can follow a target while naming zero game concepts (decision C5/D3). Others — PlayerStates, PlayerDirection, ContactPoints — physically live under engine/types/ but are named for the player and are consumed almost entirely by the game's Player. They are the project's documented "known residue": shared vocabulary parked in the engine folder. Where it matters below, the prose says honestly which is which — and points to the one place (the camera) that pointedly refuses to import the player's enum.

src/engine/types/cameraTarget.ts 25 LOC

This file IS the seam between the game's Player and the Engine Camera. It exports two things: a CameraFollowState enum (the camera's own vertical-follow modes, expressed in engine words) and a CameraTarget interface (the per-frame snapshot the camera reads). Nobody "constructs" it — the Game Director fills a fresh CameraTarget object each frame from its Player and hands it to Camera.Update. It imports nothing. Its lifecycle is the lifecycle of a value type: born in GameDirector.cameraTargetFromPlayer(), dead the moment Camera.Update returns.

Hard-won victory — C5 / D3 decouple. Before this file, camera.ts reached straight into the live Player — it read player.CurrentPosition, player.NextTranslation.x, player.IsTouching.ground, and switched on PlayerState. That made the engine camera un-reusable: it named game concepts. CameraTarget carries exactly those scalars and nothing else, so the camera's easing branches are preserved byte-for-byte while the camera now imports nothing from game/. A different game — or a multi-player follow — just fills the snapshot differently.

export const CameraFollowState = { GROUNDED, CLIMBING, FALLING, JUMPING } as const

The camera's four vertical-follow modes, in engine terms — note there is no RUNNING or IDLE here, because to the camera "standing still" and "walking" are the same grounded behaviour. The game maps its richer PlayerState onto these four. The as const freezes the literal string values so they double as a type. 

export const CameraFollowState = {
  GROUNDED: "grounded",
  CLIMBING: "climbing",
  FALLING: "falling",
  JUMPING: "jumping",
} as const;

export type CameraFollowState =
  (typeof CameraFollowState)[keyof typeof CameraFollowState];
ModeCamera behaviour (per Engine Camera Update)
GROUNDEDRest at the baseline; while truly grounded, re-baseline baselineY to the player's Y so walking up terrain pans up.
CLIMBINGTrack the player upward with a climbLeadY lead (next handhold visible); releases any CameraSensor pin.
FALLINGFollow down only once the player drops past fallBufferY below the camera; leads slightly with fallLeadY; capped rate.
JUMPINGHold the baseline — deliberately does not chase the jump arc.

The same-named const + type pattern (line 13's comment calls it "the playerStates.ts idiom") means CameraFollowState.CLIMBING is a value and followState: CameraFollowState is a type — one declaration, both jobs.

export interface CameraTarget

The per-frame follow input. Five fields, each one the exact scalar the camera used to read off the live Player. The camelCase field names are deliberate (decision D15: data-shape interfaces use camelCase, unlike the PascalCase public members on classes). 

export interface CameraTarget {
  x: number;            // world X to follow (was player.CurrentPosition.x)
  y: number;            // world Y to follow (was player.CurrentPosition.y)
  velocityX: number;    // horizontal velocity -> look-ahead direction (was NextTranslation.x)
  followState: CameraFollowState; // vertical-follow mode (mapped from the game's state)
  isGrounded: boolean;  // truly on the ground -> grounded re-baseline (was IsTouching.ground)
}
FieldTypeConsumed in Camera.Update for…
xnumberBase of targetX before the eased look-ahead is added.
ynumberClimb/fall tracking target and the grounded re-baseline value.
velocityXnumberSign (vs MOVE_EPSILON) decides the look-ahead direction; below epsilon the look-ahead eases to 0.
followStateCameraFollowStateThe switch that selects vertical-follow behaviour.
isGroundedbooleanGuards the grounded re-baseline (only re-anchor Y when truly on the ground and no sensor pin).
Producer vs consumer. Producer: Game Director — cameraTargetFromPlayer(player) maps player.State through a switch into a CameraFollowState (CLIMBING / FALLING / JUMPING, else GROUNDED) and copies player.CurrentPosition, player.NextTranslation.x, player.IsTouching.ground. Consumer: Engine Camera — Update(target?: CameraTarget). Note the ?: the camera guards against the target not being ready yet and returns early. 
flowchart LR

Modifying this: if the camera needs a new input (say a zoom hint), add a field here and read it in Camera.Update — then fill it in GameDirector.cameraTargetFromPlayer. Keep the field a plain scalar/enum; the whole point of C5 is that the camera never imports a game class, so never add a Player-typed field. To add a follow mode, extend CameraFollowState and add a case in both the camera's switch and the director's mapping switch.

src/engine/types/contactPoints.ts 34 LOC

A single default-exported interface ContactPoints: eight booleans describing which surfaces a GameObject is currently touching. It imports nothing. In practice the only thing that owns one is the game's Player (player.IsTouching), which it seeds in its constructor and rewrites every frame from shapecasts and ladder sensors. The player's state handlers and PlayerDebug read it. It carries no logic — it is the vocabulary the contact-detection code writes and the state machine reads.

export default interface ContactPoints

The fields fall into three groups, flagged by the section comments in the file.

export default interface ContactPoints {
  // ===== BASIC COLLISION DIRECTIONS =====
  ground: boolean;
  ceiling: boolean;
  leftSide: boolean;
  rightSide: boolean;
  // ===== SPECIAL PLATFORM TYPES =====
  edgePlatform: boolean;
  // ===== LADDER COLLISION ZONES =====
  ladderCore: boolean;
  ladderTop: boolean;
  ladderBottom: boolean;
}
FieldSet true when…Used by the player for…
groundDown-shapecast hits a floor.Enables jumping, resets coyote time, drives idle/run animation; also feeds CameraTarget.isGrounded.
ceilingUp-shapecast hits.Cancels upward velocity; prevents jumping through ceilings.
leftSideLeft-shapecast hits a wall.Blocks leftward movement (see applyHorizontalMovement); can enable wall slide.
rightSideRight-shapecast hits a wall.Blocks rightward movement; can enable wall slide.
edgePlatformStanding on a platform whose IsEdge is true.Edge-specific animations / mechanics.

ladderTopAt a ladder's top sensor.Transition climb → stand on the platform above.
ladderBottomAt a ladder's bottom sensor.Transition climb → stand on the ground below.
Producer vs consumer. Producer: the Player writes every field. ground/ceiling/leftSide/rightSide come from four directional shapeCast calls (Object.assign(this.IsTouching, {…})), edgePlatform from the platform under the feet, and the three ladder flags from UpdateLadderState(top, core, bottom) — which the Game Director feeds from the ladder sensors. Consumers: the player's five state handlers (handlePlayer{Idle,Running,Jumping,Falling,Climbing}), applyHorizontalMovement, and PlayerDebug (read-only display).
Gotcha — ordering. IsTouching is mutated in place each frame (the player keeps one object for its whole life). Contact detection must run before the state handlers in Player.Update, or the FSM reads last frame's surface state. The booleans are an engine-neutral surface description, but only the player ever fills one today.

Modifying this: add a boolean here only when a new surface relationship exists (e.g. a movingPlatform contact). Then set it in the player's detection pass and consume it in a handler — and remember to seed it false in the player's IsTouching initializer, or it reads undefined on frame one.

src/engine/types/gameObjectType.ts 14 LOC

A default-exported const enum of the three physics collider shapes the engine supports: CUBE, SPHERE, CAPSULE. It imports nothing. It is the contract between every game entity (which picks a shape) and GameObject (which turns that string into a Rapier collider and a Three.js debug mesh).

const GameObjectType = { CUBE: "cube", SPHERE: "sphere", CAPSULE: "capsule" } as const

const GameObjectType = {
  CUBE: "cube",     // players, platforms, walls, slopes (rotated cubes)
  SPHERE: "sphere", // rolling enemies, projectiles, round objects
  CAPSULE: "capsule", // smooth-moving characters (won't catch on ledges)
} as const;

export default GameObjectType;
ValueStringRapier collider (in GameObject.createCollider)
CUBE"cube"ColliderDesc.cuboid(width/2, height/2) — also the default fall-through.
SPHERE"sphere"ColliderDesc.ball(width/2)
CAPSULE"capsule"ColliderDesc.capsule(height/2, width)
Hard-won victory — D5 / "D-lean". The file's header records a deliberate purge: only these three primitives survive. Slopes are built from rotated cubes and complex shapes from overlapped cubes, so polyline and convex-mesh colliders were removed entirely. That keeps GameObject.createCollider a tiny three-way switch and avoids the whole class of degenerate/auto-generated mesh-collider bugs.
Producer vs consumer. Producers: the game entities pick a shape and pass it into createObjectPhysics — e.g. Platform and the Player both pass GameObjectType.CUBE; Enemy, TrashCan, WinFlag, Teleporter, CameraSensor, LadderSensor each pick the shape that fits. Consumer: the engine's GameObject — createCollider (Rapier shape) and createObjectGraphicsDebug (the matching Three.js debug geometry).
Tip. GameObject.createCollider takes a plain gameObjectType: string, not the enum type. An unrecognised string is not an error — it falls through to CUBE. That is intentional (a missing shape becomes a box rather than a crash), but it means a typo'd shape name silently becomes a cube.

Modifying this: adding a fourth shape means three edits in lockstep — a new key here, a new case in GameObject.createCollider (Rapier ColliderDesc), and a new case in createObjectGraphicsDebug (Three.js geometry). Resist it unless rotated/overlapped cubes genuinely can't express the shape — that restraint is the whole D5 point.

src/engine/types/inputSource.ts 12 LOC

A default-exported interface InputSource with a single method — the seam between whatever produces input and the entities that consume it. It is the one behavioural interface in this folder (it declares a method, not data fields), and it imports only its sibling InputState. The keyboard Input device implements it today; a touch, gamepad, network, or replay source drops in by implementing the same one method, with no edits to the Player or any other entity.

export default interface InputSource { CaptureInto(state: InputState): void }

One method. CaptureInto is handed an entity's InputState bag and fills all nine intent booleans for the current frame, mutating the bag in place (see inputState.ts below). The interface owns no state of its own — it is purely the contract "given a snapshot, write this frame's intent into it". 

import InputState from "./inputState";

export default interface InputSource {
  // Fill the given InputState snapshot from this source for the current frame.
  CaptureInto(state: InputState): void;
}
Hard-won victory — depend on the seam, not the device. Player's inputDevice field is typed InputSource, not the concrete keyboard Input class. So the movement code calls this.inputDevice.CaptureInto(this.Input) against the interface, and a different producer — touch, gamepad, a recorded replay, an AI — is a one-file drop-in: write a class that implements InputSource and assign it. Nothing in the entity changes. This is the behavioural twin of InputState's "input-as-data" decouple: the data shape stays the same, and now the source of that data is an interface too — not a concrete class the entity has to name.
Producer vs consumer. Producer (implementer): Engine Input — the keyboard device implements InputSource and supplies CaptureInto. Consumer: the game's Player (and any future controllable entity) holds an inputDevice: InputSource and calls CaptureInto(this.Input) at the top of Update — depending on this interface, never on the concrete device.

Modifying this: resist widening it. A new input source is a new class that implements InputSource, not a change here. Only touch the interface if every source must grow a new capability (say a Rumble() back-channel), and then implement it in every producer. To add a new intent, edit InputState instead (a field there, a line in each CaptureInto).

src/engine/types/inputState.ts 18 LOC

A default-exported interface InputState: a plain per-frame snapshot of input intent. It imports nothing. It is the decoupling seam in Engine Events & Input: controllers read these booleans instead of polling the live keyboard device, so the source of input (keyboard today; network/replay/AI later) can change without touching movement code.

export default interface InputState

Nine booleans. The field names mirror the Input device's intent methods exactly — so a handler reads player.Input.isLeft (a value) where it once called input.IsLeft() (a method): same answer, one fewer pair of parens. 

export default interface InputState {

  isLeft: boolean;
  isRight: boolean;
  isLeftRightCombo: boolean;
  isNeitherLeftRight: boolean;
  // Jump intent
  isJump: boolean;
  // Vertical intent (ladders)
  isUp: boolean;
  isDown: boolean;
  isUpDownCombo: boolean;
  isNeitherUpDown: boolean;
}
FieldFilled from Input…True when
isLeftIsLeft()Left held, right not.
isRightIsRight()Right held, left not.
isLeftRightComboIsLeftRightCombo()Both left+right held.
isNeitherLeftRightIsNeitherLeftRight()Neither held.
isJumpIsJump()Spacebar held.

Hard-won victory — D10 "input-as-data". The Player's constructor comment marks this: each entity owns its own InputState bag and refreshes it from the device every frame (this.inputDevice.CaptureInto(this.Input) at the top of Player.Update). Because handlers read the bag, not the device, you could swap the producer for a replay file or a network packet without changing a single line of movement logic.
Producer vs consumer. Producer: Engine Input — CaptureInto(state) writes all nine fields, mutating the bag in place so existing references stay valid. Consumer: the game's Player owns the bag (seeded in initializePlayerAttributes with isNeitherLeftRight: true / isNeitherUpDown: true) and its state handlers read it.
Gotcha — mutated in place. CaptureInto overwrites the same object; it never allocates a new one. Don't stash a reference expecting an immutable snapshot of "this frame's input" — it will change next frame. If you need history (combos, buffering), copy the fields out.

Modifying this: a new intent means three coordinated edits — a field here, a matching Is* getter on Input, and a line in Input.CaptureInto. Then seed it in the player's Input initializer. Keep names mirroring the device method (drop the parens, lowercase the first letter).

src/engine/types/levelData.ts 35 LOC

The shape the GLB parser emits and the entity factories consume. Two interfaces: LevelEntity (one spawnable object) and LevelData (a dictionary of named entities). The default export is LevelData. It imports nothing. GLB is the single permanent level source, so this is the one data contract between ResourceLoader (engine) and the game's spawn pipeline.

interface LevelEntity

One GameObject to create — its size, transform, type string, and an optional metadata bag.

interface LevelEntity {
  width: number;   // X axis (horizontal)
  depth: number;   // Y axis in this coord system (into-screen; ~1 for 2D)
  height: number;  // Z axis in this coord system (vertical)
  position: number[]; // [x, y, z] = [horizontal, depth, vertical]
  rotation: number[]; // Euler [x, y, z] radians; usually [0, rotation, 0]
  type: string;    // "Platform", "Wall", "OneWayPlatform", "...Sensor", "Enemy", ...
  meta?: Record<string, string>; // parsed texture-name tokens, e.g. "floor=1 oneway=8"
}
FieldMeaning & how the factories read it
width / height / depthCollider extents. Factories pass { width: data.width, height: data.height } (and depth for walls) into the entity constructors.
positionSlot order [gameX, gameDepth, gameUp]. Factories read position[0] as x and position[2] as y; setCameraStart reads position[1] as depth.
rotationSlot order [-, gameRotation(z), -]. Factories pass -data.rotation[1] (the parser already negated once; see the victory below).
typeThe canonical type string — the same verbatim strings as game/types/entityType.ts. The factory registry is keyed by it.
meta?Generic key=value bag parsed from the mesh's texture name. The engine never interprets a key; the game does (see below).

interface LevelData { [entityName: string]: LevelEntity }

A dictionary keyed by unique entity name. The Game Director iterates Object.values(this.LevelData) to spawn everything. The key is for identity/debugging only — entities are matched by type, never by key. 

interface LevelData {
  [entityName: string]: LevelEntity;
}

export default LevelData;
Hard-won victory — the generic meta bag. The header notes this replaced the old generic value0-3 indices. Previously every entity carried four nameless numbers; now ResourceLoader parses the collision mesh's texture name (e.g. "floor=1 oneway=8") into a Record<string,string> and the game-side factories interpret keys per type (floor/edge/oneway for platforms, dir for ladders, cam/dest for camera/teleporter). The engine parser stays generic and never reads a game-specific key — preserving the engine/game seam through the level pipeline.
Producer vs consumer. Producer: Engine Resources — ResourceLoader.ParseLevel(glbPath) returns Promise<LevelData> via convertToLevelData (which also #n-disambiguates duplicate names so none are dropped). Consumers: the Game Director (LoadLevelData stores it and loops it) and the entity factories in game config, whose EntityFactory signature takes the row as data: any and reads data.meta through metaNum / metaCoords helpers.
Gotcha — axis slots, not names. position/rotation are positional arrays in the parser's fixed slot order, not {x,y,z} objects. The parser is the single place glTF axes map to game axes; the factories rely on that exact slot order ([0]=x, [2]=up). Reorder the slots in one place and you must reorder both.

Modifying this: to teach a new entity new authored data, add a key=value token in the level's texture name — no schema change needed here, meta is already open. Only touch this interface to add a genuinely new structural field (a new size axis, etc.), and then update both ResourceLoader.convertToLevelData (producer) and the factories (consumer).

src/engine/types/playerDirection.ts 19 LOC

A default-exported const of signed numeric direction constants. It imports nothing. Unlike the string enums, these are numbers because they are used arithmetically — multiplied into velocity, fed as shapecast directions, and used to flip sprites. NEUTRAL means no directional input.

const PlayerDirection = { NEUTRAL: 0, RIGHT: 1, LEFT: -1, UP: 1, DOWN: -1 } as const

const PlayerDirection = {
  NEUTRAL: 0, // no input / vertical-only checks
  RIGHT: 1,   // positive X
  LEFT: -1,   // negative X
  UP: 1,      // positive Y
  DOWN: -1,   // negative Y
} as const;

export default PlayerDirection;
ValueNumberAxisUsed for
NEUTRAL0—No input; the "other axis" arg in a single-axis shapecast.
RIGHT+1XRightward velocity / right-side shapecast / sprite faces right.
LEFT-1XLeftward velocity / left-side shapecast / sprite faces left.
UP+1YJump / climb-up / ceiling detection shapecast.
DOWN-1YFall / climb-down / ground detection shapecast.
Gotcha — overlapping numeric values. RIGHT === UP === 1 and LEFT === DOWN === -1. The axis is implied by which argument slot the value occupies, not by the value itself. That is exactly why Player.shapeCast(xDirection, yDirection) takes two args: shapeCast(NEUTRAL, DOWN) casts down, shapeCast(RIGHT, NEUTRAL) casts right. Read these as "a signed step along whichever axis the caller is probing", never as a single global direction.
Producer vs consumer. There is no separate producer — these are constants. Consumers are all game-side: Player seeds this.Direction = PlayerDirection.NEUTRAL and builds its four shapecasts from them; applyHorizontalMovement branches on player.Direction to block movement into a touched wall; and the five player state handlers set Direction from input.

Modifying this: these five cover a 2D platformer completely; you rarely add to them. If you ever need diagonals, prefer a {x,y} vector over a sixth constant — overloading one signed scalar for two axes doesn't extend to diagonals.

src/engine/types/userData.ts 10 LOC

The identity tag attached to every Rapier physics body. A default-exported interface UserData with two strings: type (shared routing flag) and name (per-instance id). It imports nothing. This is the bridge that lets the contact system route collisions by data instead of instanceof — because Rapier callbacks hand back raw colliders, not your classes.

export default interface UserData { type: string; name: string }

export default interface UserData {
  // TYPE flag — shared routing identity; many entities can share one type.
  type: string;
  // Per-INSTANCE identifier; defaults to the type until SetName overrides it.
  name: string;
}
FieldRoleSet / read by
typeShared routing TYPE flag — many entities can carry the same one, so a single contact rule matches all of them.Set by GameObject.SetType (and the spawn default); read by matchesType / isColliderType.
namePer-instance identifier to single out one entity. Defaults to type until SetName overrides it.Set by GameObject.SetName; read for debugging/logging.

The engine treats both as opaque strings — their meaning is assigned entirely by the game layer. Crucially, per-entity numeric metadata is not stored here; it lives as named typed fields on the entity classes, set at spawn from LevelEntity.meta (see levelData.ts above). userData is identity only.

// in GameObject.createObjectPhysics — the one place userData is born

this.PhysicsBody.userData = {
  type: name,
  name: name,
} as UserData;

At creation both fields default to the same incoming name string. The game then calls SetType(EntityType.WALL) (which sets both type and name to the new type) and optionally SetName(...) afterward to give a distinct per-instance id while leaving the routing type intact.

Hard-won victory — type-routing over instanceof. Rapier collision events return a raw Collider, not your GameObject subclass, so instanceof is unavailable at the dispatch point. UserData.type survives that boundary: physicsHelpers' matchesType(gameObject, type) and isColliderType(collider, type) read it, and the Contact Registry rules (declared in game config contactRules) are written as matchesType predicates. That's how the contact table fires "when an A touches a B" without a single hard-coded class check.
Producer vs consumer. Producer: the engine GameObject stamps it in createObjectPhysics and mutates it via SetType/SetName. Consumers: Engine Helpers (getUserData returns a safe { type:"", name:"" } default when the body is undefined; matchesType/isColliderType compare the flag) and, through them, the Contact Registry and the game's contactRules.
Gotcha — SetType resets name. SetType writes the new type to both fields (so a freshly typed entity has name === type by default). If you want a distinct instance id, call SetName after SetType, never before — otherwise the SetType call clobbers your name.

Modifying this: keep it to identity. The temptation to add numeric/gameplay fields here is the thing the design explicitly rejected — put those on the entity class, fed from meta. If you genuinely need a third opaque routing channel (e.g. a faction tag), add a string and a matching matches* helper in physicsHelpers; the engine must still treat it as an opaque string it never interprets.

The big picture. These nine types are the project's lingua franca. Three flow game → engine as plain snapshots (CameraTarget, InputState filled by the game's Player; UserData read by the engine's contact dispatch). One flows engine → game (LevelData from the parser into the factories). Four are shared vocabulary the game owns but the engine folder hosts (ContactPoints, GameObjectType, PlayerDirection, PlayerStates). And one — InputSource — is a behavioural seam: an engine-defined interface the keyboard Input implements and the Player depends on. The discipline that keeps the engine reusable is visible right here: every cross-boundary value is a scalar, string, or boolean, and the one cross-boundary contract is an interface — never a concrete class reference.
