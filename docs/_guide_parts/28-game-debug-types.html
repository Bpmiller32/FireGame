Game Debug & Types — collision log, level/player panels, type enums

The game half of the debug story. Three of these files are the game-owned modules that plug into the engine's game-blind Debug coordinator (the D3 seam): a big dat.GUI player inspector, a console collision/sensor log sink, and a one-line level dropdown. The other two are the tiny identity tables every entity is built from — the EntityType string vocabulary and the Rapier CollisionGroups bitmasks. None of these files is imported by anything under engine/; the game always pushes knowledge into the engine, never the reverse.

src/game/debug/playerDebug.ts src/game/debug/collisionLogger.ts src/game/debug/levelDebug.ts src/game/types/entityType.ts src/game/types/gameCollisionGroups.ts
Where these get wired. All registration is dependency injection into the engine Debug coordinator, done once after Experience.Configure():
  • PlayerDebug — registered by the Player itself in its constructor: this.Debug.RegisterModule(new PlayerDebug(this)) (player.ts), guarded by Debug.IsActive.
  • CollisionLogger — registered in App.vue: Debug.RegisterCollisionLogSink(new CollisionLogger()).
  • LevelDebug — registered in App.vue: Debug.RegisterModule(new LevelDebug(gameDirector)).
EntityType and CollisionGroups are plain const tables imported all over game/ (factories, entities, contact rules) — no lifecycle.

src/game/debug/playerDebug.ts 396 LOC

The big dat.GUI player inspector: a nested folder tree of live readouts (state, position, velocity, contact flags, ladder sensors, jump assists, timers), a Feel Lab of live-writable tuning sliders, a click-to-teleport noclip mode, and a "dump feel values" action. It implements the engine's DebugModule interface (Init(ui) / Destroy()) so the engine can drive it without ever importing it. It is constructed and registered by the Player in its constructor (only when Debug.IsActive), and handed a reference to that same Player so every control binds to live fields.

Imports: three (raycasting for teleport), dat.gui (the panel type passed to Init), Experience (singleton — for the canvas element and the camera the raycaster needs), and the DebugModule interface from engine/debug/debug.ts.

Hard-won victory — the structural Player forward type. The file declares its own local type Player = {…} instead of importing the real class. The header comment says exactly why: Player → Experience → Debug → PlayerDebug → Player would be a circular import. By describing only the shape it touches, PlayerDebug stays a leaf. Note the deliberate casing split called out in the comment: top-level fields mirror the Player class (PascalCase — State, CurrentFloor, JumpPower…), while nested data-shape fields stay camelCase to match the real underlying types (IsTouching.ground, the Rapier vector CurrentTranslation.x/y/z).

public Init(ui: dat.GUI): void

Called by the engine Debug coordinator (immediately on register if the panel is active, see RegisterModule). Builds the entire 🎮 Player Debug folder tree against the live player. Takes the root dat.GUI as its only argument; returns nothing.

public Init(ui: dat.GUI) {
  const player = this.player;
  const experience = Experience.GetInstance();
  const folder = ui.addFolder("🎮 Player Debug");
  folder.close();

  folder.add(player, "State").name("Current State").listen();
  folder.add(player, "CurrentFloor").name("Current Floor").listen();

The key idiom throughout is .listen(): dat.gui then polls the bound field every frame and repaints the control, so a read-only readout like State tracks the live Player with zero per-frame code on our side. The root folder is created .close()d so it does not clutter the panel until expanded. experience is resolved once here only because the noclip handler needs the canvas and camera.

The 💥 Collisions and 🪜 Ladders folders mirror the player's ContactPoints flags (see engine/types/contactPoints.ts) — these are exactly the eight booleans the FSM reads each frame, so watching them here is the fastest way to debug a state-transition bug.

Tip — Feel Lab sliders edit live. The Feel Lab folders (🦘 Jump (Y), 🏃 Horizontal (X), 🅰️ Assists, 🪜 Climb) call .add(player, "JumpPower").min(0).max(200).step(0.5).listen() on writable Player fields. Because the movement handlers (baseFeel / setDkAttributes / setCelesteAttributes) read these fields fresh every frame, dragging a slider changes the game feel instantly. The base profile is still applied on level load from the level registry (the single source of truth) — the Lab only lets you tune on top of the loaded profile, then dump the result.
Note — the Feel Lab slider ranges. Each slider's .min/.max/.step encodes the sane tuning envelope: e.g. JumpPower 0–200 step 0.5, RiseGravity/FallAcceleration 1–800 step 1, JumpEndedEarlyGravityModifier ("Early-Release Mult") 1–8 step 0.1, ApexHangMult 0.05–1 step 0.05, CoyoteTime 0–0.2 step 0.005, BufferJumpRange 0–8 step 0.1, MaxClimbSpeed 0–15. There is intentionally no live profile-swap dropdown here — feel is registry/level-driven so it can't silently diverge from the source of truth.

public Destroy(): void

The DebugModule teardown hook (called by Debug.Destroy() over its registered modules). Only job: drop the teleport click listener so a leftover noclip handler can't fire after the panel is gone.

public Destroy() {
  this.removeClickToTeleport();
}

Note it does not tear down the dat.gui folders — the coordinator destroys the whole Ui in one call, so per-folder cleanup would be redundant.

private dumpFeelValues(): void

Bound to the 📋 Dump Feel Values button. Builds a paste-ready block of player.Field = value; assignments from the player's current (possibly Lab-tuned) feel, logs it, and copies it to the clipboard. Purpose: when you nail a feel in the Lab, this is how you capture it back into source.

const block = [
  "// ── baseFeel.ts (shared) ──",
  `player.MaxClimbSpeed = ${p.MaxClimbSpeed};`,
  // …CoyoteTime, AnimationScalingFactor…
  "",
  "// ── profile overrides (setDkAttributes.ts / setCelesteAttributes.ts) ──",
  `player.MaxGroundSpeed = ${p.MaxGroundSpeed};`,
  // …through JumpPower, BufferJumpRange, MinJumpTime…
].join("\n");

console.log("🎚️ Current feel values:\n" + block);
const copy = navigator.clipboard?.writeText(block);

The block is split by where each field belongs in source — the shared baseFeel.ts values vs the per-profile setters — so you paste each half into the right file. The clipboard write is defensive: navigator.clipboard?. optional-chains (absent in insecure contexts), and the returned promise's .catch falls back to "copy from the log above" when the browser blocks programmatic clipboard access. This is the seam that closes the loop with Game Feel.

private setupClickToTeleport(player: Player, experience: Experience): void

Activated by the noclip toggle's .onChange(true). Installs a window click listener that raycasts the click onto the player's Z-plane and teleports the player to the hit point. The classic level-debug "fly anywhere" tool.

this.removeClickToTeleport();         // ensure only one listener ever

this.clickHandler = (event: MouseEvent) => {
  const canvas = experience.TargetElement;
  if (!canvas) return;

  const rect = canvas.getBoundingClientRect();
  const x = event.clientX - rect.left;
  const y = event.clientY - rect.top;
  if (x < 0 || x > rect.width || y < 0 || y > rect.height) return;

  const mouse = new THREE.Vector2(
    (x / rect.width) * 2 - 1,
    -(y / rect.height) * 2 + 1,
  );
  const raycaster = new THREE.Raycaster();
  raycaster.setFromCamera(mouse, experience.Camera.Instance);

  const playerZ = player.CurrentTranslation.z ?? 0;
  const plane = new THREE.Plane(new THREE.Vector3(0, 0, 1), -playerZ);
  const worldPosition = new THREE.Vector3();
  raycaster.ray.intersectPlane(plane, worldPosition);

  player.TeleportToPosition(worldPosition.x, worldPosition.y);
};
window.addEventListener("click", this.clickHandler);

Line-level notes:

  • Bounds guard — clicks outside the canvas rect are ignored, so clicking the dat.gui panel itself doesn't teleport.
  • NDC conversion — the canvas-local pixel is mapped to normalized device coords ([-1,1], Y flipped) for raycaster.setFromCamera.
  • The Z-plane — this is a 2D game on a fixed Z, so the ray is intersected with the plane z = playerZ (normal (0,0,1), constant -playerZ). CurrentTranslation.z ?? 0 defends against an undefined Z. The result feeds player.TeleportToPosition(x, y) — the same teleport the level loader uses for respawns.
  • Self-cleanup — it calls removeClickToTeleport() first so toggling on twice never stacks two listeners.

private removeClickToTeleport(): void

Detaches the active click listener (if any) and nulls clickHandler. Called from the noclip .onChange(false), from Destroy(), and defensively at the top of setupClickToTeleport.

private removeClickToTeleport() {
  if (this.clickHandler) {
    window.removeEventListener("click", this.clickHandler);
    this.clickHandler = null;
  }
}

Storing the exact handler reference is what makes removeEventListener work — an anonymous re-creation could never be removed. This is why clickHandler is a field, not a local.

Modifying this: to surface a new readout, add the field to the local type Player first (or TypeScript rejects the .add), then a folder.add(player, "Field").name("…").listen() in the matching sub-folder. For a new tuning slider, mirror the Feel Lab pattern (.min/.max/.step/.listen()) and add a line to dumpFeelValues so the dump stays complete. Keep top-level fields PascalCase and nested data-shape fields camelCase to match the real Player.

src/game/debug/collisionLogger.ts 74 LOC

The game-owned console sink for collision and sensor events. It implements the engine's CollisionLogSink interface and is registered in App.vue via Debug.RegisterCollisionLogSink(new CollisionLogger()). The engine Physics step, while iterating Rapier's contact/intersection events, calls back through Debug.LogCollisionEvent / LogSensorEvent — which forward to this sink. This is the D3 decoupling in action: the engine knows only the shape of a log sink; the game owns the formatting.

Imports: Experience (singleton — lazily, only to read the clock for timestamps), GameObject (the engine base type both colliding objects are), and the CollisionLogSink interface from engine/debug/debug.ts. 

sequenceDiagram
    participant R as Rapier events
    participant PH as Physics.Update()
    participant D as Debug (engine)
    participant CL as CollisionLogger (game)
    R->>PH: contact / intersection pair
    PH->>PH: dispatch contact rules + callbacks
    Note over PH: if (debug.LogCollisions)
    PH->>D: LogCollisionEvent(obj1, obj2, "enter")
    D->>CL: forwarded opaquely
    CL->>CL: console.log colored line

public LogCollisions = false; public LogSensors = false;

The two enable flags, both off by default. The engine Debug coordinator proxies these via get/set that delegate to the registered sink (get LogCollisions() { return this.logSink?.LogCollisions ?? false }), and Physics gates every log call on them (if (this.debug && this.debug.LogCollisions)). So flipping logging on means setting experience.Debug.LogCollisions = true (from the console or a future checkbox), which writes straight through to this field. When no sink is registered the getter returns false, so logging is a guaranteed no-op.

public LogCollisionEvent(obj1: GameObject, obj2: GameObject, eventType: "enter" | "exit"): void

Prints one colored line for a solid-collision enter/exit. Early-returns when LogCollisions is off (cheap when disabled). Parameters are the two engine GameObjects Rapier reported and the phase. No return.

if (!this.LogCollisions) return;

const timestamp = this.getTimestamp();
let color = "#ff6666";                 // exit = red
if (eventType === "enter") color = "#00ff00"; // enter = green
let symbol = "↔️";
if (eventType === "enter") symbol = "💥";

console.log(
  `%c${symbol} [${timestamp}] COLLISION ${eventType.toUpperCase()}: %c${
    obj1.constructor.name
  } %c↔ %c${obj2.constructor.name}`,
  `color: ${color}; font-weight: bold;`,
  "color: #4fc3f7; font-weight: bold;",   // obj1 = blue
  "color: #999;",                         // ↔ = gray
  "color: #ffb74d; font-weight: bold;"    // obj2 = orange
);

The %c markers are the browser console's per-span CSS mechanism: each %c consumes the next style string argument, so the line is multicolor (green/red verb, blue obj1, gray arrow, orange obj2). The crucial detail is obj1.constructor.name / obj2.constructor.name — it logs the runtime class name (Player, Enemy, Platform, TrashCan…), not the EntityType string. That's deliberate: it reads cleanly without any lookup table, and the class names happen to line up with the EntityType values.

public LogSensorEvent(obj1: GameObject, obj2: GameObject, eventType: "enter" | "exit"): void

Identical structure for sensor (intersection) events, gated on LogSensors. Different palette/symbols so sensor traffic is visually distinct from solid collisions: enter = 📡 cyan #00ddff, exit = ⚪ magenta #ff88ff; obj1 purple, obj2 green.

if (!this.LogSensors) return;
// …
let color = "#ff88ff";
if (eventType === "enter") color = "#00ddff";
let symbol = "⚪";

Separating sensors from collisions matters because the game leans on sensors heavily (ladder top/core/bottom, camera zones, teleporters) — without the color split a ladder climb would drown the console in indistinguishable lines.

private getTimestamp(): string

Returns the elapsed engine-clock time as a string like "12.34s". Lazily resolves the Experience singleton on first use and caches it.

if (!this.experience) {
  try {
    this.experience = Experience.GetInstance();
  } catch {
    return "0.00s";
  }
}
return `${this.experience.Time.Elapsed.toFixed(2)}s`;

The lazy + try/catch pattern is the safety: CollisionLogger can be constructed before Experience is fully wired, and GetInstance() throws if the singleton isn't ready. Rather than couple construction order, it returns "0.00s" until the engine clock exists, then caches the reference forever after. It reads Time.Elapsed (the engine wall-clock, see engine/core/time) so timestamps line up with everything else in the loop.

Modifying this: to add a third category (say, "trigger" events), add another Log… flag + method to both this class and the CollisionLogSink interface, then call it from Physics behind its flag — the engine forwards opaquely, so it never needs to know what the category means. To label lines by EntityType instead of class name, swap obj.constructor.name for a getUserData(obj.PhysicsBody).type read.

src/game/debug/levelDebug.ts 30 LOC

The smallest module here: a single 🗺️ Level folder with one dropdown that loads any registry level. It implements DebugModule and is registered in App.vue with a reference to the live GameDirector. The whole trick is that the dropdown binds directly to GameDirector.CurrentLevelName with .listen(), so there is no per-frame sync code at all.

Imports: dat.gui, the DebugModule interface, LEVEL_REGISTRY from the level registry (to build the option list), and — critically — GameDirector as a type-only import (import type). The header comment flags this: a value import would create the cycle GameDirector → … → LevelDebug → GameDirector. Type-only erases at compile time, so no cycle.

constructor(director: GameDirector)

Stashes the director reference; the dropdown both reads and writes through it.

public Init(ui: dat.GUI): void

Builds the folder and the single bound dropdown.

const folder = ui.addFolder("🗺️ Level");
folder.open();

folder
  .add(director, "CurrentLevelName", LEVEL_REGISTRY.map((level) => level.name))
  .name("Level (F2 cycles)")
  .listen()
  .onChange((name: string) => director.LoadLevelByName(name));

Four lines doing four things:

  • The option list — LEVEL_REGISTRY.map(l => l.name) turns the registry into the dropdown's choices, so adding a level row automatically adds a menu entry. No duplicate list to maintain.
  • The binding — .add(director, "CurrentLevelName", …) binds the control to the director's public field. Picking an option writes that field.
  • .listen() — dat.gui polls CurrentLevelName every frame and repaints the dropdown. So when you press F2 (the switchLevel event → switchToNextLevel → loadLevelByIndex sets CurrentLevelName), the menu's selection updates itself — no callback wiring from the director back to the GUI.
  • .onChange — picking an item calls director.LoadLevelByName(name), the public registry-name load path (it findIndexes the registry and funnels into the same loadLevelByIndex guarded loader). The folder is opened by default since switching levels is a common debug action.
Gotcha — the dropdown sets the field, then .onChange loads. Both directions move through CurrentLevelName. When you pick: dat.gui writes CurrentLevelName, then fires onChange → LoadLevelByName. When F2 fires: the director sets CurrentLevelName itself and .listen() reflects it without re-firing onChange (that only fires on user input), so there's no double-load. The isLoading re-entrancy guard in loadLevelByIndex (decision D5) backstops any overlap if you mash the menu mid-load.
Note — no live feel control here. The header comment is explicit: feel is registry-driven and applied only on level load (the single source of truth), so there is deliberately no per-level feel dropdown in this folder. Live feel tweaking lives only in PlayerDebug's Feel Lab.

Modifying this: to add a level you touch only LEVEL_REGISTRY — this file rebuilds its options from it. Keep GameDirector a type-only import. If you add more level-scoped debug controls, hang them off this same folder rather than reaching into the director's internals.

src/game/types/entityType.ts 29 LOC

The single identity vocabulary for every entity. Each entity carries a string type (stamped into its Rapier collider's userData), and this const map is the canonical list of those strings. It's a pure data file — no class, no lifecycle — imported by the spawn side (entityFactories), the routing side (contactRules, the GameDirector spawn loop), and the debug palette (debugColors).

Hard-won victory — why strings, not instanceof. The header comment nails it: "Rapier callbacks return a raw Collider (not a class), so type-based routing survives where instanceof can't." When a contact fires, all you get back from Rapier is a collider handle; you can't ask "is this a Player?" with instanceof. Stamping a type string onto userData at spawn lets the contact table and debug layers route purely on data. One const map keeps the spawn side and the routing side from drifting — and the trailing warning is real: these values are the exact strings level data and the engine's resourceLoader emit, so changing a value silently breaks routing.

const EntityType = { … } as const; export type EntityType = typeof EntityType[keyof typeof EntityType];

The const-object-plus-derived-union pattern (the same shape used by PlayerStates and CollisionGroups, called out in the comment). The object gives you named constants at runtime (EntityType.PLAYER); the derived type gives you a compile-time union of the values ("Player" | "Wall" | …) so a function can demand a valid entity name. Default export is the object; named export is the type.

Modifying this: to introduce a new entity, add a member here, give it a factory in entityFactories, a wireframe color in debugColors, and (if it interacts) a rule in contactRules. Keep the value string identical to the name your level editor exports, or the spawn loop will silently skip it.

src/game/types/gameCollisionGroups.ts 30 LOC

The Rapier collision-group bitmask vocabulary. Each entity sets a membership group ("what I am") and a filter mask ("what I collide with"); Rapier only generates a contact when both sides agree. This file is the const table of those bits, imported by Player, Enemy, Platform, and TrashCan, which feed them to GameObject.setCollisionGroup/setCollisionMask.

Note — Rapier's group/mask rule. The header comment states it exactly: a contact occurs when (a.group & b.mask) AND (b.group & a.mask) are both non-zero. Both directions must agree — A must want B and B must want A. Combine multiple groups or masks with bitwise OR (PLATFORM | ENEMY).

const CollisionGroups = { … } as const;

The bit layout, written in 0b… binary for clarity:

The pairing is symmetric and intentional: Player wants PLATFORM | ENEMY and both Platform (via DEFAULT) and Enemy (which lists PLAYER_BOUNDING_BOX) want the Player back, so all those contacts fire. The interesting dynamic case is the Enemy toggling its own mask at runtime: while rolling a girder it keeps PLATFORM in the mask (so it rides the floor), but when it takes a ladder (DescendLadder) it drops PLATFORM and keeps only PLAYER_BOUNDING_BOX — so it falls straight through girders yet can still kill the player on the way down.

Hard-won victory — Bit 1 left deliberately free. The comment marks bit 1 as "was PLAYER_HIT_BOX (the player's 62.5% hit sensor), removed when death detection moved to the full bounding box via the contact table." The old design had a separate smaller sensor collider for death; once the contact table made "any touch of the Enemy registers death" trivial, that sensor and its per-frame poll became dead code. The bit isn't reused — leaving the gap documents the history and avoids accidentally colliding semantics with a brand-new group. See Player collider setup.
Gotcha — these are 16-bit groups packed into Rapier's 32-bit field. GameObject.setCollisionGroup/setCollisionMask (in engine/entities/gameObject) pack group into the low 16 bits and mask into the high 16: setting the group does collisionGroups(collisionGroups | (currentMask << 16)) after reading the current mask back via >> 16; setting the mask mirrors it with & 0xffff. That's why every constant here stays inside 16 bits (and why DEFAULT is 0xffff, not 0xffffffff) — the upper half is reserved for the mask. Set group and mask in either order; each setter preserves the other half.

Modifying this: add a new group on the next free bit (bit 1 is available, but consider leaving it as the documented gap and using bit 4) and stay within 16 bits. Then update the relevant entity's setCollisionGroup/setCollisionMask calls on both sides of any new interaction — a one-sided mask change produces a silent no-contact, the classic Rapier filtering bug. Remember this only controls whether Rapier generates a contact; what happens on contact is the contact table's job (contactRules).
