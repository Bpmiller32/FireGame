Events & Input — the mitt bus and the keyboard layer

Two of the engine's smallest files carry an outsized share of its decoupling. eventBus.ts is a single typed mitt emitter that lets any subsystem talk to any other without holding a reference — the engine half of the map never names a game concept, the game half does. input.ts is the raw keyboard device: it captures keydown/keyup into held-state booleans and snapshots them into a plain InputState bag, so controllers read input as data instead of polling the live device.

src/engine/events/eventBus.ts 45 LOC

This file IS the application's single global event bus — one mitt emitter instance, exported as a default named Emitter. Nobody constructs it; it is a module-level singleton created the first time the module is imported, and every import Emitter from ".../eventBus" across the codebase resolves to that same object. It imports exactly one thing — GameObject — and only so the gameObjectRemoved event can carry a strongly-typed payload. Its entire job is to publish a typed event map so that Emitter.emit("tick") and Emitter.on("tick", fn) are compile-checked: a typo in an event name, or a payload of the wrong shape, is a build error rather than a silent no-op.

Note. The map is deliberately split into two halves — EngineEventMap and GameEventMap — that are then merged. This is the same engine/game seam that runs through the whole project (see Engine Core — Experience and Game Director). The engine half is allowed to be imported by a fresh project with no game; the game half is the Donkey-Kong vocabulary layered on top. Keeping them as two named types documents which events a reusable engine may legitimately fire.

type AppEvents = EngineEventMap & GameEventMap;

The full event surface is the intersection-type merge of both maps. As the source comment notes, the & here "just means all the keys from both maps" — for two object types with disjoint keys, the TypeScript intersection is the union of their fields. mitt<AppEvents>() is then parameterised on this combined map so both engine and game events are typed by the one emitter. 

type AppEvents = EngineEventMap & GameEventMap;

const Emitter = mitt<AppEvents>();

export default Emitter;
Gotcha. Emitter is a true singleton because ES modules are evaluated once and cached. That is the feature — every subsystem shares one bus — but it also means listeners leak across the object lifecycle unless explicitly removed. Long-lived owners therefore store their handler refs and call Emitter.off(name, ref) in their Destroy(): see Experience (experience.ts:103-104), Camera (camera.ts:329), ResourceLoader (resourceLoader.ts:524), and GameDirector (gameDirector.ts:536-541). An anonymous inline listener cannot be removed by name — that is why these classes hoist the callback into a stored field.
Note. One asymmetry worth remembering: GameDirector.Destroy calls off for resourcesReady, gameStart, gameOver, gameReset, switchLevel, and gameObjectRemoved — but not gameWin, and manualCameraControl is owned/removed by Camera instead. If you add a new long-lived listener, add the matching off in the same owner's teardown or it will accumulate across resets.

Modifying this: to add an event, put it in the correct half of the map — engine-pure signals go in EngineEventMap (no game words allowed), game-flow signals go in GameEventMap. Give it a payload type, never leave it untyped; use void for a pure signal. Then emit with Emitter.emit("name", payload) and subscribe with Emitter.on("name", fn), and — if the subscriber outlives a single load — store the handler ref and remove it in Destroy(). Do not add a second emitter instance; the whole design depends on one shared bus.

src/engine/input/input.ts 290 LOC

This file IS the raw keyboard device. It is constructed once by Experience in Configure() (experience.ts:55, this.Input = new Input()) and exposed as Experience.Input. It imports the InputState type plus the InputSource interface it implements — it still has no idea what a "player" or a "jump" means; it only knows that certain physical keys are up or down right now. It registers two window listeners in its constructor, exposes a pile of held-state booleans and combo helper methods, and offers one snapshot method (CaptureInto) that copies the current intent into a plain data bag. Lifecycle: constructor wires listeners; there is no per-frame Update() (the device is event-driven, not polled); Destroy() unhooks the window listeners.

Note. The engine Input exposes only held-state — booleans that are true for as long as the key is down. It deliberately knows nothing about press edges (the false→true transition of a single press) and nothing about game events. Edge detection and the mapping to game meaning live one layer up, in InputBindings (see Game Config — InputBindings) and in the player state handlers.
Note (the InputSource seam). Input now formally implements InputSource — a one-method engine interface (CaptureInto(state: InputState): void, engine/types/inputSource.ts) that names the capture seam. The payoff is at the consumer, not here: Player types its inputDevice as InputSource rather than as the concrete keyboard device, so a touch, gamepad, network, or replay source drops in just by implementing CaptureInto — no entity edits, because nothing depends on the keyboard class.

interface Key { keyCode: string; isPressed: (pressed: boolean) => void; }

A private file-local interface describing one tracked key: its physical keyCode and a setter callback that writes the matching held-state field. Using KeyboardEvent.code (e.g. "KeyW", "Space", "Backquote") rather than .key makes the mapping layout-independent: W is the physical W slot regardless of QWERTY/AZERTY or the active locale. 

interface Key {
  keyCode: string; // KeyboardEvent.code, layout-independent (e.g. "KeyW", "Space")
  isPressed: (pressed: boolean) => void; // Input calls this when the key changes
}

// fields: private held-state + public per-key flags + private keys[] table

Two tiers of state. The private booleans back the directional combo logic and are never read directly by outside code — isLeftKeyPressed, isRightKeyPressed, isUpKeyPressed, isDownKeyPressed, isSpacebarPressed. The public booleans are read directly by debug/binding code that wants a single raw key — IsWKeyPressed/IsAKeyPressed/ IsSKeyPressed/IsDKeyPressed (camera fly, read by CameraDebug), IsQKeyPressed/IsEKeyPressed, Is1KeyPressed/Is2KeyPressed, IsTildePressed, and IsF1Pressed…IsF4Pressed (read by InputBindings). 

// private held-state, refined by the Is* combo checks below
private isLeftKeyPressed: boolean;
private isRightKeyPressed: boolean;
private isUpKeyPressed: boolean;
private isDownKeyPressed: boolean;
private isSpacebarPressed: boolean;

public IsWKeyPressed: boolean;  // ... A/S/D, Q/E, 1/2
public IsTildePressed: boolean;
public IsF1Pressed: boolean;    // ... F2/F3/F4

private keys: Key[]; // key-code -> pressed-flag setter table
Tip. Why is left/right/up/down private but W/A/S/D public? The arrow keys feed the combo-resolving methods (IsLeft(), IsLeftRightCombo() …) — outsiders are meant to ask the question ("am I moving left?"), not read the raw flag, so the raw flag is hidden. The WASD/F-key flags have no combo semantics; they are single raw keys, so they are exposed directly.

constructor()

No parameters. Zeroes every held-state and public flag to false, builds the keys table mapping each tracked keyCode to the closure that sets its flag, then stores bound listener references and registers them on window. The closures capture this, which is why each one is an arrow function. 

this.keys = [
  { keyCode: "ArrowLeft",  isPressed: (eventResult) => { this.isLeftKeyPressed = eventResult; } },
  { keyCode: "ArrowRight", isPressed: (eventResult) => { this.isRightKeyPressed = eventResult; } },
  // ... ArrowUp/ArrowDown, Space, KeyW/A/S/D, Digit1/2, KeyQ/E, Backquote, F1..F4
];

// Store bound references so they can be removed in destroy()
this.onKeyDownListener = (event: KeyboardEvent) => this.onKeyDown(event.code);
this.onKeyUpListener = (event: KeyboardEvent) => this.onKeyUp(event.code);

window.addEventListener("keydown", this.onKeyDownListener, false);
window.addEventListener("keyup", this.onKeyUpListener, false);
Hard-won victory. The two listeners are hoisted into onKeyDownListener / onKeyUpListener fields before being registered, precisely so Destroy() can pass the same function identity to removeEventListener. If the constructor had registered an inline arrow (window.addEventListener("keydown", e => this.onKeyDown(e.code))), the listener could never be removed — every engine teardown/rebuild would stack another live keyboard handler on window, and stale Input instances would keep mutating after they were meant to be dead.
Note. Listeners are bound to window, not the canvas, so input works regardless of focus within the page. Only fifteen physical keys are tracked; everything else falls through both loops untouched.

private onKeyDown(keyName: string) / private onKeyUp(keyName: string)

The two raw event handlers. Each receives event.code (already unwrapped by the bound listener), scans the keys table for a matching keyCode, and calls that entry's isPressed setter with true (down) or false (up). A linear scan over ~15 entries per keystroke is trivially cheap and keeps the table the single source of truth for which keys exist. 

private onKeyDown(keyName: string) {
  for (const key of this.keys) {
    if (keyName === key.keyCode) {
      key.isPressed(true);
    }
  }
}

Line note: there is no break after a match — harmless because keyCode values are unique, so at most one entry ever fires. Unrecognised keys simply complete the loop with no effect.

IsLeft() / IsRight() / IsLeftRightCombo() / IsNeitherLeftRight()

The horizontal intent resolvers. These collapse the two raw arrow flags into four mutually-exclusive questions. IsLeft() is true only when left is held and right is not — pressing both at once is its own distinct state (IsLeftRightCombo()), not "left wins." This matters because the player's running/jumping handlers treat "both pressed" and "neither pressed" the same way (decelerate), and they need to distinguish that from a clean single direction. 

public IsLeft() {
  if (this.isLeftKeyPressed && !this.isRightKeyPressed) {
    return true;
  }
  return false;
}
public IsLeftRightCombo() {
  if (this.isLeftKeyPressed && this.isRightKeyPressed) {
    return true;
  }
  return false;
}
Tip. The four methods partition the (left, right) truth table into exactly four cases — IsLeft, IsRight, IsLeftRightCombo, IsNeitherLeftRight — so a state handler can switch on intent without re-deriving the boolean algebra. IsJump() is the simple odd one out: it just mirrors isSpacebarPressed with no combo logic.

IsUp() / IsDown() / IsUpDownCombo() / IsNeitherUpDown() / IsJump()

The vertical equivalents, used for ladder climbing, plus the jump query. Same exclusive partitioning as the horizontal set: IsUp() requires up held and down clear, etc. IsJump() returns the raw spacebar held-state — edge handling (coyote time, jump buffering, variable jump height) is entirely the player layer's job, not the device's. 

public IsUp() {
  if (this.isUpKeyPressed && !this.isDownKeyPressed) {
    return true;
  }
  return false;
}
public IsJump() {
  if (this.isSpacebarPressed) {
    return true;
  }
  return false;
}

public CaptureInto(state: InputState): void

The decoupling seam. Given an existing InputState bag, it calls each of the nine intent methods and writes the result into the matching field — mutating the passed object in place, never allocating a new one. The Player calls this once per frame, before its state handlers read intent (player.ts:470, this.inputDevice.CaptureInto(this.Input)). 

// Snapshot input intent into InputState (mutated in place so existing refs stay valid).
// Per-entity input instead of the live device — the seam for network/replay/AI input.
public CaptureInto(state: InputState): void {
  state.isLeft = this.IsLeft();
  state.isRight = this.IsRight();
  state.isLeftRightCombo = this.IsLeftRightCombo();
  state.isNeitherLeftRight = this.IsNeitherLeftRight();
  state.isJump = this.IsJump();
  state.isUp = this.IsUp();
  state.isDown = this.IsDown();
  state.isUpDownCombo = this.IsUpDownCombo();
  state.isNeitherUpDown = this.IsNeitherUpDown();
}
Hard-won victory (D10 — input-as-data). Mutating in place rather than returning a fresh object means any reference already held to player.Input stays valid frame-to-frame — no churn, no dangling old snapshots. More importantly, because the player reads a plain InputState bag rather than calling the live device, the source of that bag can be swapped wholesale — a network packet, a replay buffer, or an AI controller could fill the same fields and the movement code would not change a line. The player keeps its own inputDevice ref separately (player.ts:135) purely to drive this one capture.

public Destroy()

Removes both window listeners by passing the stored bound references. Called from Experience.Destroy (experience.ts:109). No flags are reset — once destroyed the instance is discarded, so the only thing that matters is detaching from window so no zombie handler keeps firing. 

public Destroy() {
  window.removeEventListener("keydown", this.onKeyDownListener);
  window.removeEventListener("keyup", this.onKeyUpListener);
}

Modifying this: to track a new key, add one entry to the keys table (a unique KeyboardEvent.code plus a setter) and declare the backing field — public for a raw single key, private if it participates in a new combo group. If it should produce game intent consumed by entities, add a field to InputState and a line to CaptureInto; if it should fire a game event, do the edge detection in InputBindings, not here. Resist adding edge or game logic to this file — its value is that it stays a dumb, game-blind device.

src/engine/types/inputState.ts 18 LOC

This file IS the data contract between the input device and whatever consumes intent. It is a pure interface — no class, no runtime code — describing a flat bag of nine booleans. Input.CaptureInto writes it; entity controllers read it. Nobody "owns" the type; instances are created by consumers (the Player builds one in initializePlayerAttributes, player.ts:140) and then refilled each frame.

export default interface InputState { isLeft; isRight; isLeftRightCombo; isNeitherLeftRight; isJump; isUp; isDown; isUpDownCombo; isNeitherUpDown }

Three logical groups: horizontal intent (4 booleans), the single jump intent, and vertical/ladder intent (4 booleans). The field names mirror the device's intent methods exactly — IsLeft() the method becomes isLeft the field — so reading intent costs one property access and no parentheses.

Whenever you add a field here, add the matching write in Input.CaptureInto, or the field will silently stay at its initial value.
