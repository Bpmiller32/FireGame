Player States — the FSM bodies

Five tiny pure-function handlers — one per PlayerState — are the flesh on the bones of the generic StateMachine. Each runs once per frame for the player's current state: it reads the per-entity input snapshot and the shapecast contact flags, writes a desired NextTranslation, picks a sprite, and — when a condition trips — reassigns player.State to transition next frame. This is where coyote-time, the jump buffer, variable jump height, apex-hang, and ladder snapping actually live.

The generic StateMachine<Player, PlayerState> (see FSM Architecture — StateMachine) dispatches exactly one handler per fsm.Update(), keyed on player.State. A handler that reassigns player.State does not re-dispatch this frame — the new state runs next frame. All the tuning numbers (JumpPower, RiseGravity, CoyoteTime, …) come from the Feel (Attributes) profiles; the handlers never hard-code feel.

The resting state — standing still on the ground — and the engine's initial state (Player seeds State = PlayerStates.IDLE). It is also the grounded "hub": every grounded reset (coyote refill, early-jump latch clear, buffer refill) lives here and in Running. Imports Player (the host), PlayerStates (transition targets), SpriteAnimations (the frame table) and PlayerDirection (signed facing). No physics import — it nudges the body purely through player.NextTranslation, which Player.updateTranslation() later feeds to the kinematic character controller.

const handlePlayerIdle = (player: Player) => void

Runs while standing. The order of the four transition checks is deliberate: a slope/edge guard, then fall, then run, then jump, then climb — and only if none fire does the body do its grounded housekeeping and pick an idle sprite.

1. The vertical-direction edge guard (not a transition). Before any state change, it catches the case where the body still carries upward/zero vertical intent while the player presses a horizontal key:

if (
  player.NextTranslation.y >= 0 &&
  (player.Input.isLeft || player.Input.isRight)
) {
  player.NextTranslation.y = -player.MaxFallSpeed;
  return;
}

It slams vertical velocity to terminal-fall (-MaxFallSpeed) and returns — staying IDLE for one frame so the body is pressed firmly down before the next frame re-evaluates ground contact. This prevents a one-frame "float" when leaving a state with residual y >= 0.

2. Transition to FALLING — walked off something while not rising:

if (!player.IsTouching.ground && player.NextTranslation.y <= 0) {
  player.TimeFallWasEntered = player.Time.Elapsed;
  player.State = PlayerStates.FALLING;
  return;
}

Stamping TimeFallWasEntered here is what arms coyote time — the FALLING handler measures the coyote window against this timestamp.

3. Transition to RUNNING — any horizontal press:

if (player.Input.isLeft || player.Input.isRight) {
  player.State = PlayerStates.RUNNING;
  return;
}

4. Transition to JUMPING — the buffered-jump launch:

if (player.Input.isJump && player.BufferJumpAvailable) {
  // Launch impulse + reset the early-end latch so every jump starts at full
  // height (a buffered jump fired on landing skips the grounded reset below).
  player.NextTranslation.y = player.JumpPower;
  player.EndedJumpEarly = false;
  player.TimeJumpWasEntered = player.Time.Elapsed;
  player.State = PlayerStates.JUMPING;
  return;
}

Note the gate is BufferJumpAvailable, not raw "is grounded". The grounded reset further down sets that flag true each frame the player is on the ground and not holding jump, so a fresh press launches. Setting EndedJumpEarly = false here matters because a buffered jump can fire on the landing frame and skip the grounded reset below — the comment spells out exactly that race.

5. Transition to CLIMBING — on a ladder core, pressing up/down, but not pressing "off the end":

if (
  player.IsTouching.ladderCore &&
  (player.Input.isUp || player.Input.isDown) &&
  !(player.IsTouching.ladderTop && player.Input.isUp) &&
  !(player.IsTouching.ladderBottom && player.Input.isDown)
) {
  player.NextTranslation.x = 0;
  player.NextTranslation.y = 0;
  player.State = PlayerStates.CLIMBING;
  return;
}

The two negated clauses stop you from "grabbing" a ladder by pushing up at its very top or down at its very bottom (you'd just walk off). The ladder flags (ladderCore/Top/Bottom) are not shapecast here — they are pushed in each frame by GameDirector via Player.UpdateLadderState() from the Ladder Sensor. Zeroing both axes freezes the body the instant it latches on.

The grounded body (only reached if no transition fired):

// In a grounded state, give coyote and reset early jump gravity

player.EndedJumpEarly = false;
if (!player.Input.isJump) {
  player.BufferJumpAvailable = true;
}

// Simple max gravity in non-vertical state to fix downward movement on slopes, maintain touching ground

This is the canonical grounded reset, mirrored verbatim in Running: refill coyote, clear the early-end latch, and re-arm the buffer jump only when jump is released (so holding jump can't auto-rebuffer). The constant -MaxFallSpeed downward push keeps the kinematic controller's snap-to-ground glued to slopes instead of micro-hopping.

Sprite selection. A switch (player.Direction) picks IDLE_LEFT/IDLE_RIGHT directly for LEFT/RIGHT. The default (NEUTRAL) branch is the interesting one — it inspects the current sprite state to choose which idle orientation to settle into:

The rising half of the jump arc. Entered only from IDLE/RUNNING (buffered jump) or FALLING (coyote jump) — in all three the launch site sets NextTranslation.y = player.JumpPower and stamps TimeJumpWasEntered. This handler does not set the impulse; it spends it. It owns variable jump height (early-release), apex-hang, and the ceiling/apex exits. Imports add applyHorizontalMovement for air steering.

const handlePlayerJumping = (player: Player) => void

1. Transition to FALLING — ceiling bonk:

if (player.IsTouching.ceiling && player.NextTranslation.y >= 0) {
  player.NextTranslation.y = 0;
  player.TimeFallWasEntered = player.Time.Elapsed;
  player.State = PlayerStates.FALLING;
  return;
}

Hitting a ceiling while still moving up kills upward velocity and drops to FALLING immediately (stamping the fall timestamp, though coyote won't matter from a ceiling). The y >= 0 guard avoids re-triggering on a ceiling brush while already descending.

2. Transition to FALLING — the natural apex:

// Natural apex — rise gravity has decelerated the launch impulse to zero. The
// peak emerges from physics now, not a fixed MaxJumpTime timer.
if (player.NextTranslation.y <= 0) {
  player.TimeFallWasEntered = player.Time.Elapsed;
  player.State = PlayerStates.FALLING;
  return;
}

This is the only way a jump ends upward travel: rise gravity (below) bleeds vy down to zero and the very frame it crosses to <= 0, the state flips to FALLING. The comment is explicit that the apex is emergent — there is no fixed jump-duration timer. Consequence: FALLING is always entered at the apex with vy <= 0, which is why FALLING's old "early release" branch was dead code (it documents this).

Variable height — the early-end latch:

// "early end", which boosts rise gravity below so the player peaks lower.

  C --> IDLE: atLadderTop — snap up (+MaxClimbSpeed), ground = true
  C --> IDLE: atLadderBottom — snap down, ground = true
  C --> C: accel toward Direction*MaxClimbSpeed, else decel to 0
Hard-won victory — the top snap + forced ground. Setting NextTranslation.y = +MaxClimbSpeed and IsTouching.ground = true together on a top exit is what makes dismounting a ladder land you cleanly on the platform instead of flickering CLIMBING -> FALLING -> CLIMBING at the lip.
Gotcha — no sprite update. CLIMBING never calls SpriteAnimator.ChangeState (it doesn't even import the table), so the on-ladder pose is whatever was last set before grabbing. If you want a dedicated climb animation, add a climb entry to Sprite Animations and a ChangeState call here — mind the up/down/neutral facing.

Modifying this: climb feel is three shared base-feel numbers (MaxClimbSpeed, ClimbAcceleration, ClimbDeceleration). The ladder geometry (top/core/bottom) is owned by the Ladder Sensor and pushed in via Player.UpdateLadderState(), not sensed here — fix mis-snapping at the edges there, not in this handler. Keep the IsTouching.ground = true on exit or you'll reintroduce the lip flicker.

Tip — reading the whole player loop. These handlers only write intent into player.NextTranslation and flags. The actual move (kinematic computeColliderMovement, snap-to-ground, two-sided contact dispatch) and the shapecast contact sensing that fills IsTouching both happen in Player's updateTranslation() / detectCollisions(), run right after fsm.Update() each frame.
