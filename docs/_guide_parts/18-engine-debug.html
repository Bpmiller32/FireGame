Engine Debug — coordinator, Camera & Physics debug

The engine's debug layer is a tiny coordinator (Debug) plus two focused sub-modules (CameraDebug, PhysicsDebug). Debug owns the single dat.GUI panel and the stats.js monitor, gates everything behind a URL hash, and exposes dependency-injection seams so the game can plug in a debug-wireframe palette, a collision-log sink, and arbitrary GUI modules without the engine ever naming a single game concept (this is decision D3). GameObjectType is included here because it is the engine's collider-shape vocabulary that PhysicsDebug reads back when it draws.

Hard-won victory — D3, the "engine never names a game type" rule. Three seams keep engine/debug/ completely game-blind. The game pushes knowledge into the engine; the engine never reaches out for it:
  • SetPhysicsTypeColors(map) — the wireframe palette (Config: debugColors) keyed by the game's EntityType strings. PhysicsDebug looks each collider up in an opaque Map<string, THREE.Color> and falls back to gray; it never knows what "player" or "enemy" mean.
  • RegisterCollisionLogSink(sink) — the game's console-log formatter (Game Debug: collisionLogger) behind the CollisionLogSink interface; Physics routes events to it.
  • RegisterModule(module) — any game GUI panel behind the DebugModule interface (Game Debug: levelDebug / playerDebug).
All wiring happens once, in App.vue, after Experience.Configure(). The payoff: you can lift engine/ into a different game with zero edits to the debug layer.

src/engine/debug/debug.ts 160 LOC

This file is the debug coordinator. It is the first subsystem the singleton Core: Experience constructs in Configure() (this.Debug = new Debug()) — first, because every other subsystem checks this.experience.Debug.IsActive in its own constructor to decide whether to wire its debug panel. Debug owns exactly two pieces of UI — the single dat.GUI panel (Ui) and the stats.js monitor (Stats) — and delegates all domain-specific drawing to two private sub-modules it instantiates: CameraDebug and PhysicsDebug. Its imports are deliberately minimal:

  • dat from "dat.gui" and Stats from "stats.js" — the panel and FPS/MS monitor it owns directly.
  • CameraDebug, PhysicsDebug — the engine-side sub-modules, wired in-engine (not game-registered).
  • GameObject — only as a type in the CollisionLogSink method signatures.
  • * as THREE — only for the THREE.Color in SetPhysicsTypeColors's signature; it never constructs one.

Lifecycle: the constructor decides IsActive and, if active, builds the panel + monitor. There is no per-frame Update() on Debug itself — instead Physics and Camera call UpdatePhysicsDebug/UpdateCameraDebug from their own update loops, so the debug code rides along on the subsystem it observes. Destroy() tears down the panel, monitor, and registered modules.

Note — the forward-typed cycle break. Camera and Physics are aliased to any at the top of the file: 
// Forward types to avoid circular imports — Camera/Physics all depend on
// Experience which depends on Debug. Sub-modules carry their own forward types.
type Camera = any;
type Physics = any;
Importing the real classes would close the loop Camera → Experience → Debug → Camera. Debug only ever passes these through to its sub-modules, so it doesn't need the real shapes; the sub-modules each carry their own structural forward type (see below).

export interface DebugModule { Init?(ui: dat.GUI): void; Destroy?(): void }

The shape a game GUI panel must satisfy to be registered. Both methods are optional: Init(ui) builds dat.GUI folders when the panel is ready; Destroy() is optional teardown. The engine knows only this shape — never a concrete panel class.

export interface CollisionLogSink { LogCollisions: boolean; LogSensors: boolean; LogCollisionEvent(obj1: GameObject, obj2: GameObject, eventType: "enter" | "exit"): void; LogSensorEvent(obj1: GameObject, obj2: GameObject, eventType: "enter" | "exit"): void; }

The shape of the game's collision/sensor console logger. Game Debug: collisionLogger implements this; Physics forwards log events here (no-op if none registered). The two boolean flags are the live toggles a debug panel flips. Note that Physics reads these flags through the Debug coordinator, not off the sink directly — which is what the next member is for.

public IsActive: boolean; public Ui?: dat.GUI; public Stats?: Stats; private camera = new CameraDebug(); private physics = new PhysicsDebug(); private modules: DebugModule[] = []; private logSink?: CollisionLogSink;

IsActive is the master gate read everywhere. Ui/Stats are optional because they only exist when active. The two sub-modules are constructed unconditionally (they're cheap, hold no state until Init). modules is the registered-game-panel store; logSink is the single optional collision logger.

public get LogCollisions(); public set LogCollisions(v); public get LogSensors(); public set LogSensors(v);

These four accessors are a delegating facade over the registered sink. Physics reads this.debug.LogCollisions to decide whether to log; a debug panel writes it to toggle. They forward to the sink and degrade safely when no sink is registered: 

public get LogCollisions() { return this.logSink?.LogCollisions ?? false; }
public set LogCollisions(v: boolean) { if (this.logSink) this.logSink.LogCollisions = v; }
public get LogSensors() { return this.logSink?.LogSensors ?? false; }
public set LogSensors(v: boolean) { if (this.logSink) this.logSink.LogSensors = v; }

The ?? false getter is the important line: with no sink, logging is simply off, and Physics's if (this.debug.LogCollisions) guard never fires. The setter no-ops without a sink, so a GUI bound to it can't crash on a missing logger.

constructor()

Decides whether debug mode is on and, if so, builds the panel and monitor. No parameters; no return. 

// Enable debug mode with #debug in URL, or always on in dev builds
this.IsActive = import.meta.env.DEV || window.location.hash === "#debug";

if (this.IsActive) {
  this.initPanel();
  this.initStats();
}

Two ways in: Vite's import.meta.env.DEV (always on while developing) or #debug in the URL hash (so a deployed production build can be inspected by adding the fragment). The check is a one-time read at construction — toggling the hash later does nothing until reload.

public RegisterModule(module: DebugModule)

The game-GUI-panel seam. Stores the module, then — only if the panel is live — runs its Init immediately so the folder appears. 

public RegisterModule(module: DebugModule) {
  this.modules.push(module);
  if (this.IsActive && this.Ui && module.Init) {
    module.Init(this.Ui);
  }
}

It always pushes, even when inactive — that's the "no-op store" the call site comments on (App.vue: "RegisterModule is a no-op store when debug is inactive"). The game can register unconditionally at boot; the engine decides whether the panel actually materializes. The this.IsActive && this.Ui guard means a module registered while inactive is kept but never inited (and so never torn down, since Destroy() only runs when active).

Thin pass-throughs to the two in-engine sub-modules. Likewise Physics calls InitPhysicsDebug/UpdatePhysicsDebug/DestroyPhysicsDebug behind its own IsActive gate.

}
Gotcha — the this.Ui! non-null assertions. InitCameraDebug and InitPhysicsDebug assert this.Ui! is defined. That's safe only because the callers (Camera, Physics) gate their Init* calls behind Experience.Debug.IsActive, and IsActive being true is exactly the condition under which the constructor built this.Ui. Call InitCameraDebug directly while inactive and you'd hand undefined to dat.GUI. The ordering dependency — "panel exists ⇒ active" — is the contract that keeps the ! honest.

public SetPhysicsTypeColors(colors: Map<string, THREE.Color>)

The palette seam — the cleanest expression of D3. Forwards the game's entity-type→color map straight to PhysicsDebug.SetTypeColors, opaquely. 

// Inject the game's wireframe palette (entity-type -> color); engine forwards it opaquely to PhysicsDebug.
// Safe to call when inactive — PhysicsDebug only reads it while rendering, which only runs when active.
public SetPhysicsTypeColors(colors: Map<string, THREE.Color>) {
  this.physics.SetTypeColors(colors);
}

The key field is Map<string, THREE.Color> — a string key, never a typed enum. The engine cannot and does not import the game's EntityType. And unlike the Init* methods, this one is safe to call when inactive: it only stores a reference, and the stored map is read solely inside PhysicsDebug.Update(), which only ever runs when active. That's why App.vue can call it unconditionally right after Configure().

public LogCollisionEvent(obj1, obj2, eventType); public LogSensorEvent(obj1, obj2, eventType);

The forwarders Physics calls when its flags are on. Each is a single optional-chained delegation — a no-op if no sink is registered: 

public LogCollisionEvent(obj1: GameObject, obj2: GameObject, eventType: "enter" | "exit") {

public LogSensorEvent(obj1: GameObject, obj2: GameObject, eventType: "enter" | "exit") {

Note the asymmetry that keeps logging cheap: Physics first checks this.debug.LogCollisions (the delegating getter) before calling these, so the ?. here is belt-and-suspenders rather than the primary guard.

public Destroy()

Tears down the panel, removes the stats DOM node, and destroys every registered module. Called by Core: Experience's Destroy(), but only inside an if (this.Debug && this.Debug.IsActive) guard. 

public Destroy() {
  this.Ui?.destroy();
  this.Stats?.dom.parentNode?.removeChild(this.Stats.dom);
  for (const module of this.modules) {
    module.Destroy?.();
  }
}

this.Stats?.dom.parentNode?.removeChild(...) deletes the monitor element via its parent rather than assuming it's a direct child of document.body. The sub-modules' physics wireframe is not torn down here — that goes through DestroyPhysicsDebug from Physics's own Destroy(), since PhysicsDebug owns a mesh that lives in the scene.

Modifying this: to add a new game debug panel, implement DebugModule in game/debug/ and call Debug.RegisterModule(...) from App.vue — never import a game type here. To add a new engine debug sub-module (say an audio visualizer), mirror the CameraDebug/PhysicsDebug pattern: instantiate it as a private field, add Init/Update/Destroy pass-throughs that assert this.Ui!, and call them from the subsystem's own IsActive-gated loop. Keep the rule: anything the engine can't name belongs behind a registration seam, not an import.

src/engine/debug/cameraDebug.ts 167 LOC

A focused sub-module owned by Debug (private camera = new CameraDebug()). It does two jobs: build the 📷 Camera Debug folder (transform readouts + live follow-tuning knobs) in Init, and run a manual fly-cam (WASD/QE pan, 1/2 zoom-or-raise, ~ camera-type toggle) every frame in Update. It imports only * as THREE (vector math) and dat from "dat.gui". Like its sibling, it declares a structural forward type for Camera to break the import cycle.

Note — the structural Camera mirror. Rather than aliasing to any, CameraDebug spells out exactly the slice of Camera it touches: 
// Forward type — Camera → Experience → Debug → CameraDebug → Camera would cycle.
type Camera = {
  Instance: THREE.Camera; CameraType: string; ManualControl: boolean;
  FrustumSize: number; ZoomFactor: number; AspectRatio: number; LastToggleTime: number;
  OrthographicCamera?: THREE.OrthographicCamera;
  Input?: { IsWKeyPressed: boolean; /* ...IsSKey, IsAKey, ... IsTildePressed */ };
  Time: { Elapsed: number };
  SwitchCamera: () => void;
  Tuning: { lookaheadX: number; /* ...mirrors Camera.Tuning */ };
};
The field names are PascalCase to mirror the real Camera class, and the nested Input/Time/Tuning shapes mirror those classes too — so TypeScript still type-checks the access even though no import exists. If you add a knob to Camera.Tuning (see Engine Camera) you must add it here as well, or the GUI binding won't compile.

const MOVE_SPEED = 0.5; const ROTATE_SPEED = 0.02;

Module-level constants: world units of pan per frame, and radians of yaw per frame for A/D in perspective mode. Frame-based (not Time.Delta-scaled) — this is a developer fly-cam, not gameplay, so frame-rate-dependent speed is acceptable.

public Update(camera: Camera)

Per-frame manual camera movement, called from Debug.UpdateCameraDebug → from Camera.Update. It first builds a ground-plane basis, then reads the held WASD/QE/1/2 keys, then handles the ~ toggle. 

// Camera forward/right flattened onto the ground plane.
const forward = new THREE.Vector3(0, 0, -1).applyQuaternion(camera.Instance.quaternion);
forward.y = 0;
forward.normalize();

const right = new THREE.Vector3().crossVectors(forward, new THREE.Vector3(0, 1, 0)).normalize();

const isOrtho = camera.Instance instanceof THREE.OrthographicCamera;

forward.y = 0 flattens the look direction so W/S pan along the ground instead of diving toward it. The behavior then forks on isOrtho: in orthographic mode WASD pans the world axes and 1/2 zoom; in perspective mode W/S dolly along forward, A/D yaw (rotateY ± ROTATE_SPEED), Q/E strafe along right, and 1/2 raise/lower the Y position. 

if (camera.Input?.IsWKeyPressed) {
  if (isOrtho)
    camera.Instance.position.addScaledVector(new THREE.Vector3(0, 1, 0), MOVE_SPEED);
  else camera.Instance.position.addScaledVector(forward, MOVE_SPEED);
} else if (camera.Input?.IsSKeyPressed) {
  // ...mirror with -MOVE_SPEED...
} else if (camera.Input?.IsAKeyPressed) {
  if (isOrtho) camera.Instance.position.addScaledVector(right, -MOVE_SPEED);
  else camera.Instance.rotateY(ROTATE_SPEED);
}
// ...D, Q, E, 1, 2...

The whole movement block is one else if chain, so only one action fires per frame — diagonal pans aren't possible. camera.Input? is optional-chained because Camera only assigns this.Input when debug is active (see Engine Camera, constructor line 86). The tilde toggle is debounced against the engine clock: 

if (camera.Input?.IsTildePressed &&
    camera.Time.Elapsed - camera.LastToggleTime >= 0.25) {
  camera.SwitchCamera();
  camera.LastToggleTime = camera.Time.Elapsed;
  console.log(`📷 Switched to ${camera.CameraType} camera`);
}

The 0.25s gate stops a single key-hold from flickering perspective↔ortho every frame; LastToggleTime lives on Camera so it survives between frames.

Gotcha — who owns the camera while flying. This Update writes the camera transform directly, but it only takes effect because Camera.Update early-returns the moment ManualControl is true (after calling UpdateCameraDebug). So the manual fly-cam and the follow logic never fight: flip Manual Control in the panel (or the bound key) and the fly-cam fully owns the camera; flip it back and follow resumes. See Engine Camera.

private zoom(camera: Camera, zoomIn: boolean)

Orthographic-only zoom: nudges FrustumSize by ±ZoomFactor, clamps to a minimum of 1, then recomputes the ortho bounds from the aspect ratio and pushes the projection matrix. 

let delta = camera.ZoomFactor;
if (zoomIn) delta = -camera.ZoomFactor;
camera.FrustumSize = Math.max(1, camera.FrustumSize + delta);
const ortho = camera.OrthographicCamera!;
ortho.left = (-camera.FrustumSize * camera.AspectRatio) / 2;
ortho.right = (camera.FrustumSize * camera.AspectRatio) / 2;
ortho.top = camera.FrustumSize / 2;
ortho.bottom = -camera.FrustumSize / 2;
ortho.updateProjectionMatrix();

Zooming in means a smaller frustum, hence delta is negated when zoomIn. This duplicates the bound-math that Camera.Resize() uses (see Engine Camera) — the same four-line formula keeps the orthographic view square to the viewport.

Modifying this: adding a movement key means extending the else if chain and the Input field list on the structural Camera type (and the real Input class). Adding a new follow knob means adding it to Camera.Tuning, the CameraTuning interface in Engine Camera, and the structural mirror here, then one tuning.add(...) line. Because the type is structural, a typo silently drops a binding only at runtime — keep the mirror in lock-step.

src/engine/debug/physicsDebug.ts 179 LOC

The collider wireframe renderer, owned by Debug (private physics = new PhysicsDebug()). Each frame it walks the Rapier world, draws every collider's outline as line segments colored by the game's entity-type palette, and surfaces two object counters in a 🌍 World Debug folder. It owns a single THREE.LineSegments mesh — adds it to the scene, rebuilds its geometry every frame, and disposes it on teardown. Imports: * as THREE (the mesh + colors), RAPIER (collider shape accessors), dat from "dat.gui" (the counters). It declares a read-only structural Physics view — the scene it draws into and the world it walks: 

type Physics = {
  Scene: THREE.Scene;
  World: {
    colliders: { len: () => number };
    forEachCollider: (cb: (collider: RAPIER.Collider) => void) => void;
  };
};

Note Physics itself carries no debug fields — this module owns its own mesh, so the physics subsystem stays clean. This replaces Rapier's built-in debugRender, whose flat palette could only tell sensors from solids.

Hard-won victory — D3 lives here. The whole reason this hand-rolled renderer exists is to color colliders by game type while the engine stays type-blind: 
private static readonly DEFAULT_COLOR = new THREE.Color("#777777"); // gray fallback
// EntityType string -> color, injected by the game (D3: engine stays type-blind).
private typeColors = new Map<string, THREE.Color>();
The map starts empty and is filled only via SetTypeColors (← Debug.SetPhysicsTypeColors ← Config: debugColors). Any collider whose type isn't in the map draws gray. The engine never writes a single color literal for a game type.

private static readonly CIRCLE_SEGMENTS = 20; private mesh?: THREE.LineSegments<THREE.BufferGeometry, THREE.LineBasicMaterial>; private scene?: THREE.Scene; private counts = { renderObjectCount: 0, physicsObjectCount: 0 };

CIRCLE_SEGMENTS = arc resolution for ball/capsule outlines. counts is bound by reference into the GUI so .listen() reflects live updates without re-binding — the Update loop just mutates its two fields.

The receiving end of the D3 palette seam. Replaces the whole map by reference. Safe to call any time (even inactive) because the map is only read inside Update, which only runs when active.

public Init(ui: dat.GUI, physics: Physics)

Creates the wireframe mesh and the counter folder. Called by Physics.Configure() via Debug.InitPhysicsDebug, only when active. 

this.scene = physics.Scene;

this.mesh = new THREE.LineSegments(
  new THREE.BufferGeometry(),
  // vertexColors draws each segment with the per-vertex RGBA we build in
  // Update() — one color per collider, keyed by its entity type.
  new THREE.LineBasicMaterial({ vertexColors: true })
);
this.mesh.frustumCulled = false;
this.scene.add(this.mesh);

const folder = ui.addFolder("🌍 World Debug");
folder.open();
folder.add(this.counts, "renderObjectCount").name("Render Objects").listen();
folder.add(this.counts, "physicsObjectCount").name("Physics Objects").listen();

vertexColors: true is what lets one mesh carry many colors — each segment endpoint gets its own RGBA, so platforms, enemies, and sensors are different colors in a single draw call. frustumCulled = false stops Three.js from culling the mesh when its (frequently rebuilt, possibly stale) bounding sphere falls offscreen — the wireframe must always render. This folder .open()s by default (unlike the camera folder) since the counts are at-a-glance health numbers.

public Update(physics: Physics)

Recounts the scene and colliders, then rebuilds the entire wireframe from scratch. Called from Physics.Update() every frame. 

let entityCount = 0;
physics.Scene.traverse(() => entityCount++);
this.counts.renderObjectCount = entityCount;
this.counts.physicsObjectCount = physics.World.colliders.len();

renderObjectCount counts every node in the Three.js scene graph (traverse); physicsObjectCount is Rapier's live collider count — a quick way to spot leaks (numbers should fall back to baseline after a level unload). Then the rebuild: 

const positions: number[] = [];
const colors: number[] = [];

physics.World.forEachCollider((collider) => {
  const userData = collider.parent()?.userData as { type?: string } | undefined;
  const color =
    this.typeColors.get(userData?.type ?? "") ?? PhysicsDebug.DEFAULT_COLOR;
  this.appendColliderOutline(collider, color, positions, colors);
});

const geometry = this.mesh!.geometry;
geometry.setAttribute("position",
  new THREE.BufferAttribute(new Float32Array(positions), 2));
geometry.setAttribute("color",
  new THREE.BufferAttribute(new Float32Array(colors), 4));
this.mesh!.visible = true;

The type lookup is the heart of D3: collider.parent()?.userData.type is the game's EntityType string (set on the rigid body — see Entities: userData), and that string indexes typeColors, falling back to gray for any unmapped type. Two BufferAttribute item-sizes matter: position is 2 (Rapier2D is XY-only; the mesh sits at z=0) and color is 4 (RGBA, matching the four values seg() pushes per endpoint).

Gotcha — full rebuild every frame. positions/colors are fresh arrays each frame and wrapped in new Float32Arrays, so this allocates per frame. That's a deliberate simplicity-over-speed call for a dev-only overlay (it never ships to players), but it's why this is the kind of thing you'd never leave on in a perf profile.

private appendColliderOutline(collider, color, positions, colors)

Appends one collider's outline (cuboid / ball / capsule) to the shared buffers, in world pose. It reads the collider's translation and rotation once, precomputes cos/sin, and defines a local seg() closure that rotates a local-space segment into world space and pushes the color on both endpoints: 

const t = collider.translation();
const angle = collider.rotation();
const cos = Math.cos(angle);
const sin = Math.sin(angle);

const seg = (ax: number, ay: number, bx: number, by: number) => {
  positions.push(t.x + ax * cos - ay * sin, t.y + ax * sin + ay * cos);
  positions.push(t.x + bx * cos - by * sin, t.y + bx * sin + by * cos);
  colors.push(color.r, color.g, color.b, 1, color.r, color.g, color.b, 1);
};

That's a standard 2D rotation (x' = x·cos − y·sin, y' = x·sin + y·cos) plus the translation — so rotated cubes (the engine's slopes!) draw at their true angle. The colors.push writes 8 numbers: RGBA for endpoint A then RGBA for endpoint B, all opaque. Then it branches on the Rapier shape: 

switch (collider.shapeType()) {
  case RAPIER.ShapeType.Cuboid: {
    const h = collider.halfExtents();
    seg(-h.x, -h.y, h.x, -h.y); seg(h.x, -h.y, h.x, h.y);
    seg(h.x, h.y, -h.x, h.y);   seg(-h.x, h.y, -h.x, -h.y);
    break;
  }
  case RAPIER.ShapeType.Ball: {
    const r = collider.radius();
    const n = PhysicsDebug.CIRCLE_SEGMENTS;
    for (let i = 0; i < n; i++) { /* n chords around the circle */ }
    break;
  }
  case RAPIER.ShapeType.Capsule: {
    const r = collider.radius();
    const hh = collider.halfHeight();
    seg(-r, -hh, -r, hh); seg(r, -hh, r, hh);   // straight sides (axis = local Y)
    // ...two semicircle caps over +hh / under -hh...
    break;
  }
}

Cuboid = four sides from halfExtents; ball = CIRCLE_SEGMENTS chords; capsule = two straight sides plus rounded caps (axis is local Y, so caps bulge above +hh and below -hh). These three cases are exactly the three shapes the engine ever creates — see the GameObjectType article below. The comment pins this to Rapier 0.14 shape accessors (halfExtents(), radius(), halfHeight()); they've moved between versions, hence the version pin in VERSION.md.

public Destroy()

Removes the mesh from the scene and disposes both its geometry and material — the GPU-resource cleanup Three.js requires. Reached via Debug.DestroyPhysicsDebug from Physics.Destroy(). 

if (this.mesh) {
  this.scene?.remove(this.mesh);
  this.mesh.geometry.dispose();
  this.mesh.material.dispose();
  this.mesh = undefined;
}

Setting this.mesh = undefined makes the method idempotent and lets the next Init start clean. This is the one piece of debug teardown that does not go through Debug.Destroy() — it has to follow the physics world's lifecycle because the mesh lives in the shared scene.

The engine's entire collider-shape vocabulary — a frozen as const object with three members. It's the type GameObject.createCollider switches on to pick a Rapier collider, and (transitively) the set of shapes PhysicsDebug.appendColliderOutline needs to draw. Documented here because it closes the loop: the same three shapes that go in at collider creation come back out at debug-draw time.

Hard-won victory — D5 / "D-lean". The header comment records a deliberate diet: only these three primitives survive; polyline and convex-mesh colliders were purged. Slopes are rotated cubes and complex shapes are overlapped cubes. The payoff cascades into this very section — PhysicsDebug's switch only ever needs three cases, and the 2D rotation in seg() is enough to draw an angled slope correctly. A polyline collider would have meant per-vertex outline code and far more surface for bugs.
Gotcha — two different "type" axes, don't conflate them. GameObjectType is the collider's geometry (engine concept, this file). The color palette is keyed by the game's EntityType (a separate enum in Game Types: entityType, e.g. PLAYER, ENEMY, LADDER_SENSOR). So PhysicsDebug draws each collider by its GameObjectType shape (via shapeType()) but colors it by its EntityType string (via userData.type). A red ball and a red box can share a color yet differ in shape; a box can be white (player) or green (platform) yet draw identically. Keeping the two axes orthogonal is what lets the engine own shapes while the game owns identity.

The consumer side, for reference — GameObject.createCollider maps each member to a Rapier ColliderDesc, with CUBE as the catch-all default: 

switch (gameObjectType) {
  case GameObjectType.SPHERE:  return RAPIER.ColliderDesc.ball(size.width / 2);
  case GameObjectType.CAPSULE: return RAPIER.ColliderDesc.capsule(size.height / 2, size.width);
  case GameObjectType.CUBE:
  default:                     return RAPIER.ColliderDesc.cuboid(size.width / 2, size.height / 2);
}
