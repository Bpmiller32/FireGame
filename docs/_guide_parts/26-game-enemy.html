Enemy (barrels) — rolling, descending, bouncing

The Enemy is the Donkey-Kong barrel: a dynamic sphere that rolls the girders, decides whether to take a ladder, and — in its crazy flavour — bounces girder-to-girder down the screen. There is exactly one Enemy class (the old CrazyEnemy was folded in, D28): normal vs crazy is just the starting FSM state plus a bounce-drift sign. The barrel's own movement reactions (reverse at walls, edge-turn, bounce) live in its collision callbacks; the cross-entity verdicts (kill the Player, ignite the oil can) deliberately do not — they live in the contact table, which is what finally fixed the barrel/enemy bug.

A DK barrel entity, subclass of the engine GameObject. It is constructed at runtime by the GameDirector spawner (SpawnEnemy / SpawnCrazyEnemy → new Enemy(...)), pushed into the director's single Enemies[] array, and ticked once per frame by director.Update via enemy.Update(player, firstTrashCan). It imports the engine clock Time (delta drives the bounce integration), the generic StateMachine (its three action states), the GameUtils facade (collider-type checks, the take-ladder dice, the fully-inside test), the EntityType/CollisionGroups tables, and the LadderSensor/TrashCan/Player classes it touches. Lifecycle: constructor (build the sphere physics body + arm collisions) → per-frame Update (poll ladders, run the current state handler, sync graphics) → Destroy (inherited from GameObject, fired through the gameObjectRemoved event).

The defining architectural choice is at the top of the file: normal and crazy barrels are one class, discriminated by state.

// Enemy — a DK barrel, two flavors picked at spawn (D28: data-driven, old CrazyEnemy folded in here).
// NORMAL barrel → starts ROLLING; rolls the girders and may take ladders (ROLLING ⇄ DESCENDING).
// CRAZY barrel → starts BOUNCING; vertically-tossed wild barrel, bounces girder-to-girder down the screen, never takes ladders.
// A crazy barrel never leaves BOUNCING and a normal never enters it, so State === BOUNCING is the "is this crazy?" discriminator.
Hard-won victory (D28 + the contact table). There used to be a separate CrazyEnemy class and hard-coded if (other is Player) gameOver() / if (other is TrashCan) ignite() logic inside the enemies' own collision callbacks. Two problems compounded: the two classes drifted, and the death/ignite branches fired from inside entity callbacks where a barrel-vs-barrel or barrel-vs-enemy contact could mis-route. The latest refactor (commit "Replace collision routing with contact table, finally fix barrel/enemy bug") did two things at once: (1) folded CrazyEnemy into this one data-driven Enemy (D28), so there is one class, one array, one update loop; and (2) moved every cross-entity verdict out to the declarative contactRules table, matched by EntityType. Now this file owns only the barrel's reaction to the world (walls, girders, ladders); who-kills-whom is a separate, two-sided table. Because the enemy's collision mask excludes ENEMY, barrels never even generate contacts with each other — the old bug is structurally impossible.

// Feel constants (module-scope)

Eight tuning numbers split into a "roll" group (normal barrel) and a "bounce" group (crazy barrel). They are module-private — not part of the per-level feel profile — because barrel difficulty is global, not per-character.

Line-level notes: IsGrounded is "event-owned" — it is set/cleared only by the collision callbacks (enter sets it on a girder, exit clears it), never polled, so it reflects real contact state. direction and bounceDirection are independent: a normal barrel only ever uses direction, a crazy one only bounceDirection. ladderSensorValue caches the climb direction of the ladder the barrel is currently inside, so SnapDirectionToLadder can face the roll the right way when the descent ends. The last two fields are the hot-path allocation killers: scratchVel is a single reused velocity object the three setLinvel movers write into, and onLadderIntersection is the ladder-poll callback hoisted to a bound field (created once at construction) so checkLadderIntersections stops minting a fresh closure every frame. Both matter because they multiply with the on-screen barrel count.

Line-level notes: GameObjectType.SPHERE maps to a Rapier ball collider — the barrel is literally a rolling sphere. The body is RigidBodyDesc.dynamic(), so Rapier integrates it (though the handlers override velocity every frame with setLinvel). The collision group is ENEMY and the mask is PLATFORM | PLAYER_BOUNDING_BOX — so a barrel collides with girders/walls and the player box, but not with other barrels (no ENEMY bit in the mask) and not with sensors. That mask exclusion is half of why the barrel/enemy bug is gone (the other half is moving the verdicts to the contact table).

Note (auto-armed events). The constructor never calls enableContactEvents(). Because this class defines OnCollisionEnter/OnCollisionExit, GameObject.enablePhysicsEvents sees the callbacks during createObjectPhysics and turns on COLLISION_EVENTS for the collider automatically. The contact-table rules then ride the same event stream (see Physics.handleCollisionEvents, which calls both the entity callback and Contacts.Dispatch).

Line-level notes: bounceDirection is normalised to exactly ±1 here — the spawner passes Math.sign(...) which can be 0 (oil can directly below), and this collapses 0 to +1 (rightward). Collision callbacks — the barrel's reaction to the world

These two methods are the barrel's own movement reactions: reverse at walls, track grounded, bounce off girders. They are NOT where the game-rule verdicts live (kill player / ignite can) — those are in the contact table. The split is the whole point: the entity knows how to move through its world; it does not know it can kill anything.

public OnCollisionEnter(other: GameObject): void

Fired by Physics on every solid collision start. Branches first on flavour (State === BOUNCING), then on what was hit. Parameter: the other GameObject. Returns nothing.

const otherCollider = other.PhysicsBody?.collider(0);
if (!otherCollider) return;

// CRAZY barrel: bounce off girders, flip drift at walls. Never grounds/rolls.
if (this.State === EnemyStates.BOUNCING) {
  if (GameUtils.IsColliderType(otherCollider, EntityType.WALL)) {
    this.bounceDirection = this.bounceDirection * -1;
    return;
  }

  if (
    GameUtils.IsColliderType(otherCollider, EntityType.PLATFORM) ||
    GameUtils.IsColliderType(otherCollider, EntityType.ONE_WAY_PLATFORM)
  ) {
    // Only bounce when coming DOWN onto a girder (ignore contacts while
    // already rising), so a single landing produces one clean kick.
    if (this.bounceVelocityY <= 0) {
      this.bounceVelocityY = BOUNCE_IMPULSE;
    }
  }
  return;
}

// NORMAL barrel below — wall reverse + grounded tracking.

// Wall — reverse direction.
if (GameUtils.IsColliderType(otherCollider, EntityType.WALL)) {
  this.direction = this.direction * -1;
  return;
}

// One-way platform — grounded only when landing on top of it.
if (GameUtils.IsColliderType(otherCollider, EntityType.ONE_WAY_PLATFORM)) {
  if (this.PhysicsBody!.translation().y > otherCollider.translation().y) {
    this.IsGrounded = true;
  }
  return;
}

// Solid platform — grounded.
if (GameUtils.IsColliderType(otherCollider, EntityType.PLATFORM)) {
  this.IsGrounded = true;
  return;
}

Line-level notes: collisions hand you a raw Rapier Collider, not the class instance, so identity is checked by GameUtils.IsColliderType(collider, EntityType.X) (which reads collider.parent().userData.type) — never instanceof. The crazy bounce-kick is gated on bounceVelocityY <= 0: only re-kick when the barrel is descending onto the girder, so one landing yields one clean upward impulse instead of a jitter while it is still rising. The one-way-platform ground check compares the barrel's translation().y against the platform's — a barrel only "lands" on a one-way girder it is above, so it can rise through it from below.

public OnCollisionExit(other: GameObject): void

Fired when a solid contact ends. Clears grounded and, while rolling, turns the barrel around so it doesn't walk off the girder edge. Parameter: the other GameObject.

// Crazy barrels don't ground or edge-turn — nothing to do on exit.
if (this.State === EnemyStates.BOUNCING) return;

const otherCollider = other.PhysicsBody?.collider(0);
if (!otherCollider) return;

if (otherCollider.isSensor()) return;

if (
  GameUtils.IsColliderType(otherCollider, EntityType.ONE_WAY_PLATFORM) ||
  GameUtils.IsColliderType(otherCollider, EntityType.PLATFORM)
) {
  this.IsGrounded = false;

  if (this.State === EnemyStates.ROLLING) {
    this.direction = this.direction * -1;
  }
}

Line-level notes: the edge-turn is State === ROLLING-gated, NOT fired while DESCENDING. That is essential — taking a ladder intentionally leaves the platform downward, and if the exit handler flipped direction every time a descending barrel passed off a girder it would scramble the roll direction it resumes with. The isSensor() early-out means leaving a ladder sensor never affects grounded/direction (ladders are handled by the poll, not by collision events).

Gotcha (edge-turn coupling). IsGrounded and the wall/edge direction flips are split across two sources: walls flip on OnCollisionEnter, platform edges flip on OnCollisionExit. A barrel that rolls into a wall reverses immediately; a barrel that rolls off an open ledge reverses when it loses the platform. Both feed the same direction field the rolling handler reads next frame.

State-handler helpers — the verbs the handlers call

The free-function state handlers in states/ stay tiny by delegating the actual physics to these public methods on the Enemy.

public DecideTakeLadder(): boolean

Rolls the take-ladder dice. Returns: true if the barrel should descend the ladder it is currently inside. Always true while the oil can is unlit (the relentless DK opening); otherwise a flat LADDER_TAKE_CHANCE (0.75) roll.

if (!this.currentTrashCan?.IsOnFire) return true;
return GameUtils.CalculatePercentChance(LADDER_TAKE_CHANCE);

Line-level notes: currentTrashCan is the oil can the director passes into Update each frame. Until the very first crazy barrel lights it (IsOnFire flipped by the contact table), every barrel descends every ladder — the wave of barrels pours straight down. Once lit, barrels start branching probabilistically, which is what spreads them across the girders. CalculatePercentChance is the mathHelpers percentChance re-export (Math.random() < p).

public SnapDirectionToLadder()

Faces the roll the way the ladder sensor points. Used at the moment a descent ends (and at its start) so the barrel resumes rolling in a sensible direction.

if (this.ladderSensorValue > 0) {
  this.direction = 1;
} else if (this.ladderSensorValue < 0) {
  this.direction = -1;
}

ladderSensorValue was cached in checkLadderIntersections from the LadderSensor.Direction of whatever core the barrel was inside (positive = right, negative = left). A 0 value leaves direction untouched.

public RollHorizontally()

The per-frame normal-barrel roll. Sets the collision mask based on grounded-ness, then pins the velocity: horizontal by direction, vertical to a constant downward pull.

let mask: number = CollisionGroups.PLAYER_BOUNDING_BOX;
if (this.IsGrounded) {
  mask = CollisionGroups.PLATFORM | CollisionGroups.PLAYER_BOUNDING_BOX;
}
this.setCollisionMask(mask);

let vx = -this.groundSpeed;
if (this.direction >= 0) vx = this.groundSpeed;
this.scratchVel.x = vx;
this.scratchVel.y = -ROLL_FALL_SPEED;
this.PhysicsBody!.setLinvel(this.scratchVel, true);

Line-level notes: the velocity is fully authored every frame — the barrel never coasts on Rapier's integrator. y: -ROLL_FALL_SPEED is a hand-applied downward pull (≈ gravity) that keeps the sphere pressed onto the girders and sliding down slopes, but it is deterministic, not engine gravity. The mask is rewritten each frame: while grounded it keeps PLATFORM so the barrel rests on girders; the moment it is airborne (IsGrounded false) PLATFORM drops from the mask, so a barrel that has rolled off an edge falls cleanly without snagging on the girder lip below until it next lands. PLAYER_BOUNDING_BOX is always kept so the barrel can hit the player no matter what.

public DescendLadder()

The per-frame ladder-descent move: drop platform collisions and travel straight down.

// Drop PLATFORM so the barrel passes through girders; ALWAYS keep the player
// box so it can still kill the player while descending.
this.setCollisionMask(CollisionGroups.PLAYER_BOUNDING_BOX);
this.scratchVel.x = 0;
this.scratchVel.y = -this.groundSpeed * LADDER_DESCEND_FACTOR;
this.PhysicsBody!.setLinvel(this.scratchVel, true);

Line-level notes: removing PLATFORM from the mask is how the barrel passes through the girders it crosses on the way down a ladder — the same trick the player uses to climb. Keeping PLAYER_BOUNDING_BOX means a descending barrel still kills the player it lands on. Horizontal velocity is hard-zeroed so the descent is perfectly vertical regardless of the roll direction it came in with.

public Bounce()

The per-frame crazy-barrel arc. Integrates the hand-rolled gravity, clamps to terminal speed, and applies horizontal drift. Unlike the roll/descend methods it never touches the mask — a crazy barrel always keeps PLATFORM so it bounces off girders.

this.bounceVelocityY -= BOUNCE_GRAVITY * this.time.Delta;
if (this.bounceVelocityY < -BOUNCE_MAX_FALL) {
  this.bounceVelocityY = -BOUNCE_MAX_FALL;
}

let vx = -BOUNCE_DRIFT_SPEED;
if (this.bounceDirection >= 0) vx = BOUNCE_DRIFT_SPEED;
this.scratchVel.x = vx;
this.scratchVel.y = this.bounceVelocityY;
this.PhysicsBody!.setLinvel(this.scratchVel, true);

Line-level notes: this is the one place time.Delta matters — bounceVelocityY is integrated frame-rate-independently (-= BOUNCE_GRAVITY * Δt), whereas the roll's downward speed is a flat constant. The upward kick is not here; it is applied in OnCollisionEnter when the bouncing barrel lands on a girder (bounceVelocityY = BOUNCE_IMPULSE). So Bounce only ever pulls down and clamps; the bounce up is event-driven. Drift sign flips at walls (also in OnCollisionEnter), so the barrel weaves down the screen.

private checkLadderIntersections()

Refreshes this frame's ladder inputs (IsInsideLadder, IsAtLadderBottom, ladderSensorValue) by polling sensor overlaps with the hoisted onLadderIntersection callback — deliberately NOT event-driven (D9, the "fully inside" climb idiom). Runs the bottom-cooldown timer too. The method body itself is now just the cooldown tick, the two flag resets, and the poll call; every per-overlap test lives in the bound field.

if (this.ladderBottomCooldown > 0) {
  this.ladderBottomCooldown -= this.time.Delta;
}

this.IsInsideLadder = false;
this.IsAtLadderBottom = false;

// Poll intersections with the hoisted callback (this.onLadderIntersection) — no
// per-frame closure allocation. (Not event-driven: the "fully inside" climb
// idiom, D9, needs continuous polling.)
this.Physics.World.intersectionPairsWith(
  this.PhysicsBody!.collider(0),
  this.onLadderIntersection
);

The per-overlap work lives in onLadderIntersection, a bound arrow field built once in the field initializer (so this is already captured) rather than a closure re-created on every checkLadderIntersections call:

private onLadderIntersection = (otherCollider: RAPIER.Collider) => {
  const ladder = this.Physics.GetGameObjectFromCollider(otherCollider);

  // Fully inside a ladder core with a climb direction — a descent candidate.
  if (
    GameUtils.IsColliderType(otherCollider, EntityType.LADDER_CORE_SENSOR) &&
    ladder instanceof LadderSensor &&
    ladder.Direction !== 0 &&
    GameUtils.IsObjectFullyInsideSensor(otherCollider, this)
  ) {
    this.ladderSensorValue = ladder.Direction;
    this.IsInsideLadder = true;
  }

  // At the ladder bottom — flag it (the DESCENDING handler ends the descent here)
  // AND arm the anti-oscillation cooldown immediately, on first contact in ANY
  // state, so a descent through an overlapping core+bottom sensor completes.
  if (
    GameUtils.IsColliderType(otherCollider, EntityType.LADDER_BOTTOM_SENSOR) &&
    this.ladderBottomCooldown <= 0
  ) {
    this.IsAtLadderBottom = true;
    this.ladderBottomCooldown = LADDER_BOTTOM_COOLDOWN;
  }
};

Line-level notes: World.intersectionPairsWith walks every sensor the barrel's collider currently overlaps (the barrel is a solid, but sensors register intersections against solids). Physics.GetGameObjectFromCollider resolves the raw collider back to its owning GameObject so the instanceof LadderSensor + ladder.Direction reads work. Two distinct conditions: a core hit only counts when the barrel is IsObjectFullyInsideSensor (horizontally contained — the D9 gate, the same one the player's climb uses via GameUtils.IsAnySensorTriggeredObjectFullyInside); a bottom hit is gated by the cooldown.

Allocation-free per-barrel hot path. Both halves of the barrel's per-frame work used to allocate: checkLadderIntersections passed an inline (otherCollider) => {…} closure to intersectionPairsWith (a fresh function object every frame), and the three movers (RollHorizontally, DescendLadder, Bounce) each built a fresh { x, y } literal for setLinvel. Now the closure is the hoisted onLadderIntersection field and the velocity is the single reused scratchVel — Rapier reads both synchronously, so reuse is safe. With dozens of barrels on screen those were the allocations that fed GC churn at the worst moment; the hot path now mints nothing per frame.
Hard-won victory (the bottom cooldown armed on entry, not on exit). The cooldown is set true + armed on the first bottom-sensor contact in any state — including while a barrel is still rolling toward the ladder — not only when a descent ends. The source comment spells out why: arm the cooldown immediately, "on first contact in ANY state, so a descent through an overlapping core+bottom sensor completes." When a ladder's core and bottom sensors overlap, a barrel that began descending would otherwise see IsAtLadderBottom on its very first descent frame, immediately flip back to ROLLING, and never actually descend. Arming the cooldown the instant the bottom sensor is touched suppresses that spurious bottom flag long enough for the descent to take.

public Update(player: Player, trashCan?: TrashCan)

The per-frame entry point, called by GameDirector.Update as enemy.Update(this.Player, firstTrashCan). Parameters: player (the uniform update contract — but the enemy deliberately ignores it), trashCan (the oil can it reads for the ladder decision). Returns nothing.

// Uniform update contract: the enemy ignores the player (it doesn't track or
// chase) and reads the trash can (the oil can) for its ladder decision.
void player;

if (this.IsBeingDestroyed) {
  return;
}

// Stash this frame's inputs, refresh ladder state, run the current state.
// Crazy (BOUNCING) barrels never take ladders, so skip the poll for them.
this.currentTrashCan = trashCan;
if (this.State !== EnemyStates.BOUNCING) {
  this.checkLadderIntersections();
}
this.fsm.Update();

this.syncGraphicsToPhysics();

Line-level notes: void player; is an explicit "I am intentionally not using this argument" — the barrel doesn't chase, unlike a smarter enemy might, but it still conforms to the shared Update(player, trashCan?) signature so the director can tick it in the same loop as everything else. The IsBeingDestroyed guard short-circuits a barrel that has already been flagged for teardown (e.g. it hit the player or fell in the can) so it doesn't run a frame on a half-dead body. Crazy barrels skip checkLadderIntersections entirely — they never descend, so the poll would be wasted work. fsm.Update() dispatches to exactly one of the three handlers; syncGraphicsToPhysics() (inherited) then copies the body's translation/rotation onto the Three.js sprite.

Modifying this: tune barrel feel with the module constants at the top (roll speed, ladder odds, bounce height/drift) — they are global, not per-level. To add a new barrel behaviour, add a state to enemyStates and a handler in states/, then wire it into the fsm map in initializeEnemyAttributes; do not add a subclass (the D28 lesson was one class, many states). Never add "kills X / wins Y" logic to OnCollisionEnter — that belongs in the contact table; this file is strictly the barrel's reaction to the world. If you give barrels a new sensor interaction, follow checkLadderIntersections (poll, don't event) only if it needs the D9 fully-inside semantics; otherwise prefer a contact rule.

src/game/entities/enemy/enemyStates.ts 17 LOC

The enemy's state enum — a frozen string-const map plus a derived union type, mirroring PlayerStates and CollisionGroups. It is imported by enemy.ts (the State field, the FSM map keys) and by all three state handlers (to assign transitions). String values (not numeric) so debug logs and the StateMachine's Record<S,...> handler map read clearly.

const EnemyStates = {
  // Rolling along a girder; at an unjudged ladder it decides whether to descend. -> DESCENDING
  ROLLING: "rolling",

  // Taking a ladder — straight down, passing through platforms. -> ROLLING at the bottom sensor
  DESCENDING: "descending",

  // Crazy barrel: bounces girder-to-girder, never takes ladders. Terminal (marks it as crazy).
  BOUNCING: "bouncing",
} as const;

// Type for enemy state values — ensures only valid states are used.
export type EnemyState = (typeof EnemyStates)[keyof typeof EnemyStates];

export default EnemyStates;

The header comment ties the file to the project's decision log: "Enemy state machine for the barrel enemy (DK normal/crazy); game-side per ADR-0003 / D3." The D3 / ADR-0003 rule is that game-specific FSMs live in the game layer (here), driven by the engine-pure generic StateMachine — the engine never names "rolling" or "bouncing". as const freezes the values to string literals so EnemyState is the exact union "rolling" | "descending" | "bouncing", which is what makes the FSM handler map exhaustively type-checked.

Note. BOUNCING is terminal — there is no transition out of it, which is precisely why State === EnemyStates.BOUNCING doubles as the "is this a crazy barrel?" test throughout enemy.ts. A normal barrel only ever ping-pongs between ROLLING and DESCENDING.

src/game/entities/enemy/states/handleEnemyRolling.ts 31 LOC

The ROLLING state handler — a free function (enemy: Enemy) => void matching the StateMachine handler shape. The engine FSM calls it once per frame while enemy.State === ROLLING. It owns the per-frame roll and the once-per-entry ladder decision; the wall-reverse and edge-turn that also belong to rolling live in the enemy's collision callbacks, not here.

const handleEnemyRolling = (enemy: Enemy) => void

Decide-then-act. If the barrel is fully inside an un-judged ladder, roll the take-ladder dice exactly once; if it takes, snap direction, switch to DESCENDING, and begin the descent this same frame. Otherwise just roll.

// Change state
if (enemy.IsInsideLadder) {
  // Decide once per continuous occupancy of the ladder core.
  if (!enemy.DidJudgeLadder) {
    enemy.DidJudgeLadder = true;

    if (enemy.DecideTakeLadder()) {
      enemy.SnapDirectionToLadder();
      enemy.State = EnemyStates.DESCENDING;
      enemy.DescendLadder(); // begin the descent on this same frame
      return;
    }
  }
} else {
  // Off the ladder — re-arm the decision for the next ladder entered.
  enemy.DidJudgeLadder = false;
}

// Handle Rolling state
enemy.RollHorizontally();

Line-level notes: DidJudgeLadder is the "decide once" latch — without it the dice would be re-rolled every frame the barrel sits inside the core, so a barrel that declined would eventually take the ladder by repeated coin-flips. It is set true on the first judged frame and only re-armed (false) once the barrel is off the ladder again, so each continuous occupancy gets exactly one decision. Calling DescendLadder() right after setting State = DESCENDING means the velocity flips to straight-down on the same frame the decision is made (no one-frame stall), and the return skips the trailing RollHorizontally() so the two move methods never fight over the velocity in one frame.

Gotcha (transition is next-frame for the handler, this-frame for the move). Reassigning enemy.State here does not re-dispatch within this fsm.Update() — the StateMachine reads State once per update, so the DESCENDING handler first runs next frame. The handler bridges that gap by calling DescendLadder() itself, so the physics already reflect the descent before handleEnemyDescending takes over.

The DESCENDING state handler. Runs each frame while the barrel is taking a ladder. It watches for the ladder-bottom sensor and, on reaching it, lands and resumes rolling; otherwise it keeps descending.

const handleEnemyDescending = (enemy: Enemy) => void

If IsAtLadderBottom (set by the cooldown-gated poll), re-ground, switch back to ROLLING, and roll this same frame. Otherwise keep descending straight down.

// Change state
if (enemy.IsAtLadderBottom) {
  // The cooldown was already armed when the bottom sensor fired (in the
  // enemy's ladder poll); just land and resume rolling on this same frame.
  enemy.IsGrounded = true; // landed at the ladder foot, back on the girder
  enemy.State = EnemyStates.ROLLING;
  enemy.RollHorizontally();
  return;
}

// Handle Descending state
enemy.DescendLadder();

Line-level notes: the handler manually sets IsGrounded = true because at the ladder foot the platform-collision callback may not have re-fired yet (the barrel had PLATFORM dropped from its mask during descent) — without it the very next RollHorizontally() would think the barrel is airborne and drop PLATFORM again. It calls RollHorizontally() in the same frame (which restores PLATFORM to the mask and starts horizontal motion) so the resume is seamless, then returns to skip the trailing DescendLadder(). The anti-oscillation cooldown that keeps this from immediately re-triggering a descent was already armed back when the bottom sensor first fired in checkLadderIntersections — see the victory callout there.

};

Why so thin: the discriminator State === BOUNCING short-circuits the ladder poll in Update (crazy barrels never descend) and routes every girder/wall response through OnCollisionEnter's bouncing branch. So by the time this handler runs, the only thing left to do per frame is integrate the arc — which is exactly one call. This is the payoff of the D28 fold-in: the "crazy" behaviour is one extra handler plus a couple of branches, not a whole parallel class.

Modifying this: if you want crazy barrels to eventually settle into rolling, this is where you'd add the exit transition (e.g. when bounce energy decays past a threshold, set enemy.State = EnemyStates.ROLLING).
