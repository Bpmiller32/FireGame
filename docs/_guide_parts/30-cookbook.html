Cookbook — how to extend the engine

This is the hands-on companion to Game Config — the data that defines the game. The whole point of the engine/game split is that almost every gameplay change is a data edit in a config file, not a new code path. Below are six step-by-step recipes — add an entity, add a level, add a player state, tune the feel, add an input, add a contact rule — each one grounded line-by-line in the five files that drive the game: entityFactories.ts, levelRegistry.ts, inputBindings.ts, contactRules.ts, and baseFeel.ts. Every recipe lists which files to touch in order, the exact additions, and the one gotcha that will bite you if you skip a step.

src/game/config/entityFactories.ts src/game/config/levelRegistry.ts src/game/config/inputBindings.ts src/game/config/contactRules.ts src/game/attributes/baseFeel.ts
Note — the design these recipes lean on. Three identity tables are the spine of the game: EntityType (the routing string every entity carries), PlayerStates (the FSM states), and CollisionGroups (the Rapier bit masks). Each is a single const map so the spawn side and the routing side can never drift. The recipes below mostly amount to "add a row to one of these tables, then wire the one or two places that consume it."

Recipe — add a new entity type touches 5+ files

Goal: introduce a brand-new level-placeable thing (we'll use a Spike hazard that kills the player on touch). An entity needs an identity (its EntityType string), a way for the GLB parser to recognise its cube (the resourceLoader's canonical-type mirror), a class that builds its physics body, a factory that the level loader dispatches to, optionally its own collision group, and a contact rule that says what happens when it's touched. Do them in this order so each step compiles against the one before it. 

flowchart LR
  A["Blender cube named "Spike""] --> B["resourceLoader.parseScene"]
  B -->|CANONICAL_TYPES lookup| C["LevelData row type=Spike"]
  C --> D["GameDirector.LoadLevelData"]
  D -->|"ENTITY_FACTORIES[Spike]"| E["createSpike factory"]
  E --> F["new Spike(...)"]
  F --> G["contactRules: Spike+Player = gameOver"]

1. src/game/types/entityType.ts — add the identity string

Add one line to the EntityType const. The value is the verbatim string everything routes on, so pick it once and never change it. 

const EntityType = {
  PLAYER: "Player",
  WALL: "Wall",
  // ...existing...
  TELEPORTER: "Teleporter",
  SPIKE: "Spike",            // <-- NEW
  PLAYER_START: "PlayerStart",
  CAMERA_START: "CameraStart",
} as const;

2. src/engine/resources/resourceLoader.ts — mirror it in CANONICAL_TYPES

The engine deliberately cannot import the game's EntityType (it names zero game concepts), so the parser keeps a hand-synced copy. Add the lower-cased mesh-name token mapping to the canonical string. The parser takes the first _-delimited token of a cube's name, lower-cases it, and looks it up here. 

private static readonly CANONICAL_TYPES: Record<string, string> = {
  wall: "Wall",
  platform: "Platform",
  onewayplatform: "OneWayPlatform",
  // ...existing...
  teleporter: "Teleporter",
  spike: "Spike",            // <-- NEW: a cube named "Spike" (or "Spike_2") resolves here
  playerstart: "PlayerStart",
  camerastart: "CameraStart",
  graphics: "GraphicsObject",
  graphicsobject: "GraphicsObject",
};
Gotcha — the silent-drop. Skip this step and your Spike cube resolves to no type: lookupCanonicalType returns undefined, parseScene treats the leaf mesh as a mis-named cube, calls warnUnknownType, and skips it entirely. The entity just never appears. This mirror in Engine Resources — resourceLoader is the "known residue" of the engine/game split: it must be kept in sync by hand whenever you add a level-placeable type.

3. src/game/entities/spike.ts — the entity class

Patterned exactly on Teleporter (a sensor that defines no callback of its own and joins the contact table). It extends GameSensor, builds its physics body with the new EntityType.SPIKE routing flag, flags itself a sensor, and arms contact events so the declarative table can see it. 

import * as RAPIER from "@dimforge/rapier2d-compat";
import GameObjectType from "../../engine/types/gameObjectType";
import GameSensor from "../../engine/entities/gameSensor";
import EntityType from "../types/entityType";

// Spike - kills the player on contact. The "who dies" rule lives in the
// declarative contact table (contactRules.ts), not here.
export default class Spike extends GameSensor {
  constructor(
    size: { width: number; height: number },
    position: { x: number; y: number },
    rotation: number = 0
  ) {
    super();

    this.createObjectPhysics(
      EntityType.SPIKE,
      GameObjectType.CUBE,
      size,
      position,
      rotation,
      RAPIER.RigidBodyDesc.fixed()
    );

    this.setAsSensor(true);
    // Take part in the contact table even though we define no callback here.
    this.enableContactEvents();

    this.createObjectGraphicsDebug("red", 0.4);
  }
}
Hard-won victory — enableContactEvents(). GameObject only auto-arms collision events for entities that define an OnCollisionEnter/OnSensorEnter callback (enablePhysicsEvents). A contact-table-only entity defines none, so without this call Rapier emits no event for it and your contact rule never fires. Teleporter hit exactly this and the comment there spells it out: "Take part in the contact table even though we define no callback here."

4. (optional) src/game/types/gameCollisionGroups.ts — a collision group

A sensor like Spike can skip this — sensors use the default group (collides/intersects with everything), the same way Teleporter sets no group. You only need a group bit if the entity is a solid the player must physically stop against. If so, claim the free bit (bit 1 is open — it used to be PLAYER_HIT_BOX) and remember the player must be told to collide with it. 

const CollisionGroups = {
  PLAYER_BOUNDING_BOX: 0b0000000000000001, // Bit 0
  SPIKE:               0b0000000000000010, // Bit 1 (was free) — only if SOLID
  ENEMY:               0b0000000000000100, // Bit 2
  PLATFORM:            0b0000000000001000, // Bit 3
  DEFAULT:             0b1111111111111111,
} as const;
Gotcha — collision is two-way. Collision occurs only when (a.group & b.mask) AND (b.group & a.mask). Giving Spike its own group does nothing unless the player's mask includes it. The player sets setCollisionMask(CollisionGroups.PLATFORM | CollisionGroups.ENEMY, 0) in its constructor — you'd have to OR in | CollisionGroups.SPIKE there too. For a kill-on-touch hazard a sensor (no group) is simpler and avoids this entirely.

5. src/game/config/entityFactories.ts — the factory + registry row

Write a factory matching the EntityFactory signature (data, ctx, type) => GameObject and register it under the EntityType.SPIKE key. Mirror the existing one-liners (axis mapping position[0]/position[2], the -rotation[1] negation, sizes passed straight through). Read any per-entity metadata off data.meta with the metaNum/metaCoords helpers. 

import Spike from "../entities/spike";

// FROM nothing — brand-new hazard.
const createSpike: EntityFactory = (data) => {
  const spike = new Spike(
    { width: data.width, height: data.height },
    { x: data.position[0], y: data.position[2] },
    -data.rotation[1]
  );
  return spike;
};

const ENTITY_FACTORIES: Record<string, EntityFactory> = {
  [EntityType.CAMERA_SENSOR]: createCameraSensor,
  // ...existing...
  [EntityType.TELEPORTER]: createTeleporter,
  [EntityType.SPIKE]: createSpike,   // <-- NEW
};
Note — the ctx arg. Spike isn't queried by index anywhere (unlike trashCans[0] or the ladder sub-arrays), so its factory pushes into no FactoryContext view and ignores ctx. If you do need the director to iterate your entities, add a typed array to FactoryContext, push the new entity into it inside the factory, and alias it in GameDirector.getFactoryContext (and tear it down in UnloadLevelData). See Game Director for both.

6. src/game/config/contactRules.ts — what touching it does

Spike is a sensor, so it fires the "sensorEnter" phase. Declare the rule the same way the WinFlag and Teleporter rules are declared — match both sides byType, pick the phase, run the reaction. 

// Player touches a Spike -> game over.
contacts.Add(
  byType(EntityType.SPIKE),
  byType(EntityType.PLAYER),
  "sensorEnter",
  () => Emitter.emit("gameOver")
);

Step 7 — the GLB. In the level editor (BlockBench/Blender) drop a cube, name it Spike (or Spike_2 for a second instance — the parser strips the suffix), and export the .glb into src/assets/levels. The parser reads the cube's name for its type and its texture name for metadata; you don't touch any loader code. See the add a level recipe below for the drop-in mechanics.

Recipe — add a playable level touches 1 file + 1 GLB

Levels are GLB-only and the whole pipeline is data: drop a .glb into src/assets/levels and add one row to LEVEL_REGISTRY. GameDirector boots index 0, F2 cycles, and the dat.gui level dropdown selects — all three funnel through the same loadLevelByIndex, so one row is genuinely all you add.

1. Drop the GLB — auto-discovery does the rest

GLBs under src/assets/levels are Vite-bundled (hashed), not in public/. An eager import.meta.glob collects every *.glb as its bundled URL into a filename→URL map, which levelUrl() resolves against. 

const levelGlbUrls = import.meta.glob("../../assets/levels/*.glb", {
  query: "?url",
  import: "default",
  eager: true,
}) as Record<string, string>;
Warning — the glob is eager. EVERY .glb in the folder is bundled into the build whether or not a registry row references it. Delete unused level GLBs to stay lean — an orphaned 4 MB level still ships.

2. src/game/config/levelRegistry.ts — add a LEVEL_REGISTRY row

Each row is a LevelEntry: a name (the dat.gui label and GameDirector.CurrentLevelName value), a source (the bundled URL via levelUrl("file.glb")), a feelProfile (a key of FEEL_PROFILES — see the tune the feel recipe), a cameraFollow boolean, and an optional graphics overlay key. 

export const LEVEL_REGISTRY: LevelEntry[] = [
  {
    name: "Slopes Testing",
    source: levelUrl("TestLevel-Slopes.glb"),
    feelProfile: "dk",
    cameraFollow: true,
  },
  // ...existing rows...
  {
    name: "My New Level",                    // <-- NEW ROW
    source: levelUrl("MyNewLevel.glb"),
    feelProfile: "celeste",
    cameraFollow: true,
    // graphics: "dkGraphicsData",  // optional detailed-art overlay (a ResourceLoader.Items key)
  },
];

Append it and you're done — it joins the F2 cycle automatically (switchToNextLevel does (currentLevelIndex + 1) % LEVEL_REGISTRY.length). To boot into it instead, change DEFAULT_LEVEL_INDEX.

Gotcha — every level needs a PlayerStart. GameDirector.setPlayerStart scans the LevelData for a PlayerStart row and only then constructs the Player. No PlayerStart cube → the player is never spawned → Update's guard if (!this.Player?.PhysicsBody) return; freezes the whole game with no error. Include a CameraStart too, or the camera sits at its default origin.
Hard-won victory — fail loud on a missing GLB (D5 neighbour). If levelUrl() can't resolve a filename it logs the available list and returns "". loadLevelByIndex checks for that empty source and throws a clear error before handing it to the loader — otherwise gltfLoader.load("") fetches the HTML page and throws a baffling parse error far from the real cause (a typo'd filename). The surrounding try/catch/finally still closes the Vue loading overlay so the screen never hangs.

Recipe — add a player state touches 3 files

The player movement FSM is the generic StateMachine (see Architecture — Finite State Machine) driving a handler-per-state map. Adding a state — say DASHING — is three edits: the state constant, a handler file, and the registration + at least one transition into it.

1. src/engine/types/playerStates.ts — add the constant

const PlayerStates = {
  IDLE: "idle",
  RUNNING: "running",
  JUMPING: "jumping",
  FALLING: "falling",
  CLIMBING: "climbing",
  DASHING: "dashing",      // <-- NEW
} as const;

2. src/game/entities/player/states/handlePlayerDashing.ts — the handler

A handler is a plain (player: Player) => void. It runs once per frame while the player is in this state; it reads player.Input/player.IsTouching, writes player.NextTranslation, drives the sprite, and reassigns player.State to transition. Pattern it on handlePlayerIdle: guard-clause transitions first (each with an early return), then the in-state behaviour. 

import Player from "../player";
import PlayerStates from "../../../../engine/types/playerStates";

const handlePlayerDashing = (player: Player) => {
  // Transition out: dash ended -> fall back to falling/idle.
  if (player.IsTouching.ground) {
    player.State = PlayerStates.IDLE;
    return;
  }

  // In-state behaviour: drive horizontal velocity hard in the facing direction.
  const dir = player.Direction; // LEFT/RIGHT/NEUTRAL
  player.NextTranslation.x = dir * player.MaxGroundSpeed * 2;

  // ...sprite animation via player.SpriteAnimator.ChangeState(...)...
};

export default handlePlayerDashing;

3. src/game/entities/player/player.ts — register + transition IN

Import the handler and add it to the FSM map (the player constructs its StateMachine in initializePlayerAttributes). Then add a transition into DASHING from an existing handler — e.g. in handlePlayerRunning when a dash key is pressed. 

import handlePlayerDashing from "./states/handlePlayerDashing";

this.fsm = new StateMachine<Player, PlayerState>(this, {
  [PlayerStates.IDLE]: handlePlayerIdle,
  [PlayerStates.RUNNING]: handlePlayerRunning,
  [PlayerStates.FALLING]: handlePlayerFalling,
  [PlayerStates.JUMPING]: handlePlayerJumping,
  [PlayerStates.CLIMBING]: handlePlayerClimbing,
  [PlayerStates.DASHING]: handlePlayerDashing,   // <-- NEW
});
Gotcha — a state with no handler silently freezes. StateMachine.Update dispatches via this.handlers[this.host.State]?.(this.host) — that optional chaining means a state whose key is missing from the map is a no-op, not a crash. Forget step 3 and the player enters DASHING and just stops updating forever, with no error. Every PlayerStates member must have a map entry.
Note — transitions take effect NEXT frame. StateMachine.Update reads State once per frame. A handler that sets player.State = PlayerStates.DASHING does not re-dispatch into the dash handler within the same frame — the new state runs on the following Update. This is by design and is why every handler does its state write last and returns.
Tip — new feel knobs go in the attribute profiles. If your dash reads a new tunable (player.DashSpeed, say), declare it on Player (public DashSpeed!: number;) and seed it in baseFeel or a profile (next recipe) — the ! definite-assignment fields are undefined until something sets them, and a handler reading undefined produces NaN velocity.

Recipe — tune the feel baseFeel.ts + a profile

"Feel" is a set of numeric attributes written onto the Player by a profile. The layering is: every profile calls applyBaseFeel(player) FIRST (shared values), then overrides on top. baseFeel.ts holds the values common to DK and Celeste (climb speed/accel/decel, the coyote/buffer runtime flags, CoyoteTime, AnimationScalingFactor); setDkAttributes / setCelesteAttributes hold the per-game divergences (see Game Feel). 

flowchart TD
  A["level loads"] --> B["GameDirector.applyFeelProfile(entry.feelProfile)"]
  B --> C["FEEL_PROFILES[profile](player)"]
  C --> D["applyBaseFeel(player) — shared values"]
  D --> E["DK/Celeste overrides on top"]

Tune a SHARED value — edit baseFeel.ts

Values that should change for both profiles live here. For example, to give every profile a snappier climb, bump MaxClimbSpeed: 

const applyBaseFeel = (player: Player) => {
  // Climbing (shared)
  player.MaxClimbSpeed = 4;        // <-- e.g. raise to 6
  player.ClimbAcceleration = 30;
  player.ClimbDeceleration = 8;
  // ...coyote/buffer flags, CoyoteTime, AnimationScalingFactor...
  player.CoyoteTime = 0.033; // 2 frames
  player.AnimationScalingFactor = 1.6;
};

Tune a PER-PROFILE value — edit setDkAttributes.ts

Values that differ per game live in the profile, after the applyBaseFeel(player) call. To make the DK jump higher, raise JumpPower: 

const setDkAttributes = (player: Player) => {
  applyBaseFeel(player);            // shared first
  // DK overrides (differ from base feel)
  player.MaxGroundSpeed = 11.8;
  player.JumpPower = 24;            // <-- raise for a higher jump
  player.RiseGravity = 72;         // lighter rise = floatier ascent
  player.MaxFallSpeed = 40;
  // ...
};

Add a NEW profile — register it in FEEL_PROFILES

Write a setter (call applyBaseFeel first), then add it to the FEEL_PROFILES table in levelRegistry.ts. The FeelProfile type is keyof typeof FEEL_PROFILES, so a level row's feelProfile is then type-checked against the new key automatically. 

import setFloatyAttributes from "../attributes/setFloatyAttributes";

export const FEEL_PROFILES = {
  dk: setDkAttributes,
  celeste: setCelesteAttributes,
  floaty: setFloatyAttributes,   // <-- NEW; now usable as feelProfile: "floaty"
} as const;
Hard-won victory — feel is single-source, applied only on load. GameDirector.applyFeelProfile runs exactly once, in loadLevelByIndex, from the loading level's registry entry — there is deliberately no live debug swap that bypasses it. That's why the registry row is the one true place a level's feel is decided; a value you change in the Feel Lab at runtime is never the persisted truth.
Gotcha — ordering: profile beats base. Because the profile calls applyBaseFeel first and overrides after, a value set in BOTH places resolves to the profile's. A value set ONLY in baseFeel (and not overridden) is the effective value. If a new attribute reads as its base value when you expected the override, check you actually re-assigned it below the applyBaseFeel(player) line in the profile.

Recipe — add an input key input.ts + inputBindings.ts

Two layers handle input. The engine Input device exposes raw HELD-state booleans and names no game concept. inputBindings.ts (the game) watches the flow F-keys, detects the press edge, and emits a game event once per press. This recipe adds a new flow key — F4 → "togglePause" — through both layers. (Movement keys like arrows and Space go a different path: they're captured into InputState via CaptureInto and read by the player handlers directly — see Engine Events & Input.)

1. src/engine/input/input.ts — expose the held-state flag

The F4 flag already exists in this codebase (IsF4Pressed, with its F4 entry in the keys[] table) — for a genuinely new key you'd add all three pieces: the public field, its false init in the constructor, and a Key table entry whose keyCode is the layout-independent KeyboardEvent.code. 

public IsF4Pressed: boolean;   // field

this.IsF4Pressed = false;      // init in constructor

{                              // entry in the keys[] table
  keyCode: "F4",
  isPressed: (eventResult: boolean) => {
    this.IsF4Pressed = eventResult;
  },
},

2. src/game/config/inputBindings.ts — edge-detect and emit

Add a wasF4Pressed latch (init false in the constructor), then in Update emit the event on the false→true edge and store this frame's value. The existing F1/F2/F3 blocks are the exact template. 

private wasF4Pressed: boolean;

constructor(input: Input) {
  this.input = input;
  this.wasF1Pressed = false;
  // ...
  this.wasF4Pressed = false;   // <-- NEW
}

public Update() {
  // ...F1/F2/F3...

  // F4 -> togglePause
  if (this.input.IsF4Pressed && !this.wasF4Pressed) {
    Emitter.emit("togglePause");
  }
  this.wasF4Pressed = this.input.IsF4Pressed;   // remember for next frame
}
Gotcha — forget the latch and it fires every frame. The whole reason this file exists is that engine Input only tells you a key is held, not that it was just pressed. The && !this.wasF4Pressed guard plus this.wasF4Pressed = ... assignment is the edge detector. Drop the assignment and "togglePause" emits 60×/second while the key is down.

3. Listen for the event — GameDirector (or App.vue)

inputBindings.Update() runs each frame from App.vue's OnGameUpdate hook (right before gameDirector.Update()). The emit goes onto the global mitt bus; subscribe wherever the reaction belongs, the way GameDirector.setupEventHandlers subscribes to switchLevel / gameReset. 

Emitter.on("togglePause", () => {
  const p = this.experience.Physics;
  p.SetPaused(!p.IsPaused);
});

Recipe — add a contact rule contactRules.ts

The contact table is the declarative "when an A touches a B, do this" layer over the engine's physics dispatch (see Architecture — The Contact Table). registerContactRules(contacts) runs ONCE at startup in App.vue, after Configure, against webglExperience.Physics.Contacts. Each rule is one contacts.Add(matchA, matchB, phase, handler) call. There are no if/else chains — you add a row.

contacts.Add(a, b, phase, run) — the four args

Match each side with the byType helper (which wraps matchesType(go, type)). Pick the phase. Write the run handler — it receives the two matched GameObjects in the rule's declared (a, b) order, so cast to the concrete class to read typed fields.

phasefires whenused by
"enter"two SOLID bodies begin touchingEnemy↔Player = gameOver; Enemy↔TrashCan = ignite
"exit"two solid bodies stop touching(available, none declared)
"sensorEnter"a body enters a SENSORTeleporter, CameraSensor, WinFlag
"sensorExit"a body leaves a sensor(available, none declared)

Example — a solid contact that emits an event

The simplest rule: match both sides by type, fire on solid "enter", emit a bus event the game already listens for.

// Enemy touches Player -> game over. Two-sided.
contacts.Add(
  byType(EntityType.ENEMY),
  byType(EntityType.PLAYER),
  "enter",
  () => Emitter.emit("gameOver")
);

Example — a handler that reads typed state off the matched entity

When the reaction needs data from the entity, cast the GameObject arg. The handler args arrive in the rule's own declared order regardless of which body actually moved.

// Player enters a Teleporter sensor -> teleport to its destination.
contacts.Add(
  byType(EntityType.TELEPORTER),
  byType(EntityType.PLAYER),
  "sensorEnter",
  (teleporter, player) => {
    const dest = (teleporter as Teleporter).Destination;
    (player as Player).TeleportToPosition(dest.x, dest.y);
  }
);
Hard-won victory — two-sided dispatch. ContactRegistry.Dispatch tests each rule as both (x, y) and (y, x) with an else if, so a rule fires whether the player ran into the enemy or the enemy rolled into the player — "any touch registers" — yet fires once, not twice, and the handler always gets args in your declared order. You never write the rule twice.
Gotcha — match the phase to the body kind. A sensor entity emits "sensorEnter"/"sensorExit", a solid emits "enter"/"exit". Declare a Teleporter rule with "enter" and it never fires — the Teleporter is a sensor. When in doubt, check whether the entity's class calls setAsSensor(true).
Gotcha — both entities must have contact events armed. A rule can only fire if Rapier actually emits an event for the pair. Callback-defining entities are auto-armed; contact-table-only entities must call enableContactEvents() in their constructor (Teleporter, and the Spike in the first recipe). And the player side works in BOTH directions only because the kinematic controller's own blocked-by collisions are fed back into the table as "enter" contacts in Player.updateTranslation — without that, a player-as-mover collision fires no Rapier event at all.
Note — where rules do NOT belong. An entity's own movement reaction (an enemy reversing at a wall, a one-way platform toggling) lives with the entity, not in this table. The contact table is for cross-entity outcomes: who dies, wins, teleports, ignites.
