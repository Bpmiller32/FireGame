Player — body, movement & sprites

Randy, the hero. Player is the one entity that fuses a Rapier kinematic character controller, a Three.js animated sprite, the generic finite-state-machine, and the tunable "feel" attributes into a single GameObject subclass. It owns its movement State, snapshots input as data each frame, drives itself through five state handlers, and feeds the contact table from BOTH sides of every collision. The two helpers beside it are deliberately tiny: applyHorizontalMovement is the shared accel/decel/clamp the run/jump/fall handlers all call, and spriteAnimations is the flat lookup table of sprite-sheet frames keyed by state + facing.

This file IS the playable character. It extends GameObject (see Engine Entities), so it inherits the rigid-body/mesh marriage, the transform sync, the collision-group helpers and Destroy; what it adds on top is everything Donkey-Kong-specific about a controllable hero. It is constructed by the entity factories (the game's spawn registry), never by the engine. Its imports tell the whole story: engine singletons it caches (Time, the input device — held behind the InputSource interface, not the concrete keyboard device — ResourceLoader, Debug), the generic StateMachine it runs on, the five handlePlayer* state functions it dispatches to (see Player States), SpriteAnimator + the local SpriteAnimations table for rendering, setDkAttributes for feel (see Feel), GameUtils for collider-type checks, the local CollisionGroups/EntityType vocabulary, and Platform so shapecasts can read named fields. (It no longer imports Emitter — it does not announce its own removal; see Destroy().)

Lifecycle: the constructor wires attributes → sprite → character-controller → physics body → sprite mesh → collision groups → debug. Each frame the owner (the GameDirector) calls Update(), which snapshots input, syncs the mesh, runs one FSM tick, moves the kinematic body, and re-senses ground/walls. Destroy() frees the character controller and sprite animator, then defers to GameObject.Destroy() — it deliberately does not emit its own removal (see below).

Note on the names. NextTranslation is a velocity in units/second, not a position delta — updateTranslation() multiplies it by Time.Delta to get the per-frame move. CurrentPosition is the live body translation copied out for handlers/debug, and CurrentTranslation (inherited from GameObject) is the same value mirrored onto the mesh. Keep the trio straight and the rest of the file reads cleanly.

public constructor(size: { width: number; height: number }, position: { x: number; y: number })

Builds the entire player from a bounding-box size and a spawn point. There are no other parameters — feel and identity come from elsewhere. Order matters here; the sequence is deliberate. 

super();

this.initializePlayerAttributes();
this.setSpriteAnimator();
this.setCharacterController();

this.createObjectPhysics(
  EntityType.PLAYER,
  GameObjectType.CUBE,
  size,
  position,
  0,
  RAPIER.RigidBodyDesc.kinematicPositionBased().lockRotations()
);

this.createSpriteGraphics();

this.setCollisionGroup(CollisionGroups.PLAYER_BOUNDING_BOX, 0);
this.setCollisionMask(CollisionGroups.PLATFORM | CollisionGroups.ENEMY, 0);

Line-level notes:

  • super() runs GameObject.initializeAttributes() — grabs Experience, Scene, Physics and zeroes the transform vectors — before any player code touches this.experience.
  • initializePlayerAttributes() must run first because it both caches singletons and builds the FSM and IsTouching map the rest of the constructor and every frame rely on.
  • setSpriteAnimator() runs before createSpriteGraphics() because the latter reads the SpriteMaterial the animator produced.
  • setCharacterController() creates the KCC before createObjectPhysics makes the body — the controller is world-attached and independent of the body, so it can be built first.
  • createObjectPhysics (in GameObject) is handed EntityType.PLAYER as the routing name/type, GameObjectType.CUBE so the collider becomes a cuboid, and a kinematicPositionBased().lockRotations() body — kinematic because the player is driven by the character controller, not by forces, and rotation-locked so a humanoid never tips over.
  • createSpriteGraphics() must come after createObjectPhysics so there is a body translation to sync the new sprite mesh to.
  • The two collision calls run after the body exists (both early-return if PhysicsBody is missing): the player's collider 0 is tagged group PLAYER_BOUNDING_BOX and masked to collide with PLATFORM | ENEMY only.
// Enable collision events on the bounding box so it feeds the contact system
if (this.PhysicsBody) {
  this.PhysicsBody.collider(0).setActiveEvents(RAPIER.ActiveEvents.COLLISION_EVENTS);
}

if (this.experience.Debug.IsActive) {
  this.Debug = this.experience.Debug;
  this.Debug.RegisterModule(new PlayerDebug(this));
}

GameObject.enablePhysicsEvents() only auto-arms colliders that define OnCollisionEnter/Exit callbacks. The player deliberately defines none — it routes through the declarative contact table instead — so the constructor must arm collision events on collider 0 by hand, otherwise Rapier would emit nothing and Enemy↔Player death would never fire. The debug block is the engine's inversion seam: the game hands its PlayerDebug module to the engine's debug coordinator; the engine never imports it.

Hard-won victory (collider/death rework). The header comment on the collision block records that the old 62.5% hit-box sensor and its per-frame poll were deleted. Death now fires from the full bounding box via the contact table's "any touch registers" rule (Enemy↔Player ⇒ game over). One collider, one source of truth — re-add a smaller sensor only if a stomp mechanic ever needs it.

private initializePlayerAttributes(): void

Caches the engine singletons, seeds the per-entity input snapshot, sets the starting state/direction, applies the DK feel profile, builds the FSM, and zeroes the physics/contact fields. This is where the FSM state map and the IsTouching shape are defined. 

this.Time = this.experience.Time;
this.inputDevice = this.experience.Input;
this.Resources = this.experience.Resources;

// Per-entity input snapshot, refreshed from the device each frame (D10:
// input-as-data — a network/replay/AI source could fill this instead).
this.Input = { isLeft: false, isRight: false, /* ...nine flags... */ isNeitherUpDown: true };

this.State = PlayerStates.IDLE;
this.Direction = PlayerDirection.NEUTRAL;

setDkAttributes(this);

this.fsm = new StateMachine<Player, PlayerState>(this, {
  [PlayerStates.IDLE]: handlePlayerIdle,
  [PlayerStates.RUNNING]: handlePlayerRunning,
  [PlayerStates.FALLING]: handlePlayerFalling,
  [PlayerStates.JUMPING]: handlePlayerJumping,
  [PlayerStates.CLIMBING]: handlePlayerClimbing,
});
D10 — input as data. The player does not read the live keyboard inside its handlers. It owns an InputState struct and the device fills it each frame via CaptureInto. Decoupling intent (the nine booleans) from the device is the single seam that lets a network packet, a replay file, or an AI controller drive the exact same handlers — see Engine Input. The seam is now explicit in the type system too: inputDevice is typed InputSource — the one-method interface { CaptureInto(state: InputState): void } — not the concrete keyboard Input class. The player depends only on the interface, so a touch / gamepad / network / replay source drops in with zero Player changes.

Note setDkAttributes(this) mutates the player in place, writing every feel field declared above. The FSM is constructed with this as the host and a frozen map from state string → handler; the StateMachine never owns the state, the player does (the host's State field is the source of truth). The tail seeds physics constants and the contact map: 

this.ColliderOffset = 0.01;
// Increased threshold significantly for better slope detection and prevent micro-falling
this.ColliderOffsetThreshold = this.ColliderOffset + 0.015;
this.CurrentPosition = new RAPIER.Vector2(0, 0);
this.NextTranslation = new RAPIER.Vector2(0, 0);
this.CurrentFloor = 0;

this.IsTouching = {
  ground: false, ceiling: false, leftSide: false, rightSide: false,
  edgePlatform: false, ladderCore: false, ladderTop: false, ladderBottom: false,
};

ColliderOffset (0.01) is the KCC skin gap; ColliderOffsetThreshold (0.025) is the shapecast time-of-impact cutoff that counts as "touching" — bumped above the raw offset specifically to keep the player glued to the ground on downslopes and stop one-pixel micro-falls.

private setSpriteAnimator(): void

Creates the sprite-sheet animator from the loaded randy texture, declares the sheet layout, sets the opening animation, and registers the resulting material on the base class. 

this.SpriteAnimator = new SpriteAnimator(this.Resources.Items.randy as THREE.Texture, 4, 6);
this.SpriteAnimator.State = SpriteAnimations.IDLE_RIGHT;
this.setMaterial(this.SpriteAnimator.Material, 4);

The 4, 6 are the sheet's tile grid — 4 columns × 6 rows = 24 tiles — which the animator uses to compute UV offsets (see Engine Rendering → SpriteAnimator). The initial State is set directly (the very first frame), and setMaterial(..., 4) stores the animator's material on GameObject with a sprite scale of 4 for createSpriteGraphics to apply.

private createSpriteGraphics(): void

Builds the actual THREE.Sprite mesh from the animator's material, scales it, adds it to the scene, and snaps it onto the body's transform.

this.mesh = new THREE.Sprite(this.material as THREE.SpriteMaterial);
if (this.spriteScale) {
  this.mesh.scale.set(this.spriteScale, this.spriteScale, this.spriteScale);
}
this.scene.add(this.mesh);
this.syncGraphicsToPhysics();

A THREE.Sprite always faces the camera — exactly right for a 2D billboarded character in a 3D WebGL scene. The trailing syncGraphicsToPhysics() (from GameObject) copies the body's translation onto the fresh mesh so it appears at the spawn point on frame zero, not at the origin.

private setCharacterController(): void

Creates the player's own Rapier KinematicCharacterController and tunes its slope/step behaviour. The KCC is what turns a desired velocity into a collision-aware move. 

this.CharacterController = this.Physics.World.createCharacterController(this.ColliderOffset);
this.CharacterController.enableSnapToGround(0.02);
this.CharacterController.enableAutostep(0.5, 0.2, true);
this.CharacterController.setMaxSlopeClimbAngle((45 * Math.PI) / 180);
this.CharacterController.setMinSlopeSlideAngle((30 * Math.PI) / 180);
CallEffect
createCharacterController(0.01)KCC with a 0.01 skin gap (= ColliderOffset).
enableSnapToGround(0.02)Snap down to ground within 0.02 units — kills micro-falling on slopes.

This controller is created here (not in Physics) precisely because it is the player's alone, and it must be removed by hand in Destroy() — Rapier does not free it when the body goes.

public Update(): void

The per-frame entry point the GameDirector calls. Five steps in a fixed order; reordering them breaks things.

if (this.IsBeingDestroyed) {
  return;
}

// Refresh this entity's input snapshot before the state handlers read it.
this.inputDevice.CaptureInto(this.Input);

this.syncGraphicsToPhysics();
this.updatePlayerState();
this.updateTranslation();
this.detectCollisions();
flowchart TD
  A["Player.Update()"] --> B{"IsBeingDestroyed?"}
  B -- yes --> Z["return"]
  B -- no --> C["inputDevice.CaptureInto(Input)"]
  C --> D["syncGraphicsToPhysics() (mesh = last body pose)"]
  D --> E["updatePlayerState()"]
  E --> E1["fsm.Update() -> handlePlayer*(player)"]
  E1 --> E2["SpriteAnimator.Update(Time.Delta)"]
  E2 --> F["updateTranslation() (move KCC + diff contact edges)"]
  F --> G["detectCollisions() (reset + shapecast IsTouching)"]
  • Destroy guard first — a removed player must not run a frame against a freed body.
  • Capture input next — the snapshot must be fresh before the handlers read this.Input.
  • Sync before state — the mesh shows last frame's resolved pose while the new one is computed.
  • State → translation → detect — handlers write NextTranslation, the KCC consumes it, then shapecasts refresh IsTouching for next frame's handlers.
Gotcha (one-frame contact lag). detectCollisions() runs after the handlers, so IsTouching the handlers read is always last frame's sensing. That is fine — and intended — but it's why "just landed" logic checks the previous frame's ground flag.

private updatePlayerState(): void

Two lines: tick the FSM, then advance the sprite animation by the frame's delta.

this.fsm.Update();
this.SpriteAnimator.Update(this.Time.Delta);

fsm.Update() calls exactly the handler registered for the current State string (handlers[host.State]?.(host)) — one dispatch, no re-entry. If a handler reassigns player.State, the new handler runs next frame, not this one. The sprite advance is separate and unconditional: handlers pick which animation via SpriteAnimator.ChangeState(...), and this line plays whatever is current.

private updateTranslation(): void

The heart of locomotion: read the body's position, scale the handler's desired velocity by delta, ask the KCC what move is actually possible against the world, apply it, then feed the KCC's own blocked-by contacts into the contact table as proper enter/exit edges. This is the second half of the engine's two-sided contact dispatch. 

const position = this.PhysicsBody!.translation();
this.CurrentPosition.x = position.x;
this.CurrentPosition.y = position.y;

// Desired translation scaled by the (fixed) timestep — reuse a scratch object.
this.scratchDesired.x = this.NextTranslation.x * this.Time.Delta;
this.scratchDesired.y = this.NextTranslation.y * this.Time.Delta;

// Predicate is hoisted (this.moveFilter) — don't collide with sensors or active
// OneWayPlatforms; we DO collide with enemies (they stop the player).
this.CharacterController.computeColliderMovement(
  this.PhysicsBody!.collider(0),
  this.scratchDesired,
  undefined,
  undefined,
  this.moveFilter
);

// Apply the actual translation to the next kinematic translation (scratch reuse).
const correctiveMovement = this.CharacterController.computedMovement();
this.scratchNext.x = this.CurrentTranslation.x + correctiveMovement.x;
this.scratchNext.y = this.CurrentTranslation.y + correctiveMovement.y;
this.PhysicsBody!.setNextKinematicTranslation(this.scratchNext);

computeColliderMovement takes the desired move and clips it against obstacles, returning the corrective (actually-possible) move via computedMovement(). The two undefined slots are filter flags/groups the author found unreliable; the work is done by moveFilter — now a hoisted bound field (created once, not a fresh closure per frame) that lets the player pass through sensors and active one-way platforms but be stopped by everything solid, including enemies (deliberately: a roaming enemy halts the player). The move is staged in reused scratchDesired/scratchNext vectors and committed with setNextKinematicTranslation on top of CurrentTranslation — no per-frame allocation (see the allocation-free callout below). 

if (!this.Physics.IsPaused) {
  this.currentContacts.clear();
  const numCollisions = this.CharacterController.numComputedCollisions();
  for (let i = 0; i < numCollisions; i++) {
    const collision = this.CharacterController.computedCollision(i);
    if (!collision?.collider) {
      continue;
    }
    const other = this.Physics.GetGameObjectFromCollider(collision.collider);
    if (other && !other.IsBeingDestroyed) {
      this.currentContacts.add(other);
    }
  }

  // New contacts this frame → "enter".
  for (const other of this.currentContacts) {
    if (!this.previousContacts.has(other)) {
      this.Physics.Contacts.Dispatch("enter", this, other);
    }
  }
  // Contacts gone since last frame → "exit".
  for (const other of this.previousContacts) {
    if (!this.currentContacts.has(other) && !other.IsBeingDestroyed) {
      this.Physics.Contacts.Dispatch("exit", this, other);
    }
  }

  // Swap the two sets (no allocation); next frame clears `current` before refill.
  const swap = this.previousContacts;
  this.previousContacts = this.currentContacts;
  this.currentContacts = swap;
}
Hard-won victory — the KCC two-sided dispatch, now as contact EDGES. A kinematic character keeps a skin gap from solids it moves into, so Rapier fires no collision event for the player-as-mover (confirmed by runtime logging: ccBlockedBy=[Enemy] yet zero Rapier events for the player). Without this, running into an enemy would never trigger game-over — only the enemy bumping into a standing player would. So we ask the controller what it was blocked by and dispatch those contacts ourselves, making the same declarative rules (Enemy↔Player ⇒ gameOver) fire in BOTH directions. But numComputedCollisions() is a per-frame currently-touching set — dispatching "enter" from it every frame would re-fire game-over on every resting frame. So we diff it against last frame's set using two reused Set<GameObject>s: new contacts dispatch "enter", vanished ones dispatch "exit" (skipping IsBeingDestroyed objects), then the two sets swap. Now an "enter" rule fires once on contact and there is a real "exit" edge. (The contact table only has enter/sensorEnter rules today, so "exit" is a no-op there for now — but the edge is real, and a future rule can use it.) See Engine Physics / Contacts for the table.
Gotcha (pause guard). The whole diff is wrapped in if (!Physics.IsPaused) so the game-over freeze screen does zero contact bookkeeping. With the edge diff this is belt-and-braces — a resting contact no longer re-fires "enter" on its own — but skipping it while paused keeps the frozen frame from churning the sets at all.
Hard-won victory — an allocation-free hot path (the GC-stutter fix). Everything the player does every frame used to allocate: a fresh desired-move object and next-translation object here, two inline predicate closures (one in computeColliderMovement, one per castShape ×4), plus a pair of cast-origin/direction objects per cast. That garbage piled up until the GC paused mid-jump — visible stutter. Now the two predicates are hoisted to bound arrow fields (moveFilter, shapeCastFilter) created once, and four scratch vectors (scratchDesired, scratchNext, scratchCastOrigin, scratchCastDir) are reused across the KCC move and all four casts. Rapier reads the vectors synchronously, so reuse is safe — and the per-frame hot path now allocates nothing.

private detectCollisions(): void

Clears last frame's contact flags, then re-senses ground/walls/ceiling by shapecasting. Two lines:

this.resetCollisions();
this.getShapeCastCollisions();

Note the division of labour: the solid contact flags in IsTouching (ground/walls/ceiling/ edge) are sensed here by shapecast, while the KCC's numComputedCollisions() (for game-rule dispatch) is handled separately in updateTranslation. Two mechanisms, two jobs.

private resetCollisions(): void

Resets all eight IsTouching flags to false each frame — but in place.

// Use assign instead of replacing with a JS Object, fixes issue with dat.gui
Object.assign(this.IsTouching, {
  ground: false, ceiling: false, leftSide: false, rightSide: false,
  edgePlatform: false, ladderCore: false, ladderTop: false, ladderBottom: false,
});
Gotcha (dat.gui live binding). It uses Object.assign rather than this.IsTouching = {...} on purpose. The PlayerDebug panel binds dat.gui controllers to the original object reference; replacing it with a new object would leave the GUI pointed at a stale struct that never updates. Mutating in place keeps the live debug bindings valid.

public UpdateLadderState(top: boolean, core: boolean, bottom: boolean): void

Pushes the three ladder-zone flags into IsTouching from outside. Parameters map 1:1 onto ladderTop/ladderCore/ladderBottom. No return value. 

this.IsTouching.ladderTop = top;
this.IsTouching.ladderCore = core;
this.IsTouching.ladderBottom = bottom;

Unlike ground/wall sensing, ladder contact is detected by external ladder sensors (see LadderSensor), and the GameDirector relays their state into the player through this setter. The comment flags it as a stopgap "until ladder sensors use proper callbacks" — for now the director is the courier. The climbing handler reads these three flags to decide enter/exit and to clamp at the rails.

Ordering dependency. Because detectCollisions() → resetCollisions() wipes all eight flags (including the ladder trio) each Update(), the director must call UpdateLadderState(...) after the player's update each frame, or the climb flags will be zero by the time the handlers read them.

private getPlatform(collider): Platform | undefined / private isActiveOneWayPlatform(collider): boolean

Two small adapters that turn an anonymous Rapier collider into typed game data.

private getPlatform(collider: RAPIER.Collider): Platform | undefined {
  const go = this.Physics.GetGameObjectFromCollider(collider);
  if (go instanceof Platform) return go;
  return undefined;
}

private isActiveOneWayPlatform(collider: RAPIER.Collider): boolean {
  return this.getPlatform(collider)?.IsOneWayActive ?? false;
}

getPlatform resolves a collider back to its owning GameObject via the physics registry, and narrows to Platform so callers can read named fields (FloorLevel, IsEdge, IsOneWayActive) instead of poking at anonymous userData. isActiveOneWayPlatform is the typed replacement for the old userData.value3 check — a one-way platform you can currently jump up through. Both feed the shapecast predicate and the KCC filter.

private getShapeCastCollisions(): void

Casts the player's own collider shape a hair in each of the four directions and converts near hits into the ground/wall/ceiling/edge/buffer flags. Direction is encoded as a signed axis pair from PlayerDirection (down = (0,-1), left = (-1,0), etc.). 

const shapeCasts = {
  down:  this.shapeCast(PlayerDirection.NEUTRAL, PlayerDirection.DOWN),
  left:  this.shapeCast(PlayerDirection.LEFT,    PlayerDirection.NEUTRAL),
  right: this.shapeCast(PlayerDirection.RIGHT,   PlayerDirection.NEUTRAL),
  up:    this.shapeCast(PlayerDirection.NEUTRAL, PlayerDirection.UP),
};

Ground — a down hit within the touch threshold that is not a wall and not an active one-way platform:

if (downCast &&
    downCast.time_of_impact <= this.ColliderOffsetThreshold &&
    GameUtils.IsColliderType(downCast.collider, EntityType.WALL) === false &&
    this.isActiveOneWayPlatform(downCast.collider) === false) {
  this.IsTouching.ground = true;

  const platform = this.getPlatform(downCast.collider);
  if (platform) this.CurrentFloor = platform.FloorLevel;
  else this.CurrentFloor = 0;
  if (platform) this.IsTouching.edgePlatform = platform.IsEdge;
  else this.IsTouching.edgePlatform = false;
}

When grounded, the player also reads CurrentFloor and edgePlatform straight off the Platform it is standing on — named fields, no userData spelunking. The edgePlatform flag is what lets the running handler zero vertical velocity right at a ledge so the player walks off cleanly into FALLING instead of catching on the lip.

Buffer-jump range — a down hit that is close-ish (within BufferJumpRange) while descending, but not yet grounded, latches GroundWithinBufferRange so a jump pressed just before landing still fires:

if (!this.IsTouching.ground &&
    downCast &&
    downCast.time_of_impact <= this.BufferJumpRange &&
    this.NextTranslation.y <= 0) {
  this.GroundWithinBufferRange = true;
} else {
  this.GroundWithinBufferRange = false;
}

Walls and ceiling — left/right/up hits within the threshold set the matching flag, but each ignores one-way platforms (you should never be wall-stopped or head-bonked by a pass-through platform):

// left/right/up each:
if (cast &&
    cast.time_of_impact <= this.ColliderOffsetThreshold &&
    GameUtils.IsColliderType(cast.collider, EntityType.ONE_WAY_PLATFORM) === false) {
  this.IsTouching.leftSide /* rightSide / ceiling */ = true;
}

private shapeCast(xDirection: number, yDirection: number): RAPIER.ShapeCastHit | null

Casts collider 0's shape from a tiny offset of the body origin, in the given direction, and returns the first solid hit. The predicate is what makes it see through non-solids to the real ground — and it, plus the cast origin/direction, are now reused fields so the four casts per frame allocate nothing. 

// Reuse scratch vectors (Rapier reads them synchronously) so the 4 casts/frame
// allocate nothing.
this.scratchCastOrigin.x =
  this.CurrentTranslation.x + this.ColliderOffset * xDirection;
this.scratchCastOrigin.y =
  this.CurrentTranslation.y + this.ColliderOffset * yDirection;
this.scratchCastDir.x = xDirection;
this.scratchCastDir.y = yDirection;

// HOISTED predicate (this.shapeCastFilter) penetrates sensors / active one-way
// platforms / enemies to find solid ground.
const hit = this.Physics.World.castShape(
  this.scratchCastOrigin,
  0,
  this.scratchCastDir,
  this.PhysicsBody!.collider(0).shape,
  0.0, 1000, true,
  undefined, undefined,                 // filterFlags, filterGroups
  this.PhysicsBody!.collider(0),        // exclude collider 0
  this.PhysicsBody,                     // exclude the whole player body
  this.shapeCastFilter
);
return hit;
  • The small ColliderOffset * direction nudge to the cast origin fixes a quirk where an unoffset X-cast collides with the shape's own inner wall.
  • The player's own collider and body are excluded explicitly (the two args before the predicate) so a cast never reports the player hitting itself.
  • shapeCastFilter (a hoisted bound field, not a fresh closure per cast) runs for every candidate, returning false to skip — so the cast tunnels through sensors (camera/ladder), through active one-way platforms, and through enemies (an enemy must never be treated as standable ground), and reports only the solid surface beneath.

public TeleportToPosition(targetX: number, targetY: number): void inherited

Defined on GameObject, not on Player — included here because it is how the game relocates Randy (respawn, level start, teleporters). It computes the delta from CurrentTranslation to the target and hard-sets the body translation, bypassing the character controller entirely. 

public TeleportToPosition(targetX: number, targetY: number) {
  const deltaX = targetX - this.CurrentTranslation.x;
  const deltaY = targetY - this.CurrentTranslation.y;
  this.TeleportRelative(deltaX, deltaY);  // setTranslation(current + delta)
}
Note. Because this calls setTranslation (an instantaneous warp) rather than setNextKinematicTranslation (the interpolated kinematic move), it skips collision resolution — correct for a respawn, but it is the caller's job to pick a non-overlapping target.

public Destroy(): void

Tears the player down in the right order. Note what it pointedly does not do anymore: announce itself.

// NOTE: do NOT emit "gameObjectRemoved" here. The director's handler for that
// event calls .Destroy(), so self-emitting was unbounded recursion. Whatever
// removes the player owns the emit (the same way enemies are removed).
this.Physics.World.removeCharacterController(this.CharacterController);
this.SpriteAnimator.Destroy();
super.Destroy();
  • removeCharacterController — frees the KCC created in setCharacterController; Rapier won't reclaim it with the body.
  • SpriteAnimator.Destroy() — disposes the sprite texture/material (GPU memory).
  • super.Destroy() — latches IsBeingDestroyed (now idempotent: a second Destroy() early-returns), removes the mesh, disposes geometry/material, unregisters from the contact system, and removes the rigid body (and its collider) last.
Hard-won fix (no more self-emit recursion). The player used to call Emitter.emit("gameObjectRemoved", this) on its way out. But the GameDirector's handler for that very event calls .Destroy() — so a player emitting its own removal re-entered Destroy() unbounded. One pure-ish helper function, shared by three state handlers — running, jumping, falling (see Player States). It is the single place horizontal velocity is accelerated, decelerated, clamped, and wall-stopped, so the three airborne/grounded states stay consistent without duplicating the math. It imports only Player (the blackboard it mutates), PlayerDirection (the signed axis), and GameUtils.MoveTowardsPoint (the framerate-independent stepper). It has no lifecycle of its own; the handlers call it once per frame.

const applyHorizontalMovement = (player: Player, airborne = false): boolean

Accelerate player.NextTranslation.x toward the directional target speed (or decelerate toward 0 when neutral), then hard-stop on a wall. Parameters: player — the entity to move; airborne — selects the air accel/decel pair instead of the ground pair (the running handler omits it → ground; jumping and falling pass true → air). Returns true if a wall was hit this frame, which the running handler uses to keep its wall animation playing. 

let accelRate = player.GroundAcceleration;
if (airborne) accelRate = player.AirAcceleration;
let decelRate = player.GroundDeceleration;
if (airborne) decelRate = player.AirDeceleration;

let targetSpeed = 0;
let acceleration = 0;

if (player.Direction === PlayerDirection.NEUTRAL) {
  targetSpeed = 0;
  acceleration = decelRate;
} else {
  targetSpeed = player.Direction * player.MaxGroundSpeed;
  acceleration = accelRate;
}

player.NextTranslation.x = GameUtils.MoveTowardsPoint(
  player.NextTranslation.x,
  targetSpeed,
  acceleration * player.Time.Delta
);

const hitWall =
  (player.IsTouching.leftSide  && player.Direction === PlayerDirection.LEFT) ||
  (player.IsTouching.rightSide && player.Direction === PlayerDirection.RIGHT);
if (hitWall) {
  player.NextTranslation.x = 0;
}

return hitWall;
  • Air vs ground only differ on accel/decel — top speed is shared (MaxGroundSpeed is the target for both). Air control therefore tunes how quickly you reach/lose speed mid-jump, not how fast you can ultimately go, so a jump never out-runs a sprint.
  • Neutral decelerates, input accelerates — when Direction is NEUTRAL the target is 0 at the decel rate (coast to a stop); otherwise the target is Direction * MaxGroundSpeed (the sign IS the facing) at the accel rate.
  • Framerate independence — MoveTowardsPoint (= mathHelpers.moveTowards) steps toward the target by acceleration * Time.Delta and snaps exactly to it when within one step, so the curve is identical at 30 or 144 fps.
  • Wall stop is directional — velocity is zeroed only when the touched wall is on the side you're pushing into; a left wall doesn't freeze rightward escape. Returning hitWall lets the caller keep the run cycle animating against a wall instead of snapping to idle.
Tip. Because target speed is read live from player.MaxGroundSpeed every call, sliding that one value in the Feel Lab changes both ground and air top speed instantly — there is no separate air cap to forget.

Modifying this: to give air its own top speed, add a MaxAirSpeed attribute and branch targetSpeed on airborne. To add ice/momentum, swap the NEUTRAL branch's decel for a friction term. Keep it returning hitWall — the running handler depends on that boolean.
