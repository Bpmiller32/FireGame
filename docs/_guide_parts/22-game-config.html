Game Config — the data that defines the game

Seven tiny files form the engine's control panel. None of them holds per-frame logic; together they declare what entities exist (entityFactories.ts), what happens when two of them touch (contactRules.ts), which levels are playable and how each one feels (levelRegistry.ts), which keys drive game flow (inputBindings.ts), what color each entity's debug wireframe is (debugColors.ts), which assets to load at boot (assetManifest.ts), and how a level-node's name maps to a game type (levelNodeTypes.ts). The folder is the seam where the generic engine/ layer (which names zero game concepts) meets the game/ layer: almost every gameplay tuning knob lives here, so you can re-skin the Donkey-Kong rules without ever opening an engine file. The last two are recent additions — the asset manifest and the level-node vocabulary the engine used to hardcode now live here too, so a different game reuses engine/ unchanged and just hands in its own.

src/game/config/entityFactories.ts 219 LOC

This file IS the spawn table: a const Record<string, EntityFactory> keyed by the EntityType string (see Game Types — entityType). It replaces the old GameDirector per-type methods (importWalls, importPlatforms, importLadderSensors, …) with one map, so the level loader can iterate the parsed rows and dispatch by data alone. Nobody constructs it — it is a module-level singleton object, default-exported as ENTITY_FACTORIES and imported by GameDirector (gameDirector.ts:23). It imports THREE (only to build the Vector3 a camera sensor stores), GameObject as the factory return type, Camera (handed to the camera-sensor factory), and the seven concrete game entity classes it constructs (Platform, TrashCan, WinFlag, CameraSensor, LadderSensor, Teleporter) plus the EntityType identity table. Its lifecycle is "imported once, read forever" — the only runtime action is GameDirector.LoadLevelData calling ENTITY_FACTORIES[data.type](data, ctx, data.type) once per level-data row (gameDirector.ts:249–252).

Note — only level-data-placed types live here. The runtime barrel spawner (Enemy, normal & crazy — see Game Entities — Enemy) and the non-entity load steps (ambient light, the optional graphics overlay) are NOT factories; GameDirector keeps them as explicit steps. The keys of ENTITY_FACTORIES therefore are the exact list of types the GLB loader can place into a level.

export interface FactoryContext

The bag of typed query-arrays a factory pushes into, plus the engine camera the camera-sensor factory needs. These are derived views: the same array instances GameDirector holds and queries elsewhere (this.Walls, this.Platforms, the ladder sub-arrays). Each factory appends the entity it built to the matching array as a side effect, so the loop populates the director's collections in place.

The ladders are split into three arrays (top/core/bottom) rather than one because the game queries them separately: GameDirector.Update polls LadderTopSensors and LadderBottomSensors with IsAnySensorTriggered but checks LadderCoreSensors with the fully-inside test (the hard-won climb behavior, D9, that lives in Game Entities — ladderSensor). GameDirector.getFactoryContext (gameDirector.ts:262–275) builds this struct by aliasing its own public arrays.

type EntityFactory = (data: any, ctx: FactoryContext, type: string) => GameObject

The shape every factory satisfies. data is one LevelEntity row from the GLB parse (see Engine Types — levelData); ctx is the FactoryContext to push into; type is the resolved EntityType string for this row — passed so the shared platform and ladder factories can branch on which variant they're building. The return value is the constructed GameObject.

Gotcha — data: any is deliberate, not sloppy. The old import* callbacks received (data: any) verbatim, and the level rows carry a loose meta bag read untyped. Keeping any here is a true no-op versus the original behavior, not new looseness. If you tighten it to LevelEntity, every data.meta?.[...] read still works, but you inherit the optional-chaining ceremony — the comment flags this on purpose.

function metaNum(meta, key, fallback = 0): number  ·  function metaCoords(meta, key): number[]

The metadata coercers. The GLB parser hands each row a generic meta: Record<string,string> parsed from the mesh's texture name — the engine never interprets a key. The game owns that interpretation here, so each factory reads only the keys meaningful to its type. metaNum reads one numeric value (meta.floor "1" -> 1) with a finite-number guard and a fallback; metaCoords splits a "_"-separated list (meta.cam "3.5_2_-1" -> [3.5, 2, -1]), substituting 0 for any garbage component. 

function metaNum(meta, key, fallback = 0): number {
  const n = Number(meta?.[key]);
  if (Number.isFinite(n)) return n;
  return fallback;
}

function metaCoords(meta, key): number[] {
  const parts = (meta?.[key] ?? "").split("_");
  const result: number[] = [];
  for (const c of parts) {
    const n = Number(c);
    if (Number.isFinite(n)) result.push(n);
    else result.push(0);
  }
  return result;
}

These replace the old value0–value3 positional indices. The win is legibility at the call site: metaNum(data.meta, "floor") says what it reads, where value0 said nothing. Both helpers swallow missing/garbage input rather than throwing, so a hand-authored texture name that's slightly wrong degrades to a sensible default instead of crashing the level load.

Note — the axis remap is shared by every factory. The 2D world uses the glTF row's position[0] as horizontal X and position[2] as vertical Y, while position[1] is the unused depth. Rotation is always taken as -rotation[1] (negated Y-euler). Width/height/depth pass straight through — ResourceLoader already emits height = gameUp(glTF Y) and depth = gameDepth(glTF Z), so there is no second remap here.

const createPlatform: EntityFactory = (data, ctx, type) => { … }

Handles BOTH PLATFORM and ONE_WAY_PLATFORM — the only factory registered under two keys. It branches on type === EntityType.ONE_WAY_PLATFORM to set the constructor's isOneWay flag, then reads three metadata keys and pushes into ctx.platforms. 

const isOneWay = type === EntityType.ONE_WAY_PLATFORM;
const platform = new Platform(
  { width: data.width, height: data.height, depth: data.depth },
  { x: data.position[0], y: data.position[2] },
  -data.rotation[1],
  isOneWay
);

platform.SetPlatformFloorLevel(metaNum(data.meta, "floor"));
platform.SetEdgePlatform(metaNum(data.meta, "edge"));

ctx.platforms.push(platform);

Three texture-name keys drive platform behavior: floor sets the floor index enemies/players reason about, edge (> 0) marks an edge where enemies turn around, and oneway sets the Y above which a one-way platform becomes solid. The || data.position[2] fallback is the load-bearing bit: when no oneway override is authored (so metaNum returns 0, which is falsy), the enable point defaults to the platform's own placement Y — exactly what Platform's constructor already assumed. The Platform.Update one-way toggle then reads this each frame against the player's feet (see Game Entities — platform).

src/game/config/contactRules.ts 68 LOC

This file IS the declarative cross-entity contact table — the data half of the engine's ContactRegistry (see Architecture — Contacts and Engine Entities — contactRegistry). It answers "when an A touches a B, what happens?" without a single hard-coded if/else in the physics loop. It default-exports one function, registerContactRules(contacts), called exactly once at startup from App.vue (App.vue:50) with webglExperience.Physics.Contacts, AFTER the engine is configured. It imports the engine ContactRegistry + its ContactMatch type, matchesType from physicsHelpers, the global Emitter, the EntityType table, and the four entity classes it casts to (Player, Teleporter, CameraSensor, TrashCan). Note the deliberate division of labor: an entity's OWN movement reactions (an enemy reversing at a wall) live with the entity; only cross-entity consequences live here.

const byType = (type: string): ContactMatch => (go: GameObject) => matchesType(go, type)

A tiny curried predicate factory. byType(EntityType.ENEMY) returns a ContactMatch that answers "does this GameObject carry the Enemy type flag?" — delegating to matchesType, which reads the opaque userData.type off the Rapier body (physicsHelpers.ts:18). Matching by type string rather than instanceof is essential: Rapier callbacks return a raw Collider, not a class instance, so type-based routing survives where instanceof cannot. 

const byType = (type: string): ContactMatch => (go: GameObject) =>
  matchesType(go, type);

export default function registerContactRules(contacts: ContactRegistry)

Registers all five of this game's rules into the registry via contacts.Add(a, b, phase, run). Each Add pushes a { a, b, phase, run } record; later, Physics calls contacts.Dispatch(phase, x, y) for every contact event, and the registry tests each rule two-sided — (x,y) then (y,x) via an else if — so a rule fires once regardless of which body moved, and the handler always receives args in the rule's own (a, b) order (contactRegistry.ts:37–49). That two-sidedness is why the rules below can be written from one perspective only.

Rule 1 — Enemy touches Player → game over. A solid-body enter rule. Two-sided dispatch is the point: it fires whether the player ran into the barrel or the barrel rolled into the player. The handler emits a pure signal; GameDirector listens for gameOver and pauses physics + stops spawning (gameDirector.ts:112), while App.vue shows "Game Over!". 

contacts.Add(
  byType(EntityType.ENEMY),
  byType(EntityType.PLAYER),
  "enter",
  () => Emitter.emit("gameOver")
);

Rule 2 — Enemy falls into the oil can → enemy destroyed, can lit. The only rule that uses both handler args. It casts b to TrashCan to flip IsOnFire (a public boolean on the can, trashCan.ts:8), then emits gameObjectRemoved carrying the enemy — which GameDirector listens for and calls Destroy() on (gameDirector.ts:136). This is how the opening crazy barrel Kong throws lights the can. 

contacts.Add(
  byType(EntityType.ENEMY),
  byType(EntityType.TRASH_CAN),
  "enter",
  (enemy, can) => {
    (can as TrashCan).IsOnFire = true;
    Emitter.emit("gameObjectRemoved", enemy);
  }
);

Rule 3 — Player enters a Teleporter → teleport to destination. A sensorEnter rule. The handler reads the teleporter's Destination getter (a RAPIER.Vector2, teleporter.ts:45) and calls player.TeleportToPosition(dest.x, dest.y). The teleporter class owns only the destination + sensor; the "who triggers it and what happens" rule is here, by design. 

contacts.Add(
  byType(EntityType.TELEPORTER),
  byType(EntityType.PLAYER),
  "sensorEnter",
  (teleporter, player) => {
    const dest = (teleporter as Teleporter).Destination;
    (player as Player).TeleportToPosition(dest.x, dest.y);
  }
);

Rule 4 — Player enters a CameraSensor → reposition the camera. Calls the sensor's ApplyCameraOnEnter(), which pins the camera's rest anchor to the zone's authored cam=x_y (cameraSensor.ts:50). On a non-follow level the camera eases to that exact spot; on a follow level only the Y is used (the vertical baseline the follow eases around). Note the handler uses only the first arg. 

contacts.Add(
  byType(EntityType.CAMERA_SENSOR),
  byType(EntityType.PLAYER),
  "sensorEnter",
  (sensor) => (sensor as CameraSensor).ApplyCameraOnEnter()
);

Rule 5 — Player reaches the WinFlag → win. Emits gameWin. The comment in GameDirector notes the win flag "no longer updates — it wins via the contact table, not a per-frame shapecast" (gameDirector.ts:174–175): moving this into a data rule deleted a whole per-frame query. 

contacts.Add(
  byType(EntityType.WIN_FLAG),
  byType(EntityType.PLAYER),
  "sensorEnter",
  () => Emitter.emit("gameWin")
);
Gotcha — enter vs sensorEnter must match the entity. Solid bodies (Enemy, Player, TrashCan) fire enter/exit; sensor bodies (Teleporter, CameraSensor, WinFlag — all GameSensor subclasses with setAsSensor(true)) fire sensorEnter/sensorExit. A rule registered under the wrong phase never fires. The sensors above also call enableContactEvents() in their constructors so they participate in dispatch even though they define no callback of their own.

Modifying this: add a cross-entity behavior by adding one contacts.Add(...) call — pick the two byType predicates, the correct phase, and a handler that emits an event (preferred — keeps this file thin) or casts and mutates. Never put per-frame movement reactions here. Because matching is by type string, a brand-new entity only needs its EntityType registered to be routable. If a rule "doesn't fire," check the phase first (solid vs sensor) and that both bodies enabled contact events.

This file IS the single source of truth for which levels are playable and how each one is set up. It exports LEVEL_REGISTRY (the ordered list of level entries), FEEL_PROFILES + its FeelProfile type, and DEFAULT_LEVEL_INDEX. GameDirector imports all of these (gameDirector.ts:24–29) and funnels boot, the F2 cycle, and the dat.gui level select through one load path. It imports the two feel-profile setters (setDkAttributes, setCelesteAttributes) and uses Vite's import.meta.glob to bundle level GLBs. No instances, no lifecycle — pure module-level data resolved once at import.

const levelGlbUrls = import.meta.glob("../../assets/levels/*.glb", { query: "?url", import: "default", eager: true })

Auto-discovery of level art. Level GLBs live in src/assets/levels (NOT public/) so Vite bundles and content-hashes them. The eager glob collects every *.glb in that folder as its bundled URL into a Record<string,string>. The loop below flattens the full glob paths to bare filenames so a registry row can reference a level by name. 

const levelGlbUrls = import.meta.glob("../../assets/levels/*.glb", {
  query: "?url",
  import: "default",
  eager: true,
}) as Record<string, string>;

const LEVEL_SOURCES: Record<string, string> = {};
for (const path of Object.keys(levelGlbUrls)) {
  const url = levelGlbUrls[path];
  const fileName = path.split("/").pop();
  if (fileName) LEVEL_SOURCES[fileName] = url;
}
Warning — eager glob bundles EVERY GLB. The glob is eager, so every .glb in src/assets/levels ships in the build whether or not a LEVEL_REGISTRY row references it. Delete unused level GLBs from that folder to stay lean — the source comment flags this explicitly.

function levelUrl(fileName: string): string

Resolves a bare filename (e.g. "TestLevel-Slopes.glb") to its bundled hashed URL via LEVEL_SOURCES. On a miss it logs a loud console.error listing the available filenames and returns the empty string. 

function levelUrl(fileName: string): string {
  const url = LEVEL_SOURCES[fileName];
  if (!url) {
    console.error(
      `Level GLB "${fileName}" not found in src/assets/levels. Available: ${
        Object.keys(LEVEL_SOURCES).join(", ") || "(none)"
      }`,
    );
    return "";
  }
  return url;
}
Hard-won victory — fail loud, fail early. Returning "" on a miss pairs with a guard in GameDirector.loadLevelByIndex (gameDirector.ts:395–399): an empty source throws before reaching the loader. Without it, gltfLoader.load("") fetches the HTML index page and throws a confusing parse error far from the real cause — a typo'd filename now surfaces as a clear "not found in src/assets/levels" instead.

This file IS the bridge from the engine's raw F-key device state to this game's flow events. The engine Input (see Engine Events & Input) exposes only HELD-state booleans and names zero game events; this class watches IsF1Pressed/IsF2Pressed/IsF3Pressed each frame, detects the press edge (false→true since last frame), and emits the matching game event once per press. It is constructed in App.vue (new InputBindings(webglExperience.Input), App.vue:65) and its Update() is called first each frame inside OnGameUpdate, before gameDirector.Update() (App.vue:71–74). It imports just the global Emitter and the Input type.

constructor(input: Input) {
  this.input = input;

  this.wasF1Pressed = false;
  this.wasF2Pressed = false;
  this.wasF3Pressed = false;
}

public Update()

Per-frame edge detector. For each F-key: if it is held this frame AND was not held last frame, emit the event; then store this frame's held-state into the was* flag. The "store after the test" is what converts a held key into a single edge — without it, holding F2 would emit switchLevel every frame and the level would cycle wildly. 

public Update() {
  // F1 -> gameReset
  if (this.input.IsF1Pressed && !this.wasF1Pressed) {
    Emitter.emit("gameReset");
  }
  this.wasF1Pressed = this.input.IsF1Pressed;

  // F2 -> switchLevel
  if (this.input.IsF2Pressed && !this.wasF2Pressed) {
    Emitter.emit("switchLevel");
  }
  this.wasF2Pressed = this.input.IsF2Pressed;

  // F3 -> manualCameraControl
  if (this.input.IsF3Pressed && !this.wasF3Pressed) {
    Emitter.emit("manualCameraControl");
  }
  this.wasF3Pressed = this.input.IsF3Pressed;
}
