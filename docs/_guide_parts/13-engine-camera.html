Camera — engine-pure follow camera

A self-contained Three.js camera that knows how to follow a moving point with Mario-Wonder-style easing — look-ahead on X, intent-driven pan on Y — yet names zero game concepts. It never imports Player; it reads a plain CameraTarget snapshot of scalars handed to it each frame. That seam (roadmap C5) is the whole point of this section: the same camera serves a Donkey-Kong follow level, a fixed-zone level, or a future multiplayer follow without a single edit.

What it is: the engine's one camera class. It owns a PerspectiveCamera (the shipping camera) and an optional debug-only OrthographicCamera, a smoothed follow state, and a live-editable bag of tuning knobs. Who constructs it: the Experience singleton (Engine Core) news it up exactly once in Configure() — this.Camera = new Camera(new THREE.Vector3(0, 25, 100)) — and stores it as experience.Camera. Who drives it: the GameDirector (Game Layer) calls camera.Update(target) every frame; the Renderer (Engine Layer) reads camera.Instance to draw. Why these imports: THREE for the camera objects; Experience/Sizes/Time for the shared scene, viewport, and frame delta; Input/Debug only when the debug overlay is active; Emitter for the "manualCameraControl" toggle; and crucially CameraFollowState/CameraTarget from ../types/cameraTarget — the only shape it knows about its follow target. Lifecycle: constructor → (per frame) Update(target?) → Resize() on viewport change → Destroy().

Hard-won victory — C5 / D3. This file imports nothing from game/. Before the decouple, Update() reached into a concrete Player for CurrentPosition, NextTranslation, State and IsTouching.ground. Now it consumes the CameraTarget snapshot instead. What it prevents: a cyclic engine→game dependency, and the engine layer hard-coding Donkey-Kong's notion of "player." The camera's easing math is byte-for-byte the same; only its input shape changed.

interface CameraTuning & const MOVE_EPSILON

Module-level declarations (lines 16–29). CameraTuning is the live-editable feel bag — every field is surfaced in the Camera Debug GUI folder, so you can dial the pan while the game runs. The header comment encodes the design intent: X eases a look-ahead toward travel direction; Y follows only sustained / intentional vertical motion (climb, real fall, landing) and deliberately ignores jump arcs. The *Rate fields are responsiveness constants for the exponential smoother (see smooth()), and all distances are world units. 

interface CameraTuning {
  lookaheadX: number;        // horizontal offset placed ahead of the player
  lookaheadRate: number;     // how fast the look-ahead eases toward its goal
  horizontalRate: number;    // X follow responsiveness
  verticalRateGround: number;// grounded vertical follow (gentle)
  verticalRateUp: number;    // climbing — track up briskly
  verticalRateDown: number;  // falling — slower / capped descent
  climbLeadY: number;        // look-ahead ABOVE the player while climbing
  fallBufferY: number;       // player must drop this far below the camera before it follows
  fallLeadY: number;         // lead relative to the player while falling (negative = below)
}

// Below this |velocity| the player counts as "not moving" → X look-ahead eases to 0.
const MOVE_EPSILON = 0.01;

MOVE_EPSILON is the dead-zone that decides direction: in Update() a velocityX above +MOVE_EPSILON aims the look-ahead right, below -MOVE_EPSILON aims it left, and anything in between eases the offset back to zero so a standing player re-centers.

public SetBaselinePosition(x: number, y: number): void

Sets both baselines and raises hasSensorBaseline. Caller: CameraSensor.ApplyCameraOnEnter, which fires from the declarative contact table when the player enters the zone.

The behavioral split lives in Update(), and the comment on this method spells it out: on a non-follow level the camera eases to this anchor on both axes (authored x,y spots); on a follow level only Y matters — it pins the grounded vertical baseline while follow keeps driving X off the player. Crucially, an intentional vertical move (climb or real fall) releases the pin by setting hasSensorBaseline = false again inside Update().

Gotcha — argument order at the CameraStart call site. The GLB stores positions Blender-style (Z-up). setCameraStart reads x = position[0], y = position[1], z = position[2] and then calls TeleportToPosition(x, z, y) — swapping Y and Z to convert into Three.js's Y-up world. resetGameState, by contrast, passes InitialPosition.x/.y/.z straight through. private smooth(current: number, target: number, rate: number): number

The one easing primitive every follow channel runs through. Parameters: current (where we are), target (where we want to be), rate (responsiveness from Tuning). Returns: the next value, one frame closer. 

private smooth(current: number, target: number, rate: number): number {
  const alpha = 1 - Math.exp(-rate * this.Time.FrameDelta);
  return current + (target - current) * alpha;
}
Hard-won victory — frame-rate-independent lerp. A naive current + (target-current)*k with a constant k eases faster at high frame rates and slower on a laggy frame — the camera feel drifts with FPS. The 1 - e^(-rate·dt) form folds Time.FrameDelta in, so a given rate produces the same wall-clock approach speed whether the frame is 16 ms or 33 ms. It reads FrameDelta — the real per-frame wall-clock time — and deliberately not Time.Delta (the fixed 1/60 sim step), because the camera now runs in the per-FRAME render pass, easing at the display's actual refresh rate while following the interpolated player. FrameDelta is clamped to 1/30 upstream, so a huge tab-restore hitch can't make alpha overshoot toward 1 and teleport the camera.

public Update(target?: CameraTarget): void

The heart of the file — runs once per FRAME from GameDirector.RenderUpdate (the per-frame render pass, not the fixed-step sim Update where movement integrates). It builds the target via cameraTargetFromPlayer after the player's graphics have been interpolated for this frame, so the follow reads the smooth interpolated position (player.RenderTranslation) rather than the stepped sim position. Parameter: the optional CameraTarget snapshot. Returns: nothing — it mutates currentPosition and writes it into Instance.position.

Hard-won victory — render interpolation. The sim now advances in fixed 1/60 steps (0..N per frame), but the camera no longer rides that stepped position. GameDirector's per-frame RenderUpdate(alpha) first lerps each entity between its last two sim states into RenderTranslation, then calls Update with a target built off that interpolated player — so the follow is smooth above the sim rate (buttery on a 144 Hz display, no judder on a 60 Hz one) instead of snapping once per sim step.

      verticalRate = this.Tuning.verticalRateDown;

    verticalRate = this.Tuning.verticalRateGround;
    break;

Line-level notes worth re-reading: FALLING uses a buffer — the camera only starts dropping once target.y falls below currentPosition.y - fallBufferY, so a short hop-down or the apex of a fall doesn't yank the view; below the buffer it leads into the fall by the negative fallLeadY so the landing zone is visible. The engine-side vocabulary of vertical-follow modes — deliberately named in camera terms (GROUNDED/CLIMBING/FALLING/JUMPING), not player terms. The game maps its own PlayerStates onto these.

  CLIMBING: "climbing",
  FALLING: "falling",

This is the same same-named const + type idiom used by playerStates.ts and gameObjectType.ts: CameraFollowState is usable as both a value (CameraFollowState.CLIMBING in the switch) and a type annotation (followState: CameraFollowState). interface CameraTarget

The per-frame follow snapshot. A pure data shape (camelCase fields, D15) carrying exactly the scalars the camera's easing reads — no more, no less — so the branching in Camera.Update survives the decouple unchanged. 

export interface CameraTarget {
  x: number;          // world X to follow (was player.CurrentPosition.x)
  y: number;          // world Y to follow (was player.CurrentPosition.y)
  velocityX: number;  // horizontal velocity -> look-ahead direction (was NextTranslation.x)
  followState: CameraFollowState; // vertical-follow mode (mapped from the game's state)
  isGrounded: boolean;// truly on the ground -> grounded re-baseline (was IsTouching.ground)
}

Each comment is a migration breadcrumb: x/y were player.CurrentPosition, velocityX was NextTranslation.x, isGrounded was IsTouching.ground, and followState is the one field that isn't a raw copy — it's derived on the game side.

The mapping seam — GameDirector.cameraTargetFromPlayer

The companion piece lives in the game layer. Each frame — in the per-frame render pass, right after the player's graphics have been interpolated — the GameDirector translates PlayerStates → CameraFollowState and packs the scalars into a fresh CameraTarget: 

private cameraTargetFromPlayer(player: Player): CameraTarget {
  let followState: CameraFollowState;
  switch (player.State) {
    case PlayerStates.CLIMBING: followState = CameraFollowState.CLIMBING; break;
    case PlayerStates.FALLING:  followState = CameraFollowState.FALLING;  break;
    case PlayerStates.JUMPING:  followState = CameraFollowState.JUMPING;  break;
    default:                    followState = CameraFollowState.GROUNDED;
  }
  return {
    // Follow the INTERPOLATED render position (set by InterpolateGraphics just
    // before this in RenderUpdate) so the camera tracks the smooth visual player.
    x: player.RenderTranslation.x,
    y: player.RenderTranslation.y,
    velocityX: player.NextTranslation.x,
    followState,
    isGrounded: player.IsTouching.ground,
  };
}

Note the default catch-all: IDLE and RUNNING both fall through to GROUNDED, because the camera draws no distinction between standing still and walking — both rest at the baseline. This switch is the only place the two vocabularies meet; a different game (or a multiplayer follow that averages two players' positions) would write its own version of this one method and the engine camera would never know.

Tip. Want to debug what the camera is "seeing"? The values flowing through this snapshot are exactly the player position/state/velocity already surfaced in the Player Debug readouts, and the resulting camera transform is mirrored live in the Camera Debug position fields.
