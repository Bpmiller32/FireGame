Feel — tunable movement profiles

"Feel" is the complete set of scalars that decide how heavy, snappy, floaty, or forgiving the player character is — run speed, ground/air accel, the jump launch impulse, rise/fall gravity, terminal velocity, coyote time, jump buffering, climb pacing, and animation cadence. Three tiny pure functions own it: baseFeel seeds the values shared by every character, then setDkAttributes / setCelesteAttributes stamp two distinct personalities on top. Feel is the single source of truth and is applied only at level load, from the registry, via GameDirector.applyFeelProfile — see GameDirector and levelRegistry.

Note — where these scalars live and how they're consumed. None of these three files contains any logic; each one just writes public fields on a Player instance. The fields are declared in the "Player attributes" block of player.ts (e.g. public JumpPower!: number;), and they are read every frame by the player state handlers: handlePlayerRunning, handlePlayerJumping, handlePlayerFalling, handlePlayerClimbing, and the shared applyHorizontalMovement. So to understand what a number does, you read it in the setter and feel it in the handler.
Tip — the unit convention. player.NextTranslation is a velocity (world-units per second); updateTranslation() in player.ts multiplies it by Time.Delta to get the per-frame move. So a "speed" tunable (MaxGroundSpeed, MaxFallSpeed, MaxClimbSpeed, JumpPower) is a target velocity, and an "accel/decel/gravity" tunable (GroundAcceleration, FallAcceleration, RiseGravity, …) is fed in as rate * Time.Delta — i.e. units / second² — the maximum step a velocity is allowed to change this frame. Every ramp goes through GameUtils.MoveTowardsPoint(current, target, maxStep) (= MathHelpers.moveTowards): nudge current toward target by at most maxStep, snap exactly on arrival.

src/game/attributes/baseFeel.ts 39 LOC

The default movement attribute set — the values shared by every feel profile, so the common DK/Celeste numbers live in exactly one place and each profile only writes its differences on top. It exports a single pure function, applyBaseFeel, which takes a Player and assigns the shared climbing scalars, the jump/coyote/buffer runtime flags & counters, the shared timers, the coyote-time window, and the sprite-animation pacing divisor. It imports only the Player type (for the parameter signature) — it owns no behaviour and is never constructed; it is called, always first, by the two profile setters.

Note — what base feel deliberately does NOT set. Read it carefully: base feel never touches MaxGroundSpeed, the ground/air accel-decel pairs, or any jump-arc scalar (JumpPower, RiseGravity, FallAcceleration, MaxFallSpeed, ApexHang*, MinJumpTime, BufferJumpRange). Those are profile-only — the things that actually make DK feel like DK and Celeste feel like Celeste. Base feel owns the parts that are genuinely identical between the two: ladder climbing, the coyote/buffer bookkeeping flags, and animation pacing. Consequence: calling applyBaseFeel alone would leave a player unable to walk or jump (those fields would be undefined) — it is an ingredient, never a complete recipe.

const applyBaseFeel = (player: Player) => void

Purpose. Seed the shared half of a feel profile onto player. Parameter. player — the live Player whose public attribute fields are mutated in place. Returns. nothing; it is a side-effecting setter. The body is three thematic blocks.

Climbing (shared)
player.MaxClimbSpeed = 4;       // top vertical speed on a ladder
player.ClimbAcceleration = 30;  // how fast you reach MaxClimbSpeed
player.ClimbDeceleration = 8;   // how fast you stop when you let go

Read these in handlePlayerClimbing: when up/down is held the target is player.Direction * MaxClimbSpeed ramped at ClimbAcceleration * Time.Delta; when neutral the target is 0 ramped at ClimbDeceleration * Time.Delta. Ladders feel the same in both games, so these are shared. MaxClimbSpeed also doubles as the nudge velocity used to pop the player off the top/bottom of a ladder back into IDLE.

Jump / coyote / buffer runtime state (shared)
// Early-jump-ended latch (variable height): reset every jump entry & on ground.
player.EndedJumpEarly = false;

// Coyote-jump runtime flags/counters
player.CoyoteAvailable = false;
player.CoyoteCount = 0;

// Buffer-jump runtime flags/counters (the RANGE differs per profile)
player.BufferJumpAvailable = false;
player.GroundWithinBufferRange = false;
player.WasBufferJumpUsed = false;
player.BufferJumpCount = 0;

// Jump and fall timers (shared)
player.TimeJumpWasEntered = 0;
player.TimeFallWasEntered = 0;

player.CoyoteTime = 0.033; // 2 frames

These are runtime values, not personality knobs — seeding them here guarantees every flag holds a real boolean and every counter a real number before the first frame runs, so the handlers never branch on undefined. Their meanings, traced through the handlers:

  • EndedJumpEarly — the variable-height latch. Set true in handlePlayerJumping when jump is released after MinJumpTime; reset to false on every fresh launch (idle/running/coyote) and on the ground (idle/running). When true, rise gravity is multiplied by JumpEndedEarlyGravityModifier so a tapped jump peaks lower.
  • CoyoteAvailable / CoyoteCount — grace-jump state. Set CoyoteAvailable = true while grounded (idle/running); the FALLING handler lets you still jump if Time.Elapsed < TimeFallWasEntered + CoyoteTime.
  • Buffer quartet — GroundWithinBufferRange is set by the player's downward shapecast (true when ground is within BufferJumpRange and you're descending); BufferJumpAvailable arms when you're in range with jump released; WasBufferJumpUsed + BufferJumpCount are the debounce/telemetry so one press counts once.
  • TimeJumpWasEntered / TimeFallWasEntered — timestamps (Time.Elapsed seconds) stamped on entering JUMPING/FALLING; the coyote and MinJumpTime windows are measured against them.
  • CoyoteTime = 0.033 — the actual grace window, ~2 frames at 60 fps. This one is a tunable, and it's shared: both games give the same 2-frame forgiveness.
Animation pacing (shared)
player.AnimationScalingFactor = 1.6;

The divisor that ties run-cycle speed to horizontal velocity. In handlePlayerRunning it feeds SpriteAnimator.ChangeAnimationTiming(1 / (|NextTranslation.x| / AnimationScalingFactor)) — i.e. AnimationScalingFactor / |vx|. Larger |vx| → smaller per-frame hold → faster legs; the factor sets the overall relationship so the run cycle reads right at each game's top speed (the wall-hit branch reuses MaxGroundSpeed in the same formula to keep the cycle moving while pinned against a wall).

Modifying this: put a value here only if DK and Celeste genuinely agree on it (climbing, coyote window, animation pacing) — anything that should differ between feels belongs in a profile setter instead. If you add a new shared runtime flag, seed it here too so the handlers always see a defined value on frame one. Remember base feel runs first inside each profile, so a profile can always override anything you set here.

src/game/attributes/setCelesteAttributes.ts 47 LOC

The "Celeste" personality: fast, twitchy, high-flying, very floaty at the apex — a precision platformer feel. It exports setCelesteAttributes(player), which calls applyBaseFeel(player) first (the shared seed) and then overrides the horizontal-movement and jump-arc scalars with big, snappy numbers. It imports the Player type and applyBaseFeel. It is referenced by name only through the FEEL_PROFILES table in levelRegistry (key "celeste"); a level opts into it via feelProfile: "celeste".

const setCelesteAttributes = (player: Player) => void

Purpose. Apply the Celeste feel. Parameter. player — mutated in place. Returns. nothing. First line is always the shared base; everything below is a Celeste override. 

applyBaseFeel(player);

// Horizontal
player.MaxGroundSpeed = 25;
player.GroundAcceleration = 120;
player.GroundDeceleration = 30;
player.AirAcceleration = 120;   // air == ground here (no air difference)
player.AirDeceleration = 30;

Top horizontal speed is 25 (more than double DK's), and acceleration is a steep 120 — you snap to full speed almost instantly. Air control is set equal to ground control, so mid-air steering is just as responsive as on the floor; applyHorizontalMovement picks the air pair when its airborne flag is true (jumping/falling pass true). The comment notes these are deliberately equal as a starting point — diverge them in the Feel Lab for floatier or draggier air. 

// Jump arc
player.MaxFallSpeed = 80;       // terminal fall velocity
player.FallAcceleration = 440;  // fall half of the arc
player.RiseGravity = 360;       // rise half (lighter than fall = floaty ascent)
player.JumpEndedEarlyGravityModifier = 3;
player.ApexHangThreshold = 20;  // |vy| window for apex float
player.ApexHangMult = 0.5;
player.JumpPower = 128;         // upward launch velocity
player.BufferJumpRange = 4;
player.MinJumpTime = 0.06;

Tracing each through the handlers:

  • JumpPower = 128 — the launch impulse written straight into NextTranslation.y on jump entry (idle/running/coyote). A huge initial velocity.
  • RiseGravity = 360 — subtracted from vy each frame during JUMPING (NextTranslation.y -= riseGravity * Time.Delta). The apex emerges where this brings vy to zero — there is no fixed apex timer.
  • FallAcceleration = 440 — fall ramps toward -MaxFallSpeed at this rate in FALLING. It is much larger than RiseGravity, which is the asymmetry that produces the signature snappy Celeste arc: you rise lazily and drop hard.
  • MaxFallSpeed = 80 — terminal velocity; also used in idle/running as the constant downward velocity that keeps the player glued to slopes.
  • ApexHangThreshold = 20 / ApexHangMult = 0.5 — near the top, while |vy| < 20, rise gravity is halved, giving a generous hang. The window (20) is wide, so the float is pronounced.
  • JumpEndedEarlyGravityModifier = 3 — release jump early (after MinJumpTime) and rise gravity triples, cutting the jump short. Mutually exclusive with the apex hang in the handler's if / else if.
  • BufferJumpRange = 4 — a generous landing window; pressing jump up to 4 units above the ground still fires on touchdown.
  • MinJumpTime = 0.06 — the variable-height floor; a jump always lasts at least 60 ms even on the fastest tap.
Hard-won victory — the "2× time-scaled" arc. The header comment records that these numbers were not hand-guessed: they were tuned in slow-motion (the Feel Lab), then time-scaled ~2× to feel snappy on the real clock. The rule preserves apex height while halving airtime: velocities ×2, accelerations ×4, jump times ×0.5. The original labbed values were JumpPower 64 / RiseGravity 90 / Fall 110 / MaxFall 40 / ApexHang 10×0.5 / MinJumpTime 0.12. This works because apex height h = v² / (2g) is invariant under v → 2v, g → 4g (4v² / 8g = v² / 2g), while time-to-apex t = v / g → 2v / 4g halves. Same arc shape, twice as responsive.

Modifying this: change values below the applyBaseFeel(player) line only — never remove that first call or you lose climbing/coyote/animation defaults. Keep FallAcceleration > RiseGravity if you want to preserve the rise-floaty / fall-snappy character. To add a brand-new feel knob, declare the field on Player, set it in baseFeel (if shared) or in both profile setters, and read it in a handler.

src/game/attributes/setDkAttributes.ts 49 LOC

The "Donkey Kong" personality and the default the engine actually ships with: slow, heavy, deliberate, with a short controlled hop — the original arcade feel. It exports setDkAttributes(player), structurally identical to the Celeste setter (applyBaseFeel first, then overrides) but with much smaller numbers. It is referenced through FEEL_PROFILES.dk in levelRegistry (both shipped levels use feelProfile: "dk"), and it is also imported directly by player.ts, where the constructor calls setDkAttributes(this) in initializePlayerAttributes() as a construction-time default.

Gotcha — DK feel is applied twice, and that's intentional. The Player constructor runs setDkAttributes(this) so a freshly-built player is never half-initialised (it can move on frame zero even before a level finishes loading). Then GameDirector.loadLevelByIndex calls applyFeelProfile(entry.feelProfile), which re-applies the registry's chosen profile to the same player. For a DK level this is a harmless re-stamp; for a Celeste level the load-time call overrides the constructor's DK seed. The load-time application — not the constructor — is the authoritative one. See GameDirector.applyFeelProfile.

const setDkAttributes = (player: Player) => void

Purpose. Apply the DK feel. Parameter. player — mutated in place. Returns. nothing. 

applyBaseFeel(player);

// Horizontal
player.MaxGroundSpeed = 11.8;   // less than half Celeste's 25
player.GroundAcceleration = 50;
player.GroundDeceleration = 20;
player.AirAcceleration = 50;
player.AirDeceleration = 20;

// Jump arc
player.MaxFallSpeed = 40;
player.FallAcceleration = 80;
player.RiseGravity = 72;
player.JumpEndedEarlyGravityModifier = 3;
player.ApexHangThreshold = 4;   // narrow window = barely-there hang
player.ApexHangMult = 0.5;
player.JumpPower = 24;          // a short controlled hop
player.BufferJumpRange = 1.5;   // tight landing window
player.MinJumpTime = 0.05;

Same fields, same handler paths as Celeste — only the magnitudes change, and that difference is the entire point. MaxGroundSpeed = 11.8 is a measured-feeling arcade walk; JumpPower = 24 with RiseGravity = 72 gives a short hop that clears a barrel but not much more. The apex window is narrow (ApexHangThreshold = 4) so DK gets only the faintest hang — heavy, not floaty — and BufferJumpRange = 1.5 demands a more precise pre-land press than Celeste's forgiving 4. The early-release modifier (3), apex multiplier (0.5), and the shared coyote/climb values are identical to Celeste, so the mechanics are the same; only the tuning differs.

Hard-won victory — same time-scale recipe, DK numbers. Like Celeste, DK's arc was labbed slow then time-scaled ~2× (velocities ×2, accelerations ×4, jump times ×0.5; same apex height, half the airtime). The recorded originals were JumpPower 12 / RiseGravity 18 / Fall 20 / MaxFall 20 / ApexHang 2×0.5 / MinJumpTime 0.1. Doubling JumpPower 12 → 24 and quadrupling RiseGravity 18 → 72 keeps the identical hop height while making the controls feel responsive on the real frame clock.

Modifying this: this is the live default, so test changes here against the actual game, not just the Feel Lab. Keep the constructor seed and the registry profile in sync conceptually — if you want a different construction-time default, change the call in player.ts initializePlayerAttributes(), but the real authority is the level's feelProfile.

Note — feel is the single source of truth, applied only at level load. GameDirector.applyFeelProfile(profile) is the one place feel is set: it looks the profile up in FEEL_PROFILES and calls the setter on the current player — FEEL_PROFILES[profile](this.Player). There is no live debug swap that bypasses it; the dat.gui PlayerDebug can nudge individual fields for experimentation, but the authoritative profile is whatever the loading level's registry row declares. The method comment even flags the future hook: a "trigger sets the feel" mechanic would promote applyFeelProfile from private to a public entry point.
Gotcha — load order matters. applyFeelProfile guards on this.Player existing, and inside loadLevelByIndex it runs after setPlayerStart() has spawned (or teleported) the player and after the GLB has parsed — so the player object is always present when the profile is stamped. The whole load path is serialized by the isLoading re-entrancy lock (decision code D5): without it, mashing F2 or the level dropdown mid-load could interleave teardown and respawn and apply a profile to a half-built player. Feel application rides safely inside that single-flight window.
