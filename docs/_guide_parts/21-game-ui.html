App.vue — the Vue UI shell & wiring

App.vue is the single Vue component in the app and the one place where the engine Singleton and the entire game layer are assembled. It owns presentation — the loading overlay, a status line, a Reset button, and the WebGL canvas — while the game only ever signals through the event bus. Every line of its <script setup> is a deliberate ordering decision: listeners come up before the engine, and asset loading is kicked off dead last.

src/App.vue

src/App.vue 188 LOC

This is the root component mounted by main.ts via createApp(App).mount('#app') — there is no router and no other component, so App.vue is the application shell. Its job is threefold: (1) hold two DOM refs (the canvas and a status paragraph), (2) own the loading-overlay and game-status reactive state, and (3) inside a single onMounted hook, wire the engine Experience Singleton to the game's GameDirector, contact rules, debug sinks, input bindings, and the sim/render update hooks — plus a symmetric onUnmounted that tears the whole graph down. It is intentionally the only file that knows about both the engine and the game by name — the seam where the two halves are bolted together. Everything else talks through the Emitter bus.

import { onMounted, onUnmounted, ref } from "vue"; // + Experience, Emitter, GameDirector, InputBindings, registerContactRules, CollisionLogger, LevelDebug, DEBUG_TYPE_COLORS, GAME_ASSETS, LEVEL_NODE_TYPES

The import list reads like a wiring manifest. From the engine: Experience (the Singleton it boots) and Emitter (the bus it listens on). From the game: GameDirector (the central controller it constructs), InputBindings (F-key → game-event mapper), registerContactRules (declares the collision table into the engine), CollisionLogger + LevelDebug (game-owned debug plug-ins), DEBUG_TYPE_COLORS (the wireframe palette), and the two newest seam pieces — GAME_ASSETS (the boot asset manifest handed to Configure) and LEVEL_NODE_TYPES (the level-node→EntityType vocabulary injected into the engine's GLB parser). The fact that only this file imports from both ./engine and ./game is the whole decoupling story in one glance — and the engine no longer hardcodes a single game asset path or level-node name.

const isLoading = ref(true); const loadProgress = ref<{ loaded: number; total: number; item: string } | null>(null);

The loading-overlay state, and the most carefully-reasoned pair of refs in the file. isLoading starts true so the overlay covers the screen before any code runs — the boot asset load is already underway visually by the time Vue paints. loadProgress is the per-item count that drives a real progress bar; it is deliberately null except during the initial multi-asset boot load. 

// Loading screen: starts up so it covers initial asset loading, then the engine/
// game emit loadingStarted/loadingFinished around every level (GLB) load. The
// presentation layer (Vue) owns the overlay; the game just signals via events.
// `loadProgress` drives a real bar during the initial multi-asset load
// (loadingProgress events); it's cleared for level loads (a single GLB parse has
// no per-item count) so the overlay falls back to a plain "Loading…".
const isLoading = ref(true);
const loadProgress = ref<{ loaded: number; total: number; item: string } | null>(
  null
);

The shape { loaded, total, item } matches the engine's loadingProgress event payload exactly (see eventBus.ts's EngineEventMap and ResourceLoader.StartLoading, which emits a seed { loaded: 0, total, item: "" } and then advances loaded per asset). Two distinct loads share this one overlay: the multi-asset boot load (textures + GLB models, which has a per-item count → progress bar) and a per-level GLB load (a single opaque parse, no count → spinner).

Sitting beside them are seven named event-handler consts — onLoadingStarted, onLoadingFinished, onLoadingProgress, onGameWin, onGameOver, onGameReset, onGameStart. Naming them is what lets onUnmounted remove exactly these from the shared bus by reference: a bare Emitter.off("gameWin") would wipe every module's gameWin listener, and anonymous handlers can't be removed at all (they'd double-register on hot-reload and fire twice). Their bodies appear in steps 1 and 7 below.

onMounted(async () => { ... });

The entire boot-and-wire sequence. It is async because Experience.Configure awaits Rapier init. The ordering inside is not incidental — it encodes two hard-won "listener exists before the event can fire" guarantees. The flow:

Step 1 — loading listeners FIRST (lines 67–71). The three loading handlers are named consts at script scope (shown here with their bodies); onMounted registers them by reference before Configure is even called:

// Named handlers at script scope so onUnmounted can off EXACTLY these:
const onLoadingStarted = () => {
  isLoading.value = true;
  loadProgress.value = null; // level loads are a single GLB parse — no per-item bar
};
const onLoadingFinished = () => {
  isLoading.value = false;
};
const onLoadingProgress = (p: { loaded: number; total: number; item: string }) => {
  loadProgress.value = p;
};

// ...inside onMounted, FIRST — before Configure — so a fast boot load can't fire
// loadingFinished before we're listening:
Emitter.on("loadingStarted", onLoadingStarted);
Emitter.on("loadingFinished", onLoadingFinished);
Emitter.on("loadingProgress", onLoadingProgress);

loadingStarted raises the overlay and clears loadProgress (so per-level loads show the plain spinner, since a single GLB parse reports no item count). loadingFinished drops the overlay. loadingProgress stores the latest count, which makes the bar appear. These are emitted by GameDirector.loadLevelByIndex (the loadingStarted/loadingFinished bracket, the latter in its finally) and by ResourceLoader (the loadingProgress stream). App.vue never emits any of them — it is a pure consumer here.

Hard-won victory — listeners before Configure. Configure awaits async init; if the boot assets happened to resolve inside that await window and someone emitted loadingFinished before these handlers were attached, the overlay would never lift (mitt does not buffer or replay). Subscribing first makes the race impossible. The same logic, one step larger, governs StartLoading() being last.

Step 2 — boot the engine Singleton (lines 74–78).

// Non-null local for use here; the script-scoped ref is for onUnmounted.
const exp = Experience.GetInstance();
webglExperience = exp;

// The game supplies its own asset manifest; the engine names none of these.
await exp.Configure(webglRef.value, GAME_ASSETS);

GetInstance() returns the lazily-created Experience Singleton; Configure is its real (async) constructor — it builds Debug, Sizes, Time, Input, Resources, the Three.js Scene, Camera, Renderer, and awaits Physics.Configure() (Rapier init), then subscribes the engine's resize and tick handlers. Note the second argument: Configure(canvas, resources) now takes the game's GAME_ASSETS manifest and feeds it straight into the ResourceLoader it constructs — the engine no longer hardcodes a randy/enemy/dkGraphicsData import list, so engine/ ships without naming a single game asset. Passing webglRef.value is the reason this whole block is inside onMounted: the canvas element only exists once Vue has mounted the DOM. The instance is also stashed in the script-scoped webglExperience ref (via the exp local) so onUnmounted can later call its Destroy().

Reusability win — the manifest is injected, not baked in. GAME_ASSETS (see assetManifest.ts) is a plain Resource[] the game owns; the engine just loads whatever it's handed. The same engine can boot a different game by passing a different manifest — the old hardcoded ?url imports inside resourceLoader.ts are gone.

Step 3 — the game declares its rules into the engine (lines 82–88). Four injections, each pushing a game concept into an engine subsystem that names nothing game-specific:

exp.Resources.RegisterLevelTypes(LEVEL_NODE_TYPES);

registerContactRules(exp.Physics.Contacts);
exp.Debug.RegisterCollisionLogSink(new CollisionLogger());
exp.Debug.SetPhysicsTypeColors(DEBUG_TYPE_COLORS);

RegisterLevelTypes(LEVEL_NODE_TYPES) (see levelNodeTypes.ts) hands the engine's GLB parser the game's level-node-name→EntityType vocabulary, so the engine can classify the nodes it finds in a level file without ever mirroring the game's EntityType enum — it must run before StartLoading() kicks off the first level load. registerContactRules (see contactRules.ts) populates the engine's Physics.Contacts registry with this game's six cross-entity rules (Enemy↔Player = gameOver, Enemy↔TrashCan, Teleporter, CameraSensor, WinFlag) — the engine dispatches physics contacts into the table but never names a game concept. RegisterCollisionLogSink(new CollisionLogger()) hands the engine Debug coordinator a game-owned formatter implementing the CollisionLogSink interface; Physics routes enter/exit log calls to it (no-op when no sink). SetPhysicsTypeColors(DEBUG_TYPE_COLORS) injects the entity-type→color map (see debugColors.ts) which PhysicsDebug consults to tint each collider's wireframe.

Decision code D3 — the engine never names a type. The injection comment above these calls is tagged (D3). The wireframe palette is keyed by game EntityType strings, but the engine's PhysicsDebug must not import the game's enum. So the game injects the Map<string, THREE.Color> and the engine looks each collider's userData.type up in it opaquely, falling back to gray for unlisted types. D3 prevents the engine layer from ever depending on a game-defined name — the bug it forestalls is a circular/leaky dependency that would make engine/ un-reusable.

Step 4 — construct the game (lines 92–94), built AFTER Configure.

const director = new GameDirector();
gameDirector = director;
const inputBindings = new InputBindings(exp.Input);

GameDirector's constructor calls Experience.GetInstance() and caches Scene, Time, Camera, Resources — so it must run after Configure has created those subsystems. Critically, its constructor also runs setupEventHandlers(), which subscribes the resourcesReady listener that boots the default level. That subscription is the reason StartLoading() is deferred to the very end. InputBindings takes the engine Input device and converts F-key press edges into game events (F1→gameReset, F2→switchLevel, F3→manualCameraControl). The director is also captured in the script-scoped gameDirector ref so onUnmounted can call its Destroy().

Step 5 — register the debug level dropdown (line 97).

// Debug level select (dat.gui) — no-op store when debug is inactive.
exp.Debug.RegisterModule(new LevelDebug(director));

LevelDebug is a DebugModule that adds a dat.GUI "🗺️ Level" folder bound directly to GameDirector.CurrentLevelName. It is passed the director so its dropdown can call LoadLevelByName. Per Debug.RegisterModule, when debug is inactive the module is merely stored (its Init never runs); when active the folder is built immediately. App.vue doesn't branch on debug state — it always registers, and the engine decides.

Step 6 — the sim and render hooks (lines 99–107). Two late-bound seams now, not one: OnGameUpdate runs once per fixed sim step, OnRenderUpdate once per rendered frame.

exp.OnGameUpdate = () => {
  inputBindings.Update();
  director.Update();
};
exp.OnRenderUpdate = (alpha: number) => {
  director.RenderUpdate(alpha);
};

These are the late-bound seams that keep the engine free of game names. Experience's onTick now runs a fixed-step simulation loop: it calls this.OnGameUpdate?.() once per fixed step (so the sim rate is decoupled from frame rate), then — after the loop — computes an interpolation factor alpha = accumulator / dt and calls this.OnRenderUpdate?.(alpha) exactly once per frame, just before Renderer.Update(). The split maps cleanly onto the director: Update() is the SIM pass (input edges, player, enemies, spawn, ladder), while RenderUpdate(alpha) is the per-frame pass that interpolates each body's graphics between its previous and current sim positions and then moves the camera to follow the interpolated player. Order inside OnGameUpdate still matters: inputBindings.Update() runs first so an F-key press this step (e.g. F1→gameReset) is emitted before director.Update() ticks the player/enemies.

Note. App.vue assigns both hooks but never calls them. The engine's Time "tick" event drives the loop; App.vue only supplies the two function bodies. This is the inversion that lets engine/core/experience.ts ship without importing a single game file.
Why two hooks — smooth motion at any frame rate. Splitting a per-step OnGameUpdate from a per-frame OnRenderUpdate(alpha) is what lets physics tick at a fixed rate while rendering interpolates between steps — a rolling barrel slides smoothly instead of snapping a step at a time. App.vue's only job here is to wire the right pass to each hook; the interpolation math lives in GameDirector.RenderUpdate and the engine loop.

Step 7 — game-status handlers (defined at script scope, registered here — lines 48–64, 109–112). Four more named handlers translate game-flow events into UI changes; onMounted subscribes them by reference (so onUnmounted can later off exactly these):

const onGameWin = () => {
  if (statusRef.value) statusRef.value.innerText = "Game Win!";
  isResetButtonVisible.value = true;
};
const onGameOver = () => {
  if (statusRef.value) statusRef.value.innerText = "Game Over!";
  isResetButtonVisible.value = true;
};
const onGameReset = () => {
  if (statusRef.value) statusRef.value.innerText = "";
  isResetButtonVisible.value = false;
};
// A (re)start — boot or a level switch — clears any lingering game-over/win UI.
const onGameStart = () => {
  if (statusRef.value) statusRef.value.innerText = "";
  isResetButtonVisible.value = false;
};

// ...inside onMounted, after the update hooks:
Emitter.on("gameWin", onGameWin);
Emitter.on("gameOver", onGameOver);
Emitter.on("gameReset", onGameReset);
Emitter.on("gameStart", onGameStart);

gameWin and gameOver write the banner and reveal the Reset button. gameReset and gameStart both clear the banner and hide the button — the difference is semantic: gameReset is the player re-arming the current level, while gameStart fires on every successful level load (boot, F2 switch, debug select — emitted from GameDirector.loadLevelByIndex on success). Listening to gameStart is what stops a fresh level from inheriting the previous level's "Game Over!" text. None of these events originates here; they come from contactRules.ts (gameOver/gameWin), GameDirector (gameStart), and InputBindings/the Reset button (gameReset).

Step 8 — kick off asset loading LAST (line 116).

// Kick off asset loading LAST — everything that listens for the boot
// ("resourcesReady" → GameDirector) and per-asset progress is now wired.
exp.Resources.StartLoading();
Hard-won victory — StartLoading() dead last. ResourceLoader deliberately does not auto-start in its constructor (its own comment explains why). If loading began before GameDirector existed, a fast load could emit resourcesReady into zero listeners — mitt drops it — and the boot would hang on the loading screen forever. By starting only after the director (resourcesReady) and the progress listeners (step 1) are all subscribed, the completion event is guaranteed to land. This is the macro version of the step-1 race fix.
Gotcha — ordering is load-bearing top to bottom. Moving any of these calls is dangerous: listeners must precede Configure; RegisterLevelTypes must follow Configure (it reaches into Resources) yet precede StartLoading (the first level load needs the vocabulary in place); GameDirector must follow Configure (it reads subsystems) yet precede StartLoading (it owns the resourcesReady listener); the update hooks (OnGameUpdate/OnRenderUpdate) can be assigned any time before the first tick fires but are placed after the game objects exist. Treat onMounted as a fixed recipe.

const handleButtonClicked = () => { ... };

The Reset button's @click handler. It emits gameReset (which GameDirector listens for to unpause physics and call resetGameState()), then also clears the status text and hides itself locally. 

// Reset button: clear win/over status and re-arm the game
const handleButtonClicked = () => {
  Emitter.emit("gameReset");

  if (statusRef.value) {
    statusRef.value.innerText = "";
  }

  isResetButtonVisible.value = false;
};

Note the slight redundancy: emitting gameReset already triggers the gameReset handler from step 7, which clears the same status and button. Doing it inline as well is harmless and makes the click feel instant regardless of listener order. This is the only event App.vue emits.

const barWidth = () => string;

Computes the progress-bar fill percentage as a CSS width string. Returns "0%" when there is no progress payload or total is zero (guarding a divide-by-zero / NaN), otherwise loaded / total * 100 + "%". 

// Loading-bar fill percent from loaded/total (empty -> 0%)
const barWidth = () => {
  if (!loadProgress.value || !loadProgress.value.total) return "0%";
  return (loadProgress.value.loaded / loadProgress.value.total) * 100 + "%";
};

It is a plain function (not a computed), called from the template's :style binding; Vue re-invokes it whenever loadProgress changes because it reads that ref. The empty guard is what shows a 0%-wide bar the instant StartLoading emits its seed { loaded: 0, total } event.

<!-- Loading overlay — covers the canvas during any level/asset load: progress bar while initial assets load, plain "Loading…" while a level GLB parses. -->
<div

    <p class="text-white text-sm tracking-widest">

  </p>
</div>

The overlay is gated by v-if="isLoading" (a full-screen black z-50 layer over the canvas). Inside, a v-if="loadProgress" / v-else picks the presentation: when a count exists (boot load) it renders "Loading N / total · item" plus the filled bar driven by :style="{ width: barWidth() }"; otherwise (a per-level GLB parse, where loadProgress was nulled by the loadingStarted handler) it shows the pulsing "Loading…" spinner text. The item suffix is itself gated by v-if="loadProgress.item" so the seed item: "" event doesn't render a dangling separator.

onUnmounted(() => { ... });

The teardown hook, symmetric with the onMounted recipe. It removes exactly the seven named listeners this component registered — by reference, so it can't disturb other modules sharing the bus — then destroys the game and the engine, in that order. This is the path that finally exercises the engine subsystems' Destroy() at all; it fires on an HMR reload or, if App.vue ever became a mounted/unmounted child, on a route change. 

onUnmounted(() => {
  Emitter.off("loadingStarted", onLoadingStarted);
  Emitter.off("loadingFinished", onLoadingFinished);
  Emitter.off("loadingProgress", onLoadingProgress);
  Emitter.off("gameWin", onGameWin);
  Emitter.off("gameOver", onGameOver);
  Emitter.off("gameReset", onGameReset);
  Emitter.off("gameStart", onGameStart);

  gameDirector?.Destroy();
  webglExperience?.Destroy();
});

Order matters: gameDirector?.Destroy() runs before webglExperience?.Destroy() so the director can off its own bus handlers and unload its level data while the engine subsystems it borrows still exist; only then does the engine tear down its own loop, renderer, and physics world. The optional chaining (?.) guards the case where a half-failed Configure left one of the script-scoped refs null.

Hard-won victory — symmetric teardown, no leaks. Naming the handlers (script scope) and offing each by reference is what makes a clean unmount possible: a bare Emitter.off("gameWin") would wipe every module's gameWin listener, and the old anonymous handlers couldn't be removed at all — they'd survive a hot-reload and double-fire on the next mount. Pairing every Emitter.on in onMounted with an Emitter.off here, plus gameDirector.Destroy() + webglExperience.Destroy(), closes the leak the previous version openly warned about.

Modifying this: To add a new HUD element reacting to gameplay, define a fresh game event in eventBus.ts's GameEventMap, emit it from the game side, and add one Emitter.on here that flips a ref the template renders — do not reach into game objects from Vue. To wire a new engine↔game seam (another contact-rule set, debug sink, or update participant), add it to the onMounted recipe in the right slot: injection calls after Configure, anything subscribing to resourcesReady before StartLoading(), and listeners before Configure when they must catch boot events. Keep StartLoading() the final statement of onMounted. And whenever you add an Emitter.on here, define it as a named handler and add the matching Emitter.off (by that same reference) to onUnmounted so the listener tears down cleanly.
