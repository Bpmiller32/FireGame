Rendering — Renderer & SpriteAnimator

Two small, single-responsibility files own everything the player actually sees. renderer.ts is a thin lifecycle wrapper around one THREE.WebGLRenderer — it is created by the Experience singleton, rendered once per tick, resized on resize, and disposed on teardown. spriteAnimator.ts is a per-entity helper that turns a single static sprite-sheet texture into a flip-book: it slices the sheet into a tile grid and walks a THREE.SpriteMaterial's UV offset frame-by-frame. Both files share one obsessive theme — pixel-perfect, never-blurry rendering — and both encode that as inline CRITICAL comments rather than decision-codes.

src/engine/rendering/renderer.ts 58 LOC

This file IS the engine's window onto the GPU — a wrapper that owns exactly one THREE.WebGLRenderer and nothing else. It is constructed by the Experience singleton (see Engine Core — Experience): this.Renderer = new Renderer(), immediately after the Scene and Camera exist and just before Physics. It imports Experience only to pull engine singletons out of the one global wiring hub, Sizes for width/height/pixel-ratio, and Camera because rendering needs a viewpoint — it does not import the game layer at all, which is what keeps it reusable. Its lifecycle is deliberately minimal: constructor grabs refs and builds the renderer, Resize() re-fits it to the window, Update() draws one frame, Destroy() frees the GL context.

Note. The renderer never subscribes to the event bus itself. The Experience owns the tick and resize subscriptions and calls Renderer.Update() / Renderer.Resize() directly. That keeps this class a passive device the coordinator drives, not an actor that wires itself into the world — the same pattern every engine subsystem follows.

private experience; private canvas; private sizes; private scene; private camera; private Instance!: THREE.WebGLRenderer

Five cached references plus the raw renderer. Everything except Instance is read straight off the Experience in the constructor, so this object never re-resolves a singleton at runtime. Instance is the only thing this class actually owns and is marked ! (definite-assignment) because it is built in setInstance() rather than inline. 

private experience: Experience;
private canvas: HTMLElement;
private sizes: Sizes;
private scene: THREE.Scene;
private camera: Camera;

private Instance!: THREE.WebGLRenderer; // the raw THREE WebGL renderer

Note scene is a THREE.Scene (the engine-shared draw list) while camera is the engine's own Camera wrapper — so rendering reaches through this.camera.Instance for the underlying THREE camera (see Camera).

constructor()

No parameters. Resolves the singleton, caches the canvas/sizes/scene/camera off it, then builds and sizes the renderer. The order matters: setInstance() creates the GL context bound to the canvas, and Resize() immediately fits it to the current window — there is no first-frame at the wrong size. 

this.experience = Experience.GetInstance();
this.canvas = this.experience.TargetElement as HTMLElement;
this.sizes = this.experience.Sizes;
this.scene = this.experience.Scene;
this.camera = this.experience.Camera;

this.setInstance();
this.Resize();

TargetElement is cast to HTMLElement because the Experience stores it generically; the WebGL renderer wants a concrete canvas element. Reading Sizes here (not window directly) is the seam that lets Sizes own the single source of truth for dimensions and pixel ratio.

private setInstance()

Builds and configures the one THREE.WebGLRenderer. This is the most opinionated method in the file: every option here exists to make pixel art crisp. 

this.Instance = new THREE.WebGLRenderer({
  canvas: this.canvas,
  // CRITICAL: Antialiasing MUST be disabled for pixel art!
  // Antialiasing smooths edges which makes pixel art look blurry...
  antialias: false,
});

this.Instance.setClearColor("#211d20");
this.Instance.setSize(this.sizes.Width, this.sizes.Height);
this.Instance.setPixelRatio(this.sizes.PixelRatio);

Line by line: canvas binds the GL context to the existing DOM canvas (rather than letting THREE create a detached one). setClearColor("#211d20") paints the background — a near-black warm charcoal — every frame before drawing. setSize sets the drawing-buffer resolution from Sizes, and setPixelRatio applies the (capped) device pixel ratio so HiDPI displays render sharp without over-rendering.

Hard-won victory: antialias: false. This is the single most important line in the renderer. MSAA smooths triangle edges, which on a magnified pixel-art sprite reads as a shimmering, blurry halo that crawls during motion. Disabling it is what makes a 16×-scaled firefighter look like crisp pixels instead of a smear. It is the renderer-side half of a fix that the SpriteAnimator completes with NearestFilter — you need both off/nearest or the art still blurs.
Gotcha. setPixelRatio must come after (or alongside) setSize, and both read from Sizes. If you ever set the renderer size from window.innerWidth directly you fork the source of truth and the canvas can desync from the camera's aspect on resize. Always go through Sizes.

public Resize()

Re-fits the renderer to the current window. Called by the Experience's resize handler (which also calls Camera.Resize()). It simply re-applies size and pixel ratio from the freshly updated Sizes values. 

this.Instance?.setSize(this.sizes.Width, this.sizes.Height);
this.Instance?.setPixelRatio(this.sizes.PixelRatio);

The ?. optional chaining guards the one race where Resize() could fire before Instance exists. In practice the constructor builds it first, but the guard keeps a stray early resize from throwing. Pixel ratio is re-applied here too, not just in setInstance, because dragging a window between a HiDPI and a standard monitor changes window.devicePixelRatio live.

public Update()

Draws exactly one frame: the shared scene viewed through the active camera. This is the engine's render call, invoked once per tick by the Experience. 

this.Instance?.render(this.scene, this.camera.Instance);

this.camera.Instance reaches past the engine's Camera wrapper to the underlying THREE camera that render() requires. The crucial detail is when this runs in the frame: the Experience's tick handler runs Physics.Update() first (which moves bodies and lets each game object sync its mesh to its physics transform), then the game's OnGameUpdate hook, then Renderer.Update() last — so the renderer always draws the world's final, settled state for that frame. 

sequenceDiagram
    participant T as Time (tick)
    participant E as Experience.onTick
    participant P as Physics.Update
    participant G as OnGameUpdate
    participant R as Renderer.Update
    T->>E: emit "tick"
    E->>P: step world, sync meshes
    E->>G: game logic / camera follow
    E->>R: render(scene, camera.Instance)
    R-->>T: frame drawn
Gotcha: render is last for a reason. If you ever move Renderer.Update() before Physics.Update(), you draw last frame's positions — one-frame visual lag and jitter on fast-moving barrels. The ordering in Experience's onTick is load-bearing.

public Destroy()

Frees the GPU resources held by the renderer. Called by the Experience's Destroy() in the documented teardown order (camera → physics → renderer → resources). 

// Dispose of the WebGL renderer
this.Instance.dispose();

dispose() releases the WebGL context and its GPU buffers. Note this does not dispose geometries, materials, or textures in the scene — those belong to the individual GameObjects and the ResourceLoader, each of which disposes its own (e.g. the SpriteAnimator below disposes its texture and material). Renderer only owns the context, so it only frees the context.

Modifying this: treat the four pixel-art settings as a unit — if you turn antialias back on you reintroduce blur, and changing setPixelRatio away from the Sizes value forks the source of truth. To add post-processing you would introduce an EffectComposer and swap Update()'s render for composer.render() (and resize the composer in Resize()). To change the background, edit the one setClearColor hex. Keep this file game-agnostic: no game import belongs here.

src/engine/rendering/spriteAnimator.ts 139 LOC

This file IS the engine's flip-book player: it takes one static sprite-sheet THREE.Texture, treats it as a grid of equal tiles, and animates a THREE.SpriteMaterial by sliding its UV offset from tile to tile over time. It imports only three — it knows nothing about players, states, or the game. One SpriteAnimator is constructed and owned per animated entity; for the hero it is built in Player.setSpriteAnimator() as new SpriteAnimator(this.Resources.Items.randy, 4, 6) (see Player). Lifecycle: constructor caches the grid dimensions and builds the material; ChangeState() swaps which frames play; ChangeAnimationTiming() scales playback speed; Update(deltaTime) advances the frame; Destroy() disposes the texture and material.

interface SpriteState { indicies: number[]; timing: number[] }

The data shape that drives one animation. indicies is the ordered list of tile ids to show (the misspelling is kept on purpose, per the comment, so it matches the rest of the codebase). timing is the parallel array of seconds each frame holds — same length as indicies, index-aligned. The game supplies these as a static table in spriteAnimations.ts (see Player); the engine never hard-codes any frame numbers. 

interface SpriteState {
  indicies: number[]; // tile indices, left-to-right top-to-bottom from 0
  timing: number[];   // seconds per frame; same length as indicies
}

How a sheet becomes a flip-book

The animator never crops or copies pixels. The texture stays whole; what moves is a small UV window. In the constructor the material's repeat is set to (1/tilesHorizontal, 1/tilesVertical) — for Randy that is (1/4, 1/6), a window exactly one tile wide and one tile tall. Every frame, Update() recomputes the window's offset so it lands on the desired tile. Here is the real sheet the offsets walk:

Randy is a 4×6 grid (24 tiles, ids 0..23), numbered left-to-right then top-to-bottom from 0 in the top-left. So row 0 is ids 0,1,2,3; row 1 is 4,5,6,7; and so on down to row 5 = 20,21,22,23. The empty cells are intentional gaps — the game's SpriteAnimations table only references the occupied ones:

AnimationindiciesSheet locationtiming (s)
IDLE_LEFT[1, 2]row 0, cols 1&2[5, 0.1]
IDLE_RIGHT[5, 6]row 1, cols 1&2[0.1, 5]
RUN_LEFT[8, 9, 10, 11]row 2, all 4[0.25×4]
RUN_RIGHT[12, 13, 14, 15]row 3, all 4[0.25×4]
JUMP_LEFT / FALL_LEFT[17, 18]row 4, cols 1&2[0.25, 0.25]
JUMP_RIGHT / FALL_RIGHT[21, 22]row 5, cols 1&2[0.25, 0.25]
Tip: the blink. Idle is a two-frame loop where one frame holds for 5 s and the other for 0.1 s — that 0.1 s frame is the eyes-closed tile, so Randy blinks once every ~5 seconds while standing. Notice IDLE_LEFT is [5, 0.1] but IDLE_RIGHT is [0.1, 5]: the long-hold frame is the second entry on the right side because the open-eyes tile happens to be the second id (6) there. The timing array is what encodes "which tile is the resting pose."

private currentTile; elapsedTime; runningTileIndex; tilesHorizontal; tilesVertical; timingMultiplier; public State; public Material!

Two indices are easy to confuse, so the field names are precise: currentTile is the tile id on the sheet (e.g. 13), while runningTileIndex is the position within State.indicies (e.g. the 2nd entry). elapsedTime accumulates deltaTime toward the current frame's duration. tilesHorizontal/tilesVertical are the grid dims. timingMultiplier scales every frame duration to speed up or slow down playback. State and Material are the only public members — the game reads/writes State and the entity's mesh binds to Material.

constructor(spriteSheet: THREE.Texture, tilesHorizontal: number, tilesVertical: number)

Parameters: the sheet texture (Randy's randy texture from ResourceLoader), and the grid dimensions. Zeroes the playback cursors, stores the grid, initialises State to an empty animation, sets timingMultiplier = 1, then builds the material. 

this.currentTile = 0;
this.elapsedTime = 0;
this.runningTileIndex = 0;
this.tilesHorizontal = tilesHorizontal;
this.tilesVertical = tilesVertical;

this.State = { indicies: [], timing: [] };
this.timingMultiplier = 1;

this.setMaterial(spriteSheet);

The empty starting State means a freshly built animator shows tile 0 (top-left) and does not advance until the game assigns a real state — which the Player does immediately after construction: this.SpriteAnimator.State = SpriteAnimations.IDLE_RIGHT.

private setMaterial(spriteSheet: THREE.Texture)

Builds the THREE.SpriteMaterial and locks in pixel-perfect filtering — the SpriteAnimator's half of the crispness fix that Renderer starts with antialias: false. 

spriteSheet.colorSpace = THREE.SRGBColorSpace;

this.Material = new THREE.SpriteMaterial({ map: spriteSheet });

this.Material.map!.magFilter = THREE.NearestFilter; // scaling UP: hard edges
this.Material.map!.minFilter = THREE.NearestFilter; // scaling DOWN: no blur
this.Material.map!.anisotropy = 1;                  // minimum: no angle filtering

// Show one tile from the sheet at a time
this.Material.map!.repeat.set(
  1 / this.tilesHorizontal,
  1 / this.tilesVertical,
);

colorSpace = SRGBColorSpace keeps the art's colours correct through THREE's colour pipeline. repeat.set(1/4, 1/6) shrinks the sampled UV rectangle to one tile — combined with the per-frame offset in Update(), that rectangle becomes the moving "window" over the sheet.

Hard-won victory: NearestFilter on BOTH min and mag. The comment is emphatic for a reason — THREE defaults minFilter to a mip-mapped linear filter. With the default, the sprite blurs the moment it is drawn even slightly smaller than its native size (camera pull-back, sub-pixel positions). Forcing NearestFilter on magFilter (zoomed in) and minFilter (zoomed out), plus anisotropy = 1 (no oblique-angle smoothing), is what guarantees Randy stays razor-sharp at every scale and never shimmers in motion. Miss any one of these three and the blur sneaks back.

public ChangeState(newState: { indicies: number[]; timing: number[] })

Swaps the playing animation. Called by the player state handlers (handlePlayerIdle, handlePlayerRunning, handlePlayerJumping, handlePlayerFalling — see the Player states) every frame with the appropriate SpriteAnimations.* object. 

// Skip if already in this state (don't reset the animation every frame)
if (this.State === newState) {
  return;
}

this.State = newState;
this.timingMultiplier = 1;
this.runningTileIndex = 0;
this.currentTile = this.State.indicies[this.runningTileIndex];
this.elapsedTime = this.State.timing[this.runningTileIndex]; // Force play new animation instead of waiting a loop

The reference-equality guard is the load-bearing line: handlers call ChangeState unconditionally each frame, but because SpriteAnimations.RUN_RIGHT is a single shared object, passing the same one short-circuits — so the animation does not restart from frame 0 sixty times a second. It only truly changes when the object identity differs. Resetting timingMultiplier to 1 here means a new state starts at base speed (running re-applies its own multiplier afterward).

Gotcha: that last line pre-loads elapsedTime. Setting elapsedTime = timing[0] means the very next Update() immediately satisfies the advance condition and steps to indicies[1] — the new animation visibly "moves" on the first frame instead of holding indicies[0] for its full duration first. The practical consequence: the first frame actually rendered after a state change is indicies[1], not indicies[0]. That is intended ("force play"), but it means index 0 is effectively the loop's starting pose, not its first-shown pose. Keep that in mind when ordering frames in spriteAnimations.ts.

public ChangeAnimationTiming(newTimingMultiplier: number)

Scales playback speed without changing which frames play. The running handler drives this so the run cycle speeds up with the player's velocity: handlePlayerRunning calls ChangeAnimationTiming(1 / (|NextTranslation.x| / AnimationScalingFactor)). 

// Don't change timing unless needed
if (this.timingMultiplier === newTimingMultiplier) {
  return;
}

this.timingMultiplier = newTimingMultiplier;

Every duration in Update() is multiplied by timingMultiplier, so a multiplier below 1 shortens frame holds (faster animation) and above 1 lengthens them (slower). The equality guard avoids redundant writes. AnimationScalingFactor (set to 1.6 in baseFeel.ts) is the game-feel knob that tunes how fast the legs churn relative to ground speed — a game-layer concern the engine stays ignorant of.

public Update(deltaTime: number)

Advances the flip-book by one tick. Called every frame from Player.updatePlayerState() as this.SpriteAnimator.Update(this.Time.Delta) right after the FSM runs, so the sprite reflects the state the handlers just chose. Accumulates time, and when the current frame's (scaled) duration is reached, steps to the next tile and recomputes the UV offset. 

this.elapsedTime += deltaTime;

if (
  this.State.timing[this.runningTileIndex] * this.timingMultiplier > 0 &&
  this.elapsedTime >=
    this.State.timing[this.runningTileIndex] * this.timingMultiplier
) {
  this.elapsedTime = 0;

  // Move to the next tile index, looping back to the beginning if necessary
  this.runningTileIndex =
    (this.runningTileIndex + 1) % this.State.indicies.length;

  this.currentTile = this.State.indicies[this.runningTileIndex];

  const offsetX =
    (this.currentTile % this.tilesHorizontal) / this.tilesHorizontal;
  const offsetY =
    (this.tilesVertical -
      Math.floor(this.currentTile / this.tilesHorizontal) -
      1) /
    this.tilesVertical;

  this.Material.map!.offset.x = offsetX;
  this.Material.map!.offset.y = offsetY;
}

Walking the two halves of the gate condition:

  • timing[runningTileIndex] * timingMultiplier > 0 — a zero (or negative) duration disables advancing, freezing on the current tile. This is the safety valve: a 0 in a timing array means "hold this frame forever," and it also prevents a degenerate >= 0 from advancing every single frame.
  • elapsedTime >= timing[...] * timingMultiplier — the frame's time is up. Reset elapsedTime to 0 and advance.

(runningTileIndex + 1) % indicies.length is the loop: it wraps the position cursor back to the first entry, so every animation here loops indefinitely. There is no one-shot mode — a non-looping animation would be expressed by parking on a frame with timing 0 (the gate above), not by a flag.

The offset math, and why Y is flipped

Tile ids count from the top-left, but THREE texture UVs originate at the bottom-left. So column is a straight modulo, while the row must be flipped:

  • column = currentTile % tilesHorizontal, then offsetX = column / tilesHorizontal.
  • rowFromTop = Math.floor(currentTile / tilesHorizontal), then offsetY = (tilesVertical - rowFromTop - 1) / tilesVertical — the (V - row - 1) term is the top-to-bottom → bottom-to-top flip.

Worked example — tile 13 (RUN_RIGHT frame 2), grid 4×6: column = 13 % 4 = 1 → offsetX = 1/4 = 0.25; rowFromTop = floor(13/4) = 3 → offsetY = (6 - 3 - 1)/6 = 2/6 ≈ 0.333. With repeat = (0.25, 0.1667) the sampled window is x ∈ [0.25, 0.5], y ∈ [0.333, 0.5] — the 4th row from the top, 2nd column. Exactly the tile shown in the sheet above. 

flowchart TD
    A["Update(deltaTime)"] --> B["elapsedTime += deltaTime"]
    B --> C{"elapsedTime >= timing * mult ?"}
    C -- no --> Z["hold current tile"]
    C -- yes --> D["elapsedTime = 0"]
    D --> E["runningTileIndex = (i+1) % indicies.length"]
    E --> F["currentTile = indicies[i]"]
    F --> G["offsetX = (tile % W) / W"]
    G --> H["offsetY = (V - floor(tile/W) - 1) / V"]
    H --> I["map.offset = (offsetX, offsetY)"]
Note. Only the texture offset changes per frame — never the geometry, the mesh, or the repeat. That is why this is cheap enough to run on every animated entity every frame: it is two float writes when a frame boundary is crossed, and nothing at all otherwise.

public Destroy()

Releases this animator's GPU resources. Called from Player.Destroy() as this.SpriteAnimator.Destroy() before the base GameObject teardown. 

if (this.Material.map) {
  this.Material.map.dispose();
}

this.Material.dispose();

Disposing Material.map (the texture) first, then the material, frees both the uploaded texture and the material program. Because each entity owns its own SpriteAnimator, each one disposes its own texture and material — the Renderer's dispose() only frees the GL context, not these, so without this method destroyed entities would leak their sheets.

Modifying this: to animate a new character, hand the constructor that sheet's texture and its real (tilesHorizontal, tilesVertical) — the offset math is fully driven by those two numbers, so no code changes are needed for a differently-shaped grid. To add new clips, add entries to the game's spriteAnimations.ts table (engine stays untouched). To make a clip play once and stop, give its final frame a timing of 0 so the gate freezes it. If you ever want frame indicies[0] to actually be the first shown frame, change the ChangeState pre-load from elapsedTime = timing[0] to elapsedTime = 0 — but expect a brief hold before the first advance. Keep all pixel-art filtering settings as-is; they are paired with the renderer's antialias: false.
