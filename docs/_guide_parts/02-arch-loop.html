The Frame Loop & Lifecycle

Everything that moves in FireGame is driven by a single heartbeat: Time calls requestAnimationFrame and emits a "tick" event; Experience.onTick turns that one event into a fixed-timestep accumulator that advances the simulation in constant 1/60 slices, then computes an interpolation alpha and runs a single per-frame render pass that smooths visuals between the last two sim states and renders once; and the game layer plugs its work in through two late-bound hooks (OnGameUpdate per sim-step, OnRenderUpdate per frame). This section traces that pass top to bottom — who drives it, the exact ordering and why it is exactly that order, how delta-time and resize are wired, and how the whole thing tears itself down symmetrically on Destroy.

src/engine/core/time.ts 45 LOC

Time is the engine's clock and the sole driver of the frame loop. It owns a Three.js THREE.Clock, computes the clamped real per-frame FrameDelta, exposes the constant fixed simulation step FixedTimestep (1/60), and on every animation frame emits a single global "tick" event through the Event Bus (../events/eventBus). It is constructed by Experience.Configure and knows nothing about who listens — the only coupling outward is the emitted event. Lifecycle: the constructor schedules the first frame; tick reschedules itself forever — even if a "tick" listener throws, the reschedule sits outside the try/catch so the loop never dies; Destroy cancels the pending frame to stop the loop dead.

public static readonly FixedTimestep = 1 / 60; public Clock; public Elapsed; public Delta; public FrameDelta;

The fields any subsystem reads off the singleton's Time. FixedTimestep is the constant simulation step (1/60) the integrators march at; Clock is the underlying monotonic timer; Elapsed is total seconds since start. The two deltas are deliberately split: Delta is the simulation timestep the integrators read (player/enemy/physics multiply into it) — the frame loop sets it equal to FixedTimestep before each fixed step, so all integration uses a constant dt regardless of display refresh rate, and it is never 0. FrameDelta is the real, clamped wall-clock time since the last rendered frame; it drives the fixed-step accumulator only and is never fed into integration. Two private fields back the math: previous (last frame's Elapsed, used to subtract a delta) and animationFrameId (the handle Destroy cancels).

constructor()

Builds the clock, seeds Elapsed from it, sets Delta to the fixed step (Time.FixedTimestep), zeroes FrameDelta, and schedules the first frame. Crucially it does not call tick() synchronously — it waits one animation frame so the first real delta has a non-zero window to subtract against. 

constructor() {
  this.Clock = new THREE.Clock();
  this.Elapsed = this.Clock.getElapsedTime();
  this.Delta = Time.FixedTimestep; // integrators default to the fixed step
  this.FrameDelta = 0;
  this.previous = 0;

  // Instead of calling tick() immediately, wait 1 frame for delta time subtraction
  this.animationFrameId = window.requestAnimationFrame(() => {
    this.tick();
  });
}

Line-level notes: this.Delta = Time.FixedTimestep is the seed value — never 0. Physics.Update feeds Time.Delta straight into World.timestep, and a zero timestep would freeze the simulation on frame one. Deferring the first tick() by a frame means previous (0) and the first Elapsed are measured a real frame apart, so the very first computed FrameDelta is meaningful rather than 0.

Hard-won victory. Delta seeds to Time.FixedTimestep (1/60), never 0 — Physics.Update feeds Time.Delta straight into World.timestep, so any consumer that reads Time.Delta before the first tick (e.g. an entity constructed during Configure) gets a usable 16.6ms instead of a frozen 0.

private tick(): void

The heartbeat. Refreshes Elapsed, computes the clamped FrameDelta, advances previous, emits the global "tick" event inside a try/catch, then re-arms itself with another requestAnimationFrame. This is the only place "tick" is ever emitted in the whole codebase. 

private tick() {
  this.Elapsed = this.Clock.getElapsedTime();
  // Clamp so a suspended tab doesn't produce a huge catch-up delta. This clamp
  // also bounds how many fixed steps a single frame can run — no spiral of death.
  this.FrameDelta = Math.min(this.Elapsed - this.previous, 1 / 30);
  this.previous = this.Elapsed;

  // A throw in any tick listener must NOT stop the loop: the next frame is
  // scheduled below regardless, so one transient bug can't black-screen the game.
  try {
    Emitter.emit("tick");
  } catch (error) {
    console.error("Frame loop error (loop continues):", error);
  }

  // Recursively keep calling tick
  this.animationFrameId = window.requestAnimationFrame(() => {
    this.tick();
  });
}

Line-level notes: Math.min(this.Elapsed - this.previous, 1 / 30) upper-bounds FrameDelta at 1/30s (~33ms). The browser suspends rAF on a backgrounded tab, so when the tab returns the raw Elapsed - previous could be many seconds — feeding that into the accumulator would try to run a huge burst of fixed steps. Clamping bounds FrameDelta (and so the per-frame step count) to one slow frame's worth instead — the first line of defense against a spiral of death, with Experience.maxStepsPerFrame as the backstop. The "tick" emit is wrapped in a try/catch and the next-frame requestAnimationFrame sits outside it, so a throwing listener logs an error but can never stop the loop.

Gotcha. The clamp means real elapsed wall-time and the simulated time drift apart after any stall. That is intentional: the simulation advances in fixed 1/60 steps off the accumulator, not wall-clock — which is exactly why GameDirector's enemy spawner counts in Time.Delta (the fixed step) and not a setInterval (see trySpawn).

public Destroy(): void

Cancels the pending requestAnimationFrame via the stored animationFrameId. Because tick only reschedules at its own tail, cancelling the one outstanding frame is sufficient to halt the loop permanently — no flag, no in-flight callback can resurrect it. 

public Destroy() {
  // Stop the animation frame loop
  window.cancelAnimationFrame(this.animationFrameId);
}

Modifying this: the fixed-timestep accumulator that decouples render rate from physics rate now lives in Experience.onTick; Time's job is just to hand it a clean, clamped FrameDelta and the constant FixedTimestep. Experience.onTick now computes that alpha (the leftover accumulator / dt fraction) right after the step loop and hands it to OnRenderUpdate, so the per-frame render pass interpolates between the last two sim states. Keep the 1/30 clamp and never let Delta reach 0; both are load-bearing for the physics step.

Sizes is Time's sibling for screen geometry: it captures the window dimensions and device pixel ratio, and emits a "resize" event whenever the window changes. Like Time, it is constructed in Experience.Configure and is decoupled from its consumers — it just announces "resize" on the bus and lets Experience.onResize fan it out to Camera and Renderer. Lifecycle: constructor reads the initial size and registers the DOM listener; Destroy removes it.

public Width: number; public Height: number; public PixelRatio: number;

The current viewport, refreshed on every resize. Renderer reads all three (setSize / setPixelRatio) and Camera.Resize reads Width/Height for its aspect ratio. The private field onResize holds the listener reference so Destroy can unregister the exact same function.

constructor()

Seeds the three fields from window, defines the resize handler (which re-reads the same three fields and emits "resize"), and registers it on the window's "resize" DOM event. 

constructor() {
  this.Width = window.innerWidth;
  this.Height = window.innerHeight;
  this.PixelRatio = Math.min(window.devicePixelRatio, 2);

  this.onResize = () => {
    this.Width = window.innerWidth;
    this.Height = window.innerHeight;
    this.PixelRatio = Math.min(window.devicePixelRatio, 2);

    Emitter.emit("resize");
  };

  window.addEventListener("resize", this.onResize);
}

Line-level notes: Math.min(window.devicePixelRatio, 2) caps the pixel ratio at 2. On a 3x or 4x display the renderer would otherwise allocate an enormous backbuffer for marginal visual gain — the cap keeps fill-rate sane. The handler is stored on this.onResize (not an inline arrow at addEventListener) for one reason: removeEventListener needs the identical reference later.

public Destroy(): void

Removes the DOM listener using the stored handler reference, ending the resize signal.

public Destroy() {
  window.removeEventListener("resize", this.onResize);
}
Note. Sizes emits "resize"; it does not touch the camera or renderer. Experience owns the wiring (onResize => Camera.Resize(); Renderer.Resize()). That is the same decoupling pattern as Time → "tick" → onTick: the core emitter never names its listeners.

Modifying this: if you add a fit-to-container mode (rendering into a sized div rather than full window), read from the target element here instead of window.innerWidth/Height, but keep emitting the same "resize" event — downstream Camera/Renderer code needs no change.

src/engine/core/experience.ts 122 LOC

Experience is the singleton that wires every engine subsystem together and defines the canonical per-frame order. It imports and owns Debug, Sizes, Time, Input, ResourceLoader, the THREE Scene, Camera, Renderer, and Physics — but imports nothing from game/. The game plugs in through one nullable hook, OnGameUpdate. Lifecycle is deliberately split: a private constructor (singleton guard), an async Configure that does the real construction and registers the "tick"/"resize" listeners, and a symmetric Destroy that unwinds everything in reverse.

private static instance: Experience; private constructor() {}

The singleton machinery. The constructor is private so no one can new Experience(); the static instance holds the one copy. Every subsystem that needs back-references (e.g. Renderer, Physics, every GameObject) calls Experience.GetInstance() rather than receiving the world by constructor injection — the singleton is the dependency-resolution mechanism.

public OnGameUpdate: (() => void) | null = null;

One of two decoupling seams between engine and game (its sibling is OnRenderUpdate, below). The engine calls this inside the fixed-step loop — once per sim-step, 0..N times a frame, right after Physics.Update(); the game assigns it in App.vue after constructing its GameDirector. Because it is a nullable function pointer, the engine core compiles and runs with zero knowledge of any game class. 

// Late-bound game update hook. The game layer assigns this; the engine just
// calls it each frame between physics and renderer. Keeps the engine core
// from naming any game class.
public OnGameUpdate: (() => void) | null = null;

For the concrete assignment, see App.vue: OnGameUpdate = () => { inputBindings.Update(); gameDirector.Update(); }. That is where the "InputBindings before GameDirector" ordering inside the game slice is actually decided.

public OnRenderUpdate: ((alpha: number) => void) | null = null;

The second seam, and the new one. Where OnGameUpdate runs per sim-step, OnRenderUpdate runs once per frame — after the whole fixed-step loop, before Renderer.Update() — and receives the interpolation alpha (accumulator / dt, in [0, 1)). The game uses it to lerp visuals between the last two sim states and to drive the camera, so rendering stays smooth above the sim rate. 

// Late-bound per-FRAME render hook, called once after the fixed-step sim loop

public async Configure(canvas: HTMLCanvasElement | null): Promise<void>

The "real constructor." It is async because Physics.Configure must await RAPIER.init() (the WASM physics module). Parameter canvas is the DOM target the renderer mounts into, captured from the Vue ref. Construction order matters: Sizes/Time/Input first (they have no engine deps), then the Scene, then Camera (the renderer reads it), then Renderer, then Physics (awaited). The two event listeners are registered last, after everything they touch exists. 

public async Configure(canvas: HTMLCanvasElement | null) {
  this.Debug = new Debug();
  this.Sizes = new Sizes();
  this.Time = new Time();
  this.Input = new Input();
  // ... ResourceLoader([randy, enemy, dkGraphicsData]) ...
  this.TargetElement = canvas;

  this.Scene = new THREE.Scene();
  this.Camera = new Camera(new THREE.Vector3(0, 25, 100));
  this.Renderer = new Renderer();
  this.Physics = new Physics();
  await this.Physics.Configure();

Constructing Time here is what actually starts the loop — new Time() schedules the first requestAnimationFrame. But note: at this moment onTick is not registered yet, so the first few ticks emitted before the listener is attached are harmlessly dropped. The Camera is seeded at (0, 25, 100); the ResourceLoader paths are intentionally empty (the real URLs come from a bundled ?url import map keyed by name — see ResourceLoader).

The back half of Configure registers the two engine event handlers, storing each on a field for cleanup:

// Sizes resize event — store ref for cleanup
this.onResize = () => {
  this.Camera.Resize();
  this.Renderer.Resize();
};
Emitter.on("resize", this.onResize);

// Time tick event — store ref for cleanup
this.onTick = () => {
  if (this.Debug.IsActive) {
    this.Debug.Stats?.begin();
  }

  // Fixed-timestep simulation. Accumulate real frame time and advance the sim
  // (physics + game update) in constant FixedTimestep steps, then render once.
  // This makes physics and movement frame-rate-independent and the sim
  // reproducible. FrameDelta is clamped in Time, so the per-frame step count is
  // bounded; maxStepsPerFrame is a final safety net.
  this.accumulator += this.Time.FrameDelta;
  const dt = Time.FixedTimestep;
  let steps = 0;
  while (this.accumulator >= dt && steps < this.maxStepsPerFrame) {
    this.Time.Delta = dt; // integrators read this as their timestep
    this.Physics.Update();
    this.OnGameUpdate?.();
    this.accumulator -= dt;
    steps++;
  }

  // Render once per frame, interpolated between the last two sim states by how
  // far we are into the next step (alpha in [0,1)). Buttery above the sim rate,
  // no judder below — render and sim rates are fully decoupled.
  const alpha = this.accumulator / dt;
  this.OnRenderUpdate?.(alpha);
  this.Renderer.Update();

  if (this.Debug.IsActive) {
    this.Debug.Stats?.end();
  }
};
Emitter.on("tick", this.onTick);

This onTick body is the frame: accumulate the real FrameDelta, run physics + game update 0..N times at a constant dt = 1/60, then compute the interpolation alpha and run the per-frame render pass (OnRenderUpdate + Renderer.Update) once. The exact order — and the reason for it — is the whole point of this section:

Gotcha. Inside each fixed step, physics → game is not negotiable, and the single Renderer.Update() comes after the whole step loop. If Renderer.Update() ran before OnGameUpdate, the screen would lag one frame behind simulation. If physics ran after the game, the player's CharacterController would move against a stale world and contact dispatch would be a step late.
Hard-won victory — fixed timestep. Banking real time in the accumulator and stepping the sim in constant 1/60 slices makes physics and movement frame-rate-independent and reproducible: a 144Hz display and a 60Hz display run the identical sequence of sim steps, just rendered at different cadences. The FrameDelta clamp in Time plus maxStepsPerFrame (5) together bound the catch-up work, so a long stall can never spiral into an ever-growing step debt.
Hard-won victory — render interpolation. The sim still steps at a fixed 1/60, but the screen no longer snaps to those steps. After the loop, alpha = accumulator / dt measures how far we are into the next step, and the per-frame OnRenderUpdate(alpha) lerps each entity's mesh between its last two sim states. Render and sim rates are now fully decoupled — buttery on a 144Hz display, no judder when the display outruns the sim — without touching the deterministic fixed-step physics underneath.
Hard-won victory. Every Stats, OnGameUpdate, and OnRenderUpdate call uses optional chaining (?.). The engine genuinely runs with no debug panel and no game attached — that is what lets the engine/ layer be reused on its own. The handlers are stored on this.onResize/this.onTick precisely so Destroy can pass the same references to Emitter.off — an inline arrow could never be unregistered.

public Destroy(): void

The symmetric teardown. It unwinds in a deliberate order: first remove Experience's own "resize"/"tick" listeners (so no late tick can touch half-freed subsystems), then stop the signal-producers (Sizes, Time, Input), then free the heavy resources (Camera, Physics, Renderer, Resources), then the debug panel. 

public Destroy() {
  // Remove our own event listeners first
  Emitter.off("resize", this.onResize);
  Emitter.off("tick", this.onTick);

  // Clear subsystem event listeners
  this.Sizes.Destroy();
  this.Time.Destroy();
  this.Input.Destroy();

  // Camera then physics then renderer then resources
  this.Camera.Destroy();
  this.Physics.Destroy();
  this.Renderer.Destroy();
  this.Resources.Destroy();

  // Debug menu
  if (this.Debug && this.Debug.IsActive) {
    this.Debug.Destroy();
  }
}

Line-level notes: removing onTick before Time.Destroy() is belt-and-braces — even if one more frame is in flight, the handler is already detached, and then Time.Destroy() cancels the pending rAF so no further ticks are emitted at all. Physics.Destroy() calls World.free() to release the Rapier WASM world; Renderer.Destroy() disposes the WebGL context. The order matches construction reversed, which is why Camera (the renderer's dependency) is torn down alongside but the renderer's GL dispose comes after physics frees its debug wireframe.

Note. GameDirector has its own Destroy (off-ing the game events, clearing the cleanup interval, unloading the level). Experience.Destroy does not call it — the two teardown chains are parallel, mirroring the two halves of the event map (EngineEventMap vs GameEventMap in Event Bus).

Modifying this: to add an engine subsystem, construct it in Configure (after its dependencies), call it from the right slot inside onTick, and free it in Destroy in reverse. Do not import anything from game/ here — if the game needs new per-frame work, route it through OnGameUpdate, not a new field on Experience. Adding a member to the tick? Decide its slot relative to physics (reads stale or fresh world?) and the renderer (must it be visible this frame?).

GameDirector is the game layer's central controller: it owns the player and all entity arrays, drives level load/unload, runs the frame-driven enemy spawner, and — most relevant to this section — defines the internal order of game logic across two passes: Update (the SIM pass, run per fixed step) and RenderUpdate (the per-frame pass that interpolates visuals and drives the camera). public Update(): void

The game's SIM pass, called from OnGameUpdate right after InputBindings.Update() — so it runs per fixed step, 0..N times a frame. Its internal order is as load-bearing as Experience.onTick's. After two guards, it runs: player → spawn → enemies → platforms → ladder. The camera and visual interpolation are no longer here — they moved to RenderUpdate (the per-frame pass). The ordering comments in the source are the real rationale, quoted below. 

// SIM update — runs inside the fixed-timestep loop (0..N times per frame), so all
// movement integrates at the constant step. Camera + visual interpolation are NOT
// here; they run once per FRAME in RenderUpdate, decoupled from the sim rate.
public Update() {
  // Freeze game logic while a level is loading ...
  if (this.isLoading) return;

  // Guard: Player must be ready
  if (!this.Player?.PhysicsBody) return;

  this.Player.Update();

  // Spawn + update enemies (runtime-spawned, not level-data), driven by the frame
  // clock here. Compact destroyed barrels in place right after — promptly, and
  // keeping the array reference stable (was a 5s wall-clock sweep).
  this.trySpawn();
  const firstTrashCan = this.TrashCans[0];
  this.Enemies.forEach((enemy) => enemy.Update(this.Player, firstTrashCan));
  GameUtils.CompactDestroyedObjects(this.Enemies);

  // Update every level-data entity that has an update, in one uniform loop.
  // Runs AFTER player.update() so the platform one-way toggle reads this
  // frame's player state. ...
  const updatableLevelEntities: UpdatableEntity[] = this.Platforms;
  for (const e of updatableLevelEntities) {
    e.Update?.(this.Player, firstTrashCan);
  }

  // Update ladder detection (manual for now, will convert to sensor callbacks later)
  this.Player.UpdateLadderState(
    GameUtils.IsAnySensorTriggered(this.LadderTopSensors),
    GameUtils.IsAnySensorTriggeredObjectFullyInside(this.LadderCoreSensors, this.Player),
    GameUtils.IsAnySensorTriggered(this.LadderBottomSensors)
  );
}

public RenderUpdate(alpha: number): void

The per-frame pass, called from OnRenderUpdate once after the whole fixed-step loop with the interpolation alpha. Same two guards as Update, then it lerps the moving entities between their last two sim states and points the camera at the interpolated player. This is what decouples render rate from sim rate: the sim steps at a fixed 1/60, but the visuals are resampled every frame. 

public RenderUpdate(alpha: number) {
  if (this.isLoading) return;
  if (!this.Player?.PhysicsBody) return;

  this.Player.InterpolateGraphics(alpha);
  for (const enemy of this.Enemies) {
    enemy.InterpolateGraphics(alpha);
  }

  // Camera follows the interpolated visual position (C5 decouple: plain snapshot).
  this.camera.Update(this.cameraTargetFromPlayer(this.Player));
}

Order matters here too: InterpolateGraphics writes each entity's RenderTranslation before camera.Update reads it through cameraTargetFromPlayer, so the camera tracks the smooth visual player, not the stepped sim body.

The C5 mapping function. It translates the player's FSM State into the engine's CameraFollowState enum and packs only the scalars the camera needs. Return: a plain CameraTarget struct — no game types cross the boundary. 

return {

  // not the stepped sim position.
  x: player.RenderTranslation.x,
  y: player.RenderTranslation.y,
  velocityX: player.NextTranslation.x,
  followState,
  isGrounded: player.IsTouching.ground,
};

x/y read the player's RenderTranslation — the interpolated visual position written by InterpolateGraphics in RenderUpdate moments earlier — so the camera tracks the smooth visual player, not the stepped sim body. velocityX reads NextTranslation.x (the player's intended horizontal velocity) and isGrounded from the player's shapecast contact flags, both from the latest sim step.

private trySpawn(): void

Frame-driven enemy spawning, called once per frame from Update. It replaces an old wall-clock setInterval with a single clock: Time.Delta. It bails immediately if spawning is off or physics is paused, then accumulates timeSinceLastSpawn and fires when the interval elapses. 

private trySpawn() {
  if (!this.isSpawningEnemies || this.experience.Physics.IsPaused) return;

  this.timeSinceLastSpawn += this.time.Delta;

  // First barrel: Kong's opening throw — a crazy barrel aimed at the oil can.
  if (this.currentSpawnInterval === 0) {
    this.timeSinceLastSpawn = 0;
    this.currentSpawnInterval = Math.random() * (4 - 3) + 3; // 3-4 seconds
    this.spawnOilCanBarrel();
    return;
  }

  // Subsequent barrels: mostly normal, every 8th crazy.
  if (this.timeSinceLastSpawn >= this.currentSpawnInterval) {
    this.timeSinceLastSpawn = 0;
    this.currentSpawnInterval = Math.random() * (3 - 2) + 2; // 2-3 seconds
    this.enemyCount++;
    if (this.enemyCount % 8 === 0) {
      this.SpawnCrazyEnemy();
    } else {
      this.SpawnEnemy();
    }
  }
}
Hard-won victory — one clock, not two. The header comment: "Replaces the old wall-clock setInterval: one clock now (Time.Delta), and a fresh boot spawns without needing a reset. ... The cadence counters are advanced BEFORE the (async) spawn so a stutter frame can't double-spawn." Because timeSinceLastSpawn is reset to 0 before the awaited spawn call, a long frame can't satisfy the interval twice. And because cadence rides the same clamped Time.Delta as everything else, pausing physics (which freezes Delta consumption via the IsPaused guard) cleanly halts spawning too.
Note (D28). Normal and crazy barrels are one unified Enemy type now, distinguished only by their starting FSM state — "D28 fold-in of the old CrazyEnemy." The director keeps a single Enemies: Enemy[] array; SpawnEnemy vs SpawnCrazyEnemy differ only in the constructor flags.

public Destroy(): void

The game-side half of the symmetric teardown. It offs every game-flow listener it registered, clears the periodic cleanup interval, and unloads the current level. This runs independently of Experience.Destroy. 

public Destroy() {
  Emitter.off("resourcesReady");
  Emitter.off("gameStart");
  Emitter.off("gameOver");
  Emitter.off("gameReset");
  Emitter.off("switchLevel");
  Emitter.off("gameObjectRemoved");

  clearInterval(this.cleanupIntervalId);
  this.UnloadLevelData();
}
Gotcha. The setInterval cleanup (every 5s, prunes destroyed enemies) is a wall-clock timer — the one place the game still uses real time rather than Time.Delta. It deliberately keeps running after gameOver (so corpses get reaped on the freeze screen) and is only stopped here in Destroy via clearInterval. Forget this and you leak a timer holding the whole director alive.

Modifying this: when adding per-frame game logic, slot it into Update with the same question Experience asks — does it need this frame's player state (put it after Player.Update()) or last frame's? Anything time-based should count in this.time.Delta, never setInterval, so it honors pause and the 1/30 stall clamp. Register new game-flow listeners in setupEventHandlers and mirror them with an Emitter.off in Destroy.

Lifecycle: Configure → running → Destroy

Stepping back from per-frame detail, the whole app moves through three phases. App.vue's onMounted registers loading listeners, calls Experience.Configure (which starts Time's loop and registers onTick/onResize), wires the game's contact rules, debug sinks, GameDirector, InputBindings, and the OnGameUpdate / OnRenderUpdate hooks, then kicks off asset loading last. From there the frame loop runs until teardown. 

flowchart TD
    A["App.vue onMounted"] --> B["Experience.GetInstance()"]
    B --> C["await Experience.Configure(canvas)"]
    C --> C1["new Time() starts rAF loop"]
    C --> C2["Emitter.on tick / resize"]
    C --> D["registerContactRules + debug sinks"]
    D --> E["new GameDirector / InputBindings"]
    E --> F["OnGameUpdate = InputBindings.Update then GameDirector.Update; OnRenderUpdate = GameDirector.RenderUpdate"]
    F --> G["Resources.StartLoading() (last)"]
    G --> H["resourcesReady -> loadLevelByIndex -> gameStart"]
    H --> LOOP["running: each frame = accumulate FrameDelta, 0..N fixed 1/60 SIM steps (Physics + OnGameUpdate), then alpha + OnRenderUpdate (interpolate + camera), Renderer once"]
    LOOP --> LOOP
    LOOP --> Z["Experience.Destroy + GameDirector.Destroy"]
    Z --> Z1["off tick/resize, Time.Destroy cancels rAF"]
    Z --> Z2["free Physics world, dispose Renderer, clear interval"]
Tip. The ordering inside onMounted is itself a hard-won detail: loading listeners are attached before Configure, and Resources.StartLoading() is called last, so a fast boot can't fire resourcesReady/loadingFinished before anyone is listening.
