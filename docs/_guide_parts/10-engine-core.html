Engine Core — Experience, Time, Sizes

The beating heart of the engine. Experience is a lazy singleton that owns and wires every engine subsystem; Time drives the single requestAnimationFrame loop and emits "tick"; Sizes tracks the viewport and emits "resize". Nothing here names a single game concept — the game plugs itself in through two late-bound hooks, OnGameUpdate (per sim step) and OnRenderUpdate (per rendered frame), and the global event bus.

What it is: the engine's composition root — a lazily-constructed singleton that instantiates, holds, and tears down every engine subsystem (Debug, Sizes, Time, Input, ResourceLoader, Scene, Camera, Renderer, Physics). Who constructs it: nobody, directly — the private constructor is blocked; the Vue shell (src/App.vue → onMounted) calls Experience.GetInstance() then await Configure(canvas, GAME_ASSETS) — handing the engine the game's asset manifest rather than letting the engine name those assets itself. Why these imports: it imports each subsystem class because it is the one place that news them up and assembles the dependency graph; it imports Emitter (see eventBus, Engine Layer) only to subscribe its two per-frame/per-resize closures. Lifecycle: GetInstance() → Configure() (the real, async constructor) → frames flow via the "tick" subscription → Destroy() unwinds everything in a deliberate order.

Note. The header comment sums it up: "Overall handler that mounts a webgl render to a dom canvas element." Every other engine file reaches back here via Experience.GetInstance() to find the shared Scene, Sizes, Time, etc. That back-reference is the reason this is a singleton rather than a passed-around object.

Hard-won victory — the OnGameUpdate seam. This single nullable field is the one thread the engine pulls to run the game, without ever importing it: 
// Late-bound game update hook. The game layer assigns this; the engine just
// calls it each frame between physics and renderer. Keeps the engine core
// from naming any game class.
public OnGameUpdate: (() => void) | null = null;
The engine ships knowing nothing about GameDirector, InputBindings, or Donkey Kong. The game assigns OnGameUpdate in App.vue, and the tick loop calls it with optional-chaining (this.OnGameUpdate?.()) so a frame that fires before the game is wired simply does nothing instead of throwing. This is the central decoupling seam of the whole codebase.
Hard-won victory — the OnRenderUpdate sibling seam. A second late-bound field lets the engine render smoothly above the sim rate without naming the game: 
// Late-bound per-FRAME render hook, called once after the fixed-step sim loop
// with the interpolation alpha (accumulator / fixed step). The game uses it to
// interpolate visuals and drive the camera, so rendering stays smooth above the
// sim rate. Stays engine-pure — names no game class.
public OnRenderUpdate: ((alpha: number) => void) | null = null;
Where OnGameUpdate fires per fixed sim step (0..N times a frame), OnRenderUpdate fires exactly once per rendered frame, after the loop, with an alpha in [0,1) — how far the accumulator has progressed into the next step. The game uses it to lerp meshes between the last two physics states and to drive the camera off that interpolated position, so a 144 Hz display stays buttery while the sim still ticks at a constant 1/60. The same optional-chaining (this.OnRenderUpdate?.(alpha)) means a frame that fires before the game is wired no-ops instead of throwing.
Gotcha — why the handler refs exist. 
// Stored event handler references for cleanup
private onResize!: () => void;
private onTick!: () => void;
mitt's off() removes a listener by identity. If you subscribe an inline arrow and later try to Emitter.off("tick", () => {...}) with a fresh arrow, you remove nothing — different function object. So the closures are captured into onResize/onTick fields at subscribe time and the exact same references are passed to off() in Destroy(). Forgetting this leaks a RAF-driven tick handler that keeps stepping physics on a torn-down world. The same pattern recurs in Sizes and in Camera (onManualCameraControl).
Note — fixed-timestep accumulator state. Two private fields back the per-frame loop: 
// Fixed-timestep accumulator state (see onTick): carries leftover real time
// between frames; maxStepsPerFrame caps catch-up so a long stall can't freeze.
private accumulator = 0;
private maxStepsPerFrame = 5;
accumulator holds real time not yet consumed by a fixed step; maxStepsPerFrame bounds how many 1/60 steps a single frame may run. Both are read only inside onTick (covered below).

public static GetInstance(): Experience

Purpose: the only sanctioned way to obtain the Experience. Parameters: none. Returns: the one-and-only instance, creating it on first call (lazy init). 

public static GetInstance(): Experience {
  if (!Experience.instance) {
    Experience.instance = new Experience();

  return Experience.instance;
}

The classic lazy singleton. Because the constructor is empty, GetInstance() is cheap and side-effect-free — all the heavy assembly is deferred to Configure(). That split matters: subsystems like Renderer and Physics call GetInstance() in their constructors to read shared state, and those constructors run during Configure() — so the instance must already exist (it does, since App.vue calls GetInstance() first).

public async Configure(canvas: HTMLCanvasElement | null, resources: Resource[]): Promise<void>

Purpose: the real constructor. It exists as a separate async method because one subsystem — Physics — must await RAPIER.init() (a WASM load), and JavaScript constructors cannot be async. Parameters: canvas — the DOM canvas element from the Vue ref (may be null; it is stored as-is and the Renderer casts it); resources — the game's boot asset manifest (a Resource[]), injected by the caller so the engine names none of the game's assets. Returns: a promise that resolves once Rapier's WASM is initialized and both engine event handlers are subscribed.

The body opens with an idempotency guard — a second Configure() (an HMR reload, a restart) is warned-and-ignored rather than overwriting live subsystems and leaking the previous physics world:

if (this.isConfigured) {
  console.warn("Experience.Configure() ignored — already configured. Call Destroy() first.");
  return;
}
this.isConfigured = true;

The isConfigured flag is set true here and reset to false in Destroy(), so the only sanctioned way to re-run Configure() is to tear the engine down first. Then the body assembles the graph in a strict order. First the dependency-free leaves: 

this.Debug = new Debug();
this.Sizes = new Sizes();
this.Time = new Time();
this.Input = new Input();

Debug goes first because later subsystems (Physics, Camera) check this.experience.Debug.IsActive in their constructors to decide whether to wire debug overlays. Sizes and Time are pure leaves — they only touch window and the event bus.

Then the resource manifest — the one the caller passed in:

this.Resources = new ResourceLoader(resources);

The engine no longer hardcodes a randy/enemy/dkGraphicsData list here; it just forwards the injected resources array to the ResourceLoader. The actual manifest lives game-side in src/game/config/assetManifest.ts (exported as GAME_ASSETS), keeping the engine asset-name-blind. Note Configure builds the ResourceLoader but does not start loading; App.vue calls Resources.StartLoading() last, after every listener is wired.

Note — reusability seam. Because the manifest is a parameter rather than a literal, a different game can drive this same engine just by passing its own Resource[]. The Resource type is exported from resourceLoader.ts and imported here precisely so the signature can name it without naming any asset.

Then the rendering/physics quartet — order is load-bearing here:

this.Scene = new THREE.Scene();
this.Camera = new Camera(new THREE.Vector3(0, 25, 100));
this.Renderer = new Renderer();
this.Physics = new Physics();
await this.Physics.Configure();

Line-level ordering notes (each later line depends on an earlier one):

  • TargetElement must be set before new Renderer(), because the Renderer constructor reads this.experience.TargetElement to build its WebGLRenderer({ canvas }).
  • Scene must exist before new Camera(...): the Camera constructor does this.scene.add(this.PerspectiveCamera).
  • Camera must exist before new Renderer(): the Renderer caches this.camera = this.experience.Camera and renders through this.camera.Instance.
  • The camera's new THREE.Vector3(0, 25, 100) is the engine's default initial position; the game's level data later teleports it via Camera.TeleportToPosition.
Hard-won victory — why Physics.Configure() is awaited. Rapier2D ships as WebAssembly; Physics.Configure() does await RAPIER.init() before it can build new RAPIER.World(...) (see physics.ts). If Experience.Configure did not await it, control would return to App.vue and the first "tick" could fire this.Physics.Update() against an undefined World — an instant crash. Awaiting here is what lets the rest of the boot (registering contact rules, constructing GameDirector) safely assume a live physics world. It is also why Configure itself had to become async.

Finally, the two engine event subscriptions — the closures captured into the stored refs:

// Sizes resize event — store ref for cleanup
this.onResize = () => {
  this.Camera.Resize();
  this.Renderer.Resize();
};
Emitter.on("resize", this.onResize);

// Time tick event — store ref for cleanup
this.onTick = () => {
  if (this.Debug.IsActive) {
    this.Debug.Stats?.begin();
  }

  // Fixed-timestep simulation. Accumulate real frame time and advance the sim
  // (physics + game update) in constant FixedTimestep steps, then render once.
  // This makes physics and movement frame-rate-independent and the sim
  // reproducible. FrameDelta is clamped in Time, so the per-frame step count is
  // bounded; maxStepsPerFrame is a final safety net.
  this.accumulator += this.Time.FrameDelta;
  const dt = Time.FixedTimestep;
  let steps = 0;
  while (this.accumulator >= dt && steps < this.maxStepsPerFrame) {
    this.Time.Delta = dt; // integrators read this as their timestep
    this.Physics.Update();
    this.OnGameUpdate?.();
    this.accumulator -= dt;
    steps++;
  }

  // Render once per frame, interpolated between the last two sim states by how
  // far we are into the next step (alpha in [0,1)). Buttery above the sim rate,
  // no judder below — render and sim rates are fully decoupled.
  const alpha = this.accumulator / dt;
  this.OnRenderUpdate?.(alpha);
  this.Renderer.Update();

  if (this.Debug.IsActive) {
    this.Debug.Stats?.end();
  }
};
Emitter.on("tick", this.onTick);

The onTick body is the engine's per-frame pipeline — now a fixed-timestep accumulator rather than a single pass. Each frame:

  1. Debug.Stats?.begin() — opens the stats.js FPS/MS sample window (optional-chained; Stats only exists when debug is active).
  2. accumulator += Time.FrameDelta — banks the real, clamped wall-clock time elapsed since the last frame (see time.ts).
  3. The while loop — while at least one fixed step's worth of time is banked (and we're under maxStepsPerFrame), it sets Time.Delta = FixedTimestep (the constant 1/60 the integrators read), runs Physics.Update() then OnGameUpdate?.(), and drains one dt from the accumulator. So physics + game logic run 0..N times per rendered frame, always at a constant step — the sim is frame-rate-independent and reproducible.
  4. const alpha = accumulator / dt then OnRenderUpdate?.(alpha) — once per frame, after the loop drains, it computes the interpolation factor (how far the leftover accumulator has progressed into the next fixed step, in [0,1)) and hands it to the per-frame render hook. The game uses alpha to lerp meshes between the last two physics states and to drive the camera off that interpolated position — smooth motion above the sim rate, no judder below.
  5. Renderer.Update() — draws Scene through Camera.Instance once per frame, after the sim has caught up.
  6. Debug.Stats?.end() — closes the sample window.
Hard-won victory — fixed timestep, variable render. Stepping physics by a constant 1/60 instead of the raw frame delta means a 144 Hz monitor and a 30 Hz one simulate identically — jump arcs, gravity, and contact resolution no longer drift with refresh rate, and a recorded input sequence replays deterministically. The render still runs once per frame, so high-refresh displays stay smooth. The two new private fields carry the state: accumulator (leftover real time between frames) and maxStepsPerFrame = 5 (a cap that prevents a "spiral of death" if a frame stalls — the sim falls behind in slow motion instead of locking up trying to catch up).
Gotcha — physics → game → render is non-negotiable. Within each step, physics runs before the game update, and the render happens only after the loop. Reorder these and you get a one-frame lag (rendering before the game moves its meshes) or stale collisions (game logic reading last frame's contacts). The pipeline reads as: simulate the world in fixed steps, let the game react each step, then paint once.

public Destroy(): void

Purpose: unwind everything Configure() built, in an order that avoids touching a torn-down subsystem. Parameters: none. Returns: void. 

public Destroy() {
  // Remove our own event listeners first
  Emitter.off("resize", this.onResize);
  Emitter.off("tick", this.onTick);

  // Optional chaining so a Destroy after a partially-failed Configure (e.g. the
  // physics WASM init threw mid-init) tears down whatever exists instead of
  // cascading on the first missing subsystem.
  this.Sizes?.Destroy();
  this.Time?.Destroy();
  this.Input?.Destroy();

  // Camera then physics then renderer then resources
  this.Camera?.Destroy();
  this.Physics?.Destroy();
  this.Renderer?.Destroy();
  this.Resources?.Destroy();

  // Debug menu
  if (this.Debug && this.Debug.IsActive) {
    this.Debug.Destroy();
  }

  // Allow a fresh GetInstance()/Configure() after teardown (HMR, tests, restart).
  this.isConfigured = false;
  Experience.instance = undefined;
}

Line-level teardown order (the comments narrate the intent):

  • Own listeners first — off("resize") and off("tick") with the stored refs. This stops the pipeline before any subsystem it calls is freed, so no half-destroyed Physics/Renderer gets a final tick.
  • Time.Destroy() cancels the RAF (see time.ts) so no further "tick" is even emitted. Sizes.Destroy() removes the window resize listener.
  • Camera → Physics → Renderer → Resources, each optionally-chained (this.Camera?.Destroy()). Camera unsubscribes its manualCameraControl handler and removes itself from the scene; Physics frees the Rapier World (itself a guarded World?.free()); Renderer disposes the WebGL context; Resources releases GPU textures/geometries. The ?. matters because a Destroy() after a partially-failed Configure() (say the physics WASM init threw mid-boot) should tear down whatever did get built instead of cascading on the first missing subsystem.
  • Debug last, guarded by this.Debug && this.Debug.IsActive — its Destroy() tears down dat.GUI + the stats.js DOM node, which exist only when debug was active.
  • Reset the singleton last of all — isConfigured = false and Experience.instance = undefined, so the next GetInstance() news up a clean instance and Configure() can run again (HMR, tests, a restart) instead of reviving a gutted object.
Hard-won victory — a clean restart. Destroy() now resets the singleton: isConfigured = false and Experience.instance = undefined. A later GetInstance() therefore builds a fresh instance and Configure() runs cleanly again — exactly what HMR, tests, and a full restart need. Paired with the isConfigured guard at the top of Configure(), the contract is symmetric: you cannot double-configure a live engine, and you cannot get back a gutted one after teardown. (Level switching still goes through GameDirector, not here — this remains a full teardown, not a soft reset.)

Modifying this: to add a new engine subsystem, (1) declare a public Foo!: Foo field, (2) new it in Configure() at the right spot in the ordering — after anything it reads off the singleton in its constructor and before anything that reads it — and (3) call this.Foo.Destroy() in Destroy() in reverse-dependency order. If the subsystem subscribes to bus events, store the handler in a private field and off() it in its own Destroy() — never inline the arrow. Do not reach for a game class here; keep the engine name-blind by routing game logic through OnGameUpdate and the event bus.

src/engine/core/time.ts 64 LOC

What it is: the engine's single heartbeat. It wraps a THREE.Clock, owns the one and only requestAnimationFrame loop, computes a clamped per-frame FrameDelta (real wall-clock time), and emits the global "tick" event each frame. Who owns it: Experience.Configure() news up a single Time. Why these imports: THREE only for THREE.Clock (a tiny high-resolution elapsed-time helper); Emitter to broadcast "tick" so the engine never needs a reference to its listeners. Lifecycle: constructor schedules the first frame → tick() recurses every frame emitting "tick" → Destroy() cancels the RAF.

Note — the Delta split. Time now exposes two distinct durations. FrameDelta is the real, clamped time per rendered frame and drives Experience.onTick's accumulator. Delta is the simulation timestep the integrators read — the frame loop sets it to the constant FixedTimestep (1/60) before each fixed step. They are different numbers doing different jobs: FrameDelta measures wall time; Delta is the fixed step size physics integrates by.

Fields

// The fixed simulation timestep. The frame loop advances the sim in constant
// steps of this size so physics + gameplay feel are frame-rate-independent and
// reproducible (see Experience.onTick).

public Clock: THREE.Clock;
public Elapsed: number;

// loop sets this to FixedTimestep before each fixed step, so all integration uses
// a constant dt regardless of display refresh rate. Never 0.
public Delta: number;
// The REAL wall-clock time since the last rendered frame (clamped). Drives the
// fixed-step accumulator only; NOT used for integration.
public FrameDelta: number;

  • FixedTimestep — the constant 1/60s simulation step, static readonly so the frame loop and any integrator can reference the one canonical value. This is the dt physics is stepped by.
  • Elapsed — total seconds since the clock started (monotonic).
  • Delta — the simulation timestep; the frame loop assigns it FixedTimestep before each fixed step, and physics + every integration calc downstream reads this. Never 0.
  • FrameDelta — the real, clamped seconds since the previous rendered frame; it feeds the accumulator in Experience.onTick and is not used for integration.
  • previous — last frame's Elapsed, the subtrahend for FrameDelta.
  • animationFrameId — the handle returned by requestAnimationFrame, kept so Destroy() can cancel the exact pending frame.

constructor()

Purpose: seed the clock and schedule the first frame — but not run a tick yet.

constructor() {
  this.Clock = new THREE.Clock();
  this.Elapsed = this.Clock.getElapsedTime();
  this.Delta = Time.FixedTimestep; // integrators default to the fixed step
  this.FrameDelta = 0;
  this.previous = 0;

  // Instead of calling tick() immediately, wait 1 frame for delta time subtraction
  this.animationFrameId = window.requestAnimationFrame(() => {
    this.tick();
  });
}
Hard-won victory — seed Delta = FixedTimestep, never 0. The integrators read Delta as their timestep, so it must be a sane non-zero value from the very first step — even before the accumulator loop has assigned it. Seeding it to the constant 1/60 (Time.FixedTimestep) guarantees a zero timestep can never reach physics. FrameDelta starts at 0 because there is no prior frame to measure against yet; the first real tick() fills it in.
Gotcha — the deliberate one-frame delay. The constructor schedules tick() via RAF rather than calling it inline. If it ran tick() immediately, previous would still be 0 while Elapsed is already a few milliseconds, producing a wrong first delta. Waiting one frame lets the clock advance so the first computed FrameDelta is a real inter-frame duration.

private tick(): void

Purpose: the per-frame core — refresh Elapsed, compute the clamped real FrameDelta, emit "tick", and re-arm the next frame. Parameters: none. Returns: void (drives side effects via the event bus and RAF). 

private tick() {
  this.Elapsed = this.Clock.getElapsedTime();
  // Clamp so a suspended tab doesn't produce a huge catch-up delta. This clamp
  // also bounds how many fixed steps a single frame can run — no spiral of death.
  this.FrameDelta = Math.min(this.Elapsed - this.previous, 1 / 30);
  this.previous = this.Elapsed;

  // A throw in any tick listener must NOT stop the loop: the next frame is
  // scheduled below regardless, so one transient bug can't black-screen the game.
  try {
    Emitter.emit("tick");
  } catch (error) {
    console.error("Frame loop error (loop continues):", error);
  }

  // Recursively keep calling tick
  this.animationFrameId = window.requestAnimationFrame(() => {
    this.tick();
  });
}

Line-level notes:

  • Elapsed = Clock.getElapsedTime() reads the monotonic clock first.
  • FrameDelta = Math.min(Elapsed - previous, 1/30) — the raw inter-frame wall time, clamped to a maximum of 1/30s (~33ms). Note this writes FrameDelta, not Delta — Delta is owned by the accumulator loop in Experience.onTick.
  • previous = Elapsed advances the baseline for next frame's subtraction.
  • Emitter.emit("tick") — the broadcast, now wrapped in try/catch. Its single engine subscriber is Experience.onTick; the game never subscribes to "tick" directly — it rides the OnGameUpdate hook inside that handler. This is what keeps the frame pipeline in one ordered place.
  • The trailing requestAnimationFrame re-arms the loop and overwrites animationFrameId with the new pending handle. Critically it sits outside the try, so the next frame is always scheduled.
Hard-won victory — clamp FrameDelta to 1/30. The comment: "a suspended tab doesn't produce a huge catch-up delta." A backgrounded tab pauses RAF; on return, Elapsed - previous could be many seconds. Since FrameDelta feeds the fixed-step accumulator, an unclamped value would queue hundreds of physics steps in one frame — a "spiral of death." Capping it bounds the per-frame step count (and onTick's maxStepsPerFrame is the final backstop), so the worst case is a brief slow-motion catch-up, never a freeze or physics explosion.
Hard-won victory — the loop survives a throwing listener. The emit("tick") is wrapped in try/catch but the re-arming requestAnimationFrame is deliberately outside it. Before this, a single exception thrown deep inside a tick subscriber (say, a game-logic bug) would propagate out of tick() and the loop would never reschedule — the screen froze. Now the error is logged and the next frame still fires, so one transient bug degrades gracefully instead of black-screening the game.

public Destroy(): void

Purpose: stop the heartbeat. Parameters: none. Returns: void.

public Destroy() {
  // Stop the animation frame loop
  window.cancelAnimationFrame(this.animationFrameId);
}

Cancels the pending frame by its stored handle. Because tick() reassigns animationFrameId every frame, this always targets the next scheduled callback — so the recursion stops cleanly and no further "tick" is emitted. Called by Experience.Destroy() early, so the pipeline halts before any subsystem is freed.

Modifying this: the fixed-timestep accumulator already lives downstream in Experience.onTick — Time's job is just to emit one "tick" per rendered frame with an honest, clamped FrameDelta. Keep the FrameDelta clamp (it bounds the per-frame step count) and keep Delta seeded to a non-zero value so an integrator that reads it before the first fixed step can't divide by or step by 0. Leave the reschedule outside the try/catch so a throwing listener can never stop the loop. If you change the emit name, update the lone subscriber in Experience.Configure and the event map in eventBus.

src/engine/core/sizes.ts 33 LOC

What it is: the viewport tracker. It captures window width/height and a capped device pixel ratio, listens for the DOM resize event, and re-broadcasts it as the engine "resize" event. Who owns it: Experience.Configure(). Why the one import: Emitter — the only thing it talks to besides window; it never reaches into Camera or Renderer, it just announces "the viewport changed" and lets them react. Lifecycle: constructor reads initial size + subscribes the DOM listener → handler updates fields and emits "resize" → Destroy() removes the DOM listener.

Fields

public Width: number;
public Height: number;
public PixelRatio: number;

private onResize: () => void; // stored handler so we can remove it later

Width/Height are read by Renderer (setSize) and Camera (aspect ratio). onResize is the stored handler ref — same off()-by-identity reasoning as in Experience, but here it's removeEventListener rather than Emitter.off.

constructor()

Purpose: snapshot the current viewport and wire the resize listener.

constructor() {
  this.Width = window.innerWidth;
  this.Height = window.innerHeight;
  this.PixelRatio = Math.min(window.devicePixelRatio, 2);

  this.onResize = () => {
    this.Width = window.innerWidth;
    this.Height = window.innerHeight;
    this.PixelRatio = Math.min(window.devicePixelRatio, 2);

    Emitter.emit("resize");
  };

  window.addEventListener("resize", this.onResize);
}

Line-level notes:

  • PixelRatio = Math.min(window.devicePixelRatio, 2) — capped at 2. On a 3x phone or Retina display, rendering at full DPR triples the pixel count for negligible visual gain (and this is a pixel-art game where antialiasing is off anyway). Capping at 2 protects the fill-rate budget. The cap is applied both at init and in the handler, so dragging a window between monitors of different DPR is handled.
  • The handler updates all three fields then Emitter.emit("resize") — so subscribers always read fresh numbers. The sole subscriber is Experience.onResize, which calls Camera.Resize() then Renderer.Resize() (camera recomputes its projection matrix before the renderer resizes its buffer).
Note — fan-out via the bus, not direct calls. Sizes could call Camera.Resize() itself, but it doesn't — it emits. That keeps Sizes a pure, game-and-render-agnostic leaf: anything that cares about viewport changes subscribes to "resize" without Sizes ever knowing it exists.

public Destroy(): void

Purpose: detach the DOM listener so a torn-down engine stops reacting to window resizes.

public Destroy() {
  window.removeEventListener("resize", this.onResize);
}

Passes the same this.onResize reference captured in the constructor — which is exactly why it was stored on a field. An inline arrow here would silently fail to remove the listener, leaking a handler that emits "resize" into a dead bus. Called from Experience.Destroy().

Modifying this: to support rendering into a sub-element rather than the full window, swap window.innerWidth/Height for a ResizeObserver on the target element — but keep the field-update-then-emit shape and the stored-handler/Destroy() contract intact. Leave the DPR cap; raising it above 2 noticeably hurts perf for no visual gain on this pixel-art renderer.

Note — where the engine actually starts. The real boot sequence lives in App.vue → onMounted, and its ordering is itself a hard-won design. Reproduced here because it is the canonical example of consuming Experience: 
const webglExperience = Experience.GetInstance();
// The game supplies its own asset manifest; the engine names none of these.
await webglExperience.Configure(webglRef.value, GAME_ASSETS);

// The game injects its level-node type vocabulary into the engine GLB parser.
// Must run before any level loads (StartLoading below triggers the first).
webglExperience.Resources.RegisterLevelTypes(LEVEL_NODE_TYPES);

registerContactRules(webglExperience.Physics.Contacts);
webglExperience.Debug.RegisterCollisionLogSink(new CollisionLogger());
webglExperience.Debug.SetPhysicsTypeColors(DEBUG_TYPE_COLORS);

const gameDirector = new GameDirector();
const inputBindings = new InputBindings(webglExperience.Input);
webglExperience.Debug.RegisterModule(new LevelDebug(gameDirector));

webglExperience.OnGameUpdate = () => {
  inputBindings.Update();
  gameDirector.Update();
};
// ...event listeners wired...
webglExperience.Resources.StartLoading();
Hard-won victory — boot ordering in App.vue. Three ordering decisions make this robust, each guarding against a fire-before-listen race:
  • Loading-screen listeners are registered FIRST, before Configure, so a fast boot can't emit loadingFinished before anyone is listening.
  • OnGameUpdate is assigned only after GameDirector / InputBindings exist — which is only possible because Configure was awaited, so Scene/Camera/Physics already exist when the game director's constructor runs.
  • Resources.StartLoading() is called LAST, after every resourcesReady/loadingProgress listener is wired, so the completion event can't fire before its listener exists.
The whole sequence is a study in "subscribe before you can possibly emit."
