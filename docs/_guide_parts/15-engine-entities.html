Engine Entities — GameObject, GameSensor, ContactRegistry, StateMachine

This is the heart of the engine's entity layer — four classes and two type files that every game entity is built from, and not one of them names a game concept. GameObject is the base class that marries a Rapier rigid body to a Three.js mesh and syncs them every frame; GameSensor turns that body into a non-solid trigger; ContactRegistry is the declarative "when an A touches a B, run this" table that replaced a wall of hard-coded if/else; and StateMachine is the tiny generic FSM the player and enemies both run on. The two type files (gameObjectType, userData) are the opaque vocabulary the engine carries on the game's behalf.

This file IS the base class for every spawnable thing in the world — the player, enemies, platforms, ladders, teleporters, win flags, trash cans all extend GameObject (directly or via GameSensor). Nobody constructs a bare GameObject; subclasses call its protected builders from their own constructors. It imports the engine singletons it needs at birth — Experience (for the shared Scene and Physics), Physics (to create bodies/colliders and register for collision callbacks) — plus GameObjectType and UserData (the opaque routing vocabulary) and getUserData from physicsHelpers. Lifecycle: constructor → initializeAttributes() grabs singletons and zeroes transforms; a subclass then calls createObjectPhysics(...) (rigid body + collider + registration), optionally CreateObjectGraphics(...) or createObjectGraphicsDebug(...) (attach a mesh); each sim step the owner calls syncGraphicsToPhysics() to copy the body's transform onto the mesh (and record the previous one), and each render frame it calls InterpolateGraphics(alpha) to lerp the mesh between those two transforms so motion stays smooth above the fixed sim rate; finally Destroy() tears down mesh, geometry, body and collider.

Note — the generic-ization. Read this file end to end and you will not find the words "player", "barrel", "ladder" or "Donkey Kong" anywhere. The recent refactor (commits "split webgl into game and engine" and "Replace collision routing with contact table") stripped every game name out of here. Identity is carried as two opaque strings (type / name in UserData) assigned by the game layer; shape is one of three primitives (GameObjectType); collision routing is data in the ContactRegistry. The base class is now reusable by a fresh project with no game.

// fields: engine refs, render side, physics side, cached transform, destroy flag

The fields split into five tiers. Engine refs (experience, scene, Physics) are the singletons grabbed at construction. Render side (geometry, material, mesh, spriteScale) is the optional Three.js half — a GameObject can exist as pure physics with no mesh. Physics side is PhysicsBody, the Rapier RigidBody. Cached transform (InitialSize/CurrentSize/InitialTranslation/CurrentTranslation/CurrentRotation) is mirrored off the body so other code never has to dig into collider.shape.halfExtents. And render interpolation (previousRenderTranslation/previousRenderRotation plus public RenderTranslation) holds the previous sim transform and the latest, so the per-frame render pass can lerp between them — and exposes RenderTranslation, the smoothed position other systems (notably the camera) follow instead of the stepped sim position.

OnCollisionEnter?(other) / OnCollisionExit?(other) / OnSensorEnter?(other) / OnSensorExit?(other)

Four optional callback slots — declared, not implemented. A subclass that wants to react to a contact overrides the relevant one; if it does, enablePhysicsEvents() auto-arms the matching Rapier events. Physics.handleCollisionEvents calls these directly on the resolved GameObject: the solid pair fires OnCollisionEnter/Exit on both bodies, the sensor pair fires OnSensorEnter/Exit only on the side that is actually the sensor. 

public OnCollisionEnter?(other: GameObject): void;
public OnCollisionExit?(other: GameObject): void;
public OnSensorEnter?(other: GameObject): void;
public OnSensorExit?(other: GameObject): void;
Note — two ways to react to a contact. An entity's own reaction to the world it moves through (e.g. Enemy.OnCollisionEnter reverses at a wall, enemy.ts:138) lives in these overrides. Cross-entity verdicts (enemy kills player, player wins) live as data in contactRules and run through the ContactRegistry. Both are dispatched from the same Physics drain loop — the callbacks first, then Contacts.Dispatch(...).

constructor() / private initializeAttributes()

The constructor does exactly one thing: call initializeAttributes(). That method grabs the three engine singletons off Experience.GetInstance() — the shared Scene and the Physics world — and zeroes every cached transform field to a fresh Vector2(0,0), empties gameObjectType, and clears IsBeingDestroyed. It does not create a body or a mesh; that is the subclass's job, by calling createObjectPhysics after super(). 

private initializeAttributes() {
  this.experience = Experience.GetInstance();
  this.scene = this.experience.Scene;
  this.Physics = this.experience.Physics;

  this.InitialSize = new RAPIER.Vector2(0, 0);
  this.CurrentSize = new RAPIER.Vector2(0, 0);
  this.InitialTranslation = new RAPIER.Vector2(0, 0);
  this.CurrentTranslation = new RAPIER.Vector2(0, 0);
  this.CurrentRotation = 0;

  this.gameObjectType = "";
  this.IsBeingDestroyed = false;
}
Gotcha — singletons must exist first. The constructor assumes Experience is already configured and experience.Physics has run Configure() (Rapier init() is async). Any entity built before the physics world exists will read an undefined Physics. In practice entities are only ever spawned by GameDirector after resourcesReady, well past engine boot.

protected createObjectPhysics(name, gameObjectType, size, position, rotation, specifiedRigidBodyType = RAPIER.RigidBodyDesc.fixed())

The main physics builder — every subclass constructor calls it. Params: name seeds the per-instance id (defaults to "GameObject"); gameObjectType picks the collider shape (a GameObjectType value); size is full width/height; position is the spawn point; rotation is in radians; specifiedRigidBodyType defaults to a fixed body (walls/platforms) — dynamic bodies (enemies) pass RAPIER.RigidBodyDesc.dynamic().

Body of the method, in order: cache InitialTranslation/InitialSize/CurrentSize and the type; build a collider descriptor from the type via createCollider; create the rigid body in the world; set its translation and rotation (mirroring both into the Current* caches); stamp userData; attach the collider to the body; register with Physics so the collider handle resolves back to this GameObject; then auto-arm events. 

const collider = this.createCollider(size, gameObjectType);

this.PhysicsBody = this.Physics.World.createRigidBody(specifiedRigidBodyType);

this.PhysicsBody.setTranslation({ x: position.x, y: position.y }, true);
this.CurrentTranslation = this.PhysicsBody.translation();
this.PhysicsBody.setRotation(rotation, true);
this.CurrentRotation = this.PhysicsBody.rotation();

// type = shared routing flag; name = per-instance id, both default to `name`
this.PhysicsBody.userData = { type: name, name: name } as UserData;

this.Physics.World.createCollider(collider, this.PhysicsBody);
this.Physics.RegisterGameObject(this);   // collider.handle -> this
this.enablePhysicsEvents();              // arm events if callbacks exist
Note — type vs name both seeded from one arg. At creation both userData fields are set to name. The comment in the source explains the contract: type is the shared routing flag many entities can carry (one contact rule matches all of them), name is the per-instance id. They start equal; a subclass calls SetType() and/or SetName() afterward to differentiate. The engine treats both as opaque strings.
Gotcha — registration is per collider, by handle. RegisterGameObject walks numColliders() and maps each collider.handle → gameObject. The contact callbacks can only fire for a collider that is registered, which is why registration happens here, immediately after the collider is attached. Destroy() must call the matching UnregisterGameObject or the map leaks stale entries pointing at dead bodies.

private enablePhysicsEvents()

The auto-arm step. It inspects which of the four callback slots this instance defined and OR-s the matching Rapier ActiveEvents bit onto collider 0 — but only if at least one callback exists. Entities that define no callbacks (a plain wall) stay event-silent, which keeps the per-step event queue small. 

let eventsToEnable = 0;

if (this.OnCollisionEnter || this.OnCollisionExit) {
  eventsToEnable |= RAPIER.ActiveEvents.COLLISION_EVENTS;
}
// In Rapier, sensors use collision events, not separate intersection events
if (this.OnSensorEnter || this.OnSensorExit) {
  eventsToEnable |= RAPIER.ActiveEvents.COLLISION_EVENTS;
}

if (eventsToEnable > 0) {
  collider.setActiveEvents(eventsToEnable);
}
Note — there is no "stay" event. The source comment is explicit: physics only dispatches enter/exit, no per-frame "stay" (see Physics.handleCollisionEvents). Anything that needs continuous "am I overlapping right now?" must poll — which is exactly the GameSensor ladder path described below. Also note both sensor and collision callbacks arm the same COLLISION_EVENTS bit, because in Rapier a sensor reports through the collision event stream; Physics disambiguates by checking collider.isSensor().

protected enableContactEvents(colliderIndex = 0)

The manual counterpart to the auto-arm above. Entities that route purely through the declarative ContactRegistry — and therefore define no callback overrides — still need their collider to emit events, so they call this explicitly. Teleporter, CameraSensor and WinFlag all do exactly this right after setAsSensor(true) (teleporter.ts:31-33). 

protected enableContactEvents(colliderIndex: number = 0) {
  this.PhysicsBody?.collider(colliderIndex)?.setActiveEvents(
    RAPIER.ActiveEvents.COLLISION_EVENTS
  );
}
Gotcha — callback-defining entities must NOT call this. Enemy defines OnCollisionEnter, so enablePhysicsEvents() already armed it; its constructor comment literally says "no enableContactEvents needed" (enemy.ts:98). Calling it a second time is harmless (idempotent set) but signals confusion: if your entity has a callback, it is armed automatically; if it is rule-only, arm it here.

protected createCollider(size, gameObjectType)

Maps the opaque type string to a Rapier ColliderDesc. Three shapes only — ball, capsule, cuboid — and the default branch falls through to a cuboid, so any unrecognised type is a box rather than a crash. 

switch (gameObjectType) {
  case GameObjectType.SPHERE:
    return RAPIER.ColliderDesc.ball(size.width / 2);
  case GameObjectType.CAPSULE:
    return RAPIER.ColliderDesc.capsule(size.height / 2, size.width);
  case GameObjectType.CUBE:
  default:
    return RAPIER.ColliderDesc.cuboid(size.width / 2, size.height / 2);
}
Hard-won victory (D5 / "D-lean"). The source comment spells out the architectural diet: "Slopes are rotated cubes and complex shapes are overlapped cubes (no polyline or convex-mesh)." A whole class of geometry bugs — degenerate polylines, non-convex meshes that Rapier silently rejects, expensive convex decomposition — was eliminated by restricting the engine to cuboid/ball/capsule. Note the conversions: ball takes a radius (width/2), cuboid takes half-extents, and capsule takes a half-height plus radius.

private createMeshGroup() / private async addMeshesToGroupAsync(meshes, batchSize) / private disposeMeshHelper(object)

The mesh-plumbing trio used by CreateObjectGraphics. createMeshGroup lazily makes the mesh an empty THREE.Group (idempotent — returns early if one exists). addMeshesToGroupAsync adds cloned meshes to that group in batchSize chunks, yielding to the event loop with await new Promise(r => setTimeout(r, 0)) between batches, then adds the whole group to the scene once. disposeMeshHelper frees one object's geometry and material(s), normalising the single-vs-array material case. 

private async addMeshesToGroupAsync(meshes: any[], batchSize: number) {
  for (let i = 0; i < meshes.length; i += batchSize) {
    const batchOfMeshes = meshes.slice(i, i + batchSize);
    batchOfMeshes.forEach((mesh) => this.mesh!.add(mesh));
    // Allow other operations to execute
    await new Promise((resolve) => setTimeout(resolve, 0));
  }
  this.scene.add(this.mesh!);
}
Hard-won victory — async batching keeps the frame alive. The comment explains it: loading many meshes from the Blender scene synchronously blocks the main thread ("too many synchronous loads to THREE.Scene"). Slicing into batches of 5 and yielding a macrotask between them lets rendering and input breathe during a level load, so the loading overlay actually animates instead of freezing.

protected syncGraphicsToPhysics(meshOffset = { x: 0, y: 0, z: 0 })

The per-sim-step bridge: copy the physics body's transform onto the Three.js mesh. The owner calls this once per fixed sim step from its Update() (Player, Enemy). Before it overwrites the Current* caches it now records them into previousRenderTranslation/previousRenderRotation — prev = last step's transform, current = this step's — so the per-frame render pass (InterpolateGraphics) has both endpoints to lerp between. meshOffset lets a subclass nudge the visual relative to the collider — notably the z is taken straight from the offset (2D physics has no z), so sprites can be layered front-to-back. 

if (this.IsBeingDestroyed || !this.PhysicsBody) {
  return;
}

// Record the previous sim transform so the render pass can interpolate toward
// the new one (see InterpolateGraphics).
this.previousRenderTranslation.x = this.CurrentTranslation.x;
this.previousRenderTranslation.y = this.CurrentTranslation.y;
this.previousRenderRotation = this.CurrentRotation;

this.CurrentTranslation = this.PhysicsBody.translation();
this.mesh?.position.set(
  this.CurrentTranslation.x + meshOffset.x,
  this.CurrentTranslation.y + meshOffset.y,
  meshOffset.z
);

this.CurrentRotation = this.PhysicsBody.rotation();
this.mesh?.rotation.set(this.mesh?.rotation.x, this.mesh?.rotation.y, this.CurrentRotation);
Gotcha — physics is the source of truth, never the mesh. Data flows one way: body → mesh. Nothing reads the mesh back into physics. The early-return on IsBeingDestroyed || !PhysicsBody is what makes it safe to call during teardown — a frame that fires after Destroy() nulled the body simply does nothing instead of dereferencing undefined. Also note CurrentTranslation is the stepped sim position refreshed here once per sim step, so code reading it (ladder containment math) gets last-step's exact position. The smoothed camera-follow position is a different field — RenderTranslation, written by InterpolateGraphics below.

public InterpolateGraphics(alpha, meshOffset = { x: 0, y: 0, z: 0 })

The per-frame render placement, the sibling to syncGraphicsToPhysics. Where the sync runs once per fixed sim step, this runs once per rendered frame for moving entities (Player, Enemies) and lerps the mesh between previousRenderTranslation and CurrentTranslation by alpha — the fraction (0..1) of how far into the next sim step the current frame falls (the engine computes it as accumulator / dt). Static entities are placed once by the sync and never need this. It also writes the interpolated point into RenderTranslation so the camera follows the smooth visual position rather than the stepped sim position. 

public InterpolateGraphics(
  alpha: number,
  meshOffset: { x: number; y: number; z: number } = { x: 0, y: 0, z: 0 }
) {
  if (this.IsBeingDestroyed || !this.mesh) {
    return;
  }

  const x =
    this.previousRenderTranslation.x +
    (this.CurrentTranslation.x - this.previousRenderTranslation.x) * alpha;
  const y =
    this.previousRenderTranslation.y +
    (this.CurrentTranslation.y - this.previousRenderTranslation.y) * alpha;

  this.RenderTranslation.x = x;
  this.RenderTranslation.y = y;

  this.mesh.position.set(x + meshOffset.x, y + meshOffset.y, meshOffset.z);

  // Interpolate rotation along the SHORTEST arc. The Rapier 2D angle wraps at ±π,
  // so a rolling barrel's angle jumps from ~+π to ~-π between steps; a naive lerp
  // would spin the long way round (a visible one-frame backspin). atan2(sin,cos)
  // wraps the delta into (-π, π] so we always take the short way.
  let dRot = this.CurrentRotation - this.previousRenderRotation;
  dRot = Math.atan2(Math.sin(dRot), Math.cos(dRot));
  const rot = this.previousRenderRotation + dRot * alpha;
  this.mesh.rotation.set(this.mesh.rotation.x, this.mesh.rotation.y, rot);
}
Hard-won victory — shortest-arc rotation (no one-frame backspin). Position lerps cleanly, but rotation does not: the Rapier 2D angle wraps at ±π, so a rolling barrel's angle jumps from roughly +π to roughly -π between two sim steps. A naive prev + (cur - prev) * alpha would read that as a near-2π delta and spin the barrel the long way round — a visible backspin every time the angle wraps. Computing the delta and folding it through Math.atan2(Math.sin(dRot), Math.cos(dRot)) wraps it back into (-π, π], so the interpolation always takes the short way and the barrel rolls forward smoothly.
Note — two transforms, two cadences. syncGraphicsToPhysics is the sim-rate truth (it samples the body and snapshots prev/current); InterpolateGraphics is the frame-rate smoothing (it blends those two). They guard differently on purpose: the sync needs the live body (!PhysicsBody bails) because it reads from it, while the interp only needs a mesh (!this.mesh bails) because it reads from the already-snapshotted caches. The owner wires them up as a pair — see GameDirector.Update (sim) vs GameDirector.RenderUpdate (frame).

protected setCollisionGroup(collisionGroups, colliderIndex = 0) / protected setCollisionMask(collisionMask, colliderIndex = 0)

The two halves of Rapier's 32-bit interaction filter. Rapier packs membership group in the low 16 bits and the filter mask in the high 16 bits of one collisionGroups() value. These methods let a subclass set one half without clobbering the other: setCollisionGroup reads the current mask (>> 16), writes the new group, and re-OR-s the mask back; setCollisionMask does the mirror, preserving the low-16 group (& 0xffff) and writing the mask (<< 16). 

// set group bits, preserve the current mask
const currentMask = this.PhysicsBody.collider(colliderIndex).collisionGroups() >> 16;
this.PhysicsBody.collider(colliderIndex)
  .setCollisionGroups(collisionGroups | (currentMask << 16));

// set mask bits, preserve the current group
const currentGroup = this.PhysicsBody.collider(colliderIndex).collisionGroups() & 0xffff;
this.PhysicsBody.collider(colliderIndex)
  .setCollisionGroups(currentGroup | (collisionMask << 16));
Tip — call order is free. Because each method preserves the other half, a subclass can call them in either order: Platform does setCollisionGroup(PLATFORM) then setCollisionMask(DEFAULT) (platform.ts:48-49); Enemy sets ENEMY group then a PLATFORM | PLAYER_BOUNDING_BOX mask. The actual bit constants live game-side in gameCollisionGroups — the engine only knows how to splice the two 16-bit fields.

public SetType(newType?) / public SetName(newName?)

Post-construction identity edits, both reading the live userData via getUserData (from physicsHelpers). SetType sets the shared routing type and mirrors it onto name — so after a SetType the instance name is the type again; call SetName after to give it a distinct per-instance id. Both no-op on a falsy argument, so a subclass can pass an optional value straight through. 

public SetType(newType?: string) {
  if (!newType) { return; }
  const userData = getUserData(this.PhysicsBody);
  userData.type = newType;
  userData.name = newType;   // name re-mirrors to type
}
public SetName(newName?: string) {
  if (!newName) { return; }
  getUserData(this.PhysicsBody).name = newName;
}
Gotcha — order matters: SetType clobbers name. Because SetType rewrites name = newType, calling SetName("barrel_07") and then SetType(ENEMY) silently throws the name away. Always do SetType first, then SetName. getUserData returns a safe empty {type:"",name:""} if the body is undefined, so these never throw on a partially-built object — they just write into a throwaway.

private TeleportRelative(newX, newY) / public TeleportToPosition(targetX, targetY)

Movement helpers. TeleportRelative offsets the body from its cached CurrentTranslation; TeleportToPosition computes the delta to an absolute world point and defers to it, then snaps the render-interpolation state to the target so the move reads as an instant jump, not a one-frame slide. Public TeleportToPosition is what the teleporter contact rule calls (contactRules.ts:49) and what PlayerDebug's click-to-teleport uses (playerDebug.ts:379). 

public TeleportToPosition(targetX: number, targetY: number) {
  const deltaX = targetX - this.CurrentTranslation.x;
  const deltaY = targetY - this.CurrentTranslation.y;
  this.TeleportRelative(deltaX, deltaY);   // wake = true

  // Snap the render-interpolation state to the new position so the mesh JUMPS to
  // the teleport target instead of sliding across the level over one frame (reset,
  // teleporter). Without this, prev=old/curr=new would lerp the whole distance.
  this.CurrentTranslation = this.PhysicsBody!.translation();
  this.previousRenderTranslation.x = this.CurrentTranslation.x;
  this.previousRenderTranslation.y = this.CurrentTranslation.y;
  this.RenderTranslation.x = this.CurrentTranslation.x;
  this.RenderTranslation.y = this.CurrentTranslation.y;
}
Hard-won victory — teleports JUMP, they don't slide. Render interpolation lerps the mesh from previousRenderTranslation to CurrentTranslation every frame. A teleport (a reset back to spawn, a teleporter pad) sets current to the far-away target while previous still holds the old spot — so without intervention the next frame would smoothly glide the player across the whole level over one step. Collapsing all three (CurrentTranslation = previousRenderTranslation = RenderTranslation = target) zeroes the lerp distance, so the entity simply appears at the destination.
Note — relies on a fresh cache. The opening delta math reads CurrentTranslation, which is refreshed once per sim step in syncGraphicsToPhysics. Within a single frame that is last step's position; for a teleport that is fine (the delta self-corrects against the real value), but it is why setTranslation passes true to wake the body — a sleeping dynamic body must wake to honour the move. The snap at the end then re-reads the body's true post-move translation, so the interpolation endpoints are exact.

public async CreateObjectGraphics(resourceFromLoader)

Attaches real art (a loaded GLTF/Blender scene) to this body. Param: resourceFromLoader is a loaded resource whose .scene.children are the meshes — handed in by GameDirector from this.resources.Items.* (e.g. gameDirector.ts:521, enemy.CreateObjectGraphics(this.resources.Items.enemy)). It ensures the group exists, deep-clones each child mesh so the shared source asset is never reparented — and clones that mesh's geometry and material too, so this instance owns its own GPU buffers — batches them into the scene async, then syncs the group to the physics transform once. 

this.createMeshGroup();

const blenderMeshes: THREE.Mesh[] = [];
for (const child of resourceFromLoader.scene.children) {
  if (child?.isMesh) {
    const cloned: THREE.Mesh = child.clone();    // clone so the original keeps its parent
    cloned.geometry = child.geometry.clone();    // own geometry buffer
    cloned.material = Array.isArray(child.material)
      ? child.material.map((m: THREE.Material) => m.clone())
      : child.material.clone();                  // own material(s)
    blenderMeshes.push(cloned);
  }
}

await this.addMeshesToGroupAsync(blenderMeshes, 5);
this.syncGraphicsToPhysics();
Hard-won victory — clone the mesh and its GPU buffers. Two hazards stack here. First, adding a mesh to a new THREE.Group removes it from its old parent — and the shared loaded GLTF in ResourceLoader's cache is reused across many entities, so moving the original's children would empty the cached asset and leave every later spawn blank; cloning each child first keeps the source intact. Second, a plain .clone() still shares geometry and material with the source. So when a dead entity (e.g. a barrel that despawns every couple of seconds) called Destroy() and disposed those, it freed buffers still in use by the source asset and every other live clone — flickering or vanishing art. Cloning geometry and material too gives each instance its own buffers, so Destroy() frees only what that instance owns.

public Destroy()

Full teardown of both halves. The very first line is an idempotent guard: if IsBeingDestroyed is already set, it returns immediately, so a second (or re-entrant) Destroy() is a harmless no-op. Otherwise it sets IsBeingDestroyed = true, then disposes and removes the mesh and all its children from the scene, disposes the cached geometry/material, and finally unregisters from Physics and removes the rigid body. Removing the body also removes its attached colliders, so there is no manual collider cleanup. 

// Idempotent: a second Destroy() (e.g. a re-entrant "gameObjectRemoved") is a
// no-op instead of double-freeing the body or recursing.
if (this.IsBeingDestroyed) {
  return;
}

// Set immediately — guards concurrent collision callbacks
this.IsBeingDestroyed = true;

if (this.mesh) {
  this.disposeMeshHelper(this.mesh);
  this.scene.remove(this.mesh);
  this.mesh.children.forEach((child) => this.disposeMeshHelper(child));
  this.mesh.clear();
  this.mesh = undefined;
}

this.geometry?.dispose();
this.material?.dispose();

if (this.PhysicsBody) {
  this.Physics.UnregisterGameObject(this);          // drop handle -> this mapping
  this.Physics.World.removeRigidBody(this.PhysicsBody); // also frees colliders
  this.PhysicsBody = undefined;
}
Hard-won victory — the destroy flag is set first, on purpose. Physics events drain after World.step(), and a body can register a contact in the very step it is being destroyed. Physics.handleCollisionEvents bails if either party IsBeingDestroyed (physics.ts:115-121), and syncGraphicsToPhysics bails too. Setting the flag at the top of Destroy() — before the body still physically exists for the rest of the method — is what prevents a callback firing on a half-torn-down object (a use-after-free style bug that manifested as enemies "killing" the player on the same frame they fell into the trash can).
Note — the guard also makes Destroy idempotent. The early-return reuses that same IsBeingDestroyed latch: a second call returns before touching anything. That matters because a contact handler may Emitter.emit("gameObjectRemoved", ...), the director's handler calls .Destroy(), and a path that re-emits for the same object would otherwise recurse or double-free the body. Because the latch never resets, a destroyed instance is permanently dead — whoever reuses a slot (e.g. GameDirector rebuilding level graphics) constructs a fresh GameObject rather than re-destroying the old one.
Gotcha — order is load-bearing. Unregister before removeRigidBody: UnregisterGameObject reads numColliders() off the still-live body to find the handles to delete. Remove the body first and that lookup would have nothing to walk, leaking the registry entry.

protected setAsSensor(value: boolean)

Marks collider 0 as a sensor (or back to solid). A sensor produces no collision response — bodies pass through — but still emits collision events, which is how Rapier reports intersections. It also sets the active collision types so the sensor can detect the body kinds it cares about. 

protected setAsSensor(value: boolean) {
  if (!this.PhysicsBody) { return; }

  this.PhysicsBody.collider(0).setSensor(value);

  // detect kinematic AND fixed bodies (defaults miss them)
  this.PhysicsBody
    .collider(0)
    .setActiveCollisionTypes(RAPIER.ActiveCollisionTypes.KINEMATIC_FIXED);
}
Hard-won victory — KINEMATIC_FIXED active types. By default Rapier only reports dynamic-vs-dynamic intersections. The player moves under a kinematic character controller and most sensors are fixed bodies — without setActiveCollisionTypes(KINEMATIC_FIXED) a kinematic player crossing a fixed ladder/teleporter/win-flag sensor would fire nothing. This one line is what makes every sensor in the game actually trigger.
Gotcha — arming events is separate. setAsSensor only flips the sensor flag; it does not arm event emission. A sensor still needs either a defined OnSensorEnter/Exit override (auto-armed) or an explicit enableContactEvents() call. That is why Teleporter/WinFlag/CameraSensor call setAsSensor(true) then enableContactEvents(), while LadderSensor — which is polled, not event-driven — calls only setAsSensor(true) and arms nothing.

This file IS the declarative contact table that replaced the old wall of hard-coded collision if/else. It is a generic engine class that names no game concept (same spirit as StateMachine): the game declares rules as data in contactRules.ts, and Physics owns the one instance as Physics.Contacts (constructed in physics.ts:37). It imports only GameObject, for the handler signature. Lifecycle: created once in Physics.Configure; filled once at startup by registerContactRules (called from App.vue); then Dispatch-ed every step from Physics.handleCollisionEvents.

type ContactMatch = (gameObject: GameObject) => boolean type ContactPhase = "enter" | "exit" | "sensorEnter" | "sensorExit" type ContactHandler = (a: GameObject, b: GameObject) => void interface ContactRule { a; b; phase; run }

The vocabulary. A ContactMatch is a predicate that decides whether a GameObject fills one side of a rule — in practice byType(EntityType.ENEMY) built on physicsHelpers.matchesType. ContactPhase is the four physics moments a rule can listen to, matching the two solid + two sensor branches in Physics. A ContactRule bundles the two side-predicates, the phase, and the handler. 

export type ContactMatch = (gameObject: GameObject) => boolean;
type ContactPhase = "enter" | "exit" | "sensorEnter" | "sensorExit";
type ContactHandler = (a: GameObject, b: GameObject) => void;

interface ContactRule {
  a: ContactMatch;
  b: ContactMatch;
  phase: ContactPhase;
  run: ContactHandler;
}

public Add(a: ContactMatch, b: ContactMatch, phase: ContactPhase, run: ContactHandler)

Declares one rule by pushing it onto the private rules array: "when an a and a b reach phase, run run(a, b)." This is the entire write API; the game calls it once per rule at startup. There is no removal API — the table is built once and lives for the session. 

private rules: ContactRule[] = [];

public Add(a: ContactMatch, b: ContactMatch, phase: ContactPhase, run: ContactHandler) {
  this.rules.push({ a, b, phase, run });
}
Note — rules are data, verdicts are decoupled. The handlers in contactRules.ts mostly just Emitter.emit("gameOver" / "gameWin" / "gameObjectRemoved", ...) — they decide what happened and let the event bus decide who reacts. So a contact never directly reaches into game flow; it raises an event. That is the second decoupling seam stacked on top of the first.

public Dispatch(phase: ContactPhase, x: GameObject, y: GameObject)

Called by Physics for every contact event of a given phase, with the two colliding GameObjects in whatever order Rapier reported them. It scans all rules, skips those of a different phase, and for a phase-match tests the pair both ways: (x,y) against (a,b), else (y,x). The first orientation that matches fires the handler with the args in the rule's declared order. 

public Dispatch(phase: ContactPhase, x: GameObject, y: GameObject) {
  for (const rule of this.rules) {
    if (rule.phase !== phase) {
      continue;
    }
    if (rule.a(x) && rule.b(y)) {
      rule.run(x, y);
    } else if (rule.a(y) && rule.b(x)) {
      rule.run(y, x);
    }
  }
}
Hard-won victory — the else if is the whole trick. Physics has no idea who is "a" and who is "b"; it just reports two handles. Testing both orientations means a rule fires regardless of which body moved into which ("any touch registers" — the enemy-kills-player rule must work whether the player ran into the enemy or the barrel rolled into the player). Crucially it is else if, not two ifs: for a symmetric pair where both predicates could match either body, a plain second if would fire the handler twice. The else if guarantees exactly one invocation per rule per event, and always with args in the rule's own (a, b) order — so the handler can safely cast a to the enemy type and b to the player type.
Note — dispatch is two-tiered. Physics.handleCollisionEvents first calls the per-entity callbacks (OnCollisionEnter etc.) on each body, then calls Contacts.Dispatch(...). So an entity's own movement reaction and the cross-entity verdict both run from the same drained event, callbacks first. The kinematic-controller path (Physics.GetGameObjectFromCollider) also lets the player feed its own computed contacts into Dispatch so rules fire even when Rapier emits no event for the kinematic mover.

src/engine/entities/stateMachine.ts 17 LOC

This file IS the engine's entire finite-state-machine — seventeen lines, fully generic, naming no state and no entity. It is a router: given a host that owns a string State field and a map from each state to a handler function, Update() calls the handler for the host's current state, once. The Player and Enemy each construct one in their attribute init (player.ts:161, enemy.ts:128) and drive it from their per-frame Update() (fsm.Update()). The host owns the state; handlers transition by reassigning host.State.

export default class StateMachine<H extends { State: S }, S extends string>

Two type parameters tie the machine together: H is the host type, constrained to "anything with a State: S field", and S is the union of state strings. The constructor stores the host and a Record<S, (host: H) => void> handler map — a total map, so TypeScript forces a handler for every state at the call site. Player instantiates it as StateMachine<Player, PlayerState>, Enemy as StateMachine<Enemy, EnemyState>. 

export default class StateMachine<H extends { State: S }, S extends string> {
  constructor(
    private host: H,
    private handlers: Record<S, (host: H) => void>
  ) {}
Tip — the Record<S, ...> is a compile-time completeness check. Because the handler map is keyed by the full state union, adding a new value to PlayerStates without adding its handler to the new StateMachine(...) map is a build error, not a silent missing-case at runtime. The FSM can never dispatch to a state that has no handler.

public Update()

Reads host.State once and invokes the matching handler with the host as its argument. The optional-chain ?. means a host whose state has no handler simply does nothing rather than throwing (belt-and-braces on top of the compile-time Record guarantee). 

public Update() {
  // Read State once: a handler that changes host.State transitions on the NEXT
  // update, it does not re-dispatch within this one.
  this.handlers[this.host.State]?.(this.host);
}
Hard-won victory — "read State once" prevents same-frame cascades. A handler that sets host.State = next does not re-dispatch within the same Update() — the new state's handler runs next frame. This is deliberate: it gives every state exactly one tick of execution, so a chain like idle→running→jumping in one frame is impossible. Without it, a handler that transitions could fall straight into the next handler, and a poorly-ordered pair of transitions could loop forever inside a single frame.
Note — the FSM only routes; the host owns the state. There is no transition() method, no enter/exit hooks, no state stack. Transitions are just player.State = PlayerStates.FALLING inside a handler (e.g. handlePlayerJumping.ts:14). This minimalism is the point — see Architecture — the generic FSM for how the player's five states and the enemy's three states both reuse this one class.
