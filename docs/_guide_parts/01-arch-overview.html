The Big Picture: engine vs game, the singleton, the wiring

FireGame is split into two layers that never blur together: a reusable engine/ that knows nothing about Donkey Kong, and a game/ that supplies the rules. A single Experience singleton owns the one canvas, scene, and physics world; a global mitt event bus decouples everything; and the whole machine is wired together from App.vue in one onMounted pass. This section is the map you read before any other.

Orientation: the two-layer architecture

Before any one file, hold the shape of the whole thing in your head. There are two layers and a strict import rule between them:

  • engine/ — reusable, game-agnostic. The renderer, the Rapier physics world, the camera, the resource/GLB loader, the input device reader, the finite-state-machine base, the contact registry, the debug coordinator. It never imports from game/. When the engine needs something game-specific it exposes a seam — an interface, a registry, a late-bound hook — and lets the game plug in.
  • game/ — the Donkey Kong rules. The GameDirector, the player/enemy entities and their FSM states, the level registry, the contact rules, the feel profiles, the F-key bindings. It imports freely from engine/ and fills every seam the engine left open.
Note. The leak that became a seam. The engine's GLB parser (resourceLoader.ts) is type-blind: it no longer carries a hand-maintained table mirroring the game's EntityType strings. Instead the game injects its level-node vocabulary — LEVEL_NODE_TYPES, a name-token → game-type map whose values come straight from EntityType so the two cannot drift — via Resources.RegisterLevelTypes() at boot. The engine still never imports the game enum; it just looks each node up in whatever map the game handed it. What used to be the one place the engine named something game-shaped is now a proper seam like every other.

Tip. The litmus test. If you ever find yourself wanting to import something from game/ inside an engine/ file, stop: the correct move is to add a seam (a registry entry, an interface the game implements, or a late-bound callback) and let App.vue wire the game side in.

src/engine/core/experience.ts 134 LOC

Experience is the engine's root object: the one place that owns the canvas, the THREE.Scene, the Rapier world, and every engine subsystem. It is a singleton — a private constructor plus a static GetInstance() — so that anyone, anywhere (the camera, the physics world, the game director) can reach the same scene/world without it being threaded through every constructor. Construction is split in two: GetInstance() hands back the bare object; the real wiring lives in an async Configure() because the physics world needs await RAPIER.init(). It imports only engine modules — never anything from game/.

private static instance: Experience; private constructor() {}

The singleton mechanism. The constructor is private, so new Experience() is a compile error everywhere outside this file. The only way to obtain the instance is the static accessor below. 

// Class prop instance and "new" blocking constructor for Singleton
private static instance: Experience;
private constructor() {}

public static GetInstance(): Experience

Lazily creates the instance on first call and returns the same one forever after. This is what App.vue calls first, and what every engine/game module calls to reach the shared scene/world. 

public static GetInstance(): Experience {
  if (!Experience.instance) {
    Experience.instance = new Experience();
  }
  return Experience.instance;
}
Why a singleton. There is exactly one canvas, one WebGL renderer, one THREE.Scene, and one Rapier World. A singleton makes that physically true: Physics and Camera both call GetInstance() and necessarily get the same scene to bind into. The alternative — passing the scene/world through every constructor — invites two worlds drifting out of sync.

public OnGameUpdate: (() => void) | null = null;

This is the single most important field in the file and the thesis of the whole architecture. The engine's per-frame loop needs to run the game's logic, but the engine core must not name a game class. So instead of import GameDirector, the engine exposes a nullable callback that the game assigns from outside. The engine calls it; it never knows what is on the other end. 

// Late-bound game update hook. The game layer assigns this; the engine just
// calls it each frame between physics and renderer. Keeps the engine core
// from naming any game class.
public OnGameUpdate: (() => void) | null = null;

It is assigned in App.vue to a closure that drives the game's input bindings and the GameDirector, and it is invoked with optional-chaining (this.OnGameUpdate?.()) inside the tick handler so that an unconfigured engine simply renders nothing-game and does not crash.

public async Configure(canvas: HTMLCanvasElement | null, resources: Resource[])

The real constructor. Instantiates every subsystem in a deliberate order, captures the canvas, builds the scene, and subscribes the two engine-level event handlers (resize, tick). It is async purely because Physics.Configure() awaits RAPIER.init() (the WASM module). Parameters: canvas — the DOM canvas element from the Vue template ref, or null if not yet mounted; resources — the game's asset manifest, passed straight to the ResourceLoader so the engine names none of the game's assets. Returns: a promise that resolves once physics is ready. 

public async Configure(
  canvas: HTMLCanvasElement | null,
  resources: Resource[]
) {
  this.Debug = new Debug();
  this.Sizes = new Sizes();
  this.Time = new Time();
  this.Input = new Input();
  this.Resources = new ResourceLoader(resources);

  this.TargetElement = canvas;

  this.Scene = new THREE.Scene();
  this.Camera = new Camera(new THREE.Vector3(0, 25, 100));
  this.Renderer = new Renderer();
  this.Physics = new Physics();
  await this.Physics.Configure();

Line-level notes:

  • Debug first. It reads import.meta.env.DEV / the #debug hash to set IsActive; later subsystems (notably Physics.Configure) check Debug.IsActive to decide whether to attach wireframes, so Debug must exist before them.
  • The ResourceLoader is handed the game's manifest, not a hardcoded one. The engine no longer names randy/enemy/dkGraphicsData anywhere — those { name, type, path } entries live in the game's assetManifest.ts (GAME_ASSETS), where the paths are Vite ?url imports resolved to bundled, content-hashed URLs. getAssetPath now simply returns source.path; a different game supplies its own manifest and reuses the engine unchanged (reusability mile, #5).
  • The camera's start position (0, 25, 100) is the only camera-placement value baked into the engine; per-level camera starts are applied later by the GameDirector.
  • Physics is constructed then awaited. The two-step new Physics() then await ...Configure() mirrors this very class's own pattern, again because of the async RAPIER.init().

Configure then registers the two engine-level event listeners and stores the handler refs for cleanup:

this.onResize = () => {
  this.Camera.Resize();
  this.Renderer.Resize();
};
Emitter.on("resize", this.onResize);

this.onTick = () => {
  if (this.Debug.IsActive) {
    this.Debug.Stats?.begin();
  }

  this.Physics.Update();
  this.OnGameUpdate?.();
  this.Renderer.Update();

  if (this.Debug.IsActive) {
    this.Debug.Stats?.end();
  }
};
Emitter.on("tick", this.onTick);
Hard-won victory. The frame order is the whole game in one line. Physics.Update() → OnGameUpdate?.() → Renderer.Update(). Physics steps the world first, then the game reads this frame's freshly-solved positions to update entities and camera, then the renderer draws. The Stats.begin()/end() bracket wraps the whole frame so the FPS/MS monitor measures real work. The handler refs (this.onResize, this.onTick) are stored as fields specifically so Destroy() can Emitter.off the exact same function instances — an inline arrow could never be removed.
Gotcha. The engine subscribes to tick, but it does not produce it. Time owns the requestAnimationFrame loop and emits tick; Sizes owns the window listener and emits resize. Experience is a subscriber. That is the event bus doing its job: the clock does not know who listens, and the loop does not know who clocks it.

public Destroy()

Symmetric teardown. Removes its own two listeners first (using the stored refs), then tells each subsystem to clean up in a sane order, and finally tears down the debug panel only if it was active. 

public Destroy() {
  Emitter.off("resize", this.onResize);
  Emitter.off("tick", this.onTick);

  this.Sizes.Destroy();
  this.Time.Destroy();
  this.Input.Destroy();

  this.Camera.Destroy();
  this.Physics.Destroy();
  this.Renderer.Destroy();
  this.Resources.Destroy();

  if (this.Debug && this.Debug.IsActive) {
    this.Debug.Destroy();
  }
}

Note the order: stop the clock and input first (so no more frames fire mid-teardown), then the render/physics/resource chain. The Debug.IsActive guard avoids destroying a panel that was never built in a production (non-debug) build.

Modifying this: to add a new engine subsystem, declare a public X!: X; field, instantiate it in Configure() (mind the ordering — anything that reads Debug.IsActive or Scene must come after them), tick it inside the onTick handler at the right point in the physics→game→render order, and tear it down in Destroy(). Resist the urge to import anything game-specific here — if the new subsystem needs game knowledge, give it a registry/interface seam instead and let App.vue feed it.

src/main.ts 5 LOC

The literal entry point named in index.html. It does exactly one thing: mount the Vue app. It is intentionally trivial — none of the engine wiring lives here, because the engine needs a real, mounted <canvas> element to attach to, and that only exists after Vue has rendered. So bootstrap is deferred to App.vue's onMounted. 

import { createApp } from 'vue'
import './style.css'
import App from './App.vue'

createApp(App).mount('#app')
Note. The order matters even here: importing ./style.css before mounting ensures the global stylesheet (and the canvas's full-bleed layout) is present when the component renders. The #app selector is the mount root in index.html.

Modifying this: almost never. If you wanted to install a Vue plugin (router, store) it would go here as createApp(App).use(plugin).mount('#app') — but the game's own lifecycle does not belong here; it belongs in App.vue.

src/App.vue 177 LOC

The wiring harness and the presentation shell. This Vue 3 <script setup> component is the only place the two layers are stitched together. Its <template> owns the loading overlay, the status text, and the reset button; its onMounted hook runs the entire bootstrap sequence. Everything game-specific that the engine needs — the asset manifest, the level-node type vocabulary, the contact rules, the collision-log sink, the debug color palette, the GameDirector, the F-key bindings, the per-frame game hook — is injected here (the manifest into Configure itself; the rest right after). The Vue layer communicates with the game purely through the event bus; it never reaches into game internals.

onMounted(async () => { ... }) — the bootstrap sequence

This is the spine of the whole application. The ordering inside it is not incidental; almost every line is placed to avoid a specific race. Walk it top to bottom.

1. Loading-screen listeners FIRST — before Configure.

Emitter.on("loadingStarted", () => {
  isLoading.value = true;
  loadProgress.value = null;
});
Emitter.on("loadingFinished", () => {
  isLoading.value = false;
});
Emitter.on("loadingProgress", (progress) => {
  loadProgress.value = progress;
});
Gotcha. Subscribe before you can be notified. These go up first so a very fast boot load cannot fire loadingFinished before anyone is listening — mitt does not buffer or replay events, so a notification with zero listeners is simply lost forever. This is the same hazard that governs StartLoading() being called last (below).

2. Get the singleton, configure the engine, and inject the level vocabulary.

const webglExperience = Experience.GetInstance();
// The game supplies its own asset manifest; the engine names none of these.
await webglExperience.Configure(webglRef.value, GAME_ASSETS);

// Inject the game's level-node type vocabulary into the engine GLB parser.
webglExperience.Resources.RegisterLevelTypes(LEVEL_NODE_TYPES);

webglRef.value is the now-mounted canvas; GAME_ASSETS is the game's boot manifest (assetManifest.ts), so the engine names none of its own assets. The await matters: nothing below may touch webglExperience.Physics until Configure has finished RAPIER.init(). The first thing it does afterward is hand the engine's GLB parser the game's level-node vocabulary via RegisterLevelTypes(LEVEL_NODE_TYPES) — the parser stays type-blind and the game's EntityType strings live in one place. It must run before any level loads, i.e. before StartLoading (below) triggers the first one.

3. Register the game's contact rules into the engine's contact registry.

registerContactRules(webglExperience.Physics.Contacts);

contactRules.ts declares the cross-entity table — Enemy→Player = gameOver, Enemy→TrashCan = destroy + ignite, Player→Teleporter/CameraSensor/WinFlag = the matching reaction. The engine's Physics.handleCollisionEvents dispatches every physics contact into this registry; it never names a game concept. This is the "no hard-coded if/else" collision routing the engine is built around (detailed in Physics & the Contact Table).

4. Register the collision-log sink and the debug color palette.

webglExperience.Debug.RegisterCollisionLogSink(new CollisionLogger());
webglExperience.Debug.SetPhysicsTypeColors(DEBUG_TYPE_COLORS);
Hard-won victory (D3). The engine never names a game type. The debug wireframe palette maps an entity-type string → a THREE.Color. Rather than baking that map into PhysicsDebug (which would force the engine to enumerate game types), the game injects the map via SetPhysicsTypeColors, and PhysicsDebug looks each collider's type up in it opaquely. Same idea for the log sink: CollisionLogger owns the game-specific formatting; the engine's Debug just forwards events to whatever sink implements the CollisionLogSink interface. Both calls are safe even when debug is inactive — they only store; PhysicsDebug reads them solely while rendering, which only happens when active.

5. Construct the game director and the F-key bindings.

const gameDirector = new GameDirector();
const inputBindings = new InputBindings(webglExperience.Input);

webglExperience.Debug.RegisterModule(new LevelDebug(gameDirector));

GameDirector is constructed after Configure so that the scene/camera/resources it pulls from the singleton already exist. InputBindings is handed the engine's Input device reader and converts F-key press edges into game events (F1→gameReset, F2→switchLevel, F3→manualCameraControl). RegisterModule with a LevelDebug wires a dat.gui level-select dropdown — but it is a no-op store when debug is inactive, so production builds pay nothing.

6. Install the per-frame game hook — the one seam.

webglExperience.OnGameUpdate = () => {
  inputBindings.Update();
  gameDirector.Update();
};
This single assignment is the engine↔game seam. The engine's tick handler calls this.OnGameUpdate?.() between physics and render. By assigning a closure here, App.vue gives the engine a way to run game logic every frame without the engine ever importing GameDirector or InputBindings. Input bindings run first (so an F-key press is observed before the director acts on the resulting event this frame).

7. Game-flow listeners that drive the UI shell.

Emitter.on("gameWin", () => { /* status = "Game Win!"; show Reset */ });
Emitter.on("gameOver", () => { /* status = "Game Over!"; show Reset */ });
Emitter.on("gameReset", () => { /* clear status; hide Reset */ });
Emitter.on("gameStart", () => { /* clear status; hide Reset */ });

These are pure presentation reactions. Note both the GameDirector and App.vue listen to gameStart / gameOver / etc.: the director changes simulation state (pause physics, stop spawning) while App.vue changes screen state (status text, reset button). Neither calls the other; they meet only at the event name. The gameStart listener clears stale win/over UI so a freshly-loaded level never shows a leftover "Game Over!".

8. Kick off asset loading LAST.

webglExperience.Resources.StartLoading();
Hard-won victory. Load last, after every listener is wired. StartLoading() is deliberately not called inside ResourceLoader's constructor. If every asset happened to resolve during the await window in Configure (e.g. assets cached), the resourcesReady event would fire into zero listeners — and the GameDirector's boot handler (which loads the default level) subscribes to exactly that event. mitt does not replay, so the boot would hang on the loading screen forever. Loading last guarantees the listener exists before the completion event can fire.

The template

The markup is small: a status line and Reset button (top-left), the WebGL <canvas>, and a full-screen loading overlay shown v-if="isLoading". When loadProgress is set (initial multi-asset load) it renders a real loaded / total bar driven by barWidth(); otherwise it shows a pulsing "Loading…" for single-GLB level swaps. The Reset button's handleButtonClicked simply emits gameReset — it does not touch the game directly; the director and the shell both react to the event.

Modifying this: this is where new game→engine wiring goes. To add a new contact rule, edit contactRules.ts (it is already registered here). To add a new debug module, construct it and pass it to Debug.RegisterModule alongside LevelDebug. To add a new per-frame game system, fold it into the OnGameUpdate closure. The two inviolable rules: keep StartLoading() the last line, and register any listener for a boot/completion event before the thing that emits it.

src/game/gameDirector.ts 546 LOC

The game layer's central controller — the counterpart to Experience on the game/ side. It owns the player and all level entities, drives the level lifecycle (load / unload / switch), runs the frame-driven enemy spawner, and translates game-flow events into simulation state. It reaches the engine the canonical way — Experience.GetInstance() in its constructor, then caches Scene, Time, Camera, Resources — and it is driven by the engine through the OnGameUpdate seam. This section maps its role and seams; the line-by-line bodies (spawning cadence, GLB load path, camera mapping) are covered in the dedicated GameDirector and entity sections.

constructor()

Pulls the engine singleton and caches the subsystems it needs, creates the level's GraphicsObject overlay holder, then wires every game-flow listener via setupEventHandlers(). 

this.experience = Experience.GetInstance();
this.scene = this.experience.Scene;
this.time = this.experience.Time;
this.camera = this.experience.Camera;
this.resources = this.experience.Resources;
this.GraphicsObject = new GameObject();

this.setupEventHandlers();

Because App.vue constructs the director after Configure, every one of these GetInstance() reads returns a fully-built subsystem. Construct it earlier and you would cache undefined.

private setupEventHandlers()

The director's entire reaction surface lives here — it is a pure event consumer. The events it listens to:

EventReaction
resourcesReadyBoot: await loadLevelByIndex(DEFAULT_LEVEL_INDEX) (which emits gameStart on success)
gameStartPhysics.SetPaused(false); enable enemy spawning
gameOverPhysics.SetPaused(true); stop spawning
gameWinFreeze identically to game over (so a stray barrel can't flip "Win" to "Over")
gameResetUnpause; resetGameState()
switchLevelswitchToNextLevel() (F2 cycle)
gameObjectRemovedremovedGameObject.Destroy()

It also starts a 5-second setInterval that prunes destroyed enemies from the Enemies array (kept separate from the frame loop so it keeps running after game over).

Note. The boot handler is why StartLoading() must run last. This resourcesReady subscription is the listener App.vue is protecting when it defers asset loading to the final line — it is the chain resourcesReady → load default level → gameStart.

public Update() — the body of the OnGameUpdate hook

Called every frame by the engine (through the closure App.vue installed on OnGameUpdate). The ordering inside is deliberate: bail while a level is loading, guard that the player exists, then player → camera → spawn → enemies → level entities → ladder state. 

if (this.isLoading) return;
if (!this.Player?.PhysicsBody) return;

this.Player.Update();
this.camera.Update(this.cameraTargetFromPlayer(this.Player));
this.trySpawn();

const firstTrashCan = this.TrashCans[0];
this.Enemies.forEach((enemy) => enemy.Update(this.Player, firstTrashCan));
Hard-won victory (C5). The camera is engine-pure. The director never hands the camera a game Player; it maps the player into a plain CameraTarget snapshot (x, y, velocityX, followState, isGrounded) via cameraTargetFromPlayer. That is why camera.ts imports nothing from game/ — it consumes scalars, not a game type. Player is updated before the camera so the follow reads this frame's position (the reverse order left the camera one frame stale).

private async loadLevelByIndex(index: number)

The single funnel for every level load — boot, the F2 cycle, and the dat.gui select all pass through here. It brackets the load with loadingStarted / loadingFinished (so the Vue overlay covers it), tears down the current level, parses the GLB, applies the registry entry's feel profile and camera-follow mode, and on success emits gameStart.

Hard-won victory (D5). One boolean, old-school. Loads are async and callers are fire-and-forget, so an isLoading guard serializes overlapping loads (F2 mashing / selecting mid-load). Without it a second load would interleave teardown and respawn → doubled entities, a leaked ambient light, a mismatched index. The same flag freezes Update() while a level rebuilds.
Gotcha. A failed load must not reject out of here. The whole body is wrapped in try/catch/finally: a missing/corrupt GLB is logged loud but swallowed, because rejecting would skip the boot handler's gameStart (leaving physics paused) and surface as an unhandled promise rejection. The finally always closes the loading overlay, so the screen never hangs. And gameStart is emitted only on the success path, so a failed load never wrongly unpauses.

private trySpawn() / SpawnEnemy() / SpawnCrazyEnemy()

Frame-driven enemy spawning, advanced off Time.Delta rather than a wall-clock setInterval (one clock, and a fresh boot spawns without needing a reset).

Hard-won victory (D28). One unified barrel array. Normal and "crazy" barrels are both Enemy now, distinguished only by their starting FSM state — the old separate CrazyEnemy class was folded in. The cadence counters are advanced before the async spawn so a stutter frame can't double-spawn.

The spawning cadence, FSM states, and oil-can targeting are detailed in the Enemy section.

public Destroy()

Unsubscribes every game-flow listener (Emitter.off by name), clears the cleanup interval, and unloads the current level. The mirror image of setupEventHandlers().

Modifying this: add a new game-flow reaction by subscribing in setupEventHandlers() and unsubscribing in Destroy() — keep them paired. Add a new level by editing the levelRegistry, not this file. Add a new level-placed entity type by giving it an entityFactory and a typed query array, not a new branch here — the spawn pass is data-driven over ENTITY_FACTORIES. Never call Physics.SetPaused or emit flow events from elsewhere if you can route it through these handlers; one funnel keeps the state machine honest.
