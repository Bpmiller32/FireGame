The Finite-State-Machine pattern

Every moving creature in FireGame runs on the same nine-line engine primitive: a host object owns a string State field, the StateMachine holds a map of state → handler function, and once per frame it dispatches to the handler for the current state. Handlers read input and physics, mutate the body, and reassign host.State to transition next frame. The engine knows nothing about idle, jumping, or barrels — the player and the enemy each instantiate the SAME generic class with their own state set.

This is the entire generic FSM — a router, nothing more. It is a pure engine primitive: it imports nothing, references no game concept, and is parameterised over two type variables so it works for any host that exposes a string State plus a matching handler map. The game constructs one per entity: Player news up a StateMachine<Player, PlayerState> in its attribute-init (see the Player entity section), and Enemy news up a StateMachine<Enemy, EnemyState> in its init (see the Enemy section). The host owns the lifecycle; the machine has no Destroy() and holds no resources — it is just two references and a method.

The design seam here is deliberate and worth stating plainly: the host owns its State; the machine owns dispatch. A handler never asks the machine to transition — it simply reassigns host.State = SomeOtherState, and the very next Update() routes to the new handler. That single rule is what keeps every state body decoupled from every other and from the machine itself.

class StateMachine<H extends { State: S }, S extends string>

Two type parameters do all the heavy lifting. H is the host type, constrained to any object that has a State field of type S. S is the union of legal state strings (e.g. PlayerState = "idle" | "running" | ...). Because the handler map is typed Record<S, (host: H) => void>, TypeScript forces you to supply a handler for EVERY member of the union — forget one state and the construction call fails to compile. That compile-time exhaustiveness is the quiet payoff of the generics.

export default class StateMachine<H extends { State: S }, S extends string> {
  constructor(
    private host: H,
    private handlers: Record<S, (host: H) => void>
  ) {}

Line notes: both host and handlers are private constructor parameters — TypeScript's shorthand that declares and assigns the fields in one stroke, so there is no body. The host is stored by reference, so when the player's State mutates anywhere, the machine sees it on the next read — no syncing, no setter.

Note. The constraint is capital-State ({ State: S }), matching Player.State and Enemy.State. The file's top-of-file prose comment still says lowercase “host.state” in a couple of places — that is stale wording, not a second field. The only field the type system (and the dispatch below) reads is State.

public Update(): void

Called once per frame by the host's own Update() (the player calls it inside updatePlayerState(); the enemy calls this.fsm.Update() directly). It reads the host's current state ONCE, looks up the handler, and invokes it with the host. That is the whole loop.

public Update() {
  // Read State once: a handler that changes host.State transitions on the NEXT
  // update, it does not re-dispatch within this one.
  this.handlers[this.host.State]?.(this.host);
}

Line notes: the optional-chain ?. is the safety valve — if the current state has no registered handler (a typo, a half-added state), dispatch is a silent no-op rather than a crash. More importantly, because this.host.State is read exactly once per call, a handler that flips host.State mid-body does NOT cause the new state's handler to also run this frame. Transitions cost exactly one frame of latency, always. Several state bodies lean on this: they set the next state AND seed a value for that state (e.g. idle sets NextTranslation.y = JumpPower and switches to JUMPING) then return, trusting next frame to pick up the new handler.

Gotcha. The single-read rule means a handler that wants its successor to act this same frame must do that work itself before returning. The enemy does exactly this: handleEnemyRolling, on deciding to descend, sets State = DESCENDING AND immediately calls enemy.DescendLadder() so the barrel starts moving on the transition frame instead of stalling for one tick. Symmetrically handleEnemyDescending calls RollHorizontally() the instant it flips back to ROLLING. If you forget, the entity freezes for a frame at every transition.

How two entities reuse the one class

The reuse is the point of the generics — the same 17 lines drive a kinematic-character player and a dynamic-rigidbody barrel. The wiring is identical in shape, only the type arguments and handler maps differ:

// Player (game/entities/player/player.ts) — kinematic, input-driven
this.fsm = new StateMachine<Player, PlayerState>(this, {
  [PlayerStates.IDLE]: handlePlayerIdle,
  [PlayerStates.RUNNING]: handlePlayerRunning,
  [PlayerStates.FALLING]: handlePlayerFalling,
  [PlayerStates.JUMPING]: handlePlayerJumping,
  [PlayerStates.CLIMBING]: handlePlayerClimbing,
});

// Enemy (game/entities/enemy/enemy.ts) — dynamic, autonomous
this.fsm = new StateMachine<Enemy, EnemyState>(this, {
  [EnemyStates.ROLLING]: handleEnemyRolling,
  [EnemyStates.DESCENDING]: handleEnemyDescending,
  [EnemyStates.BOUNCING]: handleEnemyBouncing,
});

The handler signature (host) => void is what lets the bodies live in their own files (one file per state, under player/states/ and enemy/states/). A handler is just a free function that receives the entity and mutates it. No base class, no inheritance, no switch — the map IS the dispatch table.

The barrel / enemy FSM

The enemy proves the “works for any host” claim. A DK barrel comes in two flavors chosen at spawn (D28: data-driven — the old separate CrazyEnemy class was folded into one Enemy picked by a boolean). A NORMAL barrel starts ROLLING and oscillates ROLLING ↔ DESCENDING as it rolls girders and occasionally takes a ladder; a CRAZY barrel starts BOUNCING and never leaves it. Because the flavor is encoded purely as the starting state and a crazy barrel never enters a normal state (nor vice-versa), State === BOUNCING is itself the “is this crazy?” discriminator — no separate flag. The full bodies live in the Enemy section; here is the shape:

Note. The enemy FSM mirrors the player FSM on the game side on purpose (D3 / ADR-0003): the barrel's reactive movement — reverse at walls, turn at edges, bounce on girders — lives in its collision callbacks (OnCollisionEnter/Exit), while its cross-entity verdicts (kills the player, ignites the trash can) are NOT in the FSM at all. Those route through the declarative contact table (see the Contact rules / Contact registry sections). The FSM owns only the entity's own behavior, never how it interacts with others.
Hard-won victory (D28). Collapsing normal and crazy barrels into one state-driven class deleted an entire parallel CrazyEnemy type. The spawn site just passes crazy: boolean; the constructor maps it to a starting State. Adding a third barrel flavor later is a new state + handler, not a new class hierarchy — exactly the leverage the generic machine was built to give.

Modifying this: resist the urge to add features here. If you want entry/exit hooks, transition guards, or transition logging, the temptation is to grow this class — but every host already expresses entry work as “seed a value then return” and exit work as the next handler's first lines, so a richer machine would duplicate that. The one safe extension is purely additive: a new state is a new key in the host's handler map plus its constant in the host's state enum. Because the map is typed Record<S, ...>, the compiler will refuse to construct the machine until you supply the new handler — lean on that. Do NOT make Update() re-dispatch in a loop after a transition; the one-frame-per-transition guarantee is load-bearing for both the player and enemy bodies.

src/engine/types/playerStates.ts 24 LOC

The player's state alphabet: a frozen constant object mapping symbolic names to their string values, plus a derived union type. It is the S in StateMachine<Player, PlayerState> — the set of legal player states — and it is imported by player.ts and by all five handlePlayer* bodies so that transitions are written as player.State = PlayerStates.FALLING rather than bare strings. It lives under engine/types/ by convention even though the states themselves are game-flavored; the engine ships the type slot, the game fills it. (The enemy's equivalent, enemyStates.ts, lives game-side — see the Enemy section — reflecting that barrel states are wholly a game concept.)

const PlayerStates = { IDLE, RUNNING, JUMPING, FALLING, CLIMBING } as const

Five string constants under an as const assertion. The assertion is what makes each value a literal type ("idle", not string), which in turn makes the union below precise enough to drive the exhaustive handler map.

const PlayerStates = {
  IDLE: "idle",
  RUNNING: "running",
  JUMPING: "jumping",
  FALLING: "falling",
  CLIMBING: "climbing",
} as const;

export type PlayerState = typeof PlayerStates[keyof typeof PlayerStates];

export default PlayerStates;

Line notes: typeof PlayerStates[keyof typeof PlayerStates] is the idiom that turns the object's values into a union type — "idle" | "running" | "jumping" | "falling" | "climbing". The object is the runtime value (default export, used for PlayerStates.IDLE); the type is the named export (used wherever a PlayerState annotation is needed). The same enum-as-object pattern appears in enemyStates.ts, gameObjectType.ts, and friends — it is the project's house style for closed string sets.

Gotcha. The enum's inline comments describe a slightly fuller graph than the current handlers build. Three intended edges are not (yet) implemented: JUMPING → CLIMBING, JUMPING → IDLE/RUNNING (jumping never lands directly — it always passes through FALLING), and CLIMBING → FALLING / CLIMBING → JUMPING (climbing only ever exits to IDLE). Conversely, the handlers add an edge the comments omit: FALLING → JUMPING via coyote time. Trust the diagram above (it follows the code); treat the enum comments as the author's aspirational map. The comment on JUMPING that says “jump released or max jump time” is likewise stale — handlePlayerJumping notes the apex now “emerges from physics, not a fixed MaxJumpTime timer.”

How transitions are gated: coyote-time, jump buffering, variable height

These three feel mechanics are not states — they are flags on the Player that gate the transitions into JUMPING and shape the arc once there. The grounded states (IDLE, RUNNING) re-arm them; the airborne states (FALLING, JUMPING) spend and clear them. Conceptually:

  • Coyote-time — a short grace window (CoyoteTime = 0.033s, two frames, set in baseFeel.ts) during which you can still jump just after walking off a ledge. IDLE/RUNNING set CoyoteAvailable = true every grounded frame; on leaving the ground they stamp TimeFallWasEntered = Time.Elapsed. FALLING then permits a jump only while CoyoteAvailable && Time.Elapsed < TimeFallWasEntered + CoyoteTime, consuming it by setting CoyoteAvailable = false. That is the FALLING → JUMPING edge above.
  • Jump buffering — a jump pressed slightly before landing still fires the instant you touch down. The player shapecasts downward each frame; when ground sits within BufferJumpRange (1.5 units, DK profile) and the player is descending, GroundWithinBufferRange goes true and FALLING arms BufferJumpAvailable. IDLE and RUNNING gate their jump on isJump && BufferJumpAvailable, so a buffered press launches on the first grounded frame.
  • Variable jump height — releasing the button mid-rise cuts the jump short. JUMPING latches EndedJumpEarly = true once !isJump and you are past the MinJumpTime floor (0.05s), which boosts rise gravity by JumpEndedEarlyGravityModifier so the apex lands lower. Every jump entry (in IDLE, RUNNING, and the coyote branch in FALLING) resets EndedJumpEarly = false so the next jump starts un-cut.
Tip. The pattern to internalise: grounded handlers re-arm, airborne handlers spend. IDLE/RUNNING each end with the same three lines — CoyoteAvailable = true; EndedJumpEarly = false; if (!isJump) BufferJumpAvailable = true; — and JUMPING opens by clearing both CoyoteAvailable and BufferJumpAvailable. Because re-arming happens only on grounded frames, the flags decay naturally the moment you are airborne, which is exactly what gives coyote-time its hard time bound. The full line-by-line of each gate is in the Player States section.
Note (D10). Every handler reads the player's input snapshot (player.Input), refreshed at the top of Player.Update() via inputDevice.CaptureInto(this.Input) — not the raw device. This is the input-as-data seam: a network, replay, or AI source could fill the same struct, and the FSM would not know the difference. It is also why the FSM is testable in isolation — states are pure functions of (input snapshot, physics flags). See the Input section.

Modifying this: adding a player state is a three-touch change — add the constant here (the as const object), write its handlePlayerXxx.ts body under player/states/, and register it in the handler map in player.ts. The Record<PlayerState, ...> typing means you cannot forget the third step: omit the new handler and new StateMachine<Player, PlayerState>(...) fails to compile. Update the inline transition comments when you do — they have already drifted from the handlers once (see the gotcha), and they are the only human-readable map of the graph.
