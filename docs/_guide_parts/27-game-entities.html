World Entities — platforms, sensors, teleporter, trash can, win flag

These six classes are the level furniture: the solid floors and one-way girders the player walks on, the invisible trigger zones that move the camera and mark ladders, the teleporter, the oil can the enemies dive into, and the goal flag. Each is a thin game subclass of one of the two engine base classes — GameObject (solid) or GameSensor (pass-through) — that adds a little named state and almost no behavior. The behavior that reacts to these zones lives elsewhere: in the declarative contact table (Game Config — contactRules) and in the per-frame loop in Game Director. None of them are constructed directly; they are all built by the factory registry (Game Config — entityFactories) from parsed GLB level rows.

Note — the two base classes, in one breath. GameObject (see Engine Entities — gameObject) owns the Rapier rigid body + collider, the THREE mesh, the cached CurrentTranslation/CurrentSize mirrors, collision-group helpers, and the OnCollisionEnter/Exit + OnSensorEnter/Exit override hooks. GameSensor (see Engine Entities — gameSensor) extends it with two things only: setAsSensor(true) (turns the collider non-solid and arms KINEMATIC_FIXED active-collision-types so a fixed sensor still notices the kinematic player) and IsFullyInside(other) (X-axis AABB containment). The split below is exact: Platform and TrashCan are solid GameObjects; CameraSensor, LadderSensor, Teleporter and WinFlag are GameSensors.

Tip — how a zone turns into an action. A sensor entity here defines no callbacks of its own. Instead its constructor calls enableContactEvents() to arm Rapier collision events, and the rule "Player enters X → do Y" is registered once in Game Config — contactRules. The engine's Physics.Contacts registry (Engine Entities — contactRegistry) matches the two colliders by their type flag and fires the rule's handler. That is why these files are so short: they own data (a destination, a camera anchor, a direction, a fire flag) and expose one action — the table owns the when.

src/game/entities/platform.ts 90 LOC

The solid world geometry: every floor girder, every wall, and the special "one-way" girders you can jump up through from below but stand on from above. It is the only entity in this section with a real per-frame Update. One class covers three placed types (Platform, OneWayPlatform, and Wall) — the factory decides which by passing the isOneWayPlatform flag and, for walls, calling SetType(EntityType.WALL) afterward (see Game Config — entityFactories, createPlatform/createWall). It imports GameObject (base), GameObjectType (collider-shape enum — always CUBE here), CollisionGroups (puts it on the PLATFORM layer), EntityType (routing strings), and Player (its Update reads the player's Y and state). Lifecycle: constructed by the factory, driven each frame from GameDirector.Update, destroyed with the level via GameObject.Destroy().

// height); the player passes up through it from below.

this.setCollisionMask(CollisionGroups.DEFAULT);

Line-level notes:

  • Default enable point = position.y. The threshold defaults to the platform's own centre Y. The factory immediately overrides it with SetOneWayEnablePoint(metaNum(meta,"oneway") || data.position[2]), so an author can tune the pass-through line in Blender; absent that meta it falls back to the same Y.
  • objectName picks the routing string (Platform vs OneWayPlatform), then SetType(objectName) stamps it onto the body's userData.type so the contact table and collision logger can match it. Walls call SetType(EntityType.WALL) afterward in the factory — same class, different flag.
  • CUBE is hard-coded. The comment ties back to the engine decision that this is a cuboid/sphere-only physics world (see Engine Types — gameObjectType, the D5 lean): no polyline/convex colliders, slopes are just rotated cubes.
  • Group PLATFORM, mask DEFAULT. It lives on the platform layer and collides with everything that lists it — the player bounding box and enemies both mask in the platform bit (see Game Types — gameCollisionGroups). The group/mask helpers preserve the other half of the bitfield, so order (group then mask) is safe here.

public Update(player: Player)

The heart of the one-way mechanic, called every frame for every platform from GameDirector.Update (it pushes this.Platforms into a uniform UpdatableEntity[] loop and calls e.Update?.(this.Player, firstTrashCan)). It mutates IsOneWayActive based on where the player's feet are and whether the player is climbing. 

public Update(player: Player) {
  // Exit early if not a OneWayPlatform
  if (!this.isOneWayPlatform) {
    return;
  }

  // Solid once the player's feet rise above the enable point; pass-through below.
  if (player.CurrentTranslation.y - player.CurrentSize.y / 2 > this.oneWayEnablePoint) {
    this.IsOneWayActive = false;
  } else {
    this.IsOneWayActive = true;
  }

  // While the player is climbing, force this platform to pass-through so the
  // player can move up/down a ladder without landing on one-way platforms.
  if (player.State === "climbing") {
    this.IsOneWayActive = true;
  }
}
  • Early-out for non-one-way platforms. Regular platforms and walls share this class but must never toggle, so the guard returns before touching IsOneWayActive. This is why the field's comment promises it is "only ever true for one-way platforms".
  • Feet, not centre. The comparison uses player.CurrentTranslation.y - player.CurrentSize.y / 2 — the bottom edge of the player. Once the feet clear oneWayEnablePoint the platform becomes solid (IsOneWayActive = false) so the player lands; while the feet are still below it, it stays pass-through so the jump-up doesn't bonk.
  • Climbing override. The player.State === "climbing" string compare forces pass-through regardless of Y, so a player on a ladder never snags on a one-way girder mid-climb.
Gotcha — ordering is mandatory. GameDirector.Update calls Player.Update() first, then the platform loop, precisely so this Update reads this frame's player position and state (the inline comment at the call site spells it out). If you ever reorder the director loop and update platforms before the player, the one-way toggle goes one frame stale and the player will clip or bonce on girders.
Note — who consumes IsOneWayActive. This Update only sets the flag; the actual "let the player pass / stand" decision is made by the player/physics collision path that reads it (see Player — states). The platform is a pure state holder per frame. 
sequenceDiagram
  participant GD as GameDirector.Update
  participant P as Player.Update
  participant PL as Platform.Update
  GD->>P: Player.Update() (writes this frame pos + State)
  GD->>PL: Platform.Update(player)
  PL->>PL: feet > enablePoint ? solid : pass-through
  PL->>PL: State == "climbing" ? force pass-through

Modifying this: to add a new authored knob (say a moving platform's speed), add a private field, a SetX setter, and read the meta in createPlatform — do not revive the old value0/1 index hop. To change one-way feel, the only tuning surface is oneWayEnablePoint (per-instance via the oneway= meta) and the feet-vs-centre comparison in Update. Keep the non-one-way early-return first or you will accidentally make solid platforms pass-through.

An invisible trigger zone that re-anchors the camera when the player walks into it. It is a GameSensor that holds two things: a PositionData anchor (authored in Blender as cam=x_y_z on the mesh's texture) and a reference to the engine Camera. The "Player enters → adjust camera" rule lives in the contact table (Game Config — contactRules); this class only exposes the action. It imports THREE (for the Vector3 anchor), RAPIER (to pass an explicit fixed rigid-body desc), GameSensor, GameObjectType, Camera (engine), and EntityType. Built by createCameraSensor in the factory registry, which hands it the shared ctx.camera and fills PositionData from the meta.

public SetCameraPositionData(newPositionData: THREE.Vector3)

Stores the anchor. Called by the factory after parsing cam=x_y_z into a THREE.Vector3(camX, camY, camZ).

public ApplyCameraOnEnter()

The single action this zone performs, invoked by the contact rule ((sensor as CameraSensor).ApplyCameraOnEnter()). It pins the camera's rest anchor: 

public ApplyCameraOnEnter() {
  // Pins the camera's rest anchor to this zone's authored x,y. On a non-follow
  // level the camera eases to that spot on both axes (fixed/zone camera); on a
  // follow level only the Y is used (pins the vertical baseline the follow eases
  // around). Author the spot as `cam=x_y` on the sensor's texture.
  this.camera.SetBaselinePosition(this.PositionData.x, this.PositionData.y);
}

It forwards PositionData.x and .y to Camera.SetBaselinePosition (see Engine Camera — camera), which sets baselineX/baselineY and flips hasSensorBaseline = true. The Z component of PositionData is intentionally unused here — the 2D camera anchor is X/Y only. On a zone (non-follow) level the camera eases to both axes; on a follow level only Y is consumed and X keeps tracking the player.

Note — decoupling seam. The sensor never touches the camera's transform directly; it calls one public method. That is the same C5 "camera is engine-pure" discipline used in the director: the camera consumes plain scalars and snapshots, never game objects.

Modifying this: to add a per-zone zoom or follow-mode switch, add fields + setters here and a second line in ApplyCameraOnEnter calling a new public camera method — keep the camera ignorant of CameraSensor. Remember an enter-only rule: there is no ApplyCameraOnExit, so two overlapping zones will leave the last-entered anchor pinned.

Marks a ladder region. The smallest entity here: a GameSensor carrying a single numeric Direction. Unlike the other sensors, it is not event-driven and does not call enableContactEvents() — ladder detection is the hard-won "fully inside" polling path (D9), read every frame by GameDirector. One class serves three roles — top, core, and bottom — distinguished only by the EntityType string the factory stamps via SetType, and sorted into three separate director arrays (LadderTopSensors / LadderCoreSensors / LadderBottomSensors). Imports RAPIER (fixed body desc), GameSensor, GameObjectType, EntityType.

Hard-won victory — D9, the "fully inside" climb gate. Ladder mounting is deliberately kept out of the contact/event system. The file's header comment states it plainly: ladder detection is "the hard-won 'fully inside' climb behavior (D9), polled by GameDirector / read by the player + enemy. That path is preserved and deferred — this sensor defines no contact callbacks, so it arms no physics events." The bug D9 prevents: an enemy or player brushing the edge of a ladder collider and snapping into climb state. Instead, GameDirector.Update polls GameUtils.IsAnySensorTriggeredObjectFullyInside(this.LadderCoreSensors, this.Player), which only returns true when GameSensor.IsFullyInside confirms the player's whole X extent sits inside the core sensor's X extent. Top/bottom use the looser IsAnySensorTriggered.

// the ignite rule (contactRules.ts)

The TrashCan has no method that lights itself; the rule does. From Game Config — contactRules: 

// Enemy falls into a TrashCan -> enemy destroyed, can catches fire.
contacts.Add(
  byType(EntityType.ENEMY),
  byType(EntityType.TRASH_CAN),
  "enter",
  (enemy, can) => {
    (can as TrashCan).IsOnFire = true;
    Emitter.emit("gameObjectRemoved", enemy);
  }
);

Two effects on one solid-collision "enter": the can's IsOnFire flips, and the engine emits gameObjectRemoved on the event bus (Engine Events — eventBus) to despawn the enemy. The TrashCan itself stays put. Separately, the director passes TrashCans[0] into enemy.Update(player, firstTrashCan) so the descending barrel knows where to aim (see Enemy — states).

Note — solid vs sensor, the whole reason this is a GameObject. Every other trigger in this section is a pass-through GameSensor. The trash can is the exception because the desired physics is a real collision — the barrel must land in the can — and the ignite is keyed on a solid-collision "enter" phase, not a "sensorEnter".

Modifying this: to make the fire hurt the player, add a contact rule pairing TRASH_CAN with PLAYER and guard it on IsOnFire inside the handler — leave this class a plain state holder. If you want multiple cans the enemies target, the director currently aims at index 0 only; generalize the enemy targeting logic in Game Director, not this file.

Imports just GameObjectType, GameSensor, EntityType — the leanest entity here. Built by createWinFlag. It has no fields, no setters, and (as the director comment notes) no Update.

Tip — the shared shape of all six. Constructor → createObjectPhysics (body + collider + userData.type) → arm (solid groups, or setAsSensor + enableContactEvents) → optional setters the factory calls from meta. Reactions live in the contact table; per-frame state (only Platform) lives in Update; teardown is the inherited GameObject.Destroy() the director calls on level unload. Add a new world entity by copying this shape and registering it in Game Config — entityFactories (and a rule in Game Config — contactRules if it should do something).
