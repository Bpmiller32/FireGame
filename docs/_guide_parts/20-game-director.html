GameDirector & GameUtils — the conductor

GameDirector is the single game-layer object that owns every Donkey-Kong rule: it holds the typed entity arrays, wires the lifecycle event bus, runs the per-frame update order, drives the registry-based level pipeline, and spawns barrels off the frame clock. GameUtils is the thin static facade game code calls instead of reaching into engine helpers directly — re-exporting pure math/physics helpers and adding a handful of game-flavoured array/sensor utilities the director leans on each frame.

The "central controller for game state, entities, levels, spawning." It is the one place that knows the game has a Player, barrels, ladders, trash cans (oil cans) and a win flag — the engine layer below it knows none of that. It is constructed once (by the game bootstrap) and grabs the Experience singleton in its constructor to cache the Scene, Time, Camera and ResourceLoader handles it needs. From there its life has four phases: constructor (cache systems + setupEventHandlers), Update (the sim pass — runs inside the fixed-timestep loop, driving player, enemies, level entities and ladder polling), RenderUpdate(alpha) (the per-frame pass — interpolates the moving entities by alpha and then follows the interpolated player with the camera), and Destroy (off its event handlers by ref, unload the level). Almost everything else hangs off events: it never polls "is the game over?" — it reacts to gameStart/gameOver/gameWin/gameReset on the event bus.

Its imports tell the whole story of the seam it straddles: from engine/ it pulls Experience, Camera, Time, ResourceLoader, GameObject, Emitter, the engine-pure CameraTarget/CameraFollowState and PlayerStates types, and the seeded-PRNG helpers seedRandom/randomRange (from mathHelpers); from game/ it pulls every concrete entity class (Player, Platform, Enemy, TrashCan, WinFlag, CameraSensor, LadderSensor, Teleporter), the GameUtils facade, and the two config tables that make it data-driven: ENTITY_FACTORIES (with FactoryContext) and the LEVEL_REGISTRY/FEEL_PROFILES registry.

type UpdatableEntity = { Update?(player: Player, trashCan?: TrashCan): void; }

A structural type used by the uniform level-entity update loop. Update is optional so an array of mixed level entities can be ticked with optional chaining (e.Update?.(...)) — anything without an Update is simply skipped instead of needing a runtime guard. The second arg is the trash can the enemy reads; entities that don't need it ignore it. This is the type-level expression of the engine's "uniform update contract".

type UpdatableEntity = {
  Update?(player: Player, trashCan?: TrashCan): void;
};

// typed entity arrays

The director's whole world is a set of strongly-typed arrays — the "derived query views" the rest of the game indexes by type. Player is a single instance (definite-assignment !, spawned from level data); the rest are arrays. Note the deliberate splits: Platforms vs Walls are both Platform[] but kept separate so the enemy can reverse on walls but ground on platforms; the three ladder sub-arrays (LadderTopSensors/LadderCoreSensors/LadderBottomSensors) are split because the player and barrel ask each set a different question.

constructor()

First seeds the gameplay PRNG deterministically (and logs the seed), then caches the engine subsystems off the Experience singleton, creates the empty GraphicsObject overlay holder, then wires the event bus. No physics body or entity is created here — the world is empty until resourcesReady fires and the default level loads.

// Seed gameplay randomness deterministically (same seed -> same barrel sequence).
seedRandom(RNG_SEED);
console.log(`[FireGame] RNG seed = ${RNG_SEED} (deterministic run)`);

this.experience = Experience.GetInstance();
this.scene = this.experience.Scene;
this.time = this.experience.Time;
this.camera = this.experience.Camera;
this.resources = this.experience.Resources;
this.GraphicsObject = new GameObject();

this.setupEventHandlers();
Note. Pulling everything from Experience.GetInstance() is the decoupling seam: the director never imports or constructs the renderer/physics/camera itself, it asks the singleton. That is how the same director could run under a test harness with stub systems.
Hard-won victory (deterministic runs). The module const RNG_SEED = 1337 is fed to seedRandom here at construction, so every play replays the identical barrel sequence — and resetGameState re-seeds with the same value, so a reset replays it again from the top. All gameplay randomness (spawn cadence, the barrel's take-ladder dice) now flows through the one seeded mulberry32 PRNG instead of the global unseeded Math.random, which is what makes a run reproducible. Swap to seedRandom(Date.now()) for varied runs.

private setupEventHandlers()

The director's nervous system — it listens on the event bus and never calls these flows directly. Each handler is a tiny stored arrow field (onResourcesReady, onGameStart, onGameOver, onGameWin, onGameReset, onSwitchLevel, onGameObjectRemoved) — and setupEventHandlers just registers each one by reference. Storing the refs is what lets Destroy remove exactly these listeners (and no one else's) from the shared bus. There is no wall-clock timer anywhere in the director any more — destroyed-barrel cleanup moved into Update (see GameUtils.CompactDestroyedObjects).

EventHandler does
resourcesReadyawait this.loadLevelByIndex(DEFAULT_LEVEL_INDEX) — boots the default level. Loading emits gameStart on success.
gameStartPhysics.SetPaused(false) + isSpawningEnemies = true — go live.
gameOverPhysics.SetPaused(true) + isSpawningEnemies = false — freeze, stop barrels.
gameWinIdentical freeze to gameOver — so the win screen holds and a stray barrel can't flip "Game Win!" to "Game Over!".
gameResetPhysics.SetPaused(false) + resetGameState().
switchLevelawait this.switchToNextLevel() — the F2 cycle.
gameObjectRemovedremovedGameObject.Destroy() — the single teardown sink for any entity anyone wants gone.
 
// Stored arrow fields capture `this`; they're only invoked on events.
private onGameObjectRemoved = (removed: GameObject) => {
  removed.Destroy();
};

// setupEventHandlers() — register every stored handler BY REFERENCE.
Emitter.on("resourcesReady", this.onResourcesReady);
Emitter.on("gameStart", this.onGameStart);
Emitter.on("gameOver", this.onGameOver);
Emitter.on("gameWin", this.onGameWin);
Emitter.on("gameReset", this.onGameReset);
Emitter.on("switchLevel", this.onSwitchLevel);
Emitter.on("gameObjectRemoved", this.onGameObjectRemoved);

Line-level notes: the gameObjectRemoved handler is the decoupling pivot — entities (or resetGameState/UnloadLevelData) emit "remove me" and the director is the only thing that actually calls Destroy(), so teardown is centralized. Registering each handler by its stored ref (instead of an inline closure) is the bit that makes Destroy able to off precisely these listeners off the shared bus — a bare Emitter.off("gameStart") would wipe other modules' handlers (e.g. App.vue's) too.

Hard-won victory (bus hygiene). Every handler is a captured-this arrow field, so the on here and the off in Destroy pass the same function reference. Registering all seven (including gameWin) means Destroy can later remove all seven — no leaked listener closing over a dead instance on HMR / unmount.

public Update()

The sim pass — run inside the engine's fixed-timestep loop (0..N times per frame, see Experience.onTick), so all movement integrates at the constant step. Camera and visual interpolation are not here — they run once per frame in RenderUpdate, decoupled from the sim rate. The order of the steps is load-bearing — several downstream reads depend on running after the player has moved this step.

flowchart TD
  A["isLoading? -> return"] --> B["Player.PhysicsBody ready? -> return"]
  B --> C["Player.Update()"]
  C --> E["trySpawn()"]
  E --> F["Enemies.forEach(e.Update(Player, firstTrashCan))"]
  F --> CD["GameUtils.CompactDestroyedObjects(Enemies) — swap-remove in place"]
  CD --> G["Platforms: e.Update?.(Player, firstTrashCan)"]
  G --> H["Player.UpdateLadderState(top, core, bottom)"]
if (this.isLoading) return;
if (!this.Player?.PhysicsBody) return;

this.Player.Update();

// Spawn + update enemies, then compact destroyed barrels IN PLACE (stable ref).
this.trySpawn();
const firstTrashCan = this.TrashCans[0];
this.Enemies.forEach((enemy) => enemy.Update(this.Player, firstTrashCan));
GameUtils.CompactDestroyedObjects(this.Enemies);

const updatableLevelEntities: UpdatableEntity[] = this.Platforms;
for (const e of updatableLevelEntities) {
  e.Update?.(this.Player, firstTrashCan);
}

this.Player.UpdateLadderState(
  GameUtils.IsAnySensorTriggered(this.LadderTopSensors),
  GameUtils.IsAnySensorTriggeredObjectFullyInside(this.LadderCoreSensors, this.Player),
  GameUtils.IsAnySensorTriggered(this.LadderBottomSensors)
);

Line-level notes: the two guard returns matter — isLoading freezes all game logic while a level is being torn down/rebuilt (the Vue loading screen covers the canvas), and !this.Player?.PhysicsBody bails before the player exists (between boot and first spawn). firstTrashCan = this.TrashCans[0] is the oil can passed to both the enemies and the platform loop. GameUtils.CompactDestroyedObjects(this.Enemies) runs right after the enemy tick so a barrel flagged IsBeingDestroyed this step (hit the player, lit the can) is swept this frame — no array reallocation, the same array instance the spawner pushes into. The Platforms loop runs after Player.Update() so the one-way-platform toggle reads this step's player state. Ladder detection is the last step and is still polled manually (see the GameUtils helpers) — the comment flags it as a TODO to convert to sensor callbacks. The three arguments map to top/core/bottom: top & bottom use plain IsAnySensorTriggered, but the core uses the fully-inside variant so the player must be horizontally centred on the ladder to climb (the D9 idiom).

Hard-won victory (no per-frame allocations). Destroyed barrels used to be rebuilt into a fresh array on a 5-second setInterval (GameUtils.RemoveDestroyedObjects returned a new array, reassigned). Now CompactDestroyedObjects swap-removes in place every frame — promptly, with no GC churn, and keeping the array reference stable (which levelEntityArrays/teardown rely on). The whole director is now timer-free.
Note. WinFlag is conspicuously absent from the update loop — the comment records that it "no longer updates" because winning routes through the contact table (contactRules), not a per-frame shapecast. And the camera no longer ticks here at all: it moved to RenderUpdate so it follows the smooth, interpolated player rather than the stepped sim position.

public RenderUpdate(alpha: number)

The per-frame render pass, called by Experience.OnRenderUpdate once per rendered frame with the interpolation alpha (the fraction accumulator / dt through the next sim step). It lerps the moving entities between their last two sim states so motion is smooth above the sim rate, then follows the interpolated player with the camera. Render and sim rates are fully decoupled — buttery on a 144 Hz display, no judder on a 60 Hz one.

if (this.isLoading) return;
if (!this.Player?.PhysicsBody) return;

this.Player.InterpolateGraphics(alpha);
for (const enemy of this.Enemies) {
  enemy.InterpolateGraphics(alpha);
}

// Camera follows the interpolated visual position (C5 decouple: plain snapshot).
this.camera.Update(this.cameraTargetFromPlayer(this.Player));

Line-level notes: same two guards as Update (loading freeze, player-ready). InterpolateGraphics(alpha) on the player and each enemy lerps the mesh between the previous and current sim translation/rotation and writes each entity's RenderTranslation — the interpolated position other systems read. The camera Update runs last and after the player's interpolation, so cameraTargetFromPlayer can hand it the freshly-computed RenderTranslation.

Hard-won victory (render interpolation). Splitting the per-step sim from the per-frame render is what removes the old one-frame camera lag for good: the player is no longer "updated before the camera in the same frame" (the previous fix) — instead the sim integrates at a fixed step and the render pass interpolates toward it, and the camera follows that interpolated player. Teleports/resets still snap because TeleportToPosition collapses the interpolation state (prev = current = render = target), so they jump instead of sliding.

private cameraTargetFromPlayer(player: Player): CameraTarget

Maps the game Player into the engine's plain CameraTarget snapshot. The camera reads only these five scalars, never the game Player or PlayerState — so camera.ts imports nothing from game/. The switch translates the game's PlayerStates enum into the engine-neutral CameraFollowState (with GROUNDED as the default for idle/running). Called from RenderUpdate, it now reads the player's interpolated RenderTranslation (set by InterpolateGraphics a line earlier) so the camera tracks the smooth visual player, not the stepped sim position.

switch (player.State) {
  case PlayerStates.CLIMBING: followState = CameraFollowState.CLIMBING; break;
  case PlayerStates.FALLING:  followState = CameraFollowState.FALLING;  break;
  case PlayerStates.JUMPING:  followState = CameraFollowState.JUMPING;  break;
  default:                    followState = CameraFollowState.GROUNDED;
}

return {
  // Follow the INTERPOLATED render position (set by InterpolateGraphics just
  // before this in RenderUpdate), not the stepped sim position.
  x: player.RenderTranslation.x,
  y: player.RenderTranslation.y,
  velocityX: player.NextTranslation.x,
  followState,
  isGrounded: player.IsTouching.ground,
};
Hard-won victory (C5). This single method is the camera decouple. Before C5 the engine Camera read player.State, player.IsTouching.ground etc. directly and thus imported game types. Now the game builds the snapshot and hands it in, so a different game (or a multi-player follow) just fills the struct differently. The five fields are exactly the scalars the camera's vertical-follow easing used to read off the player, so the easing branching is preserved unchanged. See the Camera section for how followState drives the GROUNDED/CLIMBING/FALLING/JUMPING easing.

Level pipeline

Loading a level is a chain: loadLevelByIndex orchestrates — it now resources.ParseLevels the GLB first (before any teardown), validates the optional graphics asset, then unloads the old level and calls LoadLevelData (spawn), which iterates the parsed rows through the ENTITY_FACTORIES registry using a FactoryContext built by getFactoryContext. (LoadGLBLevel is no longer on this internal path — it survives only as a public convenience API.)

flowchart TD
  R["loadLevelByIndex(index)"] --> P["resources.ParseLevel(entry.source) -> LevelData<br/>(parse FIRST, before unload)"]
  P --> V["validate entry.graphics asset is loaded"]
  V --> U["UnloadLevelData (if levelLoaded)"]
  U --> L["LoadLevelData(levelData)"]
  L --> SP["setPlayerStart() + setCameraStart()"]
  SP --> CTX["getFactoryContext() aliases the typed arrays"]
  CTX --> LOOP["for each row: ENTITY_FACTORIES[type]?(data, ctx, type)"]
  LOOP --> RS["resetSpawner()"]
  RS --> OV["GraphicsObject overlay (if entry.graphics)"]
  OV --> FEEL["applyFeelProfile + camera.SetCameraFollow"]
  FEEL --> GS["emit gameStart"]
  GS -. on throw .-> RB["catch: roll back currentLevelIndex / CurrentLevelName"]

public async LoadGLBLevel(glbPath: string)

Thin adapter: hands the bundled GLB URL to the ResourceLoader and awaits parsed LevelData, then forwards to LoadLevelData. Levels are GLB/glTF only (Blender/Blockbench export); glbPath is normally the registry entry's source. No longer used on the internal load path — loadLevelByIndex now parses the GLB itself (so it can parse before unloading); this stays as a public one-call convenience (parse + spawn) for callers outside the registry flow.

const parsedLevelData = await this.resources.ParseLevel(glbPath);
await this.LoadLevelData(parsedLevelData);

public async LoadLevelData(levelData: LevelData)

The data-source-agnostic spawn pass — turns parsed rows into live entities. Stores the data, ensures a single ambient light exists (lazily created once, reused across levels), places the player and camera first (so sensors have a target), then runs the factory loop, then resets the spawn cadence.

this.LevelData = levelData;

if (!this.ambientLight) {
  this.ambientLight = new THREE.AmbientLight(0xffffff, 1);
  this.scene.add(this.ambientLight);
}

this.setPlayerStart();
this.setCameraStart();

const ctx = this.getFactoryContext();
for (const data of Object.values(this.LevelData)) {
  const factory = ENTITY_FACTORIES[data.type];
  if (factory) factory(data, ctx, data.type);
}

this.resetSpawner();

Line-level notes: the factory loop is the heart of the data-driven spawn — ENTITY_FACTORIES[data.type] resolves the right builder by the EntityType string the parser emitted; if (factory) silently skips rows with no factory (PlayerStart/CameraStart handled just above, plus GraphicsObject/Unknown). Each factory pushes into ctx's arrays — which alias the director's real query arrays (see getFactoryContext), so they're populated in place. resetSpawner() at the end means a fresh level starts the barrel cadence clean; whether it actually runs is gated separately by isSpawningEnemies.

Gotcha. Order matters: setPlayerStart() must run before the factory loop because some sensors (and the camera) are positioned/targeted relative to the player. The ambient light is created once and never per-level — UnloadLevelData removes it, and the next load recreates it.

Builds the plain struct the factories push into. Critically these are the same array instances the director queries (trashCans[0] for the enemy aim, the ladder sub-arrays for the climb checks) — not copies — so pushing in a factory mutates the director's live state.

public UnloadLevelData()

Tears down the current level. A local generic destroyAll<T extends GameObject> helper emits gameObjectRemoved for every entity (routing teardown through the central sink) then truncates the array via entities.length = 0 — and instead of hand-listing each array, it maps that helper over levelEntityArrays(), so every typed array is cleared from one place. Then the graphics overlay is destroyed and replaced with a fresh instance, and the ambient light is removed.

// emit removal for each entity, then empty the array in place (stable ref)
const destroyAll = <T extends GameObject>(entities: T[]) => {
  entities.forEach((entity) => Emitter.emit("gameObjectRemoved", entity));
  entities.length = 0;
};
this.levelEntityArrays().forEach((arr) => destroyAll(arr));

// Destroy() is idempotent now (latches IsBeingDestroyed), so swap in a fresh
// GraphicsObject for the next level rather than destroying-then-reusing the old one.
this.GraphicsObject.Destroy();
this.GraphicsObject = new GameObject();
if (this.ambientLight) {
  this.scene.remove(this.ambientLight);
  this.ambientLight = undefined;
}

Line-level notes: destroyAll is now <T extends GameObject>-generic (no more any[]) and is fed every array by levelEntityArrays(), so adding a new entity kind never means hand-editing teardown again. entities.length = 0 empties the array in place, preserving the same array instance — important because getFactoryContext handed those instances out as aliases; reassigning to [] would orphan the alias. The Player is not in levelEntityArrays() and is not destroyed here — it persists across level switches and is merely teleported by setPlayerStart. The ambient light is removed and nulled so the lazy re-create in LoadLevelData fires again on the next load.

Hard-won victory (idempotent Destroy). GameObject.Destroy() now latches IsBeingDestroyed on its first line and no-ops on a second call. That makes the long-lived GraphicsObject un-reusable after teardown (a re-load would hit a dead, permanently-latched object), so each unload replaces it with new GameObject() — a clean overlay holder per level, and no never-resetting destroy latch.

private setPlayerStart()  /  private setCameraStart()

Scan the level data for the spawn-point rows. setPlayerStart reads position[0] (X) and position[2] (the glTF Z that maps to the game's vertical) — if the player already exists it teleports, otherwise it constructs the one Player and flips playerHasBeenSpawned. setCameraStart reads all three components and teleports the camera (note the argument order (x, z, y) mirroring the camera's signature).

// setPlayerStart
if (this.playerHasBeenSpawned) {
  this.Player.TeleportToPosition(x, z);
  return;
}
this.Player = new Player({ width: 1.75, height: 4 }, { x, y: z });
this.playerHasBeenSpawned = true;

// setCameraStart
this.camera.TeleportToPosition(x, z, y);

Line-level notes: both loops return on the first matching row — there is exactly one spawn point of each kind. The axis swap (position[2] feeding the game's y) is the recurring glTF-Z-is-game-Y convention used throughout the factory registry too. The player is 1.75 × 4 world units — the DK box.

private resetGameState()  /  private resetSpawner()

resetGameState (fired by gameReset) returns the world to its level-start: re-seed the PRNG with RNG_SEED (so the reset replays the identical barrel sequence), teleport player and camera back to their initial positions, emit gameObjectRemoved for every barrel, douse every trash can (IsOnFire = false), then re-enable spawning and reset the cadence. resetSpawner just zeroes the three cadence counters.

// resetGameState
// Re-seed so a reset replays the identical sequence (deterministic restart).
seedRandom(RNG_SEED);
this.Player.TeleportToPosition(this.Player.InitialTranslation.x, this.Player.InitialTranslation.y);
this.camera.TeleportToPosition(this.camera.InitialPosition.x, this.camera.InitialPosition.y, this.camera.InitialPosition.z);
this.Enemies.forEach((enemy) => Emitter.emit("gameObjectRemoved", enemy));
this.TrashCans.forEach((trashCan) => (trashCan.IsOnFire = false));
this.isSpawningEnemies = true;
this.resetSpawner();

// resetSpawner
this.timeSinceLastSpawn = 0;
this.currentSpawnInterval = 0;
this.enemyCount = 0;

Line-level notes: dead barrels are flagged here but physically swept by the per-frame CompactDestroyedObjects in Update (and the next UnloadLevelData); reset doesn't wait. Setting currentSpawnInterval = 0 is significant — it re-arms the "opening throw" branch in trySpawn (the first barrel after a reset is again Kong's oil-can shot).

private async loadLevelByIndex(index: number)

The single load path. Boot, the F2 cycle, and the dat.gui level select all funnel through here. It applies the registry entry's per-level setup (feel profile, camera follow, optional graphics overlay), and is the only method that emits gameStart on a successful load. The ordering is the key: it parses the new GLB and validates the graphics asset BEFORE tearing down the current level, and on any failure it rolls back the index/name bookkeeping — so a failed switch is a no-op, never a teardown into emptiness.

const entry = LEVEL_REGISTRY[index];
if (!entry || this.isLoading) return;

this.isLoading = true;
Emitter.emit("loadingStarted");

// Remember the current level so a failed load can roll back to it.
const prevIndex = this.currentLevelIndex;
const prevName = this.CurrentLevelName;

try {
  if (!entry.source) {
    throw new Error(`level "${entry.name}" has no resolvable GLB source — is it in src/assets/levels?`);
  }

  // Parse the GLB FIRST, before tearing down the current level. A parse throw
  // (missing/corrupt GLB) leaves the old level fully intact — we catch + roll back.
  const parsedLevelData = await this.resources.ParseLevel(entry.source);

  // Validate the optional graphics overlay is present BEFORE the unload too, so a
  // missing/failed graphics item fails into the rollback while the old level is intact.
  if (entry.graphics && !this.resources.Items[entry.graphics]) {
    throw new Error(`level "${entry.name}" graphics asset "${entry.graphics}" failed to load`);
  }

  // Parse + validate succeeded — now it's safe to swap.
  if (this.levelLoaded) {
    this.UnloadLevelData();
  }

  this.currentLevelIndex = index;
  this.CurrentLevelName = entry.name;

  await this.LoadLevelData(parsedLevelData);

  if (entry.graphics) {
    await this.GraphicsObject.CreateObjectGraphics(this.resources.Items[entry.graphics]);
  }

  this.applyFeelProfile(entry.feelProfile);
  this.camera.SetCameraFollow(entry.cameraFollow);

  this.levelLoaded = true;
  Emitter.emit("gameStart");
} catch (error) {
  // Roll back to the last good level so a failed switch is a no-op.
  console.error(`Failed to load level "${entry.name}":`, error);
  this.currentLevelIndex = prevIndex;
  this.CurrentLevelName = prevName;
} finally {
  this.isLoading = false;
  Emitter.emit("loadingFinished");
}
Hard-won victory (failed-load rollback). Parse-and-validate happen before the unload, and the catch restores currentLevelIndex/CurrentLevelName to the values it snapshotted into prevIndex/prevName. The two throwing steps that used to live after teardown (a bad GLB, a missing graphics overlay) now both fire while the old level is still standing — so a broken registry row no longer drops the player into an empty world with mismatched bookkeeping. LoadLevelData's spawn pass is deterministic, so once parse+validate pass the swap is safe.
Hard-won victory (D5 — the re-entrancy guard). if (!entry || this.isLoading) return; plus isLoading = true is "one boolean, old-school". The GLB parse is async and callers are fire-and-forget, so without the guard a second load (mashing F2, or picking a level mid-load) would interleave teardown + respawn → double entities, a leaked ambient light, and a mismatched index. The same flag freezes Update (top of the frame loop) so logic doesn't tick against half-built state, and loadingStarted/loadingFinished bracket the window so the Vue loading screen covers the seam.
Gotcha (fail loud, then fail safe). Two deliberate error choices. (1) An empty entry.source (a levelUrl() miss) is thrown here rather than letting "" reach the loader — gltfLoader.load("") would fetch the HTML page and throw a confusing parse error far from the cause. (2) That throw is caught locally and only console.error'd — it must not reject out of this method, because the boot handler awaits it; an unhandled rejection there would skip gameStart and leave a frozen, physics-paused game. The finally always clears isLoading and emits loadingFinished, so the overlay never hangs even on a failed load. And because gameStart is inside the try after the load steps, a failed load never wrongly unpauses.
Note. This is the SINGLE place the game starts running. Switching out of a paused game-over or win screen by loading a level can't leave the new level frozen, because every entry point reaches gameStart here.

Enemy spawning

private trySpawn()

Frame-driven barrel spawning, called once per frame from Update. It replaced an old wall-clock setInterval so there is now one clock (Time.Delta) and a fresh boot spawns without a reset. Bails immediately if spawning is off or physics is paused. The cadence: the first barrel is Kong's opening throw (a crazy barrel aimed at the oil can to light it); afterwards a normal/crazy mix every few seconds, every 8th one crazy.

if (!this.isSpawningEnemies || this.experience.Physics.IsPaused) return;

this.timeSinceLastSpawn += this.time.Delta;

// First barrel: Kong's opening throw — a crazy barrel aimed at the oil can.
if (this.currentSpawnInterval === 0) {
  this.timeSinceLastSpawn = 0;
  this.currentSpawnInterval = randomRange(3, 4); // 3-4 seconds
  this.spawnOilCanBarrel();
  return;
}

// Subsequent barrels: mostly normal, every 8th crazy.
if (this.timeSinceLastSpawn >= this.currentSpawnInterval) {
  this.timeSinceLastSpawn = 0;
  this.currentSpawnInterval = randomRange(2, 3); // 2-3 seconds
  this.enemyCount++;
  if (this.enemyCount % 8 === 0) {
    this.SpawnCrazyEnemy();
  } else {
    this.SpawnEnemy();
  }
}

Line-level notes: currentSpawnInterval === 0 is the one-shot opening-throw sentinel — it is only zero at boot/reset, fires the oil-can barrel, then sets a real 3–4s interval so it never re-enters. The interval values come from the seeded randomRange(3, 4) / randomRange(2, 3) (not the global Math.random), so the spawn cadence is part of the reproducible, seeded run. The cadence counters are advanced before the (async) spawn (timeSinceLastSpawn = 0 is set first) so a stutter frame can't double-spawn. The double-guard against Physics.IsPaused belts-and-suspenders with isSpawningEnemies (both are cleared on gameOver/gameWin).

private spawnOilCanBarrel()

The opening crazy barrel Kong hurls at the oil can to light it. Spawns at the normal throw point { x: -13, y: 50 } but aims its bounce drift toward the first trash can so it reaches and ignites it (igniting also consumes the barrel, via the contact table). Drift defaults rightward (aim = 0 → SpawnCrazyEnemy treats >= 0 as +1) when the can is directly below or absent.

const spawn = { x: -13, y: 50 };
const can = this.TrashCans[0];
let aim = 0;
if (can) aim = Math.sign(can.CurrentTranslation.x - spawn.x);
this.SpawnCrazyEnemy(spawn, aim);

public Destroy()

Tears the director down: offs every event handler by reference, then unloads the level. There is no timer to clear any more.

// Remove EXACTLY our handlers (pass the ref) so we don't wipe other modules'
// listeners on the shared bus. gameWin is removed too now (it used to leak).
Emitter.off("resourcesReady", this.onResourcesReady);
Emitter.off("gameStart", this.onGameStart);
Emitter.off("gameOver", this.onGameOver);
Emitter.off("gameWin", this.onGameWin);
Emitter.off("gameReset", this.onGameReset);
Emitter.off("switchLevel", this.onSwitchLevel);

// Tear down level entities BEFORE removing gameObjectRemoved: UnloadLevelData EMITS
// gameObjectRemoved, and that handler is what turns it into the actual entity.Destroy().
this.UnloadLevelData();
Emitter.off("gameObjectRemoved", this.onGameObjectRemoved);
Hard-won victory (teardown hygiene). Two fixes here. (1) Every off now passes the stored handler ref (matching the on in setupEventHandlers), so it removes only this director's listeners — a bare Emitter.off("gameStart") would wipe other modules' handlers off the shared bus. (2) gameWin — which previously had no matching off and leaked a listener closing over the dead instance — is now removed with the rest.
Gotcha (order matters). UnloadLevelData() runs before Emitter.off("gameObjectRemoved", …) on purpose: unload emits gameObjectRemoved for every entity, and the onGameObjectRemoved handler is what actually calls each entity's Destroy(). Off it first and the teardown emit would reach nobody — leaking every entity's physics body and meshes on unmount / HMR. So: tear down, then off the sink.

Modifying this: to add a new level-data-placed entity, you almost never touch this file — add a factory in entityFactories and a typed array via getFactoryContext/the field block, and the factory loop picks it up. To add a new level, add a row to LEVEL_REGISTRY; everything funnels through loadLevelByIndex. To change game flow, add an Emitter.on(...) in setupEventHandlers (and a matching off in Destroy) rather than calling logic directly — the director's whole design is "react to events, never poll". To tune difficulty, the spawn cadence numbers live in trySpawn; the barrel physics live in enemy.

The "game utilities facade — the single entry point for game-layer helpers; game code goes through here, not engine helpers directly." It is a class with only static members — never instantiated, no lifecycle. Two jobs: (1) re-export the engine's pure helpers (mathHelpers and physicsHelpers) under game-friendly names so the whole game has one import surface, and (2) hold a few game-flavoured helpers the director and enemy call directly — array sweeping and the sensor "is it triggered / fully inside?" polling. public static RemoveDestroyedObjects<T extends GameObject>(existingArray: T[]): T[]

Returns a new array containing only the objects not flagged IsBeingDestroyed. Generic over any GameObject subtype so it preserves the element type. Returns: the filtered (kept) array.

const kept: T[] = [];
for (const object of existingArray) {
  if (!object.IsBeingDestroyed) {
    kept.push(object);
  }
}
return kept;

No longer called by the director. It used to run on a 5-second cleanup interval (this.Enemies = GameUtils.RemoveDestroyedObjects(this.Enemies)), but because it returns a fresh array — reassigning the field — it could only be used on arrays the director didn't alias out through a FactoryContext. The director now sweeps barrels every frame with the in-place CompactDestroyedObjects below (stable array ref); this filtering variant remains as a general-purpose utility.

public static CompactDestroyedObjects<T extends GameObject>(arr: T[]): void

Removes destroyed objects from an array in place via swap-remove, keeping the same array reference. Walks back-to-front, and for each entry flagged IsBeingDestroyed overwrites it with the last element and pop()s the tail. Returns nothing — it mutates the array. Cheaper than RemoveDestroyedObjects (no new array, no GC churn), and the stable reference is what lets long-lived systems keep holding the same array.

for (let i = arr.length - 1; i >= 0; i--) {
  if (arr[i].IsBeingDestroyed) {
    arr[i] = arr[arr.length - 1];
    arr.pop();
  }
}

Where the director calls it. Every frame in Update, right after the enemy tick — GameUtils.CompactDestroyedObjects(this.Enemies). This replaced the old 5-second wall-clock sweep: dead barrels are dropped promptly, and because the array instance never changes, levelEntityArrays() and the teardown path can keep relying on the stable Enemies reference. Swap-remove is order-unstable (it reorders survivors), which is harmless for the barrel pool.

Hard-won victory (allocation-free hot path). Sweeping on a per-frame loop with a fresh-array filter would allocate (and GC) an array every frame; in-place swap-remove does zero allocation and runs in one backward pass. It's part of the same Batch-2 push that hoisted the player/enemy predicates and scratch vectors out of the per-frame path — the hot loop no longer churns the heap.

public static IsObjectFullyInsideSensor<T extends RAPIER.Collider, U extends GameObject>(collider: T, gameObject: U): boolean

X-axis-only AABB containment: true when the object's left/right edges fall strictly inside the collider's left/right edges. Computes the object edges from CurrentTranslation.x ± CurrentSize.x / 2 and the collider edges from collider.translation().x ± (collider.shape as RAPIER.Cuboid).halfExtents.x.

return objectMinX > colliderMinX && objectMaxX < colliderMaxX;

Line-level notes: it casts collider.shape to RAPIER.Cuboid to read halfExtents — so it assumes the sensor collider is a box. This is the collider-based sibling of GameSensor.IsFullyInside (which works off a GameObject); it exists for Rapier intersection callbacks that hand you a raw Collider. The Enemy uses it in checkLadderIntersections to decide a barrel is centred enough on a ladder core to descend.

public static IsAnySensorTriggered<T extends GameSensor>(gameSensors: T[]): boolean

True if any sensor in the list currently has an intersection. For each sensor it skips ones without a collider or world, then uses Rapier's World.intersectionPairsWith with a callback that flips a local isIntersecting flag; returns true on the first hit.

for (const sensor of gameSensors) {
  if (!sensor.PhysicsBody?.collider(0) || !sensor.Physics?.World) continue;

  let isIntersecting = false;
  sensor.Physics.World.intersectionPairsWith(
    sensor.PhysicsBody.collider(0),
    () => { isIntersecting = true; }
  );
  if (isIntersecting) return true;
}
return false;

Where the director calls it. In Update's ladder block, for the LadderTopSensors and LadderBottomSensors arms of Player.UpdateLadderState(top, core, bottom) — top/bottom only care that the player touches the sensor at all.

public static IsAnySensorTriggeredObjectFullyInside<T extends GameObject, U extends GameSensor>(gameSensors: U[], gameObject: T): boolean

The stricter sibling: true only when a sensor is both intersecting and the given object is fully inside it (sensor.IsFullyInside(gameObject)). Same intersection scan as above, with the extra containment gate before returning true.

if (isIntersecting && sensor.IsFullyInside(gameObject)) {
  return true;
}

Where the director calls it. The core arm of the ladder check — GameUtils.IsAnySensorTriggeredObjectFullyInside(this.LadderCoreSensors, this.Player). The player must be horizontally centred on the ladder core before climbing is allowed, which is what stops you from grabbing a ladder while only clipping its edge.

Hard-won victory (D9 — the "fully inside" climb gate). The ladder climb path relies on continuous polling of the core sensor's containment, not on enter/exit events. As the GameSensor comment puts it, "the D9 ladder 'fully inside' climb path depends on that continuous polling." Both this helper (for the player) and Enemy.checkLadderIntersections (for barrels) implement the same idiom — touch isn't enough, you must be horizontally contained.
Note (TEMPORARY). Both IsAnySensorTriggered* helpers are explicitly marked TEMPORARY — remove once Player uses callbacks. They are per-frame intersection polls; the intended end state is event-driven OnSensorEnter/Exit callbacks on the sensors. They work today and the climb gate genuinely needs continuous containment, so the poll stays for now — but it is the one place in the player's ladder logic that hasn't moved to the event model yet.
