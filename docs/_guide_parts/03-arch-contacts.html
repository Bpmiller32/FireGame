Collisions: groups, the contact table & sensors

This is the spine of the big refactor (commit "Replace collision routing with contact table"). A physics contact becomes a game reaction in four hops, and the engine never learns a single game word along the way: Rapier collision groups decide which pairs are even allowed to touch; the engine ContactRegistry is a generic (typeA, typeB) -> handler table; the game registers concrete rules in contactRules.ts; and each collider carries userData (a routing identity) plus contactPoints (per-surface flags) so a handler can resolve the real entities. This killed the old hard-coded if/else routing and, with the kinematic two-sided dispatch, finally fixed the barrel/enemy bug.

That convergence is the whole fix.

  RESOLVE2 --> DISPATCH
  DISPATCH --> LOOKUP["for each rule: rule.a(x) && rule.b(y)<br/>else rule.a(y) && rule.b(x)"]
  LOOKUP --> HANDLER["game handler(a, b)<br/>reads userData.type, casts to entity"]
  HANDLER --> EMIT["Emitter.emit('gameOver' / 'gameWin' / ...)<br/>or mutate entity (can.IsOnFire, TeleportToPosition)"]
  EMIT --> GD["GameDirector / Vue react"]
Note. Two kinds of "collision sensing" coexist in this engine and they must not be conflated. (1) The contact table documented here is event-driven: enter/exit verdicts between two entities (Enemy kills Player, barrel ignites the can). (2) The player's own ground/wall/ceiling sensing is shapecast-based and fills contactPoints every frame (see Player in the Game sections). Ladders are a third path — polled "fully inside" intersection checks (D9), deliberately not in the contact table. Keep the three straight.

src/game/types/gameCollisionGroups.ts 30 LOC

The bitmask gate that runs before any contact event exists. This is a tiny game-layer const map (it lives under game/types, not engine — the engine names no group) consumed by the entity constructors via GameObject.setCollisionGroup / setCollisionMask. Rapier only generates a contact between two colliders when their group/mask bits agree; everything downstream (the registry, the rules) never fires for a pair the bitmasks already filtered out. No lifecycle of its own — it is pure data imported wherever an entity is built (Player, Enemy, Platform, TrashCan).

const CollisionGroups = { ... } as const

Four named bit constants, written in binary (0b…) so the bit positions are legible at a glance. Two halves of the Rapier interaction word: an object's group says what it is; its mask says what it collides with. The header states the exact interaction test the engine relies on. 

// Collision occurs when (a.group & b.mask) AND (b.group & a.mask); combine with bitwise OR.

const CollisionGroups = {
  // Player's main collision box — solid collisions with platforms and enemies
  PLAYER_BOUNDING_BOX: 0b0000000000000001, // Bit 0

  // Bit 1 is free — was PLAYER_HIT_BOX (the player's 62.5% hit sensor), removed
  // when death detection moved to the full bounding box via the contact table.

  // Enemy collision group — collides with the player's bounding box and platforms
  ENEMY: 0b0000000000000100, // Bit 2

  // Platform collision group — regular platforms, one-way platforms, walls, etc.
  PLATFORM: 0b0000000000001000, // Bit 3

  // Default collision group — collides with everything
  DEFAULT: 0b1111111111111111,
} as const;

Line notes. Bit 0 / Bit 2 / Bit 3 are deliberately spaced (1 is left as a documented hole — see the victory below). DEFAULT is all sixteen low bits set, used as both group and mask for bystander solids that should just behave normally (the TrashCan sets group=DEFAULT, mask=DEFAULT so an enemy can fall in). as const narrows the values to literal types so the keys cannot drift.

Entitygroup (what it IS)mask (what it HITS)set in

Read the symmetry across the table: Player's mask lists ENEMY and Enemy's mask lists PLAYER_BOUNDING_BOX — both halves of (a.group & b.mask) AND (b.group & a.mask) must be true, so a single missing bit silently disables a whole interaction. The barrel's mask is mutated at runtime, too: while DescendLadder() it drops PLATFORM (to pass through girders) but keeps PLAYER_BOUNDING_BOX so it can still kill you mid-descent.

Hard-won victory — the free Bit 1. Death used to come from a separate 62.5%-size "hit box" sensor on the player (the old PLAYER_HIT_BOX) polled every frame. When death moved onto the full bounding box and into the contact table ("any touch registers"), that sensor and its poll became dead code. The bit was not renumbered — it is left as a labelled gap so future bits keep their positions and any stale 0b…10 mask in a saved level can never accidentally re-enable a hitbox that no longer exists.

Modifying this: add a new interaction by allocating the next free bit (Bit 1, then Bit 4…) as a new 0b… constant, then OR it into the relevant entities' group/mask in their constructors. Never reuse DEFAULT's value for a specific class — that opts the object into colliding with everything. Remember both directions must agree, or no contact (and therefore no rule) ever fires.

src/engine/types/userData.ts 10 LOC

The identity stamped onto every Rapier physics body — the bridge from an anonymous collider handle back to a game meaning. Rapier callbacks hand you a raw Collider, never your TS class, so instanceof is unavailable at the dispatch boundary; this opaque string pair is how routing survives. The engine defines the shape and reads it as opaque strings; the meaning of both fields is assigned entirely by the game layer (via EntityType).

export default interface UserData { type: string; name: string }

Two fields, two jobs. type is the shared routing flag: many entities carry the same type (every barrel is "Enemy"), so one contact rule matches them all. name is the per-instance id — it defaults to the type and is only overridden via SetName when you need to single out one specific entity (e.g. logging). 

export default interface UserData {
  // TYPE flag — shared routing identity; game-assigned, engine-opaque.
  type: string;

  // Per-INSTANCE identifier; defaults to the type until SetName overrides it.
  name: string;
}

Where it is written: GameObject.createObjectPhysics seeds userData = { type: name, name: name } (both start equal); SetType(newType) overwrites both (re-stamps routing identity and mirrors it to name); SetName(newName) overwrites only name. Where it is read: getUserData and matchesType in physicsHelpers — the latter is exactly what the contact-rule predicates call.

Note. Per-entity numeric metadata is intentionally not carried here. The header is explicit: it lives as named, typed fields on the entity classes (e.g. Platform.FloorLevel, Platform.IsEdge, LadderSensor.Direction), set at spawn from the GLB texture-name meta bag by the entity factories. This replaced an older anonymous value0/1/2/3 hop — the reason Player.getPlatform() resolves a real Platform instead of reading magic indices off userData.

Modifying this: resist the urge to widen this interface with game data. If a handler needs a number or a flag, add a typed field to the entity class and resolve the object with Physics.GetGameObjectFromCollider, then read the field. Keeping UserData to two opaque strings is what lets the engine stay game-blind.

src/engine/types/contactPoints.ts 34 LOC

The per-frame surface state a character accumulates — distinct from the contact table. Where the registry answers "which two entities just touched, run a verdict", ContactPoints answers "what is the player resting against / inside this frame". It is the geometry side of collisions: a flat struct of booleans the player rebuilds every frame from shapecasts (and from polled ladder sensors), then the state handlers read to drive movement. Owned by Player as IsTouching.

export default interface ContactPoints { ground; ceiling; leftSide; rightSide; edgePlatform; ladderCore; ladderTop; ladderBottom }

Eight booleans in three groups. The basic four are cardinal surfaces; one platform-type flag; three ladder zones. Each is the input to a specific mechanic.

Lifecycle each frame: Player.resetCollisions() zeroes all eight via Object.assign(this.IsTouching, {...}) (assign, not replace, so dat.gui keeps its binding), then getShapeCastCollisions() sets ground / leftSide / rightSide / ceiling / edgePlatform, and the three ladder fields are pushed in by UpdateLadderState(top, core, bottom) from the GameDirector's polled sensor checks.

Gotcha. The ladder fields here are filled by polling, not by the contact table. A reader chasing "where does ladderCore get set?" will find nothing in contactRules.ts — by design (D9). The contact registry handles cross-entity verdicts; the player's standing surface is shapecast/poll territory. Two systems, one word ("contact") — don't cross the wires.

Modifying this: add a field here only if it's a per-frame surface the player rests against or sits inside. Then set it in getShapeCastCollisions() (and zero it in resetCollisions()). A new event between two entities does not belong here — that's a contact rule.

src/engine/entities/contactRegistry.ts 50 LOC

The generic engine half: a declarative "when an A contacts a B, run this" table layered over the physics event dispatch. It is constructed once by Physics.Configure() as Physics.Contacts = new ContactRegistry(), filled by the game at startup (contactRules.ts, via App.vue), and driven by Physics.handleCollisionEvents calling Dispatch. It imports only GameObject and names zero game concepts — same spirit as the engine StateMachine: a reusable mechanism the game configures with data. Deferred per-line physics wiring lives in the Engine "Physics" section; here is the conceptual model and its API.

export type ContactMatch = (gameObject: GameObject) => boolean

A predicate deciding whether a GameObject matches one side of a rule. The game supplies these (almost always byType("Enemy") etc.). Keeping "which objects count as the A side" as a function — not a string equality baked into the engine — is what lets the engine stay game-blind while still letting the game match by type, by name, by interface, or anything else.

type ContactPhase = "enter" | "exit" | "sensorEnter" | "sensorExit"

The four physics phases a rule can listen to. enter/exit are solid collisions; sensorEnter/sensorExit are non-solid sensor intersections. Note what's absent: there is no per-frame "stay" — Rapier only emits start/stop, so a rule that needs continuous truth (ladders) must poll instead.

type ContactHandler = (a: GameObject, b: GameObject) => void  /  interface ContactRule { a; b; phase; run }

A handler runs when both sides match and receives them in the rule's own declared (a, b) order — never reversed, no matter which body actually moved (the key ergonomic guarantee, made good by Dispatch). A ContactRule is just the bundle: two predicates, a phase, a handler.

public Add(a: ContactMatch, b: ContactMatch, phase: ContactPhase, run: ContactHandler)

Declare a rule. Pushes { a, b, phase, run } onto the private rules array. That's the entire registration API — rules are plain data, so adding game behavior never edits this file. 

private rules: ContactRule[] = [];

public Add(a: ContactMatch, b: ContactMatch, phase: ContactPhase, run: ContactHandler) {
  this.rules.push({ a, b, phase, run });
}

public Dispatch(phase: ContactPhase, x: GameObject, y: GameObject)

Called by Physics for every contact event (and by the player's KCC path). Walks all rules, skips those whose phase differs, and for the rest performs the two-sided match. This is the heart of the symmetry handling. 

public Dispatch(phase: ContactPhase, x: GameObject, y: GameObject) {
  for (const rule of this.rules) {
    if (rule.phase !== phase) {
      continue;
    }

    if (rule.a(x) && rule.b(y)) {
      rule.run(x, y);
    } else if (rule.a(y) && rule.b(x)) {
      rule.run(y, x);
    }
  }
}

Line notes — read the else if carefully, it is doing two jobs at once:

  • Order-independence. Physics has no idea whether the event arrived as (enemy, player) or (player, enemy). The first branch tries a=x, b=y; if that fails the second tries the swap. Either way the rule fires, and rule.run is called with arguments in the rule's declared order — so a handler written (enemy, can) => … always gets enemy first.
  • Fire-once for asymmetric pairs. Because the second test is an else if, a single event can only satisfy one branch. For a typical rule whose two predicates are different types (Enemy vs Player) exactly one branch can ever be true for a given (x, y), so the handler runs once per event, not twice.
Gotcha — same-type rules would only fire once. The else if means if you ever declared a rule whose both predicates match the same object (e.g. byType("Enemy") on both sides for enemy-vs-enemy), the first branch would already be true for (x, y) and the swapped branch would be skipped — you'd get one call, not a call for each ordering. None of today's rules are symmetric, so this is correct as written; just know the constraint before adding an A-equals-B rule.

Modifying this: almost never. This file is finished and game-agnostic. New behavior is added as data in contactRules.ts. The only reasons to touch ContactRegistry are a new phase (which also needs a matching call site in Physics.handleCollisionEvents) or a different matching strategy in Dispatch — both rare and both engine-level decisions.

src/game/config/contactRules.ts 68 LOC

The game half: the declarative table of who dies / wins / teleports / ignites. It imports the engine's ContactRegistry and matchesType, the game's EntityType strings, the global Emitter, and the few entity classes it needs to cast to in handlers (Player, Teleporter, CameraSensor, TrashCan). Lifecycle: a single function called once at startup from App.vue — registerContactRules(webglExperience.Physics.Contacts) — after the engine is configured. The header draws the boundary precisely: an entity's own movement reactions (a barrel reversing at a wall) live with the entity; only cross-entity verdicts live here.

const byType = (type: string): ContactMatch => (go: GameObject) => matchesType(go, type)

The single predicate factory used by every rule. It closes over a type string and returns a ContactMatch that asks "does this object carry that userData.type?" via physicsHelpers.matchesType. This is the join between the four userData identity and the registry's predicate slot — "match any entity carrying this TYPE flag (e.g. every enemy)".

export default function registerContactRules(contacts: ContactRegistry)

Registers all of this game's contact rules into the passed registry. No return value; pure side-effect on contacts. Five Add calls — the complete cross-entity rule book. Read them as a table.

Rule 1 — Enemy touches Player → game over. Solid enter phase; the handler is fire-and-forget onto the event bus.

contacts.Add(
  byType(EntityType.ENEMY),
  byType(EntityType.PLAYER),
  "enter",
  () => Emitter.emit("gameOver")
);

The handler ignores its (a, b) args — it only needs to know the touch happened. "Any touch registers": because Dispatch is two-sided, this fires whether the player ran into the barrel or the barrel rolled into the player. gameOver is a GameEventMap event consumed by GameDirector (which pauses physics — see the victory below).

Rule 2 — Enemy falls into a TrashCan → enemy destroyed, can ignites. The only handler that both mutates an entity and emits.

contacts.Add(
  byType(EntityType.ENEMY),
  byType(EntityType.TRASH_CAN),
  "enter",
  (enemy, can) => {
    (can as TrashCan).IsOnFire = true;
    Emitter.emit("gameObjectRemoved", enemy);
  }
);

Here the (a, b) order earns its keep: the rule declares a = ENEMY, b = TRASH_CAN, so the handler can name them (enemy, can) and trust the registry to pass them that way regardless of which moved. The can as TrashCan cast is safe precisely because byType(TRASH_CAN) already proved the type. Setting IsOnFire = true ripples back into Enemy.DecideTakeLadder() — barrels always take ladders until the can is lit, then roll the dice — the difficulty ramp lives in that one boolean. The gameObjectRemoved event lets GameDirector tear the barrel down.

Rule 3 — Player enters a Teleporter → teleport to its destination. First sensorEnter rule.

contacts.Add(
  byType(EntityType.TELEPORTER),
  byType(EntityType.PLAYER),
  "sensorEnter",
  (teleporter, player) => {
    const dest = (teleporter as Teleporter).Destination;
    (player as Player).TeleportToPosition(dest.x, dest.y);
  }
);

The handler reads geometry off the entity, not off userData: Teleporter.Destination is a typed Vector2 the factory filled from the texture-name dest=x_y meta. Player.TeleportToPosition lives on GameObject (computes the delta and does a relative kinematic setTranslation).

Rule 4 — Player enters a CameraSensor → reposition the camera.

contacts.Add(
  byType(EntityType.CAMERA_SENSOR),
  byType(EntityType.PLAYER),
  "sensorEnter",
  (sensor) => (sensor as CameraSensor).ApplyCameraOnEnter()
);

The cleanest decoupling in the file: the rule names no camera API at all, it just tells the zone "you were entered, do your thing." CameraSensor.ApplyCameraOnEnter() calls camera.SetBaselinePosition(x, y) from its authored cam=x_y anchor. The contact table doesn't even know the Camera exists.

Rule 5 — Player reaches the WinFlag → win the level.

contacts.Add(
  byType(EntityType.WIN_FLAG),
  byType(EntityType.PLAYER),
  "sensorEnter",
  () => Emitter.emit("gameWin")
);

Mirror of Rule 1 on the sensor phase. The WinFlag used to be a solid box that self-detected with a per-frame four-direction shapecast; once the contact system existed it became a pass-through sensor and that proof-of-concept hack was deleted. You walk into the goal now.

Hard-won victory — the barrel/enemy bug. The old routing was a hard-coded if/else ladder inside the collision handler, branching on which body was which. It missed the Enemy→Player kill when only the player moved into the enemy: the player is a kinematic character controller, which keeps a skin gap from solids and so makes Rapier emit no collision event for the player-as-mover (confirmed by runtime logging: ccBlockedBy=[Enemy] yet zero Rapier events). The fix is the pair of mechanisms feeding one table: Rapier's drainCollisionEvents covers the barrel-as-mover, and Player.updateTranslation walks CharacterController.numComputedCollisions() itself for the player-as-mover — so the same declarative rule fires in both directions. But that controller list is a per-frame "currently touching" set; dispatching it raw would re-fire gameOver every frame you leaned on the enemy. So the player now diffs this frame's contacts against last frame's (two reused Sets, swapped each frame so nothing allocates): a contact that is new dispatches "enter", one that vanished dispatches "exit" — so an enter rule fires once on contact (not every frame) and rules finally get a real exit edge (contactRules declares only enter/sensorEnter today, so exit is a no-op there for now). Destroyed objects are skipped on both edges. The whole loop is guarded by if (!this.Physics.IsPaused) so the frozen game-over screen doesn't churn. 
// Player.updateTranslation() — second producer, now dispatching enter/exit EDGES:
if (!this.Physics.IsPaused) {
  this.currentContacts.clear();
  const numCollisions = this.CharacterController.numComputedCollisions();
  for (let i = 0; i < numCollisions; i++) {
    const collision = this.CharacterController.computedCollision(i);
    if (!collision?.collider) {
      continue;
    }
    const other = this.Physics.GetGameObjectFromCollider(collision.collider);
    if (other && !other.IsBeingDestroyed) {
      this.currentContacts.add(other);
    }
  }

  // New contacts this frame → "enter".
  for (const other of this.currentContacts) {
    if (!this.previousContacts.has(other)) {
      this.Physics.Contacts.Dispatch("enter", this, other);
    }
  }
  // Contacts gone since last frame → "exit".
  for (const other of this.previousContacts) {
    if (!this.currentContacts.has(other) && !other.IsBeingDestroyed) {
      this.Physics.Contacts.Dispatch("exit", this, other);
    }
  }

  // Swap the two sets (no allocation); next frame clears `current` before refill.
  const swap = this.previousContacts;
  this.previousContacts = this.currentContacts;
  this.currentContacts = swap;
}
Note — sensors must opt in. A sensor with no callback still has to arm physics events to participate in the table. Teleporter, CameraSensor, and WinFlag each call this.enableContactEvents() in their constructors for exactly this reason (they define no OnSensorEnter, so the auto-arm in GameObject.enablePhysicsEvents wouldn't fire). LadderSensor pointedly does not — it is polled (D9), so it arms no events and appears in no rule.
Gotcha — ladders are not here, by design. Despite "sensor" sounding like contact-table territory, the three ladder zones are absent from this file. Their "fully inside" climb gate needs continuous truth every frame, and the registry only sees enter/exit edges — so ladders stay on the polling path (GameSensor.IsFullyInside / Enemy.checkLadderIntersections). Looking for a ladder rule here is the classic wrong turn.

Modifying this: to add a cross-entity interaction, (1) ensure both entities carry a stable EntityType via SetType, (2) ensure their collision group/mask bits actually let them touch (gameCollisionGroups), (3) ensure non-callback sensors call enableContactEvents(), then (4) add one contacts.Add(byType(A), byType(B), phase, handler) here. Keep handlers thin: emit an event or call a method on the resolved entity; never put movement reactions or per-frame logic in a rule — that's the entity's job. Reach for enter/exit for solids and sensorEnter/sensorExit for sensor zones; if you need "while inside", you need a poll, not a rule.
