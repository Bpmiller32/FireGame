The event catalog 14 signals

Every event that crosses the mitt bus, who fires it, who hears it, and its payload. The map is declared in eventBus.ts as two halves — EngineEventMap (must never name a game concept) and GameEventMap — combined as EngineEventMap & GameEventMap. Listeners are global, so naming discipline is the architecture here.

Hard-won victory — listen before you emit. Both App.vue and GameDirector register their listeners before Resources.StartLoading() is called last. A fast boot could otherwise fire resourcesReady/loadingFinished before anyone was subscribed, leaving the loading overlay stuck or the game un-booted. Ordering is the whole trick.
Why a bus at all. Because listeners are resolved by string, an emitter never holds a reference to its listener — that is exactly what lets the engine emit tick without importing Experience, and lets contactRules declare gameOver without the physics world knowing what "game over" means.

Modifying this: add new events to the correct half of the map in eventBus.ts (engine-pure vs game). Keep payloads plain data. Anything you Emitter.on inside a long-lived object must be paired with an Emitter.off in its Destroy (see Experience, Camera, GameDirector) or you leak listeners across level loads.

The decision-log decoder the Rosetta Stone

The comments carry terse tags like (C5 decouple) or (D3) that point at the author's private decision log. Each marks a deliberate, hard-won choice. Here is every code that appears in the source, every place it shows up, and the principle it encodes — so the comments stop being cryptic.

CodePrincipleAppears at
C5 Camera decouple. The engine Camera consumes a plain CameraTarget data snapshot each frame and reads only scalars from it — it never imports the game Player or PlayerStates. The game maps its Player → CameraTarget (PlayerState → CameraFollowState) in cameraTargetFromPlayer. A different game, or a multiplayer follow, just fills the snapshot differently. cameraTarget.ts:1, gameDirector.ts:160,189, cameraTarget.ts:18
D3 The engine never names a game type. Where the engine needs game knowledge it takes it as injected data: PhysicsDebug colors colliders from a game-supplied EntityType → color map; CameraTarget is plain data, not a Player; the enemy FSM lives game-side. Dependency inversion across the whole seam (also tracked as ADR-0003). App.vue:58, physicsDebug.ts:29, cameraTarget.ts:2, enemyStates.ts:1
D5 Stay lean — prefer the boring primitive. Two faces: (a) collider shapes are CUBE/SPHERE/CAPSULE only; slopes are built from rotated/overlapped cubes and the polyline/convex paths were purged; (b) re-entrancy is guarded with a single old-school boolean (isLoading) rather than a queue/lock library. gameObjectType.ts:1, gameDirector.ts:382
D9 The "fully inside" ladder climb. Grabbing/climbing a ladder requires the climber to be fully inside the ladder-core sensor, which depends on continuous intersection polling (not a one-shot enter event). Both the player and the barrel descend logic rely on it; it's why GameSensor exposes live intersection state. ladderSensor.ts:8, gameSensor.ts:7, enemy.ts:265
D10 Per-entity input snapshot. Each entity holds its own InputState snapshot, refreshed from the device once per frame, rather than every state handler reaching into the global input device. Decouples consumers from the keyboard and makes states trivially testable / AI-swappable. player.ts:138
D15 Data-shape interfaces use camelCase fields. Plain "bag of scalars" interfaces that cross the seam (like CameraTarget) are written as camelCase data shapes, distinguishing them from class-style APIs. cameraTarget.ts:18
D28 Data-driven enemy — one class, two flavors. The old separate CrazyEnemy class was folded into Enemy. Normal vs crazy barrels are now the same class, distinguished only by their starting FSM state and a bounceDirection drift chosen at spawn. This collapse is what let the contact table cleanly fix the barrel/enemy collision bug. gameDirector.ts:49, enemy.ts:44
ADR-0003 Game-side state machines. Architecture Decision Record backing D3: entity FSMs (player, enemy) live in the game layer and are built on the engine's generic StateMachine primitive — the engine supplies the mechanism, the game supplies the states. enemyStates.ts:1
Reading the tags in context. When you hit a (C5) or (D3) in a comment, it's the author flagging "this shape is deliberate, don't 'simplify' it back into a coupling." The cleanest demonstration of the whole philosophy is the contact table + camera pair: physics fires a generic event, the game decides what it means, and neither side imports the other.

Modifying this: if you introduce a new deliberate constraint, give it the next code and a one-line note at the point of use — future-you (and this table) will thank you. If you're about to violate one of these (e.g. importing a game type into the engine), that's the signal to inject data instead.
