Helpers — math & physics utilities

Two tiny, dependency-light modules of mostly-pure functions the engine leans on everywhere: mathHelpers.ts holds game-state-free number math (a framerate-independent move-towards, a seeded mulberry32 PRNG, and a coin-flip built on it), and physicsHelpers.ts holds the thin layer that reads opaque identity off Rapier bodies and colliders so the contact table and physics queries can match entities "by type" without ever importing a game class. The game layer never calls these directly — it reaches them through the GameUtils facade.

A leaf module: "Generic pure math utility functions, no game-state dependencies." It imports nothing and is imported by nothing in the engine layer directly — instead the game-layer GameUtils facade re-exports the two game-facing helpers (GameUtils.MoveTowardsPoint and GameUtils.CalculatePercentChance) so game code funnels through one entry point, while the seeded-PRNG controls (seedRandom and randomRange) are imported straight by the game's GameDirector to pin and reset the seed. It has no constructor or lifecycle — five named exports plus a single module-level piece of state, rngState, the value the PRNG advances. moveTowards is still stateless and side-effect-free; the PRNG functions are deliberately stateful so one fixed seed reproduces one fixed sequence.

Note. Until recently this file carried no decision-code comments; the seeded PRNG added the first — a block explaining why randomness is seeded here (a replay, a lockstep peer, or a deterministic test would all desync on the first Math.random() roll). The older design lesson still holds: math that knows nothing about the Donkey-Kong rules stays in engine/, and the game reaches it through GameUtils.

export function moveTowards(current: number, target: number, maxDelta: number): number

Steps current toward target by at most maxDelta, never overshooting. Parameters: current the present value; target the value to approach; maxDelta the largest step allowed this call (callers pass accel * Time.Delta, which is what makes the motion framerate-independent). Returns: the new value — exactly target once it is within reach, otherwise one step closer.

if (Math.abs(target - current) <= maxDelta) {
  return target;
}

return current + Math.sign(target - current) * maxDelta;

Line-level notes: the guard on line 10 is the anti-overshoot clamp — if the remaining distance is no larger than one step, snap exactly to target instead of stepping past and oscillating around it. The return on line 15 uses Math.sign(target - current) to pick the direction (+1 when target is above, -1 when below, 0 when equal — though equality is already caught by the guard) and scales maxDelta by it, so the step always heads toward the target regardless of which side current sits on.

Tip. This is the engine's equivalent of Unity's Mathf.MoveTowards. Unlike lerp, the step size is absolute (a fixed delta), not a fraction of the remaining gap — so motion is linear and reaches the target in finite time rather than asymptotically. That is exactly why it is used for acceleration toward a target speed: the "feel" attributes set the rate, and the value arrives crisply.

Where it is used. Exposed as GameUtils.MoveTowardsPoint (src/game/gameUtils.ts:14) and called three times, always with a per-frame Time.Delta step:

CallerWhat it eases
applyHorizontalMovement.ts:30player.NextTranslation.x toward targetSpeed by acceleration * player.Time.Delta — the core run/accelerate/decelerate ramp.
handlePlayerFalling.ts:113player.NextTranslation.y — easing vertical velocity during a fall.
handlePlayerClimbing.ts:61player.NextTranslation.y — easing climb speed up/down a ladder.

Gotcha. The framerate-independence lives in the caller, not here. moveTowards is a pure step; it is the * Time.Delta at every call site that makes the ramp identical at 30fps and 144fps. Forget the Delta and a value will accelerate as fast as the machine renders. See Time for the delta source and baseFeel for where acceleration comes from.

Below moveTowards sits the file's other half: a small seeded pseudo-random generator. Three exports — seedRandom, random, and randomRange — share a single module-level integer, rngState (initialised to the golden-ratio constant 0x9e3779b9), and implement mulberry32, a tiny, fast, well-distributed 32-bit PRNG. They are the only stateful code in the module, and that statefulness is exactly the point.

Hard-won victory. This is the randomness half of the determinism work; its sibling is the fixed timestep. Math.random() has no seed, so a replay, a lockstep peer, or a deterministic test would desync on the very first roll. Routing every gameplay draw through a seeded mulberry32 means the same seed always replays the same sequence — and GameDirector seeds it at boot and re-seeds on reset, so a fresh run is reproducible from frame zero.

export function seedRandom(seed: number): void

Pins the generator's state. Parameter: seed any number (the game uses a fixed constant). Returns: nothing — it overwrites the module-level rngState. Same seed in, same random() sequence out.

rngState = seed >>> 0;

Line-level note: the >>> 0 coerces the seed to an unsigned 32-bit integer — mulberry32 is all 32-bit math, so a float or a negative input still lands on a well-defined state. GameDirector calls this once in its constructor with a fixed RNG_SEED and again inside resetGameState(), so a reset replays the run identically.

export function random(): number

The seeded drop-in for Math.random(): one mulberry32 step that advances rngState and returns the next value in [0, 1). Returns: a uniform float in [0, 1), fully determined by the current seed.

rngState = (rngState + 0x6d2b79f5) | 0;
let t = rngState;
t = Math.imul(t ^ (t >>> 15), t | 1);
t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
return ((t ^ (t >>> 14)) >>> 0) / 4294967296;

Line-level notes: each call bumps rngState by the odd constant 0x6d2b79f5 (the | 0 keeps it in signed-32-bit range), then runs the mulberry32 avalanche — two Math.imul mixes interleaved with xorshifts (>>> 15, >>> 7, >>> 14) to scramble the bits. The final unsigned 32-bit result is divided by 4294967296 (2^32) to map it into [0, 1). Because the whole state and arithmetic are integer, the same seed produces the same stream on every machine and JS engine — no floating-point drift.

export function randomRange(min: number, max: number): number

A seeded uniform draw in a half-open range. Parameters: min inclusive lower bound; max exclusive upper bound. Returns: a float in [min, max).

return min + random() * (max - min);

Line-level note: a plain affine remap of random()'s [0, 1) onto [min, max) — and because it reads the same seeded stream, any interval it feeds is reproducible too. GameDirector's trySpawn() uses randomRange(3, 4) and randomRange(2, 3) for its barrel-spawn cadence, so spawn timing now replays exactly under a fixed seed.

export function percentChance(probability: number): boolean

A weighted coin flip. Parameter: probability in the range 0 to 1 (0 = never, 1 = always). Returns: true with that probability, false otherwise.

return random() < probability;

Line-level note: the seeded random() yields a uniform value in [0, 1), so < probability is true for exactly that fraction of draws. The strict < (not <=) means probability === 0 is correctly never true; and because random() never returns 1.0, probability === 1 is always true. The bounds behave exactly as you'd want with no clamping needed. Swapping the built-in Math.random() for the seeded generator is what makes the barrel's ladder-descent coin flip reproducible — same seed, same descents.

Where it is used. Exposed as GameUtils.CalculatePercentChance (src/game/gameUtils.ts:16) and used in exactly one spot: enemy.ts:213, inside DecideTakeLadder(), to decide whether a rolling barrel descends a ladder it is sitting on.

// enemy.ts — DecideTakeLadder()
if (!this.currentTrashCan?.IsOnFire) return true;
return GameUtils.CalculatePercentChance(LADDER_TAKE_CHANCE);
Note. The barrel is guaranteed to take the ladder while the oil can is unlit (the relentless DK opening), then drops to a fixed LADDER_TAKE_CHANCE roll once the first barrel lights the can. The probability constant and that condition belong to the game; this helper is only the dice. See enemy.

Modifying this: add new pure number utilities (e.g. a clamp, lerp, or range-remap) here, never in the game layer — the test is simply "does it reference any game concept?" If yes, it belongs in GameUtils or an entity. Keep every function side-effect-free apart from the seeded-PRNG family (seedRandom/random/randomRange/percentChance), which deliberately share the module-level rngState — any new randomness must draw from random(), never Math.random(), or it will break replay determinism. When you add a helper game code needs, re-export it from GameUtils so game code keeps a single import surface.

src/engine/helpers/physicsHelpers.ts 27 LOC

The other leaf module: "Helpers for Rapier bodies/colliders: extract userData, check names, collision math." It imports Collider and RigidBody from @dimforge/rapier2d-compat (the physics primitives it reads), the engine's UserData interface (the identity shape stamped onto every body), and GameObject (so one overload can take an entity directly). Its whole job is to read the opaque userData bag off a Rapier handle and answer "what type is this?" — the single seam that lets the contact registry and raycast callbacks match entities without the engine ever knowing what an "enemy" or a "wall" is. No constructor, no lifecycle: three pure functions.

Note. UserData carries two strings: type (a shared routing flag — many entities can share one type so a single contact rule matches them all) and name (a per-instance id). These helpers only ever read type. The values are assigned by the game layer; here they are just opaque strings. See UserData.

export function getUserData(physicsBody?: RigidBody): UserData

Reads the identity bag off a Rapier RigidBody, returning a safe empty default when the body is missing. Parameter: physicsBody — optional, because a GameObject may not have built its body yet. Returns: the body's live UserData reference, or a fresh { type: "", name: "" } when there is no body.

if (physicsBody) {
  return physicsBody.userData as UserData;
}

// Return empty default data
return { type: "", name: "" };

Line-level notes: line 10 casts physicsBody.userData (typed unknown by Rapier) to UserData and returns it by reference — that reference identity is load-bearing, because callers mutate it in place to rename or retype the entity. Line 14's default is a brand-new object whose mutations go nowhere, which makes a bodyless call a safe no-op rather than a null-deref.

Where it is used. Engine-internal only — both call sites live in GameObject:

// gameObject.ts — SetType()
const userData = getUserData(this.PhysicsBody);
userData.type = newType;
userData.name = newType;   // mirror name to type until SetName overrides

// gameObject.ts — SetName()
getUserData(this.PhysicsBody).name = newName;

SetType grabs the live reference and writes both fields (a fresh entity's name defaults to its type); SetName chains straight off the return to overwrite just name. Both only work because the function returns the same object the body holds.

Gotcha. The by-reference return is a double-edged sword. When PhysicsBody is undefined, getUserData hands back a throwaway default, so SetName on a body-less object silently does nothing — the write lands on an object that is immediately garbage-collected. Always set the type/name after the body exists. This is why SetType/SetName are called during entity construction once the Rapier body is built, not before.

export function matchesType(gameObject: GameObject, type: string): boolean

Answers "does this entity carry the given TYPE flag?" — the GameObject-facing matcher the contact table is built on. Parameters: gameObject the entity to test; type the routing flag to compare against. Returns: true iff the entity's userData.type equals type.

const userData = gameObject.PhysicsBody?.userData as UserData | undefined;
return userData?.type === type;

Line-level notes: it does not reuse getUserData — instead it optional-chains gameObject.PhysicsBody?.userData and casts to UserData | undefined, then optional-chains again on line 20. The difference is deliberate: matchesType only needs a boolean answer and must never allocate a default object just to compare, and a missing body must read as "no match" (undefined === type is false) rather than matching the empty-string default. Reading raw avoids both the allocation and the false-positive-on-"" trap.

Where it is used. It is the engine primitive behind the game's contact table. contactRules.ts wraps it in a one-liner byType factory that produces the ContactMatch predicates fed to contacts.Add(...):

// contactRules.ts
const byType = (type: string): ContactMatch => (go: GameObject) =>
  matchesType(go, type);

contacts.Add(byType(EntityType.ENEMY), byType(EntityType.PLAYER),
  "enter", () => Emitter.emit("gameOver"));
Hard-won victory. This one function is the whole point of the contact-table refactor (commit "Replace collision routing with contact table"). Every collision rule is now a pair of byType predicates plus a callback — there is no hard-coded if (a instanceof Enemy && b instanceof Player) ladder anywhere. New interactions are data: add a row, not a branch. The engine stays ignorant of every game type because matchesType compares opaque strings. See contactRegistry for the two-sided dispatch that consumes these predicates.

export function isColliderType(collider: Collider, type: string): boolean

The collider-based sibling of matchesType: same question, but starting from a Rapier Collider rather than a GameObject. This is the form Rapier's shapecast and intersection callbacks hand you. Parameters: collider the shape returned by a query; type the routing flag. Returns: true iff the collider's parent body's userData.type equals type.

const userData = collider.parent()?.userData as UserData | undefined;
return userData?.type === type;

Line-level notes: the only structural difference from matchesType is line 25's collider.parent()? — identity is stamped on the RigidBody, not the individual Collider, so you must hop up to the parent before reading userData. The same optional-chaining + UserData | undefined cast keeps a parentless collider (Rapier can return one) reading as "no match" instead of throwing.
