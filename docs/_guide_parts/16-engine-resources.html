Resources — loader & the GLB level parser

One engine class wears two hats. As an asset loader it batch-loads our textures and DRACO-compressed GLB models at boot, reporting per-item progress on the event bus and emitting a single resourcesReady when the manifest is done. As a level parser it walks a GLB scene graph exported from the level editor, resolves each collision cube's type from its node name and its metadata from its texture name, performs the one-and-only glTF → game axis conversion, and emits the generic LevelData record that GameDirector and the entityFactories consume. The engine names zero game concepts in either role.

The largest single file in the engine, and the one with the most moving parts. It is constructed exactly once, by Experience.Configure(), which hands it the game's asset manifest — a Resource[] the game supplies (in this game three rows: randy texture, enemy model, dkGraphicsData model) — and stores it as Experience.Resources. From there the whole game reaches it as a singleton field. It imports THREE for the TextureLoader, GLTFLoader + DRACOLoader from the Three.js addons (DRACO shrinks the model files dramatically), the Emitter to broadcast loading lifecycle, and the LevelData type it emits. Crucially it now names zero game assets: the Vite ?url imports that bundle + content-hash the models live game-side in game/config/assetManifest.ts, so the loader just loads whatever URLs it is handed.

Lifecycle. The constructor only wires the loaders; it deliberately does not start loading. StartLoading() kicks off the batch (called last by App.vue). ParseLevel() is invoked on demand by GameDirector for every level load. Destroy() disposes every loaded GPU resource.

Note. The two responsibilities are independent. The asset manifest path (constructor → StartLoading → loadResource → Items) loads the three persistent assets used by the player sprite, enemy barrels, and graphics overlay. The level parser path (ParseLevel → parseScene → parseTypedNode → convertToLevelData) re-runs on every level switch and produces LevelData — it does not go through Items. The only thing they share is the same gltfLoader instance (DRACO already configured).

Type model: Resource, ResourceType, ParsedLevelObject

Three types frame everything. ResourceType is the loader selector — "texture" | "gltfModel". A Resource is one manifest row: a unique name (the Items key), its type, and a path (public-relative, an absolute URL, or a Vite-imported ?url string). It is the one exported type here, because the game declares its own asset manifest against it (see game/config/assetManifest.ts); the other two stay internal. ParsedLevelObject is the parser's intermediate row, the bridge between the raw glTF node and the final LevelData; note its position/rotation arrays already carry the game slot order, documented inline. 

interface ParsedLevelObject {
  name: string;                          // Object name from 3D software
  type: string;                          // Game object type (Platform, Wall, etc.)
  position: [number, number, number];    // game slots: [gameX, gameDepth, gameUp]
  rotation: [number, number, number];    // game slots: [-, gameRotation(z), -]
  width: number;                         // game horizontal extent (glTF X)
  height: number;                        // game vertical extent (glTF Y)
  depth: number;                         // game depth extent (glTF Z)
  meta?: Record<string, string>;         // Metadata parsed from the mesh's texture name
}

type ResourceType = "texture" | "gltfModel";

// Exported so the game can declare its asset manifest (game/config/assetManifest.ts).
export interface Resource {
  name: string;       // unique key to fetch the loaded asset from Items
  type: ResourceType; // picks the loader
  path: string;       // public-relative path, absolute URL, or Vite-imported URL
}

Private fields & public Items store

constructor(sources: Resource[])

Stores the manifest, initializes an empty Items, sets toLoad = sources.length and loaded = 0, then builds the loaders. The GLTFLoader gets a DRACOLoader whose decoder path is import.meta.env.BASE_URL + "draco/gltf/" — that folder lives in public/ and ships draco_decoder.wasm + draco_wasm_wrapper.js, so the WASM decoder is fetched at runtime (it is not bundled). Resolving it against BASE_URL rather than an absolute "/draco/gltf/" means a sub-path deploy (e.g. GitHub Pages under /FireGame/) still finds the decoder instead of 404-ing and silently breaking every GLB load. The TextureLoader needs no configuration. 

this.gltfLoader = new GLTFLoader();
const dracoLoader = new DRACOLoader().setDecoderPath(
  import.meta.env.BASE_URL + "draco/gltf/"
);
this.gltfLoader.setDRACOLoader(dracoLoader);

this.textureLoader = new THREE.TextureLoader();
Hard-won victory — why the constructor must NOT auto-load. The closing comment is the load-bearing one: loading is deliberately not kicked off in the constructor. The caller invokes StartLoading() only after the resourcesReady listener (GameDirector) is subscribed. If every asset happened to resolve during the await window inside Experience.Configure (e.g. RAPIER.init()), the completion event would fire into zero listeners and be lost — mitt does not buffer or replay — hanging the boot on the loading screen forever. The fix is purely an ordering contract enforced across three files (see the boot sequence below).

public StartLoading(): void

Begins the batch. First it seeds a 0/total loadingProgress event so the Vue bar can paint immediately, before any asset resolves. Then it loops the manifest, resolves each row's URL via getAssetPath, warns + skips rows with no path, and dispatches each through loadResource. 

Emitter.emit("loadingProgress", { loaded: 0, total: this.toLoad, item: "" });

for (const source of this.sources) {
  const assetPath = this.getAssetPath(source);
  if (!assetPath) {
    console.warn(`⚠️ No valid path for resource: ${source.name}`);
    continue;
  }
  this.loadResource(source, assetPath);
}
Gotcha — call order is a hard contract. App.vue calls StartLoading() as the very last line of onMounted, after GameDirector has subscribed to resourcesReady and after the three loading-screen listeners are registered. Move it earlier and you re-introduce the lost-event hang the constructor note warns about.

private loadResource(source: Resource, path: string): void

Loads one asset and owns the progress/completion bookkeeping. It builds two callbacks then dispatches by type. onLoad stores the file in Items[source.name], increments loaded, emits a loadingProgress with the resolved item name, and — when loaded === toLoad — emits the single resourcesReady. onError is the resilience path: it logs, still increments loaded, emits resourceLoadFailed with the asset name, advances the bar, and also checks for completion. 

const onLoad = (file: GLTF | THREE.Texture) => {
  this.Items[source.name] = file;
  this.loaded++;
  Emitter.emit("loadingProgress", { loaded: this.loaded, total: this.toLoad, item: source.name });
  if (this.loaded === this.toLoad) {
    Emitter.emit("resourcesReady");
  }
};

const onError = (error: any) => {
  console.error(`❌ Failed to load ${source.name}:`, error);
  this.loaded++;                                   // count it so a 404 can't hang the loader
  Emitter.emit("resourceLoadFailed", source.name);
  Emitter.emit("loadingProgress", { loaded: this.loaded, total: this.toLoad, item: source.name });
  if (this.loaded === this.toLoad) {
    Emitter.emit("resourcesReady");
  }
};

if (source.type === "gltfModel") {
  this.gltfLoader.load(path, onLoad, undefined, onError);
} else if (source.type === "texture") {
  this.textureLoader.load(path, onLoad, undefined, onError);
} else {
  console.warn(`⚠️ Unknown resource type for ${source.name}`);
}
Hard-won victory — a 404 must not hang the loader. Counting failed loads toward loaded is the whole point of the onError branch: a single missing asset would otherwise leave loaded < toLoad forever, resourcesReady would never fire, and the boot would freeze on the loading screen. A failed asset is "done" as far as progress is concerned — the game starts anyway, and anyone who cares hears resourceLoadFailed.
Note — no more as any. onLoad's parameter is typed as the GLTF | THREE.Texture union — exactly what Items stores. Three's load callbacks are typed per-loader (GLTF for the GLTF loader, Texture for the texture loader), and because a handler taking the wider union accepts either narrower payload, both loaders take onLoad directly. The old onLoad as any casts (a relic of the looser Object3D | Texture field type) are gone.

private getAssetPath(source: Resource): string | undefined

Resolves a manifest row to a real URL — now a thin normalizer, because the game's manifest already supplies the URL. A bundled (Vite ?url) asset arrives as a full hashed path and an absolute URL is taken as-is; a plain public-folder path just gets a leading slash for clean resolution from public/. A row with no path returns undefined (and StartLoading warns + skips it). The old name-keyed bundled-asset map — which hardcoded dkGraphicsData/enemy/randy — is gone; the engine no longer names a single game asset. 

if (!source.path) return undefined;
if (source.path.startsWith("/") || source.path.includes("://")) {
  return source.path;
}
return `/${source.path}`;

The ?url imports that trigger Vite to emit content-hashed copies into dist/assets/ now live in game/config/assetManifest.ts, which builds the GAME_ASSETS: Resource[] array with each row's path already set to the resolved bundle URL. So getAssetPath sees a real path for every row, and the old path: "" placeholder is gone.

Tip — adding a new bundled asset. No engine edit at all now: in game/config/assetManifest.ts, import x from "../../assets/.../x.glb?url" and add a { name: "x", type, path: x } row to GAME_ASSETS. For a public/-served asset, skip the import and give the row a real public-relative path instead. The loader takes whatever URL you hand it.
Note — reusability. Hoisting the asset list out of the engine and into GAME_ASSETS is what lets a different game reuse engine/ unchanged: the loader became a pure "load these URLs by name" service that names none of this game's assets.

public async ParseLevel(glbPath: string): Promise<LevelData>

The public parser. glbPath is any URL — in practice the registry's levelUrl() hands it a hashed /assets/… bundle URL, but a raw public path or a ?url import works just as well. It awaits the GLB, walks the scene into intermediate rows, converts to LevelData, and returns. Called from GameDirector.LoadGLBLevel. 

const gltf = await this.loadGLB(glbPath);
const parsedObjects = this.parseScene(gltf.scene);
const levelData = this.convertToLevelData(parsedObjects);
return levelData;

private loadGLB(path: string): Promise<any>

A thin Promise wrapper around the callback-style gltfLoader.load, so ParseLevel can await it. Resolves with the raw GLTF object ({ scene, animations, … }); rejects on error. Reuses the same DRACO-configured gltfLoader as the asset path.

private parseScene(node: THREE.Object3D): ParsedLevelObject[]

The recursive scene-graph walk, and one of the most carefully-reasoned functions in the codebase. The rule: a node's TYPE comes from its NAME — specifically the first "_"-token resolved through lookupCanonicalType — not from its folder path. Editors flatten and rename the group hierarchy on glTF export, so the node name is the one stable channel. If the name resolves to a known type, the node becomes one game object built from its entire subtree's geometry and we do not recurse into it. If it doesn't resolve: a leaf Mesh is a genuinely mis-named cube (warn + skip), while an untyped group is structural and we walk through its children. 

const type = this.lookupCanonicalType(node.name);
if (type) {
  const parsed = this.parseTypedNode(node, type);
  if (parsed) objects.push(parsed);
  return objects;                       // typed node: do NOT recurse
}

if (node instanceof THREE.Mesh) {
  this.warnUnknownType(node.name);      // mis-named leaf cube
  return objects;
}

for (const child of node.children) {    // structural group: walk through
  for (const childObj of this.parseScene(child)) objects.push(childObj);
}
Hard-won victory — the vanishing multi-material platform. That single "stop at the typed node" rule covers two export shapes. A normal single-material cube exports as one THREE.Mesh named e.g. "Platform". But a multi-material cube (a metadata texture on only some faces) is split by Three into per-face primitive meshes under an unnamed group that keeps the "Platform" name — the pieces are named "mesh_N". Reading the group name and merging the split primitives in parseTypedNode recovers it as one collider; without this rule the unnamed pieces were silently skipped and the platform vanished.

private parseTypedNode(node, type): ParsedLevelObject | null

The heart of the parser — it turns one typed node + its subtree into a single ParsedLevelObject with correct position, rotation, size, and metadata, performing the one-and-only glTF → game axis conversion. Returns null for a GraphicsObject (no physics/gameplay) or a node with no meshes. Walk-through:

1. Early-out for graphics-only.

if (type === "GraphicsObject") {
  return null;
}

2. Collect every mesh in the subtree. For a normal cube that's just the node; for a multi-material cube it's the per-face primitives Three split out. They share the node's local frame, so they're merged below. 

const meshes: THREE.Mesh[] = [];
node.traverse((child) => { if (child instanceof THREE.Mesh) meshes.push(child); });
if (meshes.length === 0) return null;

const meta = this.readMeta(meshes);

3. Decompose the WORLD transform. Using matrixWorld (parents included), not the local matrix, so a cube nested under a translated/rotated group still resolves correctly. 

node.updateWorldMatrix(true, false);
const worldTranslation = new THREE.Vector3();
const worldQuaternion = new THREE.Quaternion();
const worldScale = new THREE.Vector3();
node.matrixWorld.decompose(worldTranslation, worldQuaternion, worldScale);

const rotation = new THREE.Euler().setFromQuaternion(worldQuaternion);
Gotcha — read the WORLD quaternion, not the local one. For a tilted cube (a slope) built in the editor's X×Y plane, the angle lands in rotation.z (rotation about glTF Z, the game's depth axis). Reading the local quaternion was the old bug — it ignored any parent-group rotation.

4. Size from the LOCAL geometry box, scaled by world-scale magnitude. Each primitive's bounding box is unioned in the shared local frame, then multiplied by abs(worldScale). Using the local extent rather than a world AABB (setFromObject) keeps a rotated slope's size true instead of inflating it to its tilted envelope; the Math.abs guards a mirrored element (negative decomposed scale) from producing a negative half-extent — a degenerate collider. 

const localBox = new THREE.Box3();
for (const mesh of meshes) {
  mesh.geometry.computeBoundingBox();
  if (mesh.geometry.boundingBox) localBox.union(mesh.geometry.boundingBox);
}

const size = new THREE.Vector3();
localBox.getSize(size).multiply(
  new THREE.Vector3(Math.abs(worldScale.x), Math.abs(worldScale.y), Math.abs(worldScale.z))
);

5. Position = geometry WORLD CENTER, not node origin. The level editor bakes pivot offsets into the vertices of stretched / multi-cube elements, so the node origin is often a corner. Centering on the geometry (local center → applyMatrix4(matrixWorld)) makes the collider land exactly where the cube is drawn — WYSIWYG — and stays correct under rotation. 

const position = new THREE.Vector3();
localBox.getCenter(position).applyMatrix4(node.matrixWorld);

6. The single glTF → game axis translation. This is the only place axes are remapped. BlockBench/glTF and this 2D game share axes: gameX = glTF X (horizontal), gameUP = glTF Y (vertical), gameDEPTH = glTF Z (into-screen). Downstream readers consume a fixed slot order, so the remap happens once here. 

const parsedObject: ParsedLevelObject = {
  name: node.name,
  type: type,
  position: [position.x, position.z, position.y],   // [ gameX, gameDEPTH, gameUP ]
  rotation: [rotation.x, -rotation.z, rotation.y],   // slope angle -> slot[1], negated
  width: size.x,   // game horizontal <- glTF X
  height: size.y,  // game vertical   <- glTF Y
  depth: size.z,   // game depth      <- glTF Z
  meta,
};

Hard-won victory — the double-negative that keeps slopes the right way round. A slope's angle is a rotation about glTF Z, landing in rotation[1]. The parser negates it here; then the entity factory applies -data.rotation[1] (see entityFactories.createPlatform), so the two negations net to +rotation.z and a BlockBench CCW rotation reads the same direction in-game. Without the parser-side negation it came out mirrored. Single-axis slopes only — flip this sign if a slope ever tilts the wrong way.
Note — the old "Z-up" bug. A previous version treated glTF Z as "up", forcing you to build levels along the editor's Z axis. The comment block here is the record of that fix: the parser now maps the editor's natural X (wide) × Y (tall) plane straight onto the game's horizontal × vertical plane.

Type resolution & metadata

public RegisterLevelTypes(types: Record<string, string>): void
private lookupCanonicalType(name: string): string | undefined

The type vocabulary is now injected by the game, not held in the engine. RegisterLevelTypes(types) stores the name-token → game-type map into this.levelTypes; App.vue calls it at boot with LEVEL_NODE_TYPES (from game/config/levelNodeTypes.ts), before any level parses. lookupCanonicalType takes name.split("_")[0], lowercases it, and reads it out of this.levelTypes — so "OneWayPlatform", "OneWayPlatform_2" (a user instance suffix), and "OneWayPlatform_1" (Three's auto-dedup of duplicate names) all resolve to the same type. Exact-token matching keeps OneWayPlatform distinct from Platform — the old fuzzy "onewayplatform".includes("platform") mis-routed it. 

private levelTypes: Record<string, string> = {};

public RegisterLevelTypes(types: Record<string, string>) {
  this.levelTypes = types;
}

private lookupCanonicalType(name: string): string | undefined {
  const token = name.split("_")[0].toLowerCase();
  return this.levelTypes[token];
}

private warnUnknownType(name: string): void

Called for a leaf mesh whose name didn't resolve. Logs the offending name + token and lists the valid type names (deduped from the injected levelTypes values), with a hint to name the cube by its type. This deliberately replaced an old fallback that spawned an invisible solid platform for any unknown cube — far more surprising than a skip + warning.

private readTextureName(mesh): string · readMeta(meshes) · parseMetadata(textureName)

The metadata pipeline. Because collision geometry and graphics are separate layers, a collision mesh's texture name is a free data carrier. Three.js copies the glTF texture name verbatim onto material.map.name (it does not sanitize it the way it does node names), so =, ., _ and spaces all survive — which is what makes key=value metadata possible.

readTextureName pulls map.name off the first material and strips a trailing image extension. readMeta merges every textured primitive of a (possibly multi-material) cube into one bag (later face wins on conflict). parseMetadata splits on whitespace / comma / semicolon, treats a bare token as a flag mapping to "1", and otherwise splits key=value with keys lower-cased. Values stay strings; per-type interpretation happens game-side. 

// readTextureName: keep "=", ".", "_" but drop a trailing image extension —
// otherwise "edge=1.png" would parse the value as "1.png" (NaN) and read as 0.
return name.replace(/\.(png|jpe?g|webp|gif|bmp|tga)$/i, "");

// parseMetadata
for (const token of textureName.trim().split(/[\s,;]+/)) {
  if (!token) continue;
  const eq = token.indexOf("=");
  if (eq === -1) {
    meta[token.toLowerCase()] = "1";                       // bare flag
  } else {
    meta[token.slice(0, eq).toLowerCase()] = token.slice(eq + 1);
  }
}
Gotcha — "_" is NOT a token separator. Multi-number values use "_" between components (e.g. cam=3.5_2_-1, dest=10_55) so they stay intact through the split. The game side (metaCoords in entityFactories) splits those on "_" later. Conversely, the type-token split in lookupCanonicalType does split on "_" — that's the instance-suffix channel. Same character, two different roles in two different name fields (texture name vs node name).
Hard-won victory — the silent ".png" zero. A texture that inherited its image's filename ("edge=1.png") would parse the value as "1.png" — not-a-number — so the flag silently read as 0. The extension strip in readTextureName is the fix.

private convertToLevelData(parsedObjects): LevelData

Folds the intermediate rows into the final LevelData dictionary keyed by name, passing width/height/depth/position/rotation/type/meta straight through. Its one piece of real logic is key disambiguation: the dictionary key must be unique per cube or the loser is silently dropped. 

let key = obj.name;
for (let n = 2; key in levelData; n++) {
  key = `${obj.name}#${n}`;
}
if (key !== obj.name) {
  console.warn(`⚠️ Duplicate cube name "${obj.name}" ... kept as "${key}" so it isn't dropped.`);
}

levelData[key] = {
  width: obj.width, height: obj.height, depth: obj.depth,
  position: obj.position, rotation: obj.rotation,
  type: obj.type, meta: obj.meta ?? {},
};
Hard-won victory — the GLTFLoader name-dedup collision. GLTFLoader dedups duplicate node names, but its counter is per-base-name. So a hand-suffixed cube ("Platform_2") can collide with a plain cube whose name auto-deduped to the same string (the 3rd "Platform" → "Platform_2"). The #n suffix guarantees every cube survives. It is only the map key — entities are matched by type, never by name — so it is purely cosmetic to gameplay.

public Destroy(): void · private disposeObject3D(root): void

Frees all loaded GPU memory. Destroy first Emitter.off("resourcesReady") (drops the boot listener), then loops Items: a THREE.Texture gets dispose(); a GLTF object (detected by the "scene" in item duck-type, since Items' union doesn't capture the GLTF wrapper) is walked by disposeObject3D, which traverses and disposes every mesh's geometry and material(s). 

Emitter.off("resourcesReady");
for (const key in this.Items) {
  const item = this.Items[key];
  if (item instanceof THREE.Texture) {
    item.dispose();
  } else if (item && typeof item === "object" && "scene" in item) {
    const gltf = item as { scene: THREE.Object3D };
    this.disposeObject3D(gltf.scene);
  }
}

Called from Experience.Destroy last, after camera/physics/renderer. Note this disposes the persistent boot assets; per-level entity teardown is GameDirector's UnloadLevelData, a separate concern.

Modifying this: Most edits no longer touch this file at all. To add a new boot asset, add a row to GAME_ASSETS in game/config/assetManifest.ts (a ?url import for a bundled asset, or a real public-relative path for a public/ one). To add a new level-placeable type, edit two game files in lockstep: add the lowercased token → type mapping to LEVEL_NODE_TYPES in game/config/levelNodeTypes.ts (its value sourced from EntityType), and add the matching entity factory keyed by that exact type string. Forget the factory and the cube parses but spawns nothing; forget the LEVEL_NODE_TYPES entry and the cube warns + skips. To add a new metadata key, no engine change is needed — the bag is generic; just read it game-side with metaNum/metaCoords. The engine here stays game-agnostic: it loads whatever URLs and resolves whatever type tokens it is handed, and interprets no game key — that boundary is the whole point.

src/engine/types/levelData.ts 35 LOC

The contract type for a parsed level: a dictionary mapping entity names to properties. It is the seam between the parser (ResourceLoader.ParseLevel emits it) and the consumers (GameDirector iterates it; the entity factories read each row). It imports nothing and is pure structure. GLB is the single permanent source of LevelData — there is no JSON path anymore.

interface LevelEntity

One GameObject to create. width / depth / height are the size components; position and rotation are number[] in the game slot order the parser emits and the factories read; type is the entity-type string (the verbatim EntityType values); meta is the optional generic key→value bag. 

interface LevelEntity {
  width: number;            // X axis; horizontal size for 2D
  depth: number;            // Y slot in this array; often minimal for 2D
  height: number;           // Z slot in this array; vertical size for 2D
  position: number[];       // [horizontal, depth, vertical]
  rotation: number[];       // Euler radians; usually [0,0,0] or [0, rotation, 0]
  type: string;             // Platform, Wall, OneWayPlatform, ...Sensor, PlayerStart, ...
  meta?: Record<string, string>;  // texture-name key=value bag, e.g. "floor=1 oneway=8"
}

interface LevelData {
  [entityName: string]: LevelEntity;
}
Note — the meta bag replaced numbered fields. The file comment records the history: meta superseded an old generic value0-3 index scheme. The engine emits this bag and never interprets a game key; the factories own per-type interpretation (floor/edge/oneway for platforms, dir for ladders, cam/dest for camera/teleporter). That keeps levelData.ts game-agnostic even though it documents the game keys for the reader's benefit.
Gotcha — the comment's axis labels describe array SLOTS, not literal X/Y/Z. The depth/height doc comments say "Y axis" / "Z axis", but that's the position-array slot meaning in this 2D system, where position = [horizontal, depth, vertical]. In real glTF terms the parser fills height from glTF Y and depth from glTF Z (see parseTypedNode). Always cross-reference the parser's slot table above rather than reading these labels as Cartesian axes.

Modifying this: This interface is consumed untyped on the factory side (data: any, by deliberate design), so adding an optional field is safe and non-breaking. Renaming or reordering position/rotation slots is not safe: the slot order is a hard contract shared by parseTypedNode, setPlayerStart/setCameraStart, and every entity factory — change one and you must change all of them together.
